    LLM Benchmark Comparison Report

    Date: 2025-12-06 Platform: Linux 5.15.167.4-microsoft-standard-WSL2 (WSL2) GPU: NVIDIA GeForce RTX 4070 (8GB VRAM)

    Summary

    This report compares LLM performance across three benchmark runs:

    • v0 NO GPU - Initial benchmark without GPU acceleration (2025-12-03)
    • v1 NO GPU - Second benchmark without GPU acceleration (2025-12-06)
    • WITH GPU - Final benchmark with GPU acceleration enabled (2025-12-06)

    🚀 GPU vs CPU Performance Comparison

    Ollama llama3.2 (3B parameters)

    Metric CPU (v1) GPU Improvement
    Avg tokens/sec 10.22 40.66 +298% 🔥
    First call (ms) 1,295 792 -39%
    Medium prompt (tps) 18.92 59.15 +213%
    Avg duration (sec) 4.37 1.15 -74%

    Ollama Mistral (7B parameters)

    Metric CPU (v1) GPU Improvement
    Avg tokens/sec 3.82 25.30 +562% 🔥🔥
    First call (ms) 10,076 3,403 -66%
    Medium prompt (tps) 6.86 27.16 +296%
    Avg duration (sec) 11.88 2.44 -79%

    Key Takeaways - GPU Impact

    • llama3.2: ~4x faster with GPU
    • Mistral: ~6.6x faster with GPU (larger model benefits more)
    • Cold start: 39-66% faster model loading with GPU
    • GPU enables real-time inference for local models

    📊 Local vs Cloud Model Comparison (WITH GPU)

    Generation Speed (tokens/second)

    Model Provider Avg tps Medium tps Notes
    llama3.2 Ollama (local) 40.66 59.15 ✅ Fastest overall
    Mistral Ollama (local) 25.30 27.16 Good for 7B model
    gpt-4o-mini OpenAI (cloud) 16.91 32.67 Network latency

    First Call Latency (cold start)

    Model Provider First Call (ms) Notes
    llama3.2 Ollama (local) 792 ✅ Fastest cold start
    gpt-4o-mini OpenAI (cloud) 3,288 Network + API overhead
    Mistral Ollama (local) 3,403 Larger model = slower load

    Observations

    1. Local llama3.2 beats OpenAI gpt-4o-mini in raw speed (40.66 vs 16.91 tps)
    2. OpenAI has consistent latency regardless of prompt size (network-bound)
    3. Local models have variable performance based on prompt complexity
    4. Cold start: Local llama3.2 is 4x faster than cloud API

    🔢 Embedding Models Comparison

    Model Provider Duration (ms) Notes
    text-embedding-3-small OpenAI (cloud) 499 ✅ Faster (network optimized)
    nomic-embed-text Ollama (local) 2,274 First call includes model load

    Note: The Ollama embedding time includes model loading overhead. Subsequent calls would be faster.

    ✅ Recommendations

    For Real-time Applications (agents, chat)

    • Use llama3.2 with GPU - 40+ tps meets real-time threshold (30 tps)
    • Mistral with GPU (25 tps) is borderline acceptable

    For Batch Processing

    • Both local models with GPU exceed batch threshold (5 tps)
    • OpenAI gpt-4o-mini is viable but more expensive

    For Embeddings

    • OpenAI text-embedding-3-small is faster for one-off calls
    • Ollama nomic-embed-text is better for high-volume local processing (no API costs)

    Cost Considerations

    Model Cost Speed (GPU)
    llama3.2 (local) $0 40.66 tps
    Mistral (local) $0 25.30 tps
    gpt-4o-mini ~$0.15/1M tokens 16.91 tps

    Local models with GPU provide better performance at zero marginal cost.

    🔧 Configuration Used

    thresholds:
  llm_realtime_min_tokens_sec: 30.0
  llm_batch_min_tokens_sec: 5.0
  embedding_max_ms: 200.0

    📝 Understanding the Test Prompts

    The benchmark uses three test prompts named by input complexity, not output length:

    Test Input Expected Output Purpose
    small "Say hello." Variable (~10-50 tokens) Cold start / first call latency
    medium "List 3 benefits of Python programming." Long (~100+ tokens) Sustained generation speed
    long Agent tool selection prompt (multi-line) Short (1-3 tokens) Complex input parsing

    Why long is faster than medium?

    The long prompt has a longer input but asks for a single word response ("ONLY the tool name"). The medium prompt has a shorter input but asks for a list (3 benefits = ~100+ tokens).

    Duration depends on output length, not input length. That's why: - medium: 2.16s (generates ~100 tokens) - long: 0.13s (generates ~3 tokens)

    What is Batch Processing?

    Batch processing = processing many requests without needing immediate response.

    Examples: - Generate summaries for 1000 documents overnight - Process historical logs - Index a knowledge base

    Since no user is waiting, latency doesn't matter. That's why the batch threshold (5 tps) is lower than real-time (30 tps).
