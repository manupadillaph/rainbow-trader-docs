    4. Overfitting Analysis — S1 vs S2 Gap

    4.1 The Problem

    With 45 forced features (Tier B corrected), train scores (S1) explode while validation scores (S2) barely change:

    Model Fold Tier A S1 TierB corr S1 S1 Increase Tier A S2 TierB corr S2 S2 Change
    MLP F1 0.293 0.890 +203% 0.214 0.233 +9%
    MLP F2 0.322 0.906 +181% 0.252 0.278 +10%
    MEX F1 — 0.853 — — 0.238 —
    MEX F3 — 0.996 — — 0.238 —
    VIT F2 0.308 0.524 +70% 0.265 0.304 +15%
    TCN F2 0.288 0.437 +52% 0.229 0.316 +38%

    4.2 Interpretation

    • S1 near 1.0 = model memorizes the training set
    • S2 barely moving = zero generalization improvement from memorization
    • This is classic overfitting amplified by high feature dimensionality (45 features)
    • TCN shows the least overfitting gap (S2 actually improves by 38% in F2)
    • MLP_EX is the worst offender (S1 = 0.996 with S2 = 0.238)

    4.3 Counterpoint: The Best Models Used Fewer Features

    Trial Model Obj Features Used S1 S2 S1-S2 Gap
    2530 TCN-F3 0.543 34 0.221 0.257 -0.036 (S2 > S1!)
    2798 VIT-F3 0.506 16 0.349 0.277 0.072 (small)
    1418 MLP-F3 0.489 2 0.322 0.241 0.081 (moderate)
    4014 VIT-F4 0.488 45 0.755 0.231 0.524 (massive!)

    The best performers either had feature auto-selection (picking fewer features) or were from Tier A (2 features). The VIT-F4 with 45 features has a massive gap despite high objective — its S3 of 0.589 is driven by its S3 backtest luck, not generalization.

    4.4 Implications for Rescue

    Since the user decision is to keep 45 features forced, the rescue config must: 1. Seed with hyperparameters that showed low overfitting (high dropout, moderate weight_decay) 2. Consider increasing regularization defaults for models that showed worst overfitting (MLP_EX) 3. Let Optuna explore regularization space more aggressively

    5.2 A Real Late-Converging Example

    The best overall trial, T2530 (TCN-F3 Trial 26), is not evidence for late convergence: in the database it ran 19 epochs and its best epoch was 4 (zero-based). A real late-converging example is VIT-F3 Trial 33 (T3987), which ran the full 40 epochs and achieved its best objective at the final epoch.

    Epoch Objective F1 Macro Entry F1 Val Loss Status
    36 0.2748 0.1997 0.3148 4.629 Improving near cap
    37 0.2751 0.1999 0.3163 4.645 New best
    38 0.2755 0.2001 0.3170 4.642 New best
    39 0.2758 0.2005 0.3172 4.646 New best
    40 0.2759 0.2010 0.3160 4.647 Best objective at final epoch

    Key observations: - Val loss worsens slightly while objective still improves at the end - F1 Macro continues to inch upward through the final epochs - Entry F1 is broadly stable, so the late gain is modest rather than explosive - This is real evidence that some capped 40-epoch studies still had room to improve, but it is not universal

    5.3 Study-Level Pattern: Do Not Generalize from One Experiment

    For the TCN-F3 experiment that produced T2530 (experiment_id=78), the database shows: - 60 total trials - 10 completed - 50 pruned - 0 failed

    Among those 10 completed TCN-F3 trials, 0 had their best epoch in the last 6 epochs. So that specific study does not support a blanket epoch increase. The late-convergence signal comes from other capped 40-epoch studies, not from the best-objective experiment itself.

    5.4 Implications for Rescue

    Recommendation: Treat epoch extension as a targeted lever, not a blanket rule. Section 11.10 shows late convergers are rare globally (2.6% of completed trials), while section 5.1 shows some capped 40-epoch studies can still have much higher local rates. The rescue should only test 50–60 epochs or higher patience for architectures/studies that repeatedly hit the epoch ceiling.

    6. Metric Analysis — Why S2 and F1 Matter

    6.1 The Scoring Pipeline

    The objective is a weighted composite: Obj = w1*S1 + w2*S2 + w3*S3

    • S1 (Stage 1 / Train score): Training performance. High S1 with low S2 = overfitting.
    • S2 (Stage 2 / Validation score): Generalization performance. This is the most honest measure of model quality because it's on unseen data but within the training distribution.
    • S3 (Stage 3 / Backtest score): Live simulation score. Most realistic but also most noisy (depends on market conditions during backtest window).

    6.2 Why S2 Is Critical

    S2 measures how well the model generalizes to unseen data. A model with: - High S3 + Low S2: may have gotten lucky in the backtest window - High S2 + Moderate S3: genuinely learned patterns that transfer

    Evidence from data: - TCN-F3 T26 (best): S2=0.257, S3=0.840 → S3 >> S2, suggesting some luck in F3 backtest - VIT-F3 T54: S2=0.277, S3=0.714 → better S2, more balanced - TCN-F2 T3698: S2=0.339, S3=0.233 → best S2 of any trial, but not a realized trading winner

    6.3 Why F1 and Entry F1 Matter

    F1 Macro (f1_score(y_true, y_pred, labels=[LONG, SHORT], average="macro")): - Computed on LONG and SHORT classes ONLY (not NONE) - Measures whether the model can distinguish LONG from SHORT - Each direction's precision/recall is computed independently, then averaged - F1=0 → model predicts all NONE (degenerate) or always gets direction wrong - F1 > 0.15 → model is making meaningful directional predictions

    Entry F1 (binary_trade_detection.overall.f1): - Binary classification: TRADE (LONG+SHORT merged) vs NO-TRADE (NONE) - tp = model predicted trade AND there was a trade opportunity - fp = model predicted trade BUT there was no real trade opportunity - fn = model predicted no-trade BUT there WAS a trade opportunity - Measures "does the model know WHEN to trade?" (ignoring direction) - Generally higher than F1 Macro (0.28-0.40 vs 0.15-0.25) because direction is harder than timing

    Both matter for seed selection: - High F1 Macro + High Entry F1 = model knows when AND which direction (ideal) - High Entry F1 + Low F1 Macro = model knows when to trade but gets direction wrong - Low both = degenerate model, should not be seeded

    The best trial (TCN-F3 T26) has entry_f1=0.216 (S2 stage) and f1_macro=0.155 (S2 stage). These are moderate — the model is selective rather than accurate.

    11.1 Trade-Level Validation — Why Higher S2 Still Fails Economically

    This subsection was added after a read-only verification pass directly against database/rainbow_trader.sqlite, using:

    • trials and backtest_sessions as the source of truth for official S2, Objective, Trades, Win%, Sharpe, and PnL
    • trades as the source of truth for exit composition (STOP_LOSS vs TAKE_PROFIT) and entry metadata (entry_combined_confidence, entry_strength, entry_quality)

    Before interpreting the trade-level evidence, one important clarification is now verified:

    • In all Script 2 thesis configs, exit_threshold=0.0 and signal_flip_threshold=1.0
    • Therefore, LOW_CONFIDENCE exits are disabled and SIGNAL_FLIP exits are effectively disabled for this study
    • The relevant realized exits in Thesis 2 are therefore dominated by STOP_LOSS / TAKE_PROFIT mechanics, not by uncertainty-based forced exits

    This matters because it removes one possible explanation for the S2-vs-PnL disconnect. The mismatch is not coming from low-confidence exits closing good trades too early. The mismatch remains even when the strategy is judged almost entirely by classic TP/SL outcomes.

    What These Real Trials Show

    Case A — T3698 (TCN-F2): this is the strongest direct counterexample to the naive reading higher S2 => better trading. It posts the best S2 in the entire audit (0.339), yet finishes at - $700.90 across 311 trades. Its trade metadata is not degenerate: average entry confidence is 0.812, average entry quality is 0.835, and even high-confidence/high-quality sample trades still hit STOP_LOSS. The economic problem is simple: 224 stop-loss exits versus only 87 take-profit exits.

    Case B — T2530 (TCN-F3): this is the mirror image. Its S2 is materially lower (0.257), yet it is profitable with + $48.42, Sharpe = 3.58, and only 17 trades. It is especially important because it remains profitable even with more stop-loss exits than take-profit exits (9 SL vs 8 TP). This means profitability is not coming from raw hit-rate dominance; it is coming from selectivity plus favorable payoff asymmetry.

    Case C — T2662 (VIT-F1): this is the clean “friction victim” style example. S2=0.296 is perfectly respectable, but 275 trades and a 94 TP / 181 SL exit mix still produce - $247.38. The metadata confirms that these are not obviously broken entries: average quality remains 0.868. The issue is not “garbage trades only”; the issue is that too many accepted entries still fail under the real TP/SL contract.

    Case D — T1525 (VIT-F1): lower S2 than T2662 (0.255 vs 0.296), but much better economics: only 40 trades, 57.5% win rate, Sharpe = 3.06, and + $81.52 net. This is the practical proof that a lower-S2 model can still be a better trader when its accepted setup set is more selective.

    Why TP/SL Still Implies a Minimum Win Rate

    The key misunderstanding corrected by this trade-level inspection is the idea that, if Thesis 2 exits are mostly TP/SL, then there are only two categories of trades and the problem must reduce to “good trades hit TP, bad trades hit SL.” In reality, even with only TP/SL exits, profitability is governed by expectancy, not by label quality alone:

    expectancy = win_rate * avg_win + loss_rate * avg_loss

    where:

    • avg_win is the average realized profit of TP-resolving trades
    • avg_loss is the average realized loss of SL-resolving trades
    • both are already shaped by slippage, commissions, sizing, and the specific TP/SL geometry of each trade

    This means a model can improve S2 by identifying more “valid-looking” opportunities and still lose money if the newly accepted trades do not satisfy the minimum win/loss mix implied by the actual payoff distribution.

    The real trials above show this directly:

    • T3698 loses with 87 TP / 224 SL and 311 total trades. Here, higher S2 did not produce a profitable TP/SL mix.
    • T2530 wins with only 8 TP / 9 SL and 17 total trades. Here, the trial remains profitable despite fewer TP exits than SL exits because the average winning trades are substantially larger than the average losing trades.

    So the correct interpretation is not:

    • higher S2 => more good trades => higher PnL

    but rather:

    • higher S2 => more accepted opportunities
    • PnL improves only if the accepted set also preserves a sufficiently favorable realized TP/SL payoff mix

    In other words, the “minimum win rate” is not one global fixed number in Script 2, because TP and SL distances adapt per trade. But the principle still holds: each trial has an implicit break-even win rate determined by its realized payoff asymmetry. Trials like T3698 fail that break-even condition; trials like T2530 and T1525 clear it.

    Updated Interpretation

    The real trade metadata supports a narrower and more precise statement than the generic “friction paradox” wording alone:

    • S2 is still useful, because it measures whether the model learned transferable signal
    • But in Script 2 Thesis 2, higher S2 does not guarantee that the additional accepted entries form an economically viable trade set
    • Even with low-confidence exits disabled, some higher-S2 trials simply accept too many entries that still resolve to stop loss under the actual TP/SL rules
    • Therefore, the failure mode is not “the backtester exits early because the model got uncertain”
    • The failure mode is: the model expands the accepted trade set faster than it improves the profitability of that marginal trade set

    This is exactly why the correlation matrix separates a signal-quality cluster (S2, Entry F1, F1 Macro) from an execution-economics cluster (Trades, PnL, Sharpe).

    Accounting Clarification for This Audit

    One additional implementation detail matters for how this subsection was written:

    • backtest_sessions.total_pnl is the official session-level source of truth
    • trades.pnl is a per-trade close-time value written when a trade closes

    These two are related, but not identical in semantics. In the backtesting engine, entry commission is deducted from session equity when the trade is opened, while the stored trades.pnl is written when the trade is closed and reflects the close-side realized value after exit commission. Therefore, summing trades.pnl does not necessarily reconstruct the exact final session total_pnl.

    For audit purposes, this implies:

    • use backtest_sessions / trials for official PnL, Sharpe, Win%, and Trades
    • use trades for exit composition and entry-metadata inspection

    Audit note: During this read-only validation pass, the direct sum of stored closed-trade pnl values did not exactly reproduce the official session totals in every inspected session. For that reason, all claims about official profitability, win rate, Sharpe, and trade count in this subsection use backtest_sessions / trials as the source of truth. The trades table is used here only for exit composition and entry-metadata inspection.

    11.2 Extra Backtests Addendum — What the Threshold Reruns Actually Proved

    The raw threshold-rerun artifact remains the auto-generated annex:

    • 20260331_SCRIPT2_SECTION_BACKTEST.md

    That annex is important because it tests the core hypothesis suggested by Section 11.1: if many high-S2 losers are accepting too many marginal entries, then raising entry_threshold should help by removing those marginal entries.

    Global Outcome of the 20260331_143054 Rerun Batch

    From the annexed backtest report:

    • 60/60 reruns completed successfully
    • Net PnL improved in 48/60
    • Gross proxy (PnL+Fric) improved in only 3/60
    • 15/60 were classified as better-than-proportional
    • Corr(Trade Red%, Delta PnL) = +0.69

    This is the most important empirical update from the reruns:

    • Raising threshold usually improves net PnL
    • But it usually does so by trading less
    • It only rarely improves the gross edge itself

    So the threshold study does not prove that the original models were secretly very profitable underneath the friction. What it proves is more modest and more useful:

    • many candidates can be partially rescued by removing marginal entries
    • only a small minority show evidence that the rerun preserved or improved the underlying gross trading edge

    Concrete Rerun Examples from the Annex

    Trial / Variant Trades Net PnL PnL+Fric Interpretation

    T3641 recommended 177 -> 45 -$188 -> +$78 +$374 -> +$182 Net rescue worked, but gross proxy fell materially. This looks more like friction removal than gross-edge improvement.

    What We Can Now Say with More Confidence

    1. The S2-vs-PnL disconnect is real even under pure TP/SL-style exits. It survives direct inspection of trades and is not explained away by LOW_CONFIDENCE exits.

    2. The main rescue lever in the current system is selectivity, not “more validation quality”. The reruns repeatedly improved PnL by removing trades, not by revealing hidden gross edge in most cases.

    3. Threshold tuning can help, but it is not automatically beneficial. Some candidates improve sharply (T3710), some improve only mechanically by trading less (T3641, T2662), and some become worse (T3874).

    4. The best rescue candidates are not simply the highest-S2 losers. The better candidates are the ones whose gross proxy remains resilient when trade count drops.

    5. This strengthens the rationale for the current rescue design. Seed selection should still include S2 and Entry F1, but the rescue search space must explicitly include mechanisms that control turnover and entry selectivity. Without that, signal-quality gains alone will keep failing economically.
