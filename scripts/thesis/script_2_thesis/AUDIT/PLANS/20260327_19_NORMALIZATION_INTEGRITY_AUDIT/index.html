    Auditoría de Features: Tiers B a E (Script 2)

    Resumen por Tier

    [!NOTE] Feature definitions = definiciones de indicadores en el tier config (cada EMA, RSI, etc. cuenta como 1 definición, sin importar cuántas columnas genera). Columnas reales = columnas de salida en los templates generados (MACD = 3 cols, BB = 5 cols, etc., incluye OHLCV raw + derivados).

    Tier Feature defs Indicadores Signals OHLCV sets Multi-symbol Columnas reales
    B 18 17 0 1 (BTC 5m) ❌ 44
    C 64 32 32 1 (BTC 5m) ❌ 100
    D 133 66 33 5 (BTC 5m/15m/1h/4h/1d) ❌ 211
    E 189 89 33 8 (BTC+ETH+BNB multi-tf) ✅ 355

    Auditoría: Relativización y Normalización por Indicador

    La normalización rolling por columna destruye los cruces entre features (ej. EMA_fast vs EMA_slow), porque cada columna tiene su propio rolling mean/std. Esto es por diseño — no es un bug.

    ¿Por qué normalizar es necesario?

    Sin normalización, features con rangos muy distintos causan problemas al modelo:

    • EMA(8)/close ≈ 0.997 (varianza ~0.001) vs RSI/100 ≈ 0.50 (varianza ~0.20)
    • El gradiente prioriza features con mayor varianza → EMA queda ignorada
    • El learning rate no puede ser correcto para todas las features a la vez

    La normalización (z-score rolling) pone todas las features en media≈0, std≈1.

    ✅ Features bien configuradas (sin normalización — ya acotadas)

    Indicador Rel. Method Rango Norm Tiers
    RSI (7/14/21) /100 [0, 1] NONE B-E
    ADX, +DI, -DI /100 [0, 1] NONE B-E
    Stochastic K%, D% /100 [0, 1] NONE B-E
    BB upper/middle/lower BAND_POSITION (÷close) ~[0.95, 1.05] NONE B-E
    BB percent_b NONE [0, 1] NONE B-E
    Drawdown NONE [-1, 0] NONE D-E
    Time (sin/cos) NONE [-1, 1] NONE C-E
    Correlation NONE [-1, 1] NONE (si se usa)

    [!TIP] Estos indicadores conservan su semántica completa. El modelo puede interpretar directamente RSI=0.7 como "sobrecomprado", ADX=0.4 como "tendencia fuerte", etc.

    ⚡ Features normalizadas (ROBUST z-score — pierden semántica directa, ganan comparabilidad)

    Indicador Rel. Method Norm Tiers Nota
    EMA (9/12/14/21/26/50/100/200) /close ROBUST B-E Cruces perdidos → compensados por signals
    MACD line /close ROBUST B-E Cruce MACD-Signal perdido en norm
    MACD signal /close ROBUST B-E —
    MACD histogram /close ROBUST B-E Mantiene signo del cruce post-norm
    ATR (14) /close ROBUST B-E Solo valor propio, no relación
    OBV DIFF/rolling_mean(vol) ROBUST B-E Solo valor propio
    VWAP /close ROBUST C-E Similar a EMA
    Volatility (10/20/50) NONE ROBUST C-E Escala varía con régimen
    BB bandwidth NONE ROBUST B-E Escala varía con régimen
    PPO (Price Oscillator) /input2 ROBUST C-E Ya es ratio de EMAs
    Ratio (cross-tf, DI) NONE ROBUST C-E
    Spread (DI, RSI disagree, net flow) DIVIDE_BY_SUM ROBUST C-E
    Net flow EMAs (5/10/20) /close ROBUST C-E

    Análisis por Tier

    Tier B — Solo Indicadores Base (18 feature defs → 44 columnas)

    Contenido: OHLCV + 8 EMAs + 3 RSIs + MACD + BB + ATR + ADX + Stochastic + OBV

    Hallazgo: Sin signals. El modelo recibe EMAs normalizadas pero no tiene señales de cruce. Depende 100% de que el modelo aprenda patrones de los z-scores individuales. El MACD histogram es la única pista de cruce de EMAs.

    [!WARNING] Tier B no tiene signals binarios. Las relaciones de cruce entre EMAs no están representadas explícitamente. El modelo solo puede aprenderlas indirectamente via MACD histogram o combinaciones de z-scores.

    Tier C — Indicadores + Signals + Extras (64 feature defs → 100 columnas)

    Agrega vs B: VWAP, Volatility (3), Time features, Drawdown, Spread (ADX DI, net flow), Ratio (ADX DI), Net flow EMAs, PPO, y 32 signals.

    Signals incluidos: - Regime: ADX trending/ranging, bull/bear, high/low vol - Trading: RSI OB/OS, MACD bull/bear - Crossovers: EMA 9/21, 21/50, 50/100, 100/200 (up + down) - MACD line/signal crossover - BB breakout up/down - Stochastic OB/OS - ADX strong/very strong/weak - Volume z-score (>2, normal) - Price above/below EMA200, golden/death cross

    [!TIP] Tier C es donde se "reparan" las relaciones perdidas. Los 32 signals binarios capturan las reglas de trading clásicas que la normalización destruye en los features continuos.

    Tier D — Multi-Timeframe BTC (133 feature defs → 211 columnas)

    Agrega vs C: MTF (15m, 1h, 4h, 1d) para EMAs, RSI, MACD, ATR, ADX, Drawdown. Cross-TF ratios (close vs EMA50 en cada TF). Cross-TF RSI disagree. Regime signals para cada TF.

    Observación: Los signals de crossover siguen siendo solo BTC 5m. No hay crossover signals para timeframes superiores.

    Tier E — Multi-Symbol (189 feature defs → 355 columnas)

    Agrega vs D: ETH (5m/15m/1h) y BNB (5m/15m/1h) con OHLCV + EMAs + RSI + MACD + BB + ATR + ADX + Stochastic + OBV.

    Observación importante: Los signals (crossovers, regimes, thresholds) siguen siendo solo de BTC. ETH y BNB no tienen signals propios. El modelo recibe features continuos normalizados de ETH/BNB pero sin señales binarias de cruce para esos activos.

    [!WARNING] ETH y BNB en Tier E no tienen signals de crossover ni regime. Sus features están normalizadas independientemente, así que el modelo no tiene forma explícita de saber si el RSI de ETH cruzó 70 o si las EMAs de BNB cruzaron.

    Veredicto Global

    Aspecto Estado
    Features ya acotados (RSI, ADX, Stoch, BB, Drawdown, Time, Correlation) sin normalizar ✅ Correcto
    Features con escala variable (EMA, MACD, ATR, OBV, VWAP, Volatility) normalizados ✅ Correcto
    Cruces de EMA compensados con signals ✅ Solo en BTC 5m (Tiers C+)
    Cruces de MACD compensados con signals ✅ Solo en BTC (Tiers C+)
    MACD hist preserva magnitud de cruce EMA ✅ En todos los tiers
    Signals de regime multi-TF ✅ Solo BTC (Tier D+)
    Signals de crossover multi-TF ❌ Solo BTC 5m
    Signals para ETH/BNB ❌ No existen en Tier E

    Sobre la pregunta de descubrimiento del modelo

    La normalización per-column sí limita la capacidad del modelo de descubrir relaciones cross-feature nuevas. Pero hay que ser preciso en qué pierde:

    Lo que el modelo SÍ puede descubrir (post-normalización): - Patrones temporales: "cuando EMA_fast está 2σ arriba de su media Y RSI > 0.7 → probable reversión" - Combinaciones no lineales de z-scores individuales - Correlaciones entre z-scores que aparecen antes de movimientos

    Lo que NO puede descubrir: - "EMA_fast está X% más arriba que EMA_slow" (la magnitud del spread entre dos features normalizados independientemente no es interpretable) - Relaciones basadas en la posición relativa de dos features que no tienen un signal o feature derivado que lo capture

    Mitigación: Cada relación cross-feature que importa debe tener su propio feature derivado (MACD, Spread, Ratio, Signal). Esto es trabajo manual de feature engineering, sí. Pero es el trade-off de normalizar: estabilidad de entrenamiento a cambio de necesitar features derivados explícitos para capturar relaciones.

    Las relaciones no-lineales complejas entre features que un humano no anticipó NO pueden ser descubiertas si dependen del valor relativo directo entre columnas. Esto es inherente a la normalización per-column, no un bug del sistema.

    Cómo se compensan los cruces perdidos

    Los cruces entre features se capturan con:

    1. Signals binarios (0/1): Cruces EMA, RSI>70, MACD>0, Stoch OB/OS, ADX strong/weak, BB breakout, golden/death cross
    2. MACD histogram: Es literalmente EMA_fast - EMA_slow, captura dirección Y magnitud del cruce
    3. Spread/Ratio indicators: spread_adx_di captura +DI vs -DI, ratio_adx_di igual

    Limitación: Los signals binarios solo dicen sí/no, no magnitud. El MACD histogram es el único que captura la amplitud del cruce de EMAs, y luego se normaliza (pero mantiene el signo = dirección del cruce).

    Esto incluye raw open, high, low, close, volume_base, etc. — todas reciben ROBUST z-score rolling.

    ¿Se normalizan los derivados (return_close, etc.)?

    Sí. Están en la misma meta: ROBUST para todo. Y tiene sentido:

    Columna derivada ¿Necesita norm? Por qué
    return_close Sí (ROBUST) Su magnitud varía con la volatilidad del mercado
    hl_pct Sí (ROBUST) Rango muy dependiente del régimen
    oc_pct Sí (ROBUST) Igual que return_close
    volume_ratio Sí (ROBUST) Puede tener outliers enormes (>10x)
    quote_volume_return Sí (ROBUST) Outliers aún más extremos
    aggressive_buy_ratio Sí (ROBUST) Ya acotado [0,1], pero norm lo centra en 0
    aggressive_sell_ratio Sí (ROBUST) Ídem

    [!NOTE] aggressive_buy_ratio y aggressive_sell_ratio están naturalmente en [0,1] y suman ~1 cuando hay volumen. En principio podrían ir sin normalización (como RSI), pero el sistema los normaliza con ROBUST. El efecto es que se centran en ~0 con std~1, lo cual no es incorrecto pero pierde la semántica de "ratio" (0.5 = equilibrado).

    Auditoría: Mandatory, Weight Range y Feature Selection

    [!IMPORTANT] Fuente de verdad: templates generados en experiments/v20260325_tierB_mlp_ex/. Código verificado: hyperparams_feature_selection.py líneas 196-258.

    ¿Qué significa weight_range = None?

    weight_range: None NO significa que el modelo aprende el peso libremente. Significa:

    1. Optuna no optimiza el peso de esa columna (no registra hyperparámetro de peso).
    2. La columna recibe peso fijo 1.0 (neutral).
    3. Ese 1.0 se normaliza junto con todos los demás pesos para que sumen 1.0.
    4. Resultado: todas las columnas con weight_range=None arrancan con peso igual entre sí.
    5. Luego el modelo tiene un feature_scale learnable que parte de ese peso normalizado.

    En la práctica, como todas las columnas de todos los tiers tienen weight_range=None, esto significa que todas arrancan con peso 1/N donde N = número de columnas seleccionadas por Optuna.

    ¿Qué es mandatory?

    mandatory: True → la columna siempre se incluye en el modelo. Optuna no puede excluirla durante feature selection. Pero su peso arranca igual que el resto (1/N).

    mandatory: False → Optuna puede excluirla durante feature selection. Si queda fuera, no participa en absoluto.

    ¿Qué es feature_selection_config?

    Mecanismo completamente independiente de weight_range. Controla cuántas columnas selecciona Optuna, no cómo se ponderan.

    • min_features: mínimo de columnas totales que Optuna debe seleccionar (mandatory + optional).
    • max_features: máximo de columnas totales. null = sin tope (puede usar todas las disponibles).
    • "auto": valor especial — se resuelve a total_regular (el total exacto de columnas disponibles del tier). Ver sección abajo.
    • Optuna sugiere un feature_columns_count entre [max(min_features, mandatory_count), max_features].
    • Luego selecciona las optional columns por scoring + grouping strategy.

    ¿Qué significa "auto"?

    Cuando min_features o max_features es "auto", el generador corre un preflight antes de crear los templates. El preflight ejecuta MEGA para contar cuántas columnas regulares (total_regular) produce el tier. Después:

    min_features = "auto"  →  se resuelve a  total_regular
max_features = "auto"  →  se resuelve a  total_regular

    Si ambos son "auto" → min == max == total_regular → se usan TODAS las columnas disponibles, sin feature selection. Optuna no puede excluir ninguna.

    Esto es intencional para las categorías baseline: - rule_based: necesita las columnas exactas que sus reglas referencian. - random_models: son baselines de referencia, no optimizan feature selection. - always_none: predice siempre NONE, el feature set no importa.

    Alternativas de Normalización: Investigación

    El problema fundamental

    La normalización per-column rolling (z-score ROBUST) destruye las relaciones cross-feature que dependen de la magnitud relativa directa entre columnas. Ejemplo: después de normalizar independientemente EMA_9 y EMA_21, la comparación z_ema9 > z_ema21 no equivale a ema9 > ema21 porque cada columna tiene su propio rolling mean/std.

    El sistema actual compensa esto con signals binarios y features derivados (MACD, Spread, Ratio). Pero la pregunta es: ¿hay alternativas de normalización que preserven estas relaciones sin sacrificar estabilidad de entrenamiento?

    Alternativa 1: Group Normalization (normalización por grupo)

    Concepto: En vez de normalizar cada columna independientemente, agrupar features relacionadas y calcular estadísticas compartidas (mean/std) para todo el grupo.

    Ejemplo: Todas las EMAs (9, 12, 14, 21, 26, 50, 100, 200) comparten el mismo rolling mean y std. Si antes de normalizar EMA_9 > EMA_21, después de normalizar con stats compartidas EMA_9_norm > EMA_21_norm sigue siendo cierto.

    Pros: - Preserva el orden relativo dentro del grupo (cruces de EMAs sobreviven) - Cada grupo queda en escala ~N(0,1) para el modelo - Implementación relativamente simple

    Contras: - Relaciones entre grupos diferentes siguen destruidas (EMA vs RSI) - Los features del grupo deben tener escalas similares pre-normalización (funciona para EMAs relativizadas, no para mezclar EMA con ATR) - Si un feature del grupo tiene outliers extremos, afecta a todos

    Aplicabilidad a rainbow-trader: Alta. Los EMAs ya están relativizados (/close), así que comparten escala. Se podría aplicar group norm al grupo EMA sin cambiar el resto del pipeline. El MACD (line, signal, histogram) también podría compartir stats.

    Alternativa 2: Cross-Sectional Rank Normalization

    Concepto: En cada timestep, rankear todos los N features y convertir a percentiles [0, 1] o [-1, 1]. No usa estadísticas históricas — solo la posición relativa de cada feature respecto a los demás en ese instante.

    Ejemplo: Si en t=100 tienes 18 features, el feature con mayor valor recibe rank 1.0, el menor 0.0, y los demás se interpolan.

    Pros: - Completamente no-paramétrica (sin rolling window, sin mean/std) - Robusto a outliers extremos - Preserva el ordenamiento relativo entre features en cada timestep - Usado en competencias serias: Numerai (el hedge fund crowd-sourced más grande) usa rank normalization como estándar

    Contras: - Pierde información de magnitud completamente (solo queda el ranking) - No distingue entre "EMA_9 está 5% arriba de EMA_21" vs "EMA_9 está 0.01% arriba" - Sensible al número de features (rankear 18 features vs 189 da granularidad muy diferente) - No es estacionaria en el sentido temporal (la distribución de ranks puede cambiar)

    Aplicabilidad a rainbow-trader: Media-baja. Pierde demasiada información de magnitud. Las EMAs relativizadas tienen valores cercanos (~0.99-1.01), y la magnitud de la diferencia importa (un cruce de 0.01% vs 2% es muy diferente para trading). Podría funcionar como feature adicional (rank como columna extra), no como reemplazo.

    Alternativa 3: Adaptive Input Normalization (DAIN)

    Concepto: Una capa neuronal adicional al inicio del modelo que aprende los parámetros de normalización de forma adaptativa. Propuesto en el paper "Deep Adaptive Input Normalization for Financial Time Series" (Passalis et al., 2020).

    Arquitectura: Un mini-network que recibe el input raw y produce: (1) un shift adaptativo, (2) un scale adaptativo, (3) opcionalmente un power transform. Los parámetros se aprenden durante el entrenamiento.

    Pros: - La normalización se adapta al régimen de mercado - Puede aprender a preservar relaciones cross-feature si son útiles para el loss - No requiere feature engineering manual para compensar - Referenciado en literatura financiera específica (Springer JSPS 2020)

    Contras: - Añade complejidad al modelo (más parámetros, más riesgo de overfitting) - Necesita más datos para aprender la normalización además de la tarea principal - En la práctica, el mini-network puede converger a algo similar a z-score - No hay garantía de que preserve relaciones útiles — depende del loss

    Aplicabilidad a rainbow-trader: Media. Interesante como experimento futuro. Requeriría modificar la arquitectura del modelo (agregar una capa DAIN antes del encoder). Podría ser un Tier experimental.

    Alternativa 4: Solo Features Bounded (todo como ratios/retornos)

    Concepto: Diseñar TODOS los features como cantidades naturalmente acotadas (ratios, retornos, indicadores 0-1), eliminando la necesidad de normalización.

    Ejemplo: En vez de EMA raw + normalización, usar solamente: - ema_9 / ema_21 (ratio directo, cruce = ratio cruzando 1.0) - (ema_9 - ema_21) / close (spread normalizado por precio) - RSI ya es [0,1], no necesita nada

    Pros: - No hay normalización → no hay destrucción de relaciones - Features directamente interpretables - Simple y robusto

    Contras: - Requiere feature engineering muy cuidadoso - No todos los features se expresan fácilmente como ratios (volumen, OBV) - Escala de retornos varía con volatilidad (un return de 1% en BTC cripto vs stock) - Algunos ratios tienen outliers extremos (volume_ratio puede ser >10x)

    Aplicabilidad a rainbow-trader: Parcialmente implementada. El sistema YA usa RELATIVE_INDICATOR (divide por close/input/100). El problema es que los retornos y ratios resultantes aún tienen escalas variables (por eso se normaliza después). Para eliminar norm completamente, habría que acotar artificialmente o usar clipping.

    Alternativa 5: Instance Normalization / Layer Normalization interna

    Concepto: Delegar la normalización al modelo. En cada forward pass, el modelo normaliza internamente: - LayerNorm: Normaliza across features per sample (un timestep, todas las features) - Instance Norm: Similar pero por canal/feature across tiempo

    Pros: - El modelo aprende qué normalización es útil - LayerNorm across features preserva relaciones relativas entre features en cada timestep - Estándar en Transformers (el sistema ya usa ViT/Axial ViT)

    Contras: - No resuelve la no-estacionariedad del input (features con drift temporal) - LayerNorm per-sample no tiene memory across time - Si el input tiene outliers extremos, pueden dominar el gradiente antes de que LayerNorm actúe - BatchNorm tiene problemas con batches pequeños

    Aplicabilidad a rainbow-trader: Ya parcialmente presente. Los modelos ViT/Axial ViT internamente usan LayerNorm. Pero el input al modelo ya está normalizado externamente. La pregunta sería: ¿qué pasa si se alimenta el input sin normalización externa y se confía solo en LayerNorm? Probablemente inestable por los outliers de volumen y la escala raw de precios (~70,000 para BTC).

    Alternativa 6: Quantile Normalization

    Concepto: Transformar cada feature a una distribución uniforme [0,1] o gaussiana usando su CDF empírica rolling. Cada valor se mapea a su percentil histórico.

    Pros: - Robusto a outliers (todo queda en [0,1] o N(0,1)) - No asume distribución específica - Preserva el ordenamiento temporal de cada feature

    Contras: - Per-column → mismo problema de destrucción de relaciones cross-feature - Computacionalmente más costoso que z-score (requiere mantener CDF empírica rolling) - Non-trivial con datos no estacionarios (la CDF cambia con el régimen)

    Aplicabilidad a rainbow-trader: Baja como reemplazo, útil como complemento. Es per-column, así que no resuelve el problema fundamental. Pero podría ser más robusto que z-score para features con distribuciones muy asimétricas (volumen, retornos en crashes).
