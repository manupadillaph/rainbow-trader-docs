    Script 2 Ensemble and Regime Ideas

    • Date: 2026-04-01
    • Scope: First structured draft of two new thesis-side ideas for Script 2
    • Context: The current Script 2 results suggest that models learn some signal quality, but very little of that signal survives as consistent net profitability. This note captures two possible next research directions before any implementation work starts.
    • Status: Early idea draft. Not implemented. Intended to be refined iteratively.

    1. Core Motivation

    The current thesis evidence suggests two important possibilities:

    1. A single trained trial may be too noisy or too fragile to represent the best available signal for a fold.
    2. The signal may not be globally useful across the whole validation window, but only inside specific market regimes or regime families.
    3. A more selective operating mode may be necessary, because excessive trade count is already documented as a major driver of negative net PnL in Script 2.

    This leads to two complementary ideas:

    1. Multi-trial ensemble backtesting: combine predictions from several already-trained trials instead of using only one trial at a time.
    2. Trade outcome analysis by regime buckets: study whether profitability depends on the regime in which each trade was opened, managed, or closed.

    The central intuition is that the current problem may not be "models know nothing", but rather:

    • different models or trials may know different parts of the market
    • useful signal may be conditional on regime
    • profitability may require both aggregation and filtering
    • a stricter ensemble rule may improve results partly by reducing the number of trades

    2. Idea A - Multi-Trial Ensemble Backtesting

    2.1 Basic Idea

    Instead of running a backtest from a single trained trial, run a backtest from a set of trained trials that all produce predictions for the same validation period.

    At each inference step:

    1. Load one trained model and generate its prediction.
    2. Load the next trained model and generate its prediction.
    3. Repeat for all selected trials.
    4. Aggregate all predictions into one ensemble signal.
    5. Feed the aggregated signal into the normal backtesting flow.

    The first ensemble version should stay simple:

    • use already-trained trials only
    • do not retrain anything
    • aggregate by average first
    • compare the ensemble against the original single-trial runs

    An additional stricter version should also be considered:

    • trade only when all ensemble members agree on the direction
    • treat disagreement as abstention or NONE

    This stricter version is relevant not only as a signal-quality experiment, but also as a selectivity experiment designed to reduce trade count.

    2.2 Main Hypothesis

    Many weak or moderately useful trials may produce a stronger signal together than individually.

    Possible reasons:

    • averaging may reduce idiosyncratic noise
    • one trial may detect bullish opportunities while another is better at bearish or selective behavior
    • different architectures may fail differently, which may make their combination more robust
    • a consensus rule may reject marginal setups and therefore reduce friction-heavy overtrading

    2.3 What Can Be Averaged

    This still needs design clarification, but the initial candidates are:

    • class probabilities
    • combined confidence
    • direction scores for LONG and SHORT
    • optional strength and quality outputs if they are useful in the stored prediction artifacts

    The simplest first version is:

    • average the per-class probabilities across trials
    • convert the averaged probabilities into the final trading signal

    2.4 Candidate Ensemble Definitions

    Several ensemble definitions look feasible:

    1. Same model, same fold, top-k trials Example: top 3 or top 5 vit-F1 trials.
    2. Mixed models, same fold Example: best mlp, tcn, and vit trials for F3.
    3. Same tier, same fold Example: best trials from all completed Tier B model families for F1.
    4. Cross-run, same model/fold Example: best baseline + rescue + corrected variants for one model/fold pair.

    Several ensemble decision rules also look feasible:

    1. Average rule Average probabilities or scores across all members, then trade from the aggregated output.
    2. Unanimity rule Trade only when all members agree on the same direction.
    3. Majority rule Trade only when most members agree, with optional confidence thresholds.

    The likely safest starting point is:

    • same fold
    • already completed trainable trials
    • small ensemble sizes such as 3 and 5
    • start with average, then compare against a stricter all-must-agree rule

    2.5 Minimum Viable Experiment

    The minimum useful test would be:

    1. Select one fold.
    2. Select one ensemble rule.
    3. Rebuild predictions for the selected already-trained trials.
    4. Average the predictions.
    5. Run the normal backtest on the aggregated signal.
    6. Compare against:
    7. the best single trial
    8. the mean of the member trials
    9. random or trivial baselines if needed

    2.6 Key Research Questions

    • Does the ensemble improve net PnL?
    • Does the ensemble improve Sharpe or win rate?
    • Does the ensemble reduce overtrading?
    • Does the ensemble become more selective than the member trials?
    • Does the ensemble help mainly in some folds and not others?
    • Is it better to ensemble within one architecture or across architectures?
    • Is any gain coming mainly from better signal aggregation or from trading less often?

    3. Idea B - Validation Trade Analysis by Regime Buckets

    3.1 Basic Idea

    For each fold, take the validation period and analyze the resulting trades through a regime lens.

    The goal is to connect trade outcomes with market regime context.

    This analysis should be done at multiple levels of granularity:

    • 27 buckets: full combined regime taxonomy
    • 9 buckets: intermediate grouped taxonomy
    • 3 buckets: coarse regime families

    This is meant to answer whether models have regime-conditional edge even when their aggregate fold result is weak.

    3.2 Main Hypothesis

    A model may fail globally while still having a useful edge inside a subset of regimes.

    Examples:

    • a model may work well only in trending bullish states
    • a model may be profitable only in bearish high-volatility states
    • a model may collapse during transition or unstable regimes

    If this is true, the next step would not necessarily be "train better everywhere", but possibly:

    • trade only in favorable regimes
    • avoid hostile regimes
    • build regime-aware ensemble selection or regime-aware gating

    3.3 Two Complementary Analyses

    Analysis 1 - Fold-Specific Validation Behavior

    For models trained on Fold 1, study their trades on the validation period of Fold 1.

    Then ask:

    • which regime buckets concentrate positive trades?
    • which regime buckets concentrate losses?
    • is there a clear profitable signature by regime for that fold?

    Repeat the same logic for F2, F3, and F4.

    This view asks:

    When a model is trained for a specific fold, what regime behavior does it reveal inside its own validation window?

    Analysis 2 - Cross-Model, Cross-Tier Regime Response

    Ignore where the model was trained for a moment and focus on the trade behavior itself.

    For each model family and tier:

    • collect their validation trades
    • group them by regime buckets
    • measure how each model family responds to each regime group

    This view asks:

    How does each model architecture or tier behave across regime families, regardless of training origin?

    3.4 Why Use 27, 9, and 3 Buckets

    The 27-state taxonomy is the most detailed view, but it may be too sparse for stable operational conclusions.

    That is why the analysis should exist in three parallel forms:

    • 27: maximum detail, but higher sparsity
    • 9: more compact families, likely easier to interpret
    • 3: coarse but robust regime grouping

    The expected value of this multi-scale view is:

    • use 27 for discovery
    • use 9 for operational interpretation
    • use 3 for broad validation and sanity checks

    3.5 Minimum Viable Experiment

    The minimum useful regime study would be:

    1. Choose one completed fold.
    2. Extract validation trades for a small set of selected trained trials.
    3. Tag each trade with the regime bucket active at entry time.
    4. Summarize per bucket:
    5. trade count
    6. win rate
    7. average PnL
    8. total PnL
    9. average holding time if available
    10. Render the same summary under 27, 9, and 3 bucket schemes.

    3.6 Key Research Questions

    • Are profitable trades concentrated in a few regimes?
    • Are losing trades concentrated in transition or mixed regimes?
    • Do some folds have clearly different regime-profitability maps?
    • Do some model families show stable regime preferences across folds?
    • Does Tier B learn better regime specialization than Tier A?
    • Can regime buckets explain why some folds produce zero profitable trials?

    4. Why These Two Ideas Fit Together

    These ideas are separate, but they can reinforce each other.

    Possible progression:

    1. Use regime analysis to discover which regime families are favorable or hostile.
    2. Use ensemble backtesting to test whether combining trials improves signal quality inside those families.
    3. Eventually test regime-aware ensemble logic, such as:
    4. one ensemble for trending markets
    5. another ensemble for ranging markets
    6. abstention during unstable or low-confidence regimes

    In other words:

    • the regime analysis explains where edge may exist
    • the ensemble test explores how to combine weak edge sources into a stronger one

    5. Initial Practical Proposal

    The first implementation phase should stay intentionally small.

    5.1 Ensemble First Pass

    • start with one fold only
    • use already-trained trials only
    • use small ensemble sizes (3, then 5)
    • begin with probability averaging only
    • then compare it against a stricter all-members-must-agree rule
    • compare against the best single trial and the average member performance

    5.2 Regime Analysis First Pass

    • start with entry-time regime tagging only
    • begin with one fold and one tier
    • compute the same report under 27, 9, and 3 buckets
    • focus first on trade count, win rate, total PnL, and average PnL per trade

    5.3 Why This Order Makes Sense

    This sequence keeps the early work manageable:

    • it avoids large refactors
    • it produces interpretable outputs quickly
    • it can reveal whether the ideas are promising before deeper implementation

    6. Ensemble Implementation Architecture

    6.1 Compatibility Rules for Source Trials

    The ensemble should be allowed to combine different trials, including trials from different experiments, as long as their execution context is compatible.

    At minimum, the selected source trials should share:

    • the same base_symbol
    • the same base_timeframe
    • the same effective prediction or backtest range
    • the same execution semantics in bot_config
    • the same operational assumptions required by the bot and backtesting flow

    This means the ensemble may combine different models, different tiers, and different source experiments, but it must not combine runs that would produce incompatible execution behavior.

    The important distinction is:

    • model-side configuration may differ across source trials
    • execution-side configuration must match

    In practice, one compatible experiment can be reused as the market-data carrier for:

    • open
    • close
    • atr
    • base runtime bot configuration

    The source trials do not need to share the same training structure, but they do need to agree on the market execution context.

    6.2 Synthetic Experiment, Synthetic Study, Synthetic Trials

    The current system is structurally centered around:

    • experiment
    • study
    • trial
    • backtest_session

    For that reason, the first implementation should not try to force a fully trial-free backtest design.

    Instead, the ensemble flow should create:

    1. one synthetic experiment
    2. one synthetic study
    3. multiple synthetic trials

    The synthetic experiment exists to store:

    • the common execution context
    • the compatibility snapshot
    • source trial membership
    • the ensemble run metadata

    The synthetic study exists to group ensemble variants coherently.

    The synthetic trials exist to represent alternative ensemble decision rules, for example:

    • average
    • all_must_agree
    • majority_vote

    This approach has several practical advantages:

    • it matches the current database model cleanly
    • it avoids foreign-key breakage
    • it keeps dashboard and reporting flows understandable
    • it provides a natural place to attach aggregated prediction artifacts
    • it prevents ensemble prediction parquets from becoming orphan artifacts

    6.3 Why Not Start With Trial-Free Backtests

    Conceptually, a backtest does not always need to come from a trainable trial.

    However, in the current codebase the persistence model is still trial-centric:

    • BacktestSession is tied to trial_id
    • Trade is tied to trial_id
    • backtest creation services also assume study_id and trial_id

    Because of that, the first version should stay aligned with the current design and use synthetic trial records.

    A future refactor may later allow:

    • manual backtests without trial_id
    • prediction-driven backtests as first-class entities
    • a more domain-pure separation between model training and execution studies

    That future refactor is reasonable, but it should not block the first ensemble implementation.

    6.4 Prediction Artifact Refactor

    The current prediction persistence format is not rich enough for the intended ensemble work.

    For the ensemble logic, especially for average-based aggregation, the stored prediction artifact should preserve the full prediction payload, not just a reduced visualization subset.

    The ensemble-ready prediction artifact should include at least:

    • timestamp
    • signal
    • confidence
    • prob_none
    • prob_long
    • prob_short
    • strength
    • quality
    • combined_confidence

    This is important because:

    • average should ideally operate on probabilities, not only on final labels
    • the bot exit flow relies on directional probabilities
    • ensemble debugging is easier when the full prediction state is preserved
    • downstream analysis can compare raw probabilities versus final ensemble decisions

    6.5 Regeneration Requirement

    Because the currently saved prediction parquets do not yet persist the full ensemble-ready payload, the ensemble pipeline should assume that prediction regeneration will often be necessary.

    This is not ideal, but it is manageable if the workflow is explicit and automated.

    The first ensemble implementation should therefore include a dedicated prediction preparation layer instead of assuming that all existing prediction artifacts are immediately usable.

    6.6 Automatic Reuse-or-Regenerate Policy

    Prediction preparation should use a deterministic policy:

    1. Look for an existing prediction parquet for the source trial and required range.
    2. Validate that the parquet covers the needed timestamps.
    3. Validate that the parquet contains all required ensemble fields.
    4. Validate that the execution context still matches what the ensemble expects.
    5. If all checks pass, reuse the parquet.
    6. If any check fails, regenerate predictions automatically.

    This keeps the workflow practical:

    • reuse when possible
    • regenerate when necessary
    • avoid silent partial compatibility

    The validation should be strict and fail-safe.

    If a parquet is missing even one required field for ensemble execution, it should be treated as non-reusable and the system should regenerate it automatically.

    6.7 Proposed Ensemble Pipeline

    The ensemble workflow can be thought of as six linked modules:

    1. Selection Choose source trials and the ensemble rule to test.
    2. Compatibility validation Verify that source trials can legally share one ensemble execution context.
    3. Synthetic context creation Create synthetic experiment, study, and synthetic trials for the ensemble variants.
    4. Prediction preparation Reuse existing full prediction parquets when valid, otherwise regenerate them.
    5. Prediction merge Align predictions by timestamp and produce one ensemble prediction stream.
    6. Execution Run a backtest using:
    7. the synthetic execution context
    8. market data from a compatible experiment
    9. the aggregated prediction stream

    This makes the architecture explicit:

    • source trials provide predictive information
    • the synthetic experiment provides execution context and traceability
    • the aggregated prediction artifact becomes the real execution input

    6.8 New Backtest Entry Point

    The ensemble design likely needs a new execution entry point.

    This entry point should not behave like normal run_with_trial, because the prediction source is no longer one model checkpoint attached to one real trainable trial.

    Instead, the new entry point should accept:

    • an execution experiment context
    • a synthetic trial context
    • an aggregated prediction stream
    • the aligned market data needed for prices and ATR

    Conceptually, this is:

    • backtest from external predictions
    • with official experiment and trial traceability
    • without standard model inference inside the loop

    6.9 First-Pass Practical Recommendation

    The first implementation should keep scope tight:

    • one synthetic study per ensemble run family
    • three synthetic trials at most for the first pass:
    • average
    • all_must_agree
    • majority_vote
    • one strict compatibility checker
    • one prediction schema upgrade
    • one automatic reuse-or-regenerate layer
    • one new ensemble backtest entry point

    This is large enough to be useful, but still far smaller than a full domain refactor.

    7. Expected Outputs

    If implemented, the work should ideally produce:

    • one ensemble plan document
    • one ensemble result report per run
    • one regime bucket analysis report per fold or per study scope
    • machine-readable artifacts for downstream comparison

    Candidate output types:

    • plan.json
    • results.csv
    • results.json
    • report.md

    This aligns well with the existing extra_backtests workflow style.

    8. Main Risks and Open Design Questions

    8.1 Ensemble Questions

    • Which trials qualify for the ensemble?
    • Should all members carry equal weight, or should some be weighted by S2, Objective, or validation PnL?
    • Should aggregation happen from stored prediction parquets, or by loading checkpoints and recomputing predictions?
    • How should disagreements between LONG and SHORT signals be resolved?
    • How should thresholding work after averaging?
    • Should the synthetic experiment copy one source execution config verbatim, or build a canonical merged snapshot?

    8.2 Prediction Artifact Questions

    • Which exact prediction fields must become mandatory for ensemble-ready persistence?
    • Should the upgraded prediction parquet schema replace the old one, or version it explicitly?
    • How should reuse validation detect stale or partially compatible prediction artifacts?
    • Should aggregated ensemble predictions be stored as a separate parquet category or as normal predictions attached to the synthetic backtest session?

    8.3 Regime Analysis Questions

    • What exact definition should the 9 and 3 bucket taxonomies use?
    • Should trades be tagged by entry regime only, or also by exit regime and holding-period dominant regime?
    • How should regime transitions be handled?
    • What minimum sample size is required before a bucket can be interpreted?
    • Should the first study use best trials only, or a wider set of completed trials?

    8.4 Interpretation Risk

    Both ideas can produce attractive but fragile findings if sample sizes are too small.

    Because of that, any positive result should be interpreted with care:

    • small profitable pockets may still be noise
    • sparse regime buckets may overfit interpretation
    • ensemble improvement may come from selectivity changes rather than genuinely better signal quality

    9. Current Recommendation

    Both ideas are worth exploring.

    If only one should be implemented first, the likely best order is:

    1. Regime bucket analysis first
    2. Small ensemble backtest second

    Reason:

    • the regime study is more diagnostic
    • it can reveal whether edge is conditional before building new execution logic
    • its findings can later guide smarter ensemble construction

    The ensemble idea still looks promising, but the regime study may tell us which ensembles are worth building instead of averaging blindly.
