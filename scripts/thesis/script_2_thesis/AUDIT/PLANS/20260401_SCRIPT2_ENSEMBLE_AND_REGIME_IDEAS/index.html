    Script 2 Ensemble and Regime Status

    • Date: 2026-04-01
    • Last Updated: 2026-04-03
    • Scope: Current status record for the Script 2 ensemble workflow and the still-pending regime work
    • Context: This document records what was actually built, which strategic decisions were taken, how the workflow currently operates, and what remains pending.
    • Status: Ensemble first pass implemented and usable. Regime bucket work still pending.

    1. Purpose and Scope

    This note exists to keep one practical record of the current Script 2 thesis-side work around:

    1. Ensemble backtesting across trained trials
    2. Trade analysis by regime buckets

    The two topics remain connected, but they are at different maturity levels:

    • the ensemble workflow now exists end-to-end
    • the regime workflow still needs design finalization and implementation

    This document focuses on:

    • what is already implemented
    • the strategic decisions that now define the ensemble workflow
    • the current operating model
    • the real pending work that still matters

    2. Why This Work Exists

    The current Script 2 evidence still suggests the same core thesis problem:

    • models appear to learn something
    • that signal rarely survives as robust net profitability
    • excessive trade count and friction remain major pain points

    The current working intuition is:

    • different trained trials may capture different partial edges
    • a single trial may be too fragile to represent the best available signal for a fold
    • useful selectivity may come from stricter agreement rules
    • if edge is conditional on market structure, regime analysis may later explain where ensembles should or should not trade

    That leads to a two-part roadmap:

    1. build a serious ensemble execution workflow first
    2. use regime buckets later to study whether edge depends on market context

    3. Ensemble Work Completed

    The following ensemble components now exist and are usable.

    3.1 Selection and Planning

    A thesis-oriented planner now exists under ensemble_backtests/ and selects source trials through:

    • one or more --summary-csv inputs
    • one required --fold
    • one required --top-per-model
    • optional --tier
    • optional model filtering
    • configurable ranking modes

    Primary planner inputs are:

    • one or more --summary-csv
    • one required --fold
    • one required --top-per-model

    Its current selection model is:

    • same fold
    • cross-model
    • cross-run
    • top N per model
    • strict compatibility bucket validation after candidate gathering

    This matches the intended thesis workflow and keeps selection aligned with the artifacts already used in the Script 2 reporting flow.

    3.2 Compatibility Validation

    Selected source trials are now validated through a strict execution-compatibility check.

    At minimum, source members must share:

    • base_symbol
    • base_timeframe
    • effective stage-3 execution range
    • bot_config
    • simulation_config
    • combined_confidence_weights

    The ensemble is therefore allowed to mix:

    • different model families
    • different runs
    • different experiments
    • optionally different tiers

    but only when their execution context is truly compatible.

    3.3 Synthetic Experiment, Study, and Trials

    The ensemble workflow now creates:

    1. one synthetic experiment
    2. one synthetic study
    3. one synthetic trial per merge configuration

    This keeps the workflow aligned with the current database and reporting model, which remains trial-centric.

    The synthetic study folder now:

    • uses a proper checkpoint-style name
    • is marked as synthetic
    • includes a README.md
    • exists for traceability, not for real training checkpoints

    3.4 Prediction Preparation

    The ensemble workflow now has an explicit prediction preparation layer:

    1. look for a reusable predictions parquet
    2. validate strict ensemble schema
    3. validate coverage for the exact range
    4. reuse if valid
    5. regenerate if invalid or missing

    The strict ensemble-ready persisted prediction payload now includes:

    • timestamp
    • signal
    • confidence
    • prob_none
    • prob_long
    • prob_short
    • strength
    • quality
    • combined_confidence

    This is now the contract used by the ensemble merge logic.

    3.5 Standalone Prediction Generation Improvements

    The standalone prediction generation path used for regeneration was improved to make it practical:

    • it now calibrates batch sizes similarly to the backtesting engine
    • it persists once per outer batch
    • it keeps parquet fragmentation under control
    • it has better progress visibility during prediction generation

    3.6 Backtesting From Precomputed Predictions

    The backtesting engine now supports running from precomputed prediction frames for the synthetic ensemble trials.

    This keeps ensemble execution focused on merged predictions rather than normal per-model inference inside the main backtest loop.

    Conceptually, the workflow is now:

    1. select compatible trained trials
    2. prepare each source predictions artifact
    3. merge predictions into one synthetic prediction stream
    4. backtest directly from that merged prediction stream

    3.7 Reporting and Artifacts

    The ensemble workflow now produces a structured artifact bundle similar in spirit to extra_backtests:

    • plan.json
    • plan.md
    • results.json
    • results.csv
    • comparisons.csv
    • report.md
    • ensemble_backtests.log

    The report now includes:

    • selection context
    • prediction preparation
    • selected source trials
    • ensemble executions
    • ranking across all runs
    • pairwise ensemble-vs-source comparisons
    • friction-aware metrics
    • optional selection-diversity diagnostics based on prediction correlation

    4. Strategic Decisions

    These are the decisions that currently define the ensemble workflow.

    4.1 Selection Strategy

    The primary selection strategy is now:

    • same fold
    • cross-model
    • cross-run
    • top N per model

    The planner is centered on choosing a compatible fold-wide source pool.

    The fold is the main comparability anchor because it fixes the validation window and makes source trials more directly comparable.

    4.2 Input UX

    The main planner input is one or more script2_experiment_ids.csv files passed through --summary-csv.

    This keeps planner usage aligned with the real thesis workflow, where experiment summaries are already the main selection artifact.

    4.3 Baseline Policy

    The ensemble runner does not rerun source baselines by default.

    Instead, the default source baseline mode is:

    • use the original stage-3 backtest sessions from the selected trials

    This keeps the workflow lighter and preserves meaningful comparisons.

    An explicit source rerun remains available when needed.

    4.4 Prediction Payload Policy

    The main prediction payload is strict and explicit.

    Important explicit decisions:

    • trade_confidence is not persisted in the ensemble prediction schema
    • full class probabilities are preserved
    • trade_probability may exist locally as a derived value, but not as a persisted redundant field

    4.5 Synthetic Trial Policy

    The workflow keeps synthetic experiment/study/trial records as the main execution-traceability model.

    This was chosen because:

    • the current DB model remains trial-centric
    • the first-pass ensemble should fit the existing infrastructure
    • traceability matters more than domain purity at this stage

    4.6 Merge Rule Policy

    The merge system now supports the following family of rules:

    • average
    • weighted_average
    • all_must_agree
    • majority_vote[:threshold]
    • weighted_majority_vote[:threshold]

    Important semantic decision:

    For vote-based rules, the winner is resolved first, and the threshold is applied after winner resolution.

    This means:

    • majority_vote
    • majority_vote:0
    • majority_vote:0.0

    are equivalent by the new unified flow, not through legacy compatibility code.

    Weighted rules use planner-derived selection_score values normalized into positive merge weights.

    4.7 Prediction Diversity Policy

    Correlation-aware selection is now treated as an important extension of the ensemble workflow.

    The current design decision is:

    • prediction diversity should be measured on real predictions, not on proxy trial metrics
    • similarity should be evaluated on the real merge-relevant probabilities: prob_none, prob_long, and prob_short
    • the planner keeps its existing top_per_model structure
    • when correlation-aware selection is enabled, each new candidate is compared against the full selected pool
    • overly redundant candidates are skipped and replaced by the next ranked candidate from that same model family

    This keeps the current planner shape intact while reducing the risk of building an ensemble from near-duplicate members.

    4.8 Friction Lens

    The ensemble reporting layer now includes a friction-aware lens aligned with the extra-backtests workflow.

    Current ensemble reporting now tracks:

    • estimated friction
    • PnL + friction
    • gross-per-trade proxy

    This is important because net PnL improvements alone are not sufficient if they come only from friction or trade-count artifacts.

    5. Current Workflow

    The current ensemble workflow is:

    1. choose one or more --summary-csv files from real Script 2 runs
    2. choose one --fold
    3. optionally restrict by --tier and/or --models
    4. choose --top-per-model
    5. rank candidates by one configured selection metric
    6. split candidates into strict compatibility buckets
    7. keep the best bucket
    8. select the top N members per model inside that bucket
    9. optionally prepare predictions during planning when prediction-correlation diversity is enabled
    10. persist the full plan into plan.json
    11. prepare reusable or regenerated prediction artifacts
    12. create synthetic experiment/study/trials
    13. run one ensemble backtest per configured merge rule
    14. compare the merged runs against the original stage-3 source baselines
    15. render a final report

    5.1 Current Ranking Modes

    The planner currently supports:

    • promise
    • friction
    • objective
    • pnl
    • stage2

    Current default:

    • promise

    The friction mode intentionally follows the same scoring family used by extra_backtests.

    5.2 Current Correlation-Aware Selection

    The planner can now operate in two selection styles:

    • score-only shortlist selection
    • score-first shortlist selection with prediction-correlation filtering

    When the correlation-aware mode is enabled:

    • the planner may load reusable predictions during planning
    • missing predictions may be generated during planning
    • Pearson correlation is evaluated separately on prob_none, prob_long, and prob_short
    • the largest positive probability correlation is used as the redundancy check
    • redundancy is checked against the full selected pool

    This is intentionally heavier than pure score-only planning, but it gives the ensemble workflow a real way to prefer behavioral diversity instead of only architectural diversity.

    5.3 Current Default Merge Rule Set

    The default exploratory merge-rule set currently includes:

    • average
    • majority_vote
    • all_must_agree
    • majority_vote:0.75
    • majority_vote:0.85
    • weighted_average
    • weighted_majority_vote:0.75
    • weighted_majority_vote:0.85

    This is not the final thesis conclusion set.

    It is the current working battery for broader experimentation.

    5.4 Current Output Style

    The workflow is now designed so that:

    • plan.md explains how the source pool was selected
    • report.md explains how the ensemble executions behaved
    • results.json/csv and comparisons.csv provide machine-readable outputs

    The report is meant to be human-readable first while still preserving enough structure for downstream comparisons.

    6. Non-Final Exploratory Status

    Initial exploratory runs have already been executed to validate the workflow end-to-end.

    In addition, the first systematic discovery batch has now been executed:

    • 20260403_021500_phase1_block_a
    • 72 completed executions
    • 576 ensemble rule runs

    That batch should still be treated as preliminary rather than final thesis evidence.

    However, it is already strong enough to support two practical conclusions:

    • the ensemble direction is economically promising and worth continuing
    • the current workflow can now produce meaningful campaign-level evidence, not just infrastructure validation

    These runs are useful for:

    • checking that the planner works
    • checking that synthetic context creation works
    • checking that prediction preparation works
    • checking that the report and comparison outputs are coherent
    • checking that merge rules can be explored safely

    These runs are not treated as thesis findings.

    Current ensemble execution results should be considered:

    • exploratory
    • infrastructure validation
    • useful for workflow iteration

    Formal interpretation should still come only after broader, multi-block ensemble runs are executed under a deliberate experiment plan.

    The first systematic batch already exists, but it should be treated as:

    • preliminary evidence
    • selectivity-oriented validation
    • support for continuing to later blocks, not as a final merge-rule verdict

    One especially important caution from that batch is:

    • merge-rule rankings are still weak when the selected pools are very small
    • with only 2 or 3 members, several merge rules can collapse to very similar behavior
    • stronger merge-rule conclusions should wait for larger and broader multi-member ensembles

    7. Pending Work

    The following work remains pending and important.

    7.1 Regime Buckets

    The regime side is still pending implementation.

    Work still required:

    • the exact 27 -> 9 -> 3 bucket mapping
    • the operational meaning of each grouped bucket
    • the trade-tagging strategy
    • the report format for regime summaries

    7.2 Regime Mining

    The next regime-oriented layer goes beyond bucket counting.

    It should eventually answer questions like:

    • which regimes concentrate profitable trades
    • which regimes concentrate losses
    • whether some model families have stable regime preferences
    • whether ensemble gating should later be made regime-aware

    7.3 Large-Scale Ensemble Batch

    The current workflow still needs a larger, deliberate ensemble run campaign.

    That future batch should be used to:

    • stress the current merge rules across multiple folds and candidate pools
    • compare ranking modes more systematically
    • test stricter vote thresholds more broadly
    • test whether weighted rules outperform unweighted rules anywhere
    • test whether prediction-correlation-aware selection produces better source pools than score-only selection
    • inspect whether gains come from signal quality, selectivity, or both

    7.4 Merge Rule Exploration

    The merge framework is flexible enough to support broader experimentation, but the exploration itself is still pending.

    Important open execution questions:

    • which thresholds deserve full study
    • whether weighted rules help in practice
    • whether cross-model pools beat narrower pools
    • whether some merge rules mainly improve by reducing trades

    7.5 Workflow Cleanup

    Some practical workflow cleanup is still worth doing as larger batches are executed:

    • log noise reduction
    • report polish
    • better run naming conventions where useful
    • batch execution ergonomics

    These are not the thesis question, but they matter because they affect the speed and reliability of iteration.

    8. Next Steps

    The next recommended sequence is:

    1. use the current ensemble workflow to run a broader backtest battery
    2. compare merge modes, thresholds, weighting, and ranking strategies at larger scale
    3. only then treat any ensemble result as a real finding
    4. after that, implement regime buckets and regime mining
    5. use the regime work to guide a later regime-aware ensemble phase if the evidence supports it

    In short:

    • the ensemble infrastructure is ready for real experimentation
    • the regime work is the main pending research extension
    • the immediate next milestone is a larger and more deliberate ensemble campaign
