    Script 2 Ensemble Implementation Plan

    • Date: 2026-04-01
    • Scope: Concrete implementation plan for the first Script 2 ensemble backtesting workflow
    • Context: This document translates the ensemble idea into an execution-oriented checklist so progress can be tracked without losing intermediate decisions.
    • Status: Implementation in progress. Prediction preparation, merge rules, synthetic context creation, the backtest-from-predictions entry point, and the new same-fold cross-model planner are already implemented.
    • Related idea document: 20260401_SCRIPT2_ENSEMBLE_AND_REGIME_IDEAS.md

    1. Goal

    Implement a first ensemble backtesting workflow for Script 2 that:

    1. selects source trials from one or more script2_experiment_ids.csv files
    2. validates that those trials are ensemble-compatible
    3. creates a synthetic experiment context for execution traceability
    4. loads or regenerates full prediction artifacts for each source trial
    5. merges predictions into one ensemble prediction stream
    6. runs a backtest from the merged prediction stream
    7. compares multiple ensemble rules inside one synthetic study

    The first implementation target is not regime-aware execution.

    The first implementation target is:

    • one shared ensemble pipeline
    • three ensemble rules
    • one synthetic study with one synthetic trial per rule
    • one planner mode centered on same_fold_cross_models

    Target rules:

    • average
    • all_must_agree
    • majority_vote

    2. Current Understanding

    The current backtesting system is still trial-centric.

    Important consequences:

    • BacktestSession is linked to trial_id
    • Trade is linked to trial_id
    • prediction parquets currently persist only a reduced subset of the prediction payload
    • the bot decision flow requires directional probabilities, not just final signal labels

    Therefore, the first implementation should use:

    • one synthetic experiment
    • one synthetic study
    • three synthetic trials

    This allows the new workflow to fit the existing persistence and reporting structure without immediately refactoring the database model.

    3. High-Level Architecture

    3.1 Input

    • one or more source script2_experiment_ids.csv files from real runs
    • one required fold
    • one optional tier
    • one top-per-model
    • one ranking mode
    • optional execution-range override

    3.2 Shared Requirements

    Source trials must be compatible in execution context.

    Required compatibility checks:

    • same symbol
    • same timeframe
    • same effective backtest range
    • same execution-side bot semantics
    • same market-data context needed by the bot

    3.3 Main Pipeline

    1. Load one or more source summary CSVs.
    2. Filter to one comparable fold and optional tier/model subset.
    3. Load completed trials from the referenced experiments.
    4. Split the pool by strict compatibility bucket.
    5. Keep the top N trials per model in the best bucket.
    6. Create synthetic experiment.
    7. Create synthetic study.
    8. Create synthetic trials for:
    9. average
    10. all_must_agree
    11. majority_vote
    12. Prepare full predictions for each source trial:
    13. reuse existing parquet if valid
    14. regenerate if missing or incomplete
    15. Merge source predictions by timestamp.
    16. Write one merged prediction artifact per ensemble rule.
    17. Run one backtest per synthetic trial using the merged prediction stream.
    18. Render summary and comparison artifacts.

    4. Milestones

    M1. Source Trial Compatibility Layer

    Deliverables:

    • source trial selection input contract
    • compatibility validator
    • compatibility report or payload

    Definition of done:

    • the validator rejects incompatible source trials with explicit reasons
    • the validator identifies one canonical execution context for the ensemble run

    M2. Synthetic Context Creation

    Deliverables:

    • synthetic experiment creator
    • synthetic study creator
    • synthetic trial creator

    Definition of done:

    • one synthetic study can hold three synthetic trials for the three ensemble rules
    • synthetic records store enough metadata to reconstruct source membership and rule type

    M3. Prediction Schema Upgrade

    Deliverables:

    • upgraded persisted prediction payload
    • validation helper for ensemble-ready prediction parquets

    Required persisted fields:

    • timestamp
    • signal
    • confidence
    • prob_none
    • prob_long
    • prob_short
    • strength
    • quality
    • combined_confidence

    Definition of done:

    • regenerated predictions store the full payload
    • validation can distinguish reusable vs non-reusable old prediction parquets

    M4. Reuse-or-Regenerate Prediction Preparation

    Deliverables:

    • helper that searches for existing prediction artifacts
    • schema validation for those artifacts
    • automatic fallback to regeneration

    Definition of done:

    • if a parquet exists and is valid, it is reused
    • if a parquet is missing required fields or range coverage, it is regenerated automatically

    M5. Prediction Merge Engine

    Deliverables:

    • timestamp alignment logic
    • merge rule implementations for:
    • average
    • all_must_agree
    • majority_vote

    Definition of done:

    • each rule produces one deterministic merged prediction stream
    • merged predictions preserve enough fields for the bot to operate normally

    M6. Ensemble Backtest Entry Point

    Deliverables:

    • new execution path for backtesting from merged predictions

    Definition of done:

    • the backtest loop can consume precomputed merged predictions
    • no model checkpoint inference is needed during ensemble execution
    • each run is tied to one synthetic trial and one backtest session

    M7. Reporting and Comparison

    Deliverables:

    • machine-readable ensemble run outputs
    • one summary comparison across rules

    Definition of done:

    • we can compare synthetic trials average, all_must_agree, and majority_vote
    • outputs are easy to fold into the thesis audit later

    6. Implementation Order

    Recommended coding order:

    1. Define the synthetic metadata model and naming convention.
    2. Implement compatibility validation.
    3. Upgrade prediction persistence schema.
    4. Implement reuse-or-regenerate prediction preparation.
    5. Implement merge logic for all three ensemble rules.
    6. Implement synthetic experiment/study/trial creation.
    7. Implement the new ensemble backtest execution path.
    8. Add reporting helpers and first-run summary rendering.
    9. Run one small pilot with a controlled list of source trials.

    Reason for this order:

    • prediction preparation is a prerequisite for merging
    • merging is a prerequisite for new execution
    • synthetic context is needed for traceability before the first real run

    7. Resolved Implementation Decisions

    These decisions are considered fixed for the first implementation pass unless a new explicit change is requested.

    D1. Compatibility Strictness

    Decision:

    • use strict compatibility
    • require exact equality of the full bot_config

    Reason:

    • this simplifies the first implementation
    • it avoids ambiguous execution semantics between source trials
    • it reduces the risk of hidden behavior drift during early ensemble testing

    D2. Canonical average Rule

    Decision:

    • average prob_none, prob_long, and prob_short
    • average strength and quality
    • derive final signal from the averaged probabilities
    • recompute local tradeability from prob_long + prob_short when needed
    • recompute combined_confidence from the merged values

    This is the canonical average definition for the first pass.

    D3. majority_vote Tie Policy

    Decision:

    • if vote counts tie, break the tie using summed probabilities

    This means majority_vote will not default to NONE immediately on ties.

    D4. Prediction Parquet Category

    Decision:

    • reuse the normal predictions category
    • do not create a new parquet category for ensemble predictions in the first pass

    If ensemble-specific distinction becomes necessary later, it should be handled through metadata rather than a new top-level category for now.

    D5. DB and Backtest Relationship Scope

    Decision:

    • do not refactor backtest persistence to support trial-free backtests in the first pass
    • do not change current BacktestSession or Trade trial-link fields in the first pass
    • use synthetic experiment + synthetic study + synthetic trials

    Reason:

    • this keeps the implementation aligned with the current schema
    • it avoids a broader persistence refactor
    • it preserves traceability cleanly for prediction artifacts and backtest results
