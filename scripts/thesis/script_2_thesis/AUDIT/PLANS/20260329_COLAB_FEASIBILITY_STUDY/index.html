    4. Theoretical Model Size Estimates for Remaining Tier B Models

    4.1 Memory Estimation Methodology

    From factory.py::_build_model_memory_estimate:

    • Model state = parameter bytes + buffer bytes
    • Training param-related = params + buffers + gradients + optimizer (AdamW: 2× param bytes for m, v)
    • Total ≈ model state × 4 (params + grad + m + v) for trainable params

    From factory.py::_build_training_memory_estimate:

    • Working VRAM = training param bytes + input batch bytes + training activation bytes
    • Activation = layer_count × (attention_elements + non_attention_elements) × dtype_bytes × 2 (backward)
    • Reserved low = working × 1.10
    • Reserved high = working × 1.25 (no cudagraphs) or × 1.45 (with cudagraphs)

    4.2 TCN — Estimated Tier B Memory

    Worst-case TCN config: channels=128, kernel_size=7, num_layers=4

    • Input dim: ~25 features (Tier B standard set)
    • Seq len: 60 (max window)
    • Batch: 512 (max)

    Approximate parameter count: ~200K–500K (TCN is lightweight)

    Based on Tier A real data (tcn avg: 66K params, 0.008 GB work VRAM with 2 features), Tier B TCN with ~25 features will have more parameters but TCN does NOT have attention, so scaling is linear in features/channels.

    Estimated Tier B TCN:

    • Trainable params: 100K–500K
    • Work VRAM: 0.02–0.10 GB
    • Reserve VRAM high: 0.03–0.15 GB

    Verdict: TCN fits trivially on any Colab GPU.

    4.3 ViT — Estimated Tier B Memory

    Worst-case ViT config: hidden_dim=264, num_heads=16, num_layers=6, dim_feedforward=1024, patch_size=1

    • Input dim: ~25 features
    • Seq len: 60
    • Batch: 512

    Key: attention is O(T²) where T = temporal tokens. With patch_size=1, T=60.

    From Tier A real data (vit avg: 2.1M params, 0.506 GB work VRAM with 2 features, seq_len=40), Tier B increases input_dim from 2→25 and seq_len from 20-40→45-60.

    ViT attention complexity: batch × heads × T × T (does NOT scale with input_dim directly, only through projection layers).

    Estimated Tier B ViT:

    • Trainable params: 2M–4M (larger hidden dims, more layers)
    • Work VRAM: 0.5–1.5 GB
    • Reserve VRAM high: 0.6–2.0 GB

    Verdict: ViT fits comfortably on T4 (15 GB) and all paid GPUs.

    4.4 AxialViT — Estimated Tier B Memory

    Worst-case AxialViT config: hidden_dim=256, time_num_heads=8, feature_num_heads=8, num_layers=6, dim_feedforward=512, patch_size=1

    • Input dim: ~25 features
    • Seq len: 60
    • Batch: 512

    Key: feature-axis attention is O(F²) where F = input_dim. With 25 features: 25²=625 attention elements per batch×time position. Additionally, time-axis attention is O(T²) with T up to 60 tokens.

    From factory.py:

    time_attention = batch × input_dim × time_heads × T × T
feature_attention = batch × T × feature_heads × F × F

    With batch=512, T=60, F=25, time_heads=8, feature_heads=8:

    • time_attention = 512 × 25 × 8 × 60 × 60 = 368,640,000 elements/layer
    • feature_attention = 512 × 60 × 8 × 25 × 25 = 153,600,000 elements/layer
    • Total attention per layer = 522,240,000 elements
    • × 6 layers × 2 bytes (bf16) × 2 (backward) = ~12.5 GB just for attention activations

    From Tier A real data (axial_vit with 2 features, F2 best: 2.87 GB reserve). With 25 features, feature attention scales quadratically: (25/2)² ≈ 156× increase in feature attention. However, total VRAM is NOT purely attention — there is also a significant non-attention component.

    Estimated Tier B AxialViT (worst case):

    • Trainable params: 1M–3M
    • Work VRAM: 5–15 GB (driven by activation memory, not param count)
    • Reserve VRAM high: 6–20 GB

    Verdict: AxialViT worst-case configs MAY NOT fit on T4 (15 GB). Smaller configs (hidden_dim=64, num_layers=2, batch_size=128) will fit. Larger configs need L4/A100.

    1. Opens the Colab snapshot as an ATTACHED database
    2. Copies only the rows for the target experiment_id from:
    3. trials (WHERE experiment_id = ?)
    4. optuna_studies (WHERE experiment_id = ?)
    5. parquets (WHERE experiment_id = ? AND category IN ('SMALL', 'PREDICTIONS'))
    6. checkpoints (WHERE trial_id IN (SELECT id FROM trials WHERE experiment_id = ?))
    7. events (WHERE experiment_id = ?)
    8. Updates the experiments row (status, best_trial_id, best_score, trials counts, updated_at)
    9. Does NOT touch any other experiment's data

    A4.12 Merge Colab DB Into Local DB (NEW SCRIPT NEEDED)

    This is the critical missing piece. A Python script that selectively merges experiment-specific rows from the Colab snapshot into the local DB.

    Script: scripts/gcp/sync_state/merge_experiment_db.py

    What it must do:

    INPUT:
  --local-db    database/rainbow_trader.sqlite
  --colab-db    database/colab_snapshot_exp99.sqlite
  --experiment-id 99

OPERATIONS:
  1. ATTACH colab_db AS colab
  2. UPDATE local experiments SET (status, best_trial_id, best_score,
     trials_completed, trials_pruned, trials_failed, updated_at)
     FROM colab.experiments WHERE id = ?
  3. INSERT OR REPLACE INTO local trials
     SELECT * FROM colab.trials WHERE experiment_id = ?
  4. INSERT OR REPLACE INTO local optuna_studies
     SELECT * FROM colab.optuna_studies WHERE experiment_id = ?
  5. INSERT OR REPLACE INTO local parquets
     SELECT * FROM colab.parquets WHERE experiment_id = ?
     AND category IN ('SMALL', 'PREDICTIONS')
  6. INSERT OR REPLACE INTO local checkpoints
     SELECT * FROM colab.checkpoints
     WHERE trial_id IN (SELECT id FROM colab.trials WHERE experiment_id = ?)
  7. INSERT OR IGNORE INTO local events
     SELECT * FROM colab.events WHERE experiment_id = ?
  8. DETACH colab

OUTPUT:
  - Local DB now has all training results for experiment_id=99
  - No other experiments are affected

    A4.13 Update Summary CSV Row

    After the DB merge, extract metrics and update the summary CSV:

    source .venv/bin/activate

EXPERIMENT_ID="99"
CODE="S2-TB-VIT-F1"
RUN_TAG="20260328_tierB_corrected"
SUMMARY_CSV="scripts/thesis/script_2_thesis/results/${RUN_TAG}/script2_experiment_ids.csv"

# The run_script2.sh function query_experiment_metrics does exactly this.
# We can reuse it by sourcing the script's functions, or use inline Python:

.venv/bin/python - "$EXPERIMENT_ID" "$CODE" "$SUMMARY_CSV" "database/rainbow_trader.sqlite" <<'PYEOF'
import csv
import json
import sqlite3
import sys
from datetime import datetime

experiment_id = int(sys.argv[1])
code = sys.argv[2]
summary_csv = sys.argv[3]
db_path = sys.argv[4]

conn = sqlite3.connect(db_path)
conn.row_factory = sqlite3.Row

exp = conn.execute("""
    SELECT best_trial_id, best_score, trials_completed, trials_pruned,
           trials_failed, created_at, updated_at
    FROM experiments WHERE id = ?
""", (experiment_id,)).fetchone()

if exp is None:
    print(f"ERROR: experiment_id={experiment_id} not found in DB")
    sys.exit(1)

best_trial_id = str(exp["best_trial_id"] or "")
best_objective = str(exp["best_score"] or "")
completed = int(exp["trials_completed"] or 0)
pruned = int(exp["trials_pruned"] or 0)
failed = int(exp["trials_failed"] or 0)
best_sharpe = ""
best_return_pct = ""
checkpoint_dir = ""

if best_trial_id:
    trial = conn.execute("""
        SELECT objective_value, stage3_return_pct, checkpoint_path, training_metrics
        FROM trials WHERE id = ?
    """, (int(best_trial_id),)).fetchone()
    if trial:
        if trial["objective_value"] is not None:
            best_objective = str(trial["objective_value"])
        if trial["stage3_return_pct"] is not None:
            best_return_pct = str(trial["stage3_return_pct"])
        if trial["checkpoint_path"] is not None:
            checkpoint_dir = str(trial["checkpoint_path"])
        tm_raw = trial["training_metrics"]
        if tm_raw:
            try:
                tm = json.loads(tm_raw) if isinstance(tm_raw, str) else tm_raw
                s3 = tm.get("stage3_metrics", {}) if isinstance(tm, dict) else {}
                if isinstance(s3, dict) and s3.get("sharpe_ratio") is not None:
                    best_sharpe = str(s3["sharpe_ratio"])
            except (json.JSONDecodeError, TypeError):
                pass

duration_seconds = ""
if exp["created_at"] and exp["updated_at"]:
    try:
        fmt = "%Y-%m-%d %H:%M:%S"
        c = str(exp["created_at"]).replace("T", " ").split("+")[0].split(".")[0]
        u = str(exp["updated_at"]).replace("T", " ").split("+")[0].split(".")[0]
        delta = datetime.strptime(u, fmt) - datetime.strptime(c, fmt)
        duration_seconds = str(int(delta.total_seconds()))
    except ValueError:
        pass

conn.close()

# Read CSV, update the matching row, write back
rows = []
with open(summary_csv, "r", encoding="utf-8") as f:
    reader = csv.reader(f)
    header = next(reader)
    rows.append(header)
    for row in reader:
        if row[0].strip() == code:
            row[10] = "7"               # last_completed_step
            row[11] = "true"            # training_started
            row[12] = "true"            # training_completed
            row[13] = best_trial_id     # best_trial_id
            row[14] = best_objective    # best_objective
            row[15] = best_sharpe       # best_sharpe
            row[16] = best_return_pct   # best_return_pct
            row[17] = str(completed)    # completed_trials
            row[18] = str(pruned)       # pruned_trials
            row[19] = str(failed)       # failed_trials
            row[20] = duration_seconds  # total_duration_seconds
            row[21] = checkpoint_dir    # checkpoint_dir
        rows.append(row)

with open(summary_csv, "w", encoding="utf-8", newline="") as f:
    writer = csv.writer(f)
    writer.writerows(rows)

print(f"Updated {code} in {summary_csv}")
print(f"  best_objective={best_objective}")
print(f"  best_sharpe={best_sharpe}")
print(f"  trials: completed={completed} pruned={pruned} failed={failed}")
print(f"  duration={duration_seconds}s")
PYEOF

    A8. Known Gaps and Limitations

    1. merge_experiment_db.py does not exist yet — must be created before the first Colab experiment is brought back. This is the single blocking prerequisite.

    2. colab/03_script2_training.ipynb does not exist yet — a convenience notebook wrapping the pull/train/push flow with Script2-specific variables. Not blocking (can use raw bash cells) but recommended for repeatability.

    3. Experiment IDs are auto-incremented — if someone creates experiments on the local machine while Colab is running, the Colab DB snapshot will have stale data for other experiments. The merge script must ONLY touch the target experiment_id.

    4. Optuna study names embed experiment_id — this ties the study to a specific ID. Do not change experiment_id between local and Colab.

    5. Evolution log is local-only — the Colab run will not append to the local evolution_log.jsonl. Consider manually appending an event after the merge, or accept the gap.

    6. Training log file — the _07_training.log file is generated on Colab at a custom path. It is NOT synced via GCS. If you want it locally, copy it manually from the Colab /content/rainbow-trader/training_*.log or add a gsutil cp step to push it.
