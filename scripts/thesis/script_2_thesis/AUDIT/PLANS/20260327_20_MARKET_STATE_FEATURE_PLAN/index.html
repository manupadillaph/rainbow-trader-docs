    Market State Feature Plan

    What This Document Is

    A design plan for a new MARKET_STATE indicator type and market_state feature block that replaces the current approach of 6 binary regime signals with a richer set of continuous columns covering regime strength, persistence, transitions, and meta-stability.

    This plan does not touch Tier A or the current Tier B. It defines a new extended Tier B (B_ext) and upgrades Tier C/D/E.

    Why

    The current market_regime block is too lossy

    The existing market_regime block (available in Tier C+) produces 6 binary SIGNAL features:

    Signal Logic Output
    trending ADX >= 25 0 or 1
    ranging ADX <= 20 0 or 1
    bullish plus_di > minus_di AND ema_14 > ema_50 0 or 1
    bearish minus_di > plus_di AND ema_14 < ema_50 0 or 1
    high_vol atr_ratio > 1.2x rolling_avg 0 or 1
    low_vol atr_ratio < 0.8x rolling_avg 0 or 1

    Three problems:

    1. Information destruction. Continuous values are discretized into 0/1. The model cannot distinguish ADX=25.1 (barely trending) from ADX=50 (extremely strong trend). The model cannot learn that its optimal threshold might be ADX=22 instead of 25. We impose our hardcoded thresholds and throw away the gradient.

    2. Volatility ratio is computed and discarded. The market_regime block computes atr / close relative to its 2016-bar rolling average to produce the high_vol and low_vol signals. This continuous ratio is explicitly marked as an internal dependency (generate_mega_experiment.py line 3189: "We intentionally do NOT add it as a top-level feature, so it does not become a model input column"). The model receives the binary result but never sees the underlying continuous value. This is the only regime dimension whose raw input is not available to the model from any other block.

    3. Missing dimensions entirely. The 6 signals answer what the current regime is, but nothing about:

    4. How long it has been active (persistence)
    5. How confidently it is classified (stability)
    6. Whether it just changed (transition awareness)
    7. Whether the regime itself is stable or volatile (meta-stability)
    8. Whether higher timeframes agree (cross-timeframe coherence)

    Evidence from regime research

    The Regime Training Options Audit found that:

    • 5m -> daily dominant regimes are too twitchy for fold construction (max contiguous period: 8 days for 27-state taxonomy).
    • 1d native regimes are the best current backbone (max contiguous: 69 days for trend+direction).
    • Models break most during transitions between regimes.
    • The audit explicitly recommends Phase 3: implement persistence, candidate vs confirmed, and MIXED/UNSTABLE states.

    The Regime Folds Follow-Up Plan items #3, #4, #5 request:

    • 3: Persistence state machine (observed, candidate, confirmed states with counters)

    • 4: Confirmation rules (minimum duration, consecutive-day confirmation, majority vote, confidence threshold)

    • 5: Hysteresis and transition friction (asymmetric thresholds, weak candidate vs confirmed switch)

    A new indicator type that outputs persistence and transition columns directly addresses these requirements at the feature level.

    What

    A new MARKET_STATE indicator type that produces multiple continuous columns from a single coherent computation. It reuses existing indicator outputs (ADX, ATR, EMA, Bollinger Bands) as inputs and adds stateful regime tracking on top.

    The columns are organized in four groups. Each group can be independently enabled or disabled in the block configuration.

    Group 1: Continuous regime components

    These values are already computed by existing blocks or internally by the market_regime block. The goal is to expose them as continuous model features instead of discretizing them into binary signals.

    Column What it is Range Source
    trend_strength Raw ADX value 0-100 Already available from adx block. Only needs to be ensured present even when adx block is not in the group.
    direction_score (plus_di - minus_di) / (plus_di + minus_di) [-1, +1] Normalized directional dominance. Already computed by derived.di_dominance in Tier C+ as a SPREAD, but not available in lower tiers.
    ema_trend (ema_fast - ema_slow) / ema_slow continuous Continuous trend alignment. EMA periods configurable (default: 14 fast, 50 slow).
    volatility_ratio (atr / close) / rolling_avg(atr / close) ~1.0 center Currently computed internally by market_regime and discarded. Centered around 1.0. Values > 1.2 mean high vol, < 0.8 mean low vol. This is the highest-value column to expose because it is the only regime dimension whose continuous form is not available from any existing block.
    bb_squeeze bb_bandwidth / rolling_avg(bb_bandwidth) ~1.0 center Bollinger Band squeeze indicator. Centered around 1.0. Low values signal volatility compression (often precedes breakouts).

    Implementation complexity: Low. These can be built with existing IndicatorType values (RATIO, SPREAD) or by ensuring existing block outputs are included. No new pipeline types needed.

    Group 2: Persistence and transitions

    These are completely new computations that require stateful regime tracking. They answer questions that no existing feature can answer: how long has this regime been active? and how stable is it?

    Column What it is Range Notes
    bars_since_trend_change Number of bars since the trend component (TRENDING/NEUTRAL/RANGING) last changed 0 to N Log-scaled or normalized by rolling window to keep values in a learnable range.
    bars_since_direction_change Same for the direction component (BULLISH/NEUTRAL/BEARISH) 0 to N Same scaling.
    bars_since_vol_change Same for the volatility component (HIGH_VOL/NORMAL_VOL/LOW_VOL) 0 to N Same scaling.
    regime_stability Fraction of the last N bars (configurable, e.g. 288 = 1 day of 5m) that agree on the current combined regime [0, 1] High values mean the regime is well-established. Low values mean the market is in transition or mixed. Directly addresses FOLLOW_UP_PLAN #6 (MIXED/low-confidence detection).
    transition_recency Exponential decay from the last regime change. exp(-lambda * bars_since_last_change) [0, 1] High = just changed (transition zone). Low = stable for a long time. Lambda is configurable. Directly addresses FOLLOW_UP_PLAN #5 (hysteresis).

    Implementation complexity: Medium. Requires a new IndicatorType (e.g., MARKET_STATE) because these computations are stateful: they depend on the sequence of regime classifications, not just the current bar's indicator values. The closest existing precedent is the DRAWDOWN indicator, which also tracks state across bars (peak-to-trough).

    These columns, combined with the existing 6 binary signals, provide all the information that the FOLLOW_UP_PLAN items #3, #4, #5, #6 describe:

    FOLLOW_UP_PLAN item Covered by
    #3 Persistence state machine (observed/candidate/confirmed, counters) bars_since_*_change columns give the counters. The model learns its own observed/candidate/confirmed thresholds from data instead of us hardcoding them.
    #4 Confirmation rules (min duration, consecutive-day, majority vote, confidence) bars_since_*_change gives duration. regime_stability gives the majority-vote / confidence measure. The model learns the confirmation rule that works best.
    #5 Hysteresis and transition friction transition_recency explicitly encodes how recent the last change was. Combined with regime_stability, the model can learn to distrust recent transitions with low stability.
    #6 MIXED / low-confidence days regime_stability directly measures this. Low stability = mixed/unstable. The model can learn to reduce position sizing or avoid trading when stability is low.

    The key insight: instead of hardcoding a persistence state machine with arbitrary thresholds (e.g., "confirmed after 3 consecutive days"), we give the model the raw persistence and stability signals and let it learn the optimal thresholds from training data.

    Group 3: Meta-regime

    Standard quantitative finance practice for characterizing regime dynamics.

    Column What it is Why it matters
    trend_momentum Rate of change of ADX over N bars Is the trend strengthening or weakening? A high ADX that is falling means the trend is dying. A low ADX that is rising means a new trend may be forming.
    vol_of_vol Rolling standard deviation of volatility_ratio changes Measures the stability of the volatility regime itself. High vol-of-vol means the market is alternating rapidly between calm and volatile states. Low vol-of-vol means the current volatility state is persistent and trustworthy.

    Implementation complexity: Low. Rate of change can be a SPREAD or simple arithmetic on existing values. Rolling std is a VOLATILITY-type computation.

    Group 4: Cross-timeframe regime agreement (deferred)

    Column What it is
    regime_agreement_htf Do higher-timeframe regimes agree with the base timeframe regime?

    Status: Deferred. This requires multi-timeframe data to be available, which only applies when the experiment uses multiple timeframes (Tier D/E in the thesis). The market_state block configuration should support this group, but implementation is postponed until after Groups 1-3 are validated.

    Design note: When implemented, this should be configurable by specifying which higher timeframes to compare against, not hardcoded to specific tiers.

    Architecture: New MARKET_STATE IndicatorType

    Why a single new indicator type instead of many small ones

    Groups 1, 3 could technically be built from existing indicator types (RATIO, SPREAD, VOLATILITY). But there are strong reasons to unify everything into one indicator type:

    1. Coherent computation. The regime classification (TRENDING/RANGING, BULLISH/BEARISH, HIGH_VOL/LOW_VOL) must be computed once before the Group 2 persistence columns can count bars since last change. If we split the computation, the regime classification happens in multiple places or needs to be passed between features. A single indicator type computes the regime classification once and derives all columns from it.

    2. Single configuration surface. One block config with clear group enable/disable flags. No need to wire up 12 separate features with cross-references.

    3. Consistency with regime detector. The thresholds and logic (ADX >= 25, DI crossover, ATR ratio baseline) live in one place. If we change a threshold, all derived columns update consistently.

    4. Expandable. Adding Group 4 (cross-timeframe) or future groups (e.g., HMM-based regime probabilities) means adding columns to one indicator, not designing new feature types.

    Block configuration schema

    "market_state": {
    "enabled": true,
    "mandatory": false,
    "groups": {
        "continuous": true,      // Group 1: trend_strength, direction_score, etc.
        "persistence": true,     // Group 2: bars_since_*, stability, transition
        "meta": true,            // Group 3: trend_momentum, vol_of_vol
        "cross_tf": false        // Group 4: regime_agreement_htf (deferred)
    },
    "per_tf": [
        {
            "tfs": ["5m"],
            // Regime classification parameters (same as current market_regime)
            "adx_period": 14,
            "adx_strong_trend": 25.0,
            "adx_weak_trend": 20.0,
            "ema_fast_period": 14,
            "ema_slow_period": 50,
            "atr_period": 14,
            "vol_baseline_days": 7,
            "vol_multipliers": {"high": 1.2, "low": 0.8},
            // Persistence parameters (Group 2)
            "stability_window": 288,  // bars for regime_stability rolling window (~1 day of 5m)
            "transition_decay_lambda": 0.01,  // exp decay rate for transition_recency
            // Meta parameters (Group 3)
            "trend_momentum_window": 20,  // bars for ADX rate of change
            "vol_of_vol_window": 50  // bars for rolling std of volatility_ratio
        }
    ]
}

    This configuration is generic and tier-agnostic. Tiers are a thesis concept. The block itself is a reusable indicator that can be configured for any experiment.

    Relationship to existing market_regime block

    The existing market_regime block and its 6 binary signals remain unchanged and available. The new market_state block is a separate, complementary block.

    In the MEGA generator block_groups, tiers that want both would list both blocks:

    "block_groups": {
    // Current Tier B: no regime features
    "standard": ["ohlcv", "ema", "rsi", "macd", "bollinger_bands", "atr", "adx", "stochastic", "obv"],

    // New extended Tier B: adds market_state (groups 1+3 only)
    "standard_ext": ["ohlcv", "ema", "rsi", "macd", "bollinger_bands", "atr", "adx", "stochastic", "obv", "market_state"],

    // Tier C: full single-tf (existing blocks + market_state with all single-tf groups)
    "full_single_tf": [
        "ohlcv", "ema", "rsi", "macd", "bollinger_bands", "atr", "adx", "stochastic",
        "obv", "vwap", "volatility", "time_features", "drawdown", "derived", "signals",
        "market_regime", "market_state"
    ],

    // Tier D/E: same + multi-tf (market_state cross_tf group enabled)
    "full_multi_tf": [
        "ohlcv", "ema", "rsi", "macd", "bollinger_bands", "atr", "adx", "stochastic",
        "obv", "vwap", "volatility", "time_features", "drawdown", "derived", "signals",
        "market_regime", "market_state"
    ]
}

    Group enable/disable per tier would be controlled via block_overrides in the tier config:

    Tier market_state groups enabled Total new columns
    A (not included) 0
    B_ext continuous + meta (Groups 1+3) 7
    C continuous + persistence + meta (Groups 1+2+3) 12
    D/E All groups including cross_tf (Groups 1+2+3+4, when implemented) 12+

    What This Enables

    For Tier B_ext

    The model receives 7 continuous regime features where before it had 0. This gives it the raw ingredients to learn:

    • Trend strength and whether the market is trending, ranging, or in between (without hardcoded 25/20 thresholds)
    • Directional dominance as a continuous score, not a binary flag
    • Volatility relative to its recent baseline (the value that was previously computed and discarded)
    • Bollinger squeeze as a breakout anticipation signal
    • Whether the current trend is strengthening or weakening
    • Whether the volatility regime is stable or itself volatile

    For Tier C

    Everything from Tier B_ext plus 5 persistence columns. This means the model can learn:

    • To trust a regime that has been active for many bars
    • To distrust a regime that just changed
    • To reduce confidence when regime stability is low (MIXED/transition zones)
    • To behave differently in the first bars of a new regime vs an established regime
    • The optimal confirmation thresholds from data instead of hardcoded rules

    For the thesis

    The market_state block directly enables the training options proposed in the Regime Training Options Audit:

    • Option D (global model plus regime features): Add market_state columns to the global training set and test whether regime information helps the model specialize.
    • Option E (regime-aware do-not-trade): Use regime_stability and transition_recency to learn when NOT to trade.
    • Phase 3 (transition safety): The persistence columns implement the candidate/confirmed concept without hardcoding the state machine.

    Implementation Phases

    Phase 1: Group 1 + Group 3 (continuous + meta)

    • 7 columns, all implementable with existing indicator types or simple arithmetic
    • No new IndicatorType required yet
    • Can be built as individual RATIO/SPREAD indicators composed together in the MEGA generator
    • Deliverable: new market_state block in blocks_catalog with continuous and meta groups
    • Risk: low

    Phase 2: Group 2 (persistence)

    • 5 columns requiring stateful computation
    • New IndicatorType.MARKET_STATE added to the pipeline
    • The indicator receives regime classification parameters, classifies each bar, and tracks state
    • Deliverable: persistence columns working in the pipeline, enabled for Tier C+
    • Risk: medium (new indicator type in the data pipeline)

    Phase 3: Group 4 (cross-timeframe)

    • Deferred until Groups 1-3 are validated
    • Requires multi-timeframe experiments to be running
    • Design note: configurable by listing which higher timeframes to compare against

    Open Questions

    1. Should Group 1 columns go through RELATIVE_INDICATOR normalization? Some of them are already relative (volatility_ratio, bb_squeeze are centered around 1.0). Others are bounded (trend_strength 0-100, direction_score -1 to +1). The normalization pipeline may need to be configured per-column.

    2. Scaling for bars_since_* columns. Raw bar counts can be very large (thousands). Options: log-scale, divide by a configurable window, or let the normalization pipeline handle it. Log-scale is most common in quant literature for duration features.

    3. Should the 6 existing binary signals be kept when market_state is active? Current recommendation: yes, keep both. The binaries are cheap and serve as easy shortcuts. The continuous columns provide the gradient. They complement each other.

    4. Interaction with derived.di_dominance. The direction_score column overlaps with the existing di_dominance derived feature. When both market_state and derived blocks are active, we should either skip the duplicate or let feature selection handle it.

    How This Gets Built: The Indicator In Detail

    Everything above describes what we want. Everything below describes how we build it, based on how indicators actually work in this codebase.

    Anatomy of an indicator in this system

    Every indicator in rainbow-trader follows the same pattern. Before designing MARKET_STATE, here is the pattern that ADX, DRAWDOWN, Bollinger Bands, and all 17 existing indicators follow.

    Files per indicator

    # Module File What it contains
    1 rainbow-common types/configs/indicators/{name}_config.py Pydantic config class extending BaseIndicatorGenerated. Declares column_outputs_generated: ClassVar[list[str]] (fixed list of output column names), instance parameters (periods, windows, thresholds), test config, maybe-config for optional overrides.
    2 rainbow-common types/configs/__init__.py Import config classes. Add to IndicatorConfigUnion type alias. Add to __all__.
    3 rainbow-common types/enums.py Add value to IndicatorType enum. Add entry to INDICATOR_GROUPS dict (maps indicator to FeatureGroup for feature selection taxonomy).
    4 rainbow-core types/composed/indicators.py Pydantic composed type extending BaseIndicator. Declares relative_meta (how each column is relativized), normalization_meta (how each column is normalized), get_warm_up_periods(), get_config(). Also: add {name}_config field to IndicatorConfig wrapper. Update BaseIndicator.indicator_type Literal union.
    5 rainbow-data-processing indicator/{name}.py NEW FILE. The actual computation: process_{name}(inputs, config, ...) -> ProcessResult. Pure Polars expressions. Validates inputs, filters by range, computes columns, trims warm-up, cleans NaN/Inf, validates output.
    6 rainbow-data-processing helpers/indicator_dispatch.py Add one isinstance(config, XxxConfig): return process_xxx(...) branch to the dispatch chain.
    7 MEGA generator generate_mega_experiment.py Add block handler (reads config from JSON, calls builder.ensure_* for dependencies, calls builder.add_indicator with input refs).
    8 MEGA generator generate_mega_experiment.json Add entry to blocks_catalog with default config. Add to relevant block_groups.

    Total per indicator: 6 existing files to modify + 1 new file + 1 JSON config.

    Concrete example: DRAWDOWN indicator

    Config (drawdown_config.py):

    class DrawdownConfig(BaseIndicatorGenerated):
    """Drawdown. Columns: drawdown, max_drawdown"""
    segement_path: ClassVar[Path | str | None] = Path("indicators")
    column_outputs_generated: ClassVar[list[str]] = ["drawdown", "max_drawdown"]
    period: int
    test: DrawdownTestConfig

    Composed type (indicators.py):

    class DrawdownIndicator(BaseIndicatorOneInput):
    indicator_type: Literal[IndicatorType.DRAWDOWN] = IndicatorType.DRAWDOWN
    feature_group: ClassVar[FeatureGroup] = FeatureGroup.VOLATILITY
    config_type: ClassVar[type[BaseConfig]] = DrawdownConfig
    relative_meta: ClassVar = {
        "drawdown": RelativeIndicatorMeta(method=RelativeMethod.NONE),
        "max_drawdown": RelativeIndicatorMeta(method=RelativeMethod.NONE),
    }
    normalization_meta: ClassVar = {
        "drawdown_rel": NormalizationMethod.NONE,
        "max_drawdown_rel": NormalizationMethod.NONE,
    }
    def get_warm_up_periods(self) -> int:
        return self.get_config().period * 2

    Process function (drawdown.py): validates inputs → filters time range → computes rolling_max → drawdown = (price - rolling_max) / rolling_max → max_drawdown = rolling_min(drawdown) → trims warm-up → cleans NaN/Inf → validates → returns ProcessResult.

    Dispatch (indicator_dispatch.py): one branch in the isinstance chain:

    if isinstance(indicator_config, DrawdownConfig):
    return process_drawdown(indicator_inputs, indicator_config, ...)

    MARKET_STATE follows this exact pattern. The only difference is: more output columns (12 vs 2), more inputs (6 vs 1), and the computation includes regime classification + run-length counting.

    MarketStateConfig design

    The Pydantic config for the indicator itself. This is what lives in rainbow-common/types/configs/indicators/market_state_config.py.

    class MarketStateConfig(BaseIndicatorGenerated):
    """Market State indicator. Produces 12 continuous columns characterizing
    regime strength, persistence, transitions, and meta-stability.

    Columns:
        Group 1 (continuous): trend_strength, direction_score, ema_trend,
            volatility_ratio, bb_squeeze
        Group 2 (persistence): bars_since_trend_change, bars_since_direction_change,
            bars_since_vol_change, regime_stability, transition_recency
        Group 3 (meta): trend_momentum, vol_of_vol
    """

    segement_path: ClassVar[Path | str | None] = Path("indicators")

    # All 12 columns, always computed. Which ones become experiment features
    # is decided by the MEGA generator, not the indicator.
    column_outputs_generated: ClassVar[list[str]] = [
        # Group 1: continuous regime components
        "trend_strength",
        "direction_score",
        "ema_trend",
        "volatility_ratio",
        "bb_squeeze",
        # Group 2: persistence and transitions
        "bars_since_trend_change",
        "bars_since_direction_change",
        "bars_since_vol_change",
        "regime_stability",
        "transition_recency",
        # Group 3: meta-regime
        "trend_momentum",
        "vol_of_vol",
    ]

    # --- Regime classification thresholds ---
    adx_strong_trend: float = 25.0    # ADX >= this → TRENDING
    adx_weak_trend: float = 20.0      # ADX <= this → RANGING (between = NEUTRAL)
    ema_fast_period: int = 14         # For ema_trend and direction classification
    ema_slow_period: int = 50         # For ema_trend and direction classification

    # --- Volatility baseline ---
    vol_baseline_window: int = 2016   # Rolling mean window for ATR/Close ratio baseline.
                                      # Default 2016 = 7 days of 5m bars.
                                      # For 1h: 168. For 1d: 30.
    vol_high_multiplier: float = 1.2  # volatility_ratio > this → HIGH_VOL regime
    vol_low_multiplier: float = 0.8   # volatility_ratio < this → LOW_VOL regime

    # --- Persistence parameters (Group 2) ---
    stability_window: int = 288       # Rolling window for regime_stability.
                                      # Default 288 = 1 day of 5m bars.
                                      # For 1h: 24. For 1d: 7.
    transition_decay_lambda: float = 0.01  # Exponential decay rate for transition_recency.
                                           # 0.01 → half-life ~69 bars (~5.75h at 5m).

    # --- Meta parameters (Group 3) ---
    trend_momentum_window: int = 20   # Window for ADX rate of change.
    vol_of_vol_window: int = 50       # Window for rolling std of volatility_ratio changes.

    test: MarketStateTestConfig


class MarketStateTestConfig(BaseConfig, BaseIndicatorValidation):
    """Market State validation configuration."""


class MaybeMarketStateConfig(IndicatorColumnMeta):
    """Optional per-feature Market State overrides."""
    adx_strong_trend: float | None = None
    adx_weak_trend: float | None = None
    ema_fast_period: int | None = None
    ema_slow_period: int | None = None
    vol_baseline_window: int | None = None
    vol_high_multiplier: float | None = None
    vol_low_multiplier: float | None = None
    stability_window: int | None = None
    transition_decay_lambda: float | None = None
    trend_momentum_window: int | None = None
    vol_of_vol_window: int | None = None

    Why all 12 columns are always computed (not configurable at indicator level)

    Every indicator in this system has column_outputs_generated as a ClassVar — a class-level constant. ADX always produces 3 columns (adx, plus_di, minus_di). Bollinger always produces 5 (upper_band, middle_band, lower_band, bandwidth, percent_b). Not all may be used by every experiment.

    MARKET_STATE follows the same rule: always 12 columns. Reasons:

    1. Pipeline consistency. The pipeline assumes column_outputs_generated is fixed. The relative_indicator and normalization stages iterate over this list. Dynamic columns would break this assumption.
    2. One code path. process_market_state has one implementation. No conditional logic for "which groups are enabled."
    3. Cheap. The computation is vectorized Polars executing in Rust. 12 columns vs 7 is negligible. The expensive part is the parent indicators (ADX, EMA, ATR, BB), already computed and cached as parquets.
    4. The MEGA generator controls what becomes a feature. The generator reads feature_groups flags from the block config and only adds the selected columns as experiment features. Unused columns exist in the indicator parquet but are never loaded into the FAT window.

    vol_of_vol_window=50 ~4.2h at 5m. Enough samples to be meaningful, not so many it smooths everything.

    MarketStateIndicator design (composed type)

    This is what lives in rainbow-core/types/composed/indicators.py:

    class MarketStateIndicator(BaseIndicator):
    """Market State. 12 continuous columns for regime characterization."""

    indicator_type: Literal[IndicatorType.MARKET_STATE] = IndicatorType.MARKET_STATE
    feature_group: ClassVar[FeatureGroup] = FeatureGroup.TREND
    config_type: ClassVar[type[BaseConfig]] = MarketStateConfig

    # Per-column relativization. Most columns are already bounded or relative.
    relative_meta: ClassVar[dict[str, RelativeIndicatorMeta]] = {
        "trend_strength": RelativeIndicatorMeta(method=RelativeMethod.DIVIDE_BY_100),
            # ADX 0-100 → 0-1
        "direction_score": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Already [-1, +1]
        "ema_trend": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Already relative: (fast-slow)/slow
        "volatility_ratio": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Already centered ~1.0
        "bb_squeeze": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Already centered ~1.0
        "bars_since_trend_change": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Log-scaled in computation
        "bars_since_direction_change": RelativeIndicatorMeta(method=RelativeMethod.NONE),
        "bars_since_vol_change": RelativeIndicatorMeta(method=RelativeMethod.NONE),
        "regime_stability": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Already [0, 1]
        "transition_recency": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Already [0, 1]
        "trend_momentum": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Small continuous, normalization handles it
        "vol_of_vol": RelativeIndicatorMeta(method=RelativeMethod.NONE),
            # Non-negative, normalization handles it
    }

    # Per-column normalization (after relative_indicator pipeline).
    # Suffix _rel because normalization runs after the relative_indicator step.
    normalization_meta: ClassVar[dict[str, NormalizationMethod]] = {
        "trend_strength_rel": NormalizationMethod.NONE,      # 0-1 after DIVIDE_BY_100
        "direction_score_rel": NormalizationMethod.NONE,      # [-1, +1]
        "ema_trend_rel": NormalizationMethod.ROBUST,          # Unbounded, needs scaling
        "volatility_ratio_rel": NormalizationMethod.NONE,     # ~1.0 center, bounded in practice
        "bb_squeeze_rel": NormalizationMethod.NONE,           # ~1.0 center, bounded in practice
        "bars_since_trend_change_rel": NormalizationMethod.NONE,   # Log-scaled [0, ~1]
        "bars_since_direction_change_rel": NormalizationMethod.NONE,
        "bars_since_vol_change_rel": NormalizationMethod.NONE,
        "regime_stability_rel": NormalizationMethod.NONE,     # [0, 1]
        "transition_recency_rel": NormalizationMethod.NONE,   # [0, 1]
        "trend_momentum_rel": NormalizationMethod.ROBUST,     # Unbounded
        "vol_of_vol_rel": NormalizationMethod.ROBUST,         # Unbounded
    }

    market_state_config: MaybeMarketStateConfig | None = None

    def get_warm_up_periods(self) -> int:
        config = self.get_config()
        return self.get_warm_up_periods_static(config)

    @classmethod
    def get_warm_up_periods_static(cls, config: MarketStateConfig) -> int:
        """Warm-up dominated by vol_baseline_window. Uses 2x conservative rule."""
        _ = cls
        max_window = max(
            config.vol_baseline_window,
            config.stability_window,
            config.trend_momentum_window,
            config.vol_of_vol_window,
        )
        return max_window * 2

    Warm-up analysis

    Window Default (5m) Bars Warm-up (2x)
    vol_baseline_window 2016 7 days 4032 bars (~14 days)
    stability_window 288 1 day 576 bars (~2 days)
    trend_momentum_window 20 ~1.7h 40 bars
    vol_of_vol_window 50 ~4.2h 100 bars

    Dominant: vol_baseline_window → 4032 bars warm-up at 5m. This is the same cost as the existing market_regime block (which already uses a 2016-bar rolling mean for the ATR/Close baseline). No new warm-up penalty compared to existing regime features.

    The parent indicators (ADX, EMA, ATR, BB) have their own warm-up handled by the pipeline before MARKET_STATE runs. ADX warm-up is ~42 bars, EMA slow is ~150 bars — much smaller than MARKET_STATE's own 4032.

    process_market_state computation (pseudocode)

    This is the core function in rainbow-data-processing/indicator/market_state.py. All computations are vectorized Polars — no Python loops.

    def process_market_state(
    inputs: list[IndicatorInput],
    config: MarketStateConfig,
    timeframe: str, range_start: datetime, range_end: datetime, ...
) -> ProcessResult:

    # ── 1. VALIDATE AND JOIN INPUTS ──
    validated = validate_and_get_indicator_inputs(
        inputs, expected_count=6,
        input_names=["input_adx", "input_ema_fast", "input_ema_slow",
                     "input_atr", "input_close", "input_bb"]
    )
    frame = join_all_inputs_on_timestamp(validated)
    # Columns available: adx, plus_di, minus_di, ema (x2), atr, close, bandwidth

    # ── 2. GROUP 1: CONTINUOUS REGIME COMPONENTS ──

    frame = frame.with_columns(
        # trend_strength: raw ADX passthrough
        pl.col("adx").alias("trend_strength"),

        # direction_score: normalized DI dominance, bounded [-1, +1]
        pl.when(pl.col("plus_di") + pl.col("minus_di") > 0)
        .then((pl.col("plus_di") - pl.col("minus_di"))
              / (pl.col("plus_di") + pl.col("minus_di")))
        .otherwise(0.0)
        .alias("direction_score"),

        # ema_trend: relative EMA alignment
        pl.when(pl.col("input_ema_slow_ema") != 0)
        .then((pl.col("input_ema_fast_ema") - pl.col("input_ema_slow_ema"))
              / pl.col("input_ema_slow_ema"))
        .otherwise(0.0)
        .alias("ema_trend"),
    )

    # volatility_ratio: ATR/Close relative to its rolling mean
    frame = frame.with_columns(
        (pl.col("atr") / pl.col("close")).alias("_atr_close_ratio")
    ).with_columns(
        pl.col("_atr_close_ratio")
        .rolling_mean(window_size=config.vol_baseline_window,
                      min_periods=config.vol_baseline_window)
        .alias("_atr_close_baseline")
    ).with_columns(
        pl.when(pl.col("_atr_close_baseline") > 0)
        .then(pl.col("_atr_close_ratio") / pl.col("_atr_close_baseline"))
        .otherwise(1.0)
        .alias("volatility_ratio")
    )

    # bb_squeeze: bandwidth relative to its rolling mean
    frame = frame.with_columns(
        pl.col("bandwidth")
        .rolling_mean(window_size=config.vol_baseline_window,
                      min_periods=config.vol_baseline_window)
        .alias("_bb_baseline")
    ).with_columns(
        pl.when(pl.col("_bb_baseline") > 0)
        .then(pl.col("bandwidth") / pl.col("_bb_baseline"))
        .otherwise(1.0)
        .alias("bb_squeeze")
    )

    # ── 3. INTERNAL REGIME CLASSIFICATION ──
    # These intermediate columns are NOT output. They are used to compute Group 2.
    # Classifies each bar into discrete regime states.

    frame = frame.with_columns(
        # Trend: 0=RANGING, 1=NEUTRAL, 2=TRENDING
        pl.when(pl.col("adx") >= config.adx_strong_trend).then(pl.lit(2))
        .when(pl.col("adx") <= config.adx_weak_trend).then(pl.lit(0))
        .otherwise(pl.lit(1))
        .cast(pl.Int8).alias("_trend_regime"),

        # Direction: 0=BEARISH, 1=NEUTRAL, 2=BULLISH
        pl.when(
            (pl.col("plus_di") > pl.col("minus_di"))
            & (pl.col("input_ema_fast_ema") > pl.col("input_ema_slow_ema"))
        ).then(pl.lit(2))
        .when(
            (pl.col("minus_di") > pl.col("plus_di"))
            & (pl.col("input_ema_fast_ema") < pl.col("input_ema_slow_ema"))
        ).then(pl.lit(0))
        .otherwise(pl.lit(1))
        .cast(pl.Int8).alias("_direction_regime"),

        # Volatility: 0=LOW_VOL, 1=NORMAL_VOL, 2=HIGH_VOL
        pl.when(pl.col("volatility_ratio") > config.vol_high_multiplier).then(pl.lit(2))
        .when(pl.col("volatility_ratio") < config.vol_low_multiplier).then(pl.lit(0))
        .otherwise(pl.lit(1))
        .cast(pl.Int8).alias("_vol_regime"),
    )

    # Combined regime: unique integer 0-26 (3×3×3 = 27 possible states)
    frame = frame.with_columns(
        (pl.col("_trend_regime") * 9
         + pl.col("_direction_regime") * 3
         + pl.col("_vol_regime"))
        .cast(pl.Int16).alias("_combined_regime")
    )

    # ── 4. GROUP 2: PERSISTENCE AND TRANSITIONS ──
    #
    # The key technique: detect changes → create run groups via cum_sum
    # → count bars within each group. Fully vectorized, no Python loops.

    # Detect changes (0 = same as previous bar, 1 = changed)
    frame = frame.with_columns(
        (pl.col("_trend_regime") != pl.col("_trend_regime").shift(1))
            .cast(pl.Int32).alias("_trend_changed"),
        (pl.col("_direction_regime") != pl.col("_direction_regime").shift(1))
            .cast(pl.Int32).alias("_dir_changed"),
        (pl.col("_vol_regime") != pl.col("_vol_regime").shift(1))
            .cast(pl.Int32).alias("_vol_changed"),
        (pl.col("_combined_regime") != pl.col("_combined_regime").shift(1))
            .cast(pl.Int32).alias("_combined_changed"),
    )

    # bars_since_*_change:
    #   Step 1: cum_sum(change_flag) → group ID that increments at each change
    #   Step 2: cum_count() within each group → bars since last change
    #   Step 3: Log-scale: log1p(counter) / log1p(stability_window)
    #           This maps [0, ∞) to [0, ~1+] where 1.0 = stable for stability_window bars
    scale = config.stability_window

    for prefix, change_col, output_col in [
        ("trend", "_trend_changed", "bars_since_trend_change"),
        ("dir", "_dir_changed", "bars_since_direction_change"),
        ("vol", "_vol_changed", "bars_since_vol_change"),
    ]:
        run_group = f"_{prefix}_run_group"
        raw_counter = f"_{prefix}_raw_counter"
        frame = frame.with_columns(
            pl.col(change_col).cum_sum().alias(run_group)
        ).with_columns(
            pl.lit(1).cum_sum().over(run_group).alias(raw_counter)
        ).with_columns(
            (pl.col(raw_counter).cast(pl.Float64).log1p()
             / math.log1p(scale))
            .alias(output_col)
        )

    # regime_stability: fraction of last N bars without combined regime changes
    # = 1.0 - (number_of_changes_in_window / window_size)
    # Approximation of "fraction of window with same regime". Fully vectorized.
    frame = frame.with_columns(
        (1.0 - pl.col("_combined_changed")
         .rolling_sum(window_size=config.stability_window, min_periods=1)
         .cast(pl.Float64) / config.stability_window)
        .alias("regime_stability")
    )

    # transition_recency: exp(-lambda * bars_since_combined_change)
    frame = frame.with_columns(
        pl.col("_combined_changed").cum_sum().alias("_combined_run_group")
    ).with_columns(
        pl.lit(1).cum_sum().over("_combined_run_group").alias("_combined_raw_counter")
    ).with_columns(
        (-config.transition_decay_lambda
         * pl.col("_combined_raw_counter").cast(pl.Float64))
        .exp()
        .alias("transition_recency")
    )

    # ── 5. GROUP 3: META-REGIME ──

    # trend_momentum: rate of change of ADX
    frame = frame.with_columns(
        ((pl.col("adx") - pl.col("adx").shift(config.trend_momentum_window))
         / config.trend_momentum_window)
        .alias("trend_momentum")
    )

    # vol_of_vol: rolling std of volatility_ratio changes
    frame = frame.with_columns(
        pl.col("volatility_ratio").diff()
        .rolling_std(window_size=config.vol_of_vol_window,
                     min_periods=config.vol_of_vol_window)
        .alias("vol_of_vol")
    )

    # ── 6. CLEANUP ──
    # Drop internal columns (prefix _), keep 12 output columns + timestamp + inputs
    # Standard: trim_warm_up, clean_indicator_output, validate
    output_columns = ["timestamp", ...inputs...,
                      *MarketStateConfig.column_outputs_generated, SYNTHETIC_COL]
    # ... trim, clean, validate, return ProcessResult ...

    Key implementation notes

    1. No Python loops. The bars_since_*_change counters use cum_sum to create run-length groups and cum_sum().over(group) to count within groups. These are pure Polars expressions executing in Rust.

    2. regime_stability approximation. The exact definition ("fraction of last N bars with same regime as current bar") would require rolling_map with a Python callback (~100x slower). We use the approximation 1 - change_frequency: 1.0 - rolling_sum(change_flag, window) / window. This measures "what fraction of the window had no regime change" — highly correlated with the exact definition, and fully vectorized.

    3. Log-scaling for counters. Raw bars_since_* can be thousands (a regime lasting 1 day at 5m = 288 bars, lasting 1 week = 2016). Log-scaling: log1p(counter) / log1p(stability_window) compresses to ~[0, 1+]. Value of 1.0 means "stable for at least stability_window bars." Values >1.0 for very long regimes — fine, the model can learn what these mean.

    4. Division safety. All divisions guard against zero denominators with pl.when(...).then(...).otherwise(...).

    5. First-bar handling. shift(1) produces null for the first row. != null evaluates to null, which .cast(pl.Int32) maps to 0. So the first bar counts as "no change" — correct behavior.
