    Script 2 Ensemble Discovery Campaign

    • Date: 2026-04-03
    • Scope: Phase 1 systematic ensemble campaign for Script 2
    • Goal: find which model families and cross-model combinations have the best chance of producing net profit
    • Status: ready for execution

    1. Purpose

    This document defines the first large ensemble campaign to answer one practical thesis-side question:

    Which models, and which combinations of models, have the best chance of making money under real friction?

    This is not a generic grid search.

    The campaign is intentionally centered on:

    • the folds where positive net PnL already appeared
    • the runs that currently define the strongest Tier B frontier
    • merge rules that explicitly test selectivity versus permissiveness
    • diversity-aware selection as a first-class axis, not as a side experiment

    2. Evidence Base

    The campaign is driven by the following evidence.

    2.1 Folds

    From 20260401_SCRIPT2_PRELIMINARY_ANALYSIS.md:

    • only 14 / 1185 trainable trials have positive PnL
    • all positive trials are in F1 or F3
    • F2 and F4 produced zero positive trials

    Decision: Phase 1 targets F1 and F3 only.

    2.2 Core problem

    From 20260331_SCRIPT2_SECTION11_REGEN.md:

    • Trades vs PnL = -0.81
    • S2 vs PnL = -0.01
    • friction drag dominates monetization

    From 20260331_TRADE_SELECTIVITY_ANALYSIS.md:

    • the current system already shows that selectivity is the main lever
    • stricter gating is a plausible path to monetization

    Decision: merge-rule coverage must include permissive, selective, and highly selective rules.

    2.3 Runs to use

    From 20260331_SCRIPT2_SECTION2_REGEN.md:

    • 20260325_tierB_mlp_ex contributes heavily to the current best frontier
    • 20260328_tierB_corrected is the corrected forced-feature Tier B run and must be part of the search frontier

    From 20260328_SCRIPT2_V4_TIER_B_CORRECTED_AUDIT.md:

    • 20260328_tierB_corrected is structurally different from 20260325_tierB_mlp_ex
    • its role is not redundant; it is the forced-feature contrast

    Decision: Phase 1 uses the combined source pool:

    • 20260325_tierB_mlp_ex
    • 20260328_tierB_corrected

    This maximizes the candidate frontier while staying inside the strongest currently audited Tier B sources.

    2.4 Model families

    From 20260331_SCRIPT2_TIER_B_RESCUE_AUDIT.md:

    • TCN is the strongest overall money-oriented family in Tier B
    • VIT is consistently strong and appears repeatedly in top rankings
    • MLP remains a credible money candidate
    • logistic is useful as a baseline/simple family but not a core money leader
    • mlp_ex is powerful but overfitting-prone
    • axial_vit is exploratory

    Decision: Phase 1 centers on:

    • mlp
    • tcn
    • vit

    and includes one broader cross-model pool with logistic to test whether a simple family helps stabilize voting.

    2.5 Diversity-aware selection

    From the current ensemble work in 20260401_SCRIPT2_ENSEMBLE_AND_REGIME_IDEAS.md:

    • prediction-diversity-aware planning is now implemented
    • it can constrain:
    • probability correlation on prob_none, prob_long, prob_short
    • signal agreement on final LONG/SHORT/NONE decisions

    Decision: diversity is a primary axis of Phase 1, not an optional afterthought.

    3. Phase 1 Design

    Phase 1 is a discovery campaign.

    Its job is:

    • identify which families can monetize as single-family ensembles
    • identify which cross-model combinations look stronger than single-family pools
    • test whether diversity-aware selection helps or hurts monetization

    Phase 1 does not try to answer every possible question.

    It does not include:

    • Tier A
    • 20260401_tierB_rescue_v5
    • axial_vit
    • mlp_ex as a core Phase 1 family
    • F2 or F4

    Those can come in Phase 2 after we see what looks alive.

    4. Campaign Axes

    4.1 Fixed source pool

    All Phase 1 plans use these two summary CSVs together:

    • scripts/thesis/script_2_thesis/results/20260325_tierB_mlp_ex/script2_experiment_ids.csv
    • scripts/thesis/script_2_thesis/results/20260328_tierB_corrected/script2_experiment_ids.csv

    4.2 Folds

    • F1
    • F3

    4.3 Ranking modes

    • promise
    • friction

    4.4 Diversity modes

    Three planner modes are used:

    1. off No diversity filter.

    2. mild

    3. --max-prob-corr 0.85

    4. --max-signal-agreement 0.60

    5. tight

    6. --max-prob-corr 0.60
    7. --max-signal-agreement 0.45

    These are not claimed as optimal values. They are simply the first coarse ladder for answering:

    • does diversity-aware selection help at all?
    • and does a tighter diversity constraint help more than a softer one?

    4.5 Merge rules

    Every plan uses the current full 8-rule set:

    • average
    • majority_vote
    • all_must_agree
    • majority_vote:0.75
    • majority_vote:0.85
    • weighted_average
    • weighted_majority_vote:0.75
    • weighted_majority_vote:0.85

    This is important because Phase 1 is not only about source selection. It is also about the merge rule itself.

    5. Block Structure

    Phase 1 contains two blocks.

    5.1 Block A — Single-Family Monetization

    Purpose:

    • test whether multiple trials from the same family can stabilize and monetize that family's edge

    Families:

    • mlp
    • tcn
    • vit

    Top-per-model:

    • 2
    • 3

    Count:

    • 3 families
    • 2 folds
    • 2 rankings
    • 2 top-per-model values
    • 3 diversity modes

    Total:

    • 3 * 2 * 2 * 2 * 3 = 72 plans

    5.2 Block B — Cross-Model Monetization

    Purpose:

    • test whether certain family combinations are stronger than any one family alone

    Model pools:

    1. tcn,vit
    2. mlp,tcn
    3. mlp,vit
    4. mlp,tcn,vit
    5. logistic,mlp,tcn,vit

    Top-per-model:

    • 1

    Rationale:

    • tcn,vit tests the two strongest money/frontier families together
    • mlp,tcn tests whether mlp complements tcn
    • mlp,vit tests whether mlp complements vit
    • mlp,tcn,vit is the main three-family money pool
    • logistic,mlp,tcn,vit tests whether the simple linear family adds stabilizing value rather than just noise

    Count:

    • 5 model pools
    • 2 folds
    • 2 rankings
    • 1 top-per-model value
    • 3 diversity modes

    Total:

    • 5 * 2 * 2 * 1 * 3 = 60 plans

    5.3 Total Phase 1 Size

    • Block A: 72 plans
    • Block B: 60 plans

    Phase 1 total:

    • 132 plans

    With 8 merge rules each:

    • 132 * 8 = 1056 ensemble backtests

    6. Command Template

    All Phase 1 commands use the same shell helpers.

    source .venv/bin/activate

PLANNER="scripts/thesis/script_2_thesis/ensemble_backtests/generate_ensemble_backtest_plan.py"
RUNNER="scripts/thesis/script_2_thesis/ensemble_backtests/run_ensemble_backtests.py"
ROOT="scripts/thesis/script_2_thesis/ensemble_backtests/results/20260403_phase1_discovery"

SRC_A="scripts/thesis/script_2_thesis/results/20260325_tierB_mlp_ex/script2_experiment_ids.csv"
SRC_B="scripts/thesis/script_2_thesis/results/20260328_tierB_corrected/script2_experiment_ids.csv"

MERGE8="average,majority_vote,all_must_agree,majority_vote:0.75,majority_vote:0.85,weighted_average,weighted_majority_vote:0.75,weighted_majority_vote:0.85"

gen_plan() {
  local name="$1"
  local fold="$2"
  local rank_by="$3"
  local top_per_model="$4"
  local models="$5"
  local diversity="$6"

  shift 6

  local extra=()
  if [ "$diversity" = "mild" ]; then
    extra=(--max-prob-corr 0.85 --max-signal-agreement 0.60)
  elif [ "$diversity" = "tight" ]; then
    extra=(--max-prob-corr 0.60 --max-signal-agreement 0.45)
  fi

  .venv/bin/python "$PLANNER" \
    --summary-csv "$SRC_A" \
    --summary-csv "$SRC_B" \
    --fold "$fold" \
    --tier B \
    --models "$models" \
    --top-per-model "$top_per_model" \
    --rank-by "$rank_by" \
    --merge-rules "$MERGE8" \
    "${extra[@]}" \
    --output-dir "$ROOT/$name"
}

run_plan() {
  local name="$1"
  .venv/bin/python "$RUNNER" --plan-json "$ROOT/$name/plan.json"
}

mkdir -p "$ROOT"

    7. Block A Commands — Single-Family Monetization

    for fold in 1 3; do
  for rank_by in promise friction; do
    for top_per_model in 2 3; do
      for model in mlp tcn vit; do
        for diversity in off mild tight; do
          name="A_${model}_f${fold}_${rank_by}_top${top_per_model}_${diversity}"
          gen_plan "$name" "$fold" "$rank_by" "$top_per_model" "$model" "$diversity"
          run_plan "$name"
        done
      done
    done
  done
done

    This block produces 72 plans.

    What it answers:

    • can one family monetize better when we ensemble several of its own best trials?
    • does that answer differ between F1 and F3?
    • does friction ranking select better members than promise ranking?
    • does diversity-aware selection help even inside one family?

    8. Block B Commands — Cross-Model Monetization

    for fold in 1 3; do
  for rank_by in promise friction; do
    for diversity in off mild tight; do
      name="B_tcn_vit_f${fold}_${rank_by}_top1_${diversity}"
      gen_plan "$name" "$fold" "$rank_by" 1 "tcn,vit" "$diversity"
      run_plan "$name"

      name="B_mlp_tcn_f${fold}_${rank_by}_top1_${diversity}"
      gen_plan "$name" "$fold" "$rank_by" 1 "mlp,tcn" "$diversity"
      run_plan "$name"

      name="B_mlp_vit_f${fold}_${rank_by}_top1_${diversity}"
      gen_plan "$name" "$fold" "$rank_by" 1 "mlp,vit" "$diversity"
      run_plan "$name"

      name="B_mlp_tcn_vit_f${fold}_${rank_by}_top1_${diversity}"
      gen_plan "$name" "$fold" "$rank_by" 1 "mlp,tcn,vit" "$diversity"
      run_plan "$name"

      name="B_log_mlp_tcn_vit_f${fold}_${rank_by}_top1_${diversity}"
      gen_plan "$name" "$fold" "$rank_by" 1 "logistic,mlp,tcn,vit" "$diversity"
      run_plan "$name"
    done
  done
done

    This block produces 60 plans.

    What it answers:

    • which pair or trio of money-relevant families combine best?
    • does logistic add stabilizing signal or just dilute the pool?
    • does diversity-aware selection improve the cross-model pools more than it improves single-family pools?

    9. Recommended Execution Order

    Do not run the 132 plans in arbitrary order.

    Use this order:

    1. Block B first, rank_by=promise, diversity=off
    2. Block B, rank_by=promise, diversity=mild
    3. Block B, rank_by=friction, diversity=off
    4. Block B, rank_by=friction, diversity=mild
    5. Block A, rank_by=promise, diversity=off
    6. Block A, rank_by=promise, diversity=mild
    7. Add all tight runs only after the earlier groups finish cleanly

    Why:

    • the cross-model pools are the main thesis question
    • promise is the main ranking hypothesis
    • friction is the main challenger ranking
    • tight diversity is important, but it should not block the first read of the study

    10. What To Audit After Phase 1

    For every completed run family, compare:

    • best ensemble net PnL
    • best ensemble PnL+Fric
    • best ensemble gross/trade
    • whether the winner came from:
    • single-family
    • cross-model
    • diversity off
    • diversity mild
    • diversity tight
    • promise
    • friction

    The most important questions after Phase 1 are:

    1. Which family or family-combination appears most often among the least-bad or profitable runs?
    2. Does diversity-aware selection improve monetization, or only reduce trade count without real edge preservation?
    3. Which merge rules win most often for the strongest source pools?
    4. Does friction ranking beat promise ranking when judged by net PnL and gross proxy?

    11. Explicitly Deferred To Phase 2

    Phase 2 should come only after Phase 1 has been audited.

    Deferred items:

    • F2 and F4 control folds
    • source provenance ablation:
    • 20260325 only
    • 20260328 only
    • combined
    • mlp_ex expansion
    • axial_vit exploration
    • tighter diversity ladders beyond off / mild / tight
    • ranking sensitivity beyond promise / friction
    • any use of 20260401_tierB_rescue_v5

    12. Bottom Line

    This Phase 1 campaign is intentionally money-oriented.

    It is not trying to prove every abstract ensemble idea.

    It is trying to answer:

    • whether mlp, tcn, vit, and their combinations can be turned into better net trading outcomes
    • whether diversity-aware selection helps those pools
    • and which merge rules are most compatible with the surviving pools

    If Phase 1 produces signal, Phase 2 can then expand and validate.
