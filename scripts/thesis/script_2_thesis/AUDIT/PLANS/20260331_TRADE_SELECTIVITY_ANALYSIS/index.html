    Trade Selectivity Analysis — Levers for Reducing Trade Count in Tier B Rescue

    • Date: 2026-03-31
    • Scope: Identify all available mechanisms to make models more selective (fewer, higher-quality trades) within the Script 2 thesis framework
    • Context: The Tier B Rescue Audit (Section 11.2, "Friction Paradox") showed that high trade count destroys PnL even when signal quality (S2) is strong. Only 14/1185 trials achieved positive net PnL. Profitable trials average ≤200 trades vs 300+ for unprofitable ones.
    • Status: Analysis complete. One actionable lever identified. Others documented for future use.

    1. The Problem — Why Trade Reduction Matters

    From the Tier B Rescue Audit, Section 11.2 (The Friction Paradox):

    • Best S2 trials are the worst PnL performers. The top-15 S2 ranking has ZERO positive-PnL trials.
    • Friction kills edge: The gap between best gross edge (+$1,681) and best net PnL (+$186) is ~$1,500 destroyed by transaction costs.
    • Profitable trials are selective: 12 of 14 profitable trials have ≤200 trades. T2530 (TCN-F3, best overall) made only 17 trades.
    • F2 and F4 produce zero profitable trials out of 612 combined — a regime + friction effect.

    Conclusion: The model needs to be more selective about WHEN to trade. The question is: what levers exist to achieve this within the thesis experimental framework?

    2. The Combined Confidence Formula

    The bot's entry decision is governed by:

    # File: modules/model-trainer/src/rainbow/model_trainer/helpers/combined_confidence.py
c = probability ** probability_weight
sq = (strength ** strength_weight) * (quality ** quality_weight)
combined = c + alpha * (1 - c) * sq

    Parameters:

    Parameter Type Optuna Range Current Script 2 Value Effect
    alpha Optuna hyperparameter [0.0, 1.0] FIXED at 0.0 Blend factor: how much strength/quality lift combined_confidence above probability alone
    probability_weight Optuna hyperparameter [0.1, 1.0] FIXED at 1.0 Exponent on classification probability
    strength_weight Optuna hyperparameter [0.1, 1.0] FIXED at 1.0 Exponent on predicted strength (irrelevant when alpha=0)
    quality_weight Optuna hyperparameter [0.1, 1.0] FIXED at 1.0 Exponent on predicted quality (irrelevant when alpha=0)

    With Script 2 fixed values (alpha=0.0, probability_weight=1.0):

    combined_confidence = probability^1.0 = probability

    Strength and quality have ZERO contribution to the entry signal. Combined confidence equals raw classification probability.

    Source of ranges: config/experiments/optuna.yaml, lines 212-230 (combined_confidence_weights_config).

    Script 2 fixation mechanism: Each tier config JSON sets min == max to collapse the range:

    "alpha": { "type": "float", "min": 0.0, "max": 0.0 },
"probability_weight": { "type": "float", "min": 1.0, "max": 1.0 }

    Confirmed in all 4 tier configs: script2_tiers_config_ORIG.json, *_v2_tierA_rescue.json, *_v3_tierB_mlp_ex.json, *_v4_tierB_corrected.json.

    3. The Bot Entry Gate Sequence

    When the model makes a prediction, the bot applies these gates in order before entering a trade:

    ┌─ Gate 1: SIGNAL DIRECTION CHECK
│  └─ Reject if signal NOT in {LONG, SHORT} (model predicts NONE → no trade)
│
├─ Gate 2: COMBINED CONFIDENCE THRESHOLD
│  └─ Reject if combined_confidence < entry_threshold (0.55)
│  └─ With alpha=0: this is just "probability < 0.55"
│
├─ Gate 3: MINIMUM QUALITY THRESHOLD (INDEPENDENT from combined_confidence)
│  └─ Reject if quality < min_quality_threshold (0.30)
│  └─ Quality is checked SEPARATELY even though alpha=0 removes it from combined_confidence
│
└─ Gate 4: POSITION SCALING RULES
   └─ Cooldown, duplicate entry checks, confidence-beating rules

    Source: modules/bot-logic/src/rainbow/bot_logic/bot_logic.py, lines 483-496.

    Critical detail: The quality gate (Gate 3) operates independently — quality still filters trades even with alpha=0. However, min_quality_threshold=0.30 is a very permissive filter.

    4. Available Levers — Full Inventory

    4.1 Lever: combined_confidence_weights (alpha, probability_weight, etc.)

    Status: LOCKED — DO NOT CHANGE FOR THESIS

    Why it was fixed (Decision #18, EXPERIMENT_PLAN_THESIS_VALIDATION.md Section 8.6.19):

    1. For non-trainable models (rule_based, random, always_none): These produce fixed strength=0.5 and quality=0.5 constants. With fixed constants, varying alpha is mathematically equivalent to varying entry_threshold — a confounded variable that defeats the thesis purpose. Mathematical proof:

      sq = 0.5^str_w * 0.5^qual_w = CONSTANT combined = c + alpha * (1 - c) * CONSTANT

      The ranking of predictions by combined_confidence is preserved (monotonic in c) for any alpha ≥ 0. Only the absolute values shift, which is equivalent to sliding the entry threshold.

    2. For trainable models: Even with learned strength/quality that vary per input, optimizing these weights confounds "signal quality" (what the thesis measures) with "filter tuning" (an optimization trick). A model with mediocre classification but a well-tuned entry filter could beat a model with superior classification but an untuned filter. The thesis must isolate model signal quality as the single variable.

    3. Multi-task regularization is preserved: Even with alpha=0, the strength and quality heads still provide gradient signal during training via their respective loss weights. The shared representation benefits from multi-task learning — the outputs are simply not used at inference/entry time.

    Reference: docs/plans/EXPERIMENT_PLAN_THESIS_VALIDATION.md, Section 8.6.19 ("Optuna Parameter Control Matrix by Model Category"), Decision Log entries #17-#18.

    For future use (post-thesis): Unlocking alpha to [0.0, 0.5] and probability_weight to [0.5, 1.0] would allow the entry signal to incorporate strength and quality, potentially reducing trade count by requiring "confident AND strong AND clean" signals. This is the highest-impact lever for a production system but is incompatible with the thesis comparison framework.

    4.2 Lever: entry_threshold (bot.yaml)

    Status: NOT RECOMMENDED FOR TIER B RESCUE — COMPARABILITY CONCERN

    Current value: 0.55 (from config/experiments/bot.yaml)

    What it controls:

    • Training: The entry gate loss uses it as reference: BCE((combined_confidence - entry_threshold) × 10, trade_target). The model learns to separate trade/no-trade predictions around this threshold.
    • Backtest: The bot rejects entries where combined_confidence < entry_threshold.
    • Live: Same gate as backtest.

    Full data flow (verified):

    config/experiments/bot.yaml (entry_threshold: 0.55)
  → BotConfig.strategy_config.entry_threshold
    → training.py line 213: float(experiment_config.bot_config.strategy_config.entry_threshold)
      → MultiOutputCriterion.__init__(entry_threshold=...) line 799
        → _entry_gate_loss(): entry_logits = (combined_confidence - 0.55) * 10.0
          → BCE_with_logits(entry_logits, trade_targets)

    Why it would help: Raising to 0.65 would:

    • Shift the training target to require higher confidence for trade signals
    • Filter out the 0.55-0.65 "marginal confidence" trades
    • Estimated 20-40% trade reduction based on confidence distributions

    Why we are NOT changing it: Changing entry_threshold in bot.yaml is a global config change that affects BOTH training AND backtesting fundamentally. Models trained with entry_threshold=0.65 learn a different objective than models trained with entry_threshold=0.55. This means:

    • Rescue results would not be comparable to previous Tier A, Tier B, and Tier B corrected runs
    • Seeds extracted from prior runs were trained with 0.55 — retraining neighbors at 0.65 changes the loss landscape, potentially invalidating the seed transfer hypothesis
    • The thesis comparison framework requires identical bot configuration across all experiments to isolate the model as the only variable

    For future use (post-thesis): If the thesis establishes which models have genuine signal quality, a follow-up study could test entry_threshold ∈ {0.55, 0.60, 0.65, 0.70} as a controlled experiment to quantify the selectivity/profitability tradeoff. This is a one-line config change with full training + backtest propagation.

    4.3 Lever: min_quality_threshold (bot.yaml)

    Status: NOT RECOMMENDED FOR TIER B RESCUE — COMPARABILITY CONCERN

    Current value: 0.30 (from config/experiments/bot.yaml)

    What it controls: A post-combined-confidence gate in bot_logic.py that rejects trades where predicted quality (path cleanliness) is below the threshold. This is INDEPENDENT from combined_confidence — it operates as a separate filter.

    Why we are NOT changing it: Same comparability argument as entry_threshold. Although min_quality_threshold does NOT affect training (it's purely a backtest/live filter), changing it would mean rescue backtest results use a different entry gate pipeline than all prior runs. The thesis must compare model quality under identical conditions.

    Note: With alpha=0 and min_quality_threshold=0.30, this gate is extremely permissive. Quality would need to be below 0.30 (very choppy path prediction) to be filtered. In practice, most models produce quality > 0.30 for most inputs, making this gate nearly inactive.

    For future use: Raising to 0.45-0.50 would filter "medium quality" entries without affecting training. This is a pure selectivity improvement for production use.

    4.4 Lever: entry_gate Loss Weight (Optuna [0.1, 2.0]) — THE VIABLE LEVER

    Status: ALREADY TUNABLE — THIS IS THE MAIN LEVER WE HAVE

    Current range: [0.1, 2.0] (Optuna-tuned per trial)

    What it controls: The entry_gate weight in the multi-output loss determines how much the training gradient prioritizes the entry gate BCE loss relative to classification, strength, and quality losses:

    total_loss = cls_weight × classification_loss
           + str_weight × strength_loss
           + qual_weight × quality_loss
           + gate_weight × entry_gate_loss     ← THIS ONE

    How it affects selectivity:

    • Low gate_weight (0.1): Model barely trains the entry gate head. The entry gate loss contributes tiny gradients compared to classification. Result: the model's combined_confidence distribution is diffuse — many predictions cluster around the 0.55 threshold, producing high trade counts with ambiguous confidence.
    • High gate_weight (1.5-2.0): Model strongly trains the entry gate head. The BCE loss (combined_confidence - 0.55) × 10 receives strong gradients. Result: the model learns to push predictions decisively above or below 0.55, producing a bimodal confidence distribution — fewer trades but with higher conviction.

    Evidence from audit data (Section 7, Optuna Parameter Importance):

    Experiment entry_gate Importance Rank
    Logistic-F4 (45 features) 31.3% #1 most important parameter
    Logistic-F2 (45 features) 11% Top 3
    MLP studies Lower (~2-5%) Moderate

    The entry_gate weight being the #1 most important parameter for Logistic-F4 is remarkable — it means the trading-signal calibration boundary matters MORE than any feature or training hyperparameter for that model/fold combination.

    Why it is thesis-compatible:

    1. It is already part of the Optuna search space — all prior runs tuned it
    2. It does not change the bot configuration — all models see the same entry_threshold (0.55)
    3. It does not change what combined_confidence means — it only changes how strongly the model optimizes for the entry gate during training
    4. Optuna can discover the best value per model/fold combination automatically

    Recommendation for Tier B Rescue: No range change needed. The [0.1, 2.0] range already allows Optuna to explore high selectivity (gate_weight → 2.0). The rescue run's 3-block structure (seeds → neighbors → TPE) should naturally discover whether high gate_weight helps or hurts for each model/fold.

    Optional enhancement (if desired): The rescue could bias the neighbor generation to explore higher gate_weight values for seed trials that had low gate_weight. For example, if a seed trial had gate_weight=0.19 (like the best VIT trial T2798), a neighbor variant with gate_weight=1.5 would test whether that same model configuration becomes more selective with stronger gate training. This is within the existing search space and does not require any code or config changes.

    4.5 Lever: classification, strength, quality Loss Weights (Optuna [0.1, 2.0])

    Status: ALREADY TUNABLE — INDIRECT EFFECT

    These are tuned by Optuna within [0.1, 2.0]. Their effect on selectivity is indirect through multi-task learning:

    • High classification weight: Model focuses on getting direction right → stronger probability predictions → more decisive signals
    • High quality weight: Model focuses on predicting path cleanliness → better internal representations → indirectly helps classification through shared features
    • High strength weight: Model focuses on predicting move magnitude → same mechanism

    These do NOT directly control trade count but influence the confidence distribution through training dynamics. They are already optimized and do not require changes.

    4.6 Lever: Model Architecture Hyperparameters (Optuna)

    Status: ALREADY TUNABLE — STRUCTURAL EFFECT

    Regularization parameters affect selectivity indirectly:

    • Dropout: Higher dropout → more conservative predictions → fewer extreme confidences → potentially fewer trades but also fewer correct entries
    • Weight decay: Higher weight decay → simpler learned function → potentially cleaner decision boundaries

    From audit data (Section 3):

    • Best trial T2530 (TCN-F3, 17 trades): dropout=0.3, weight_decay=1.13e-06
    • Best VIT T2798 (194 trades): dropout=0.1, weight_decay=5.21e-03

    The relationship between regularization and trade count is not monotonic — Optuna should search the space rather than us prescribing values.

    5. Summary — What We Can and Cannot Touch

    Cannot Touch (Thesis Integrity)

    Parameter Value Reason
    alpha 0.0 Confounds signal quality with filter tuning. combined_confidence = probability ensures all models are evaluated on classification quality alone.
    probability_weight 1.0 Same. No distortion of probability.
    strength_weight 1.0 Irrelevant (alpha=0). Fixed for cleanliness.
    quality_weight 1.0 Irrelevant (alpha=0). Fixed for cleanliness.
    entry_threshold 0.55 Changes training loss target AND backtest gate. Rescue results would be non-comparable to prior runs.
    min_quality_threshold 0.30 Changes backtest gate. Rescue results would be non-comparable to prior runs.

    Can Touch (Already Tunable, Thesis-Compatible)

    Parameter Range Mechanism
    entry_gate loss weight [0.1, 2.0] Primary lever. Controls how strongly the model trains its entry gate boundary. High values → bimodal confidence → fewer but more decisive trades. Was #1 most important param for Logistic-F4 (31.3%).
    classification loss weight [0.1, 2.0] Indirect. Stronger classification gradients → more decisive probabilities.
    strength loss weight [0.1, 2.0] Indirect. Multi-task regularization benefit.
    quality loss weight [0.1, 2.0] Indirect. Multi-task regularization benefit.
    Architecture hyperparams per model Indirect. Regularization (dropout, weight_decay) affects confidence distribution.

    6. Actionable Recommendation for Tier B Rescue

    1. No config changes required. The existing Optuna search space already contains the viable selectivity lever (entry_gate weight).
    2. Neighbor generation strategy: When extracting seeds and generating neighbors, consider including entry_gate weight as a dimension to perturb — especially for seeds with low gate weight values. If a seed had gate_weight=0.19, create neighbors at gate_weight ∈ {0.5, 1.0, 1.5} to test whether stronger gate training improves that configuration's selectivity.
    3. Analysis after rescue run: Track the correlation between entry_gate weight and trade count across all rescue trials. If high gate_weight consistently reduces trades while maintaining S2/S3, this validates the mechanism and informs future production config (where alpha, entry_threshold, and min_quality_threshold CAN be unlocked).

    7. Future Work (Post-Thesis)

    Once the thesis experiments are complete and the comparison framework is no longer a constraint, the following levers become available for a production optimization run:

    Lever Priority Expected Impact Change Required
    Unlock alpha to [0.0, 0.5] HIGH Allows strength+quality to boost/attenuate entry signal. Most powerful selectivity mechanism. Tier config JSON: change alpha.max from 0.0 to 0.5
    Raise entry_threshold to 0.60-0.65 HIGH Changes training target + backtest gate. Estimated 20-40% trade reduction. config/experiments/bot.yaml: 1 line change
    Raise min_quality_threshold to 0.45 MEDIUM Filters choppy-path entries. Post-training only. config/experiments/bot.yaml: 1 line change
    Unlock probability_weight to [0.5, 1.0] LOW Minor selectivity effect. Exponent on probability. Tier config JSON: change probability_weight.min from 1.0 to 0.5

    These are documented here for reference. They should NOT be applied to any Script 2 thesis experiment.
