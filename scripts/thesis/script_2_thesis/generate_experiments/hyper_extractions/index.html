    Script 2 Hyper Extractions

    This folder contains a small, self-contained workflow to generate ordered hyperparameter JSON artifacts for the Script 2 rescue process.

    The goal is simple:

    • Read completed Script 2 trials from the application SQLite DB.
    • Use script2_experiment_ids.csv files from results/ as the source of truth for simple_code, tier, fold, and model_type.
    • Select top trials by metric x source_fold x model with systematic same-category backfill.
    • Emit two families of JSON files:
    • exact: fully fixed hyperparameter payloads
    • neighbor: Optuna-ready ranged hyperparameter payloads
    • Compile one final rescue artifact for generate_script2_experiments.py.

    The output is written per model in a clean folder layout.

    Files

    • script2_hyper_extraction_input.json
    • Main input configuration.
    • Defines source runs, top-N selection, and per-model neighbor policies.
    • generate_script2_hyper_extractions.py
    • Generator script.
    • Reads the input config, queries SQLite, ranks trials, and writes output JSONs.

    Why CSV Instead of DB Metadata

    The SQLite snapshot does not reliably expose simple_code as a stable top-level column for the Script 2 rescue workflow. The result CSV files already contain the operational metadata we care about:

    • simple_code
    • tier
    • fold
    • model_type
    • experiment_id

    So this tool uses those CSVs to map experiments cleanly, and the DB only for the actual trial metrics and resolved hyperparameters.

    Input Config Shape

    The input JSON is organized in three blocks:

    1. Global paths
    2. Selection rules
    3. Per-model neighbor policies

    Example:

    {
  "db_path": "database/rainbow_trader.sqlite",
  "source_run_csv_paths": [
    "scripts/thesis/script_2_thesis/results/20260319_035820/script2_experiment_ids.csv",
    "scripts/thesis/script_2_thesis/results/20260325_tierB_mlp_ex/script2_experiment_ids.csv",
    "scripts/thesis/script_2_thesis/results/20260328_tierB_corrected/script2_experiment_ids.csv"
  ],
  "output_dir": "scripts/thesis/script_2_thesis/generate_experiments/hyper_extractions/generated",
  "selection": {
    "metrics": ["s3", "s2", "pnl", "sharpe", "entry_f1"],
    "folds": [1, 2, 3, 4],
    "top_n_exact_per_metric_source_fold": 2,
    "top_n_neighbor_per_metric_source_fold": 1
  },
  "models": {
    "tcn": {
      "source_tiers": ["A", "B"],
      "window_size_policy": { "kind": "int_clamp", "radius": 4, "min": 45, "max": 60 },
      "training_policies": {
        "learning_rate": { "kind": "float_pct", "pct": 0.2, "min": 1e-05, "max": 0.0003, "log_scale": true }
      }
    }
  }
}

    Selection Rules

    The current workflow implements:

    • Metrics:
    • s3
    • s2
    • pnl
    • sharpe
    • entry_f1
    • Folds:
    • 1
    • 2
    • 3
    • 4
    • Exact selection:
    • top 2 per metric x source_fold
    • Neighbor selection:
    • top 1 per metric x source_fold

    Selection/backfill means:

    1. Iterate categories in configured order.
    2. For one destination fold and category, the base exact quota is:
    3. len(folds) * top_n_exact_per_metric_source_fold
    4. with the current config: 4 * 2 = 8
    5. For neighbors, the base quota is:
    6. len(folds) * top_n_neighbor_per_metric_source_fold
    7. with the current config: 4 * 1 = 4
    8. Walk rank depth i = 1, 2, 3, ... inside the same category.
    9. At each i, fetch the #i candidate from every source fold.
    10. Offer that candidate to every destination fold except its own source fold.
    11. Deduplicate per destination fold by exact hyperparameter signature.
    12. Stay in the same category until either:
    13. all destination-fold quotas are full, or
    14. there are no more ranked candidates in any source fold.
    15. Any deficit carries to the next category.

    That gives the current operational split per destination fold:

    • 40 exact enqueue trials:
    • 5 categories x 4 source folds x top-2
    • 20 neighbor anchors:
    • 5 categories x 4 source folds x top-1
    • 80 total rescue trials once Tier B rescue adds:
    • 20 warmup random trials inside the merged neighbor space
    • 20 TPE-guided trials after warmup

    Metrics Source

    The generator extracts:

    • s2 from trials.stage2_result
    • s3 from trials.stage3_result
    • pnl from trials.stage3_total_pnl
    • sharpe from training_metrics.stage3_metrics.sharpe_ratio
    • entry_f1 from training_metrics.stage2_metrics.prediction_health.binary_trade_detection.overall.f1

    Only completed trials with non-null required fields are considered.

    Output Layout

    The generator writes:

    generated/
  logistic/
    exact/
      01_f1_s3_t1234.json
      ...
    neighbor/
      01_f1_s3_t1234.json
      ...
    manifest.json
  mlp/
    ...
  script2_tierb_rescue_compiled.json

    Each output JSON has:

    • selection metadata
    • final hyperparameters_config
    • requested_fold and source_fold separated for same-fold exclusion

    Example exact payload:

    {
  "kind": "exact",
  "selection": {
    "source_trial_id": 2530,
    "experiment_id": 78,
    "simple_code": "S2-TB-TCN-F3",
    "tier": "B",
    "requested_fold": 3,
    "source_fold": 3,
    "model_type": "tcn",
    "metric": "sharpe",
    "rank_within_metric_fold": 1,
    "metric_value": 3.58
  },
  "hyperparameters_config": {
    "window_size": { "type": "fixed", "value": 52 },
    "training_hyperparameters_config": {
      "learning_rate": { "type": "fixed", "value": 0.00024 }
    }
  }
}

    Example neighbor payload:

    {
  "kind": "neighbor",
  "selection": {
    "source_trial_id": 2530,
    "experiment_id": 78,
    "simple_code": "S2-TB-TCN-F3",
    "tier": "B",
    "requested_fold": 3,
    "source_fold": 3,
    "model_type": "tcn",
    "metric": "sharpe",
    "rank_within_metric_fold": 1,
    "metric_value": 3.58
  },
  "hyperparameters_config": {
    "window_size": { "type": "int", "min": 48, "max": 56 },
    "training_hyperparameters_config": {
      "weight_decay": {
        "type": "float_ranges",
        "ranges": [
          { "min": 1e-06, "max": 3e-06, "log_scale": true },
          { "min": 3e-06, "max": 9e-06, "log_scale": true }
        ]
      }
    }
  }
}

    The compiled rescue JSON contains, per model:

    • enqueue_trials_by_fold
    • exact trials already filtered and ordered for each destination fold
    • hyperparameters_config
    • one merged Optuna space built from neighbors and exact anchors

    Policy Kinds

    The current script supports these policy kinds for neighbor generation:

    • fixed
    • float_pct
    • float_ranges_split
    • int_clamp
    • categorical_adjacent
    • categorical_clamped

    These are enough to express:

    • exact anchors
    • local search around anchors
    • split search ranges where the rescue logic wants disjoint numeric bands

    How To Run

    From repo root:

    source .venv/bin/activate
python scripts/thesis/script_2_thesis/generate_experiments/hyper_extractions/generate_script2_hyper_extractions.py

    Or with an explicit config path:

    source .venv/bin/activate
python scripts/thesis/script_2_thesis/generate_experiments/hyper_extractions/generate_script2_hyper_extractions.py \
  --config-path scripts/thesis/script_2_thesis/generate_experiments/hyper_extractions/script2_hyper_extraction_input.json

    When To Edit The Input JSON

    Edit script2_hyper_extraction_input.json when you want to change:

    • source runs
    • top-N selection
    • output folder
    • tiers allowed per model
    • neighbor search shape per model
    • regularization floors or split ranges

    Expected Workflow

    1. Adjust script2_hyper_extraction_input.json.
    2. Run the generator.
    3. Inspect generated/<model>/manifest.json.
    4. Review exact/ and neighbor/.
    5. Copy or merge the final hyperparameters_config payloads into the downstream rescue assembly workflow.
