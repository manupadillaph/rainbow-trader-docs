    Script 2 Ensemble Backtests

    This folder contains the thesis workflow for testing whether a group of compatible Script 2 models can produce a better trading signal than the individual source trials.

    The workflow is intentionally split into three steps:

    1. generate a reproducible plan from one or more real Script 2 runs
    2. execute that plan and persist the raw artifacts
    3. regenerate the markdown report when the report logic evolves

    The planner is built around one main question:

    • given one or more real Script 2 runs,
    • for one comparable validation fold,
    • which models should enter the ensemble pool?

    The runner is built around a second question:

    • once the pool is fixed,
    • which merge rule produces the best execution?

    Files

    • generate_ensemble_backtest_plan.py
    • run_ensemble_backtests.py
    • run_ensemble_backtest_batch.py
    • audit_ensemble_batch_results.py
    • render_ensemble_backtest_report.py

    Core Idea

    This workflow starts from the same run sidecars already used across the Script 2 thesis exporters and merged views:

    • scripts/thesis/script_2_thesis/results/<RUN_TAG>/script2_experiment_ids.csv

    Then you choose one fold. That fold defines the comparable validation window.

    Inside that fold, the planner:

    1. loads only kind=real experiment entries
    2. loads completed trials with checkpoints from the DB
    3. applies optional metric filters
    4. ranks candidates inside each model family
    5. keeps the top N per model
    6. enforces strict execution compatibility
    7. writes a full plan.json

    If you enable --max-prob-corr and/or --max-signal-agreement, the planner keeps the same structure but changes the take top N per model step:

    • it still walks candidates inside each model_type in ranking order
    • it compares each new candidate against the full pool already selected
    • if the candidate is too correlated, it is skipped
    • the planner then tries the next ranked candidate from that same model family
    • to decide this, the planner may reuse or regenerate predictions during planning

    The runner then uses that plan to:

    1. reuse or regenerate predictions for each selected source trial
    2. merge them with one or more ensemble rules
    3. run one backtest per merge rule
    4. compare the merged runs against the original stage-3 baselines
    5. write report artifacts

    Strict Compatibility

    The planner and runner only mix trials that match exactly on:

    • experiment_type
    • base_symbol
    • base_timeframe
    • stage-3 validation range
    • full bot_config
    • full simulation_config
    • combined_confidence_weights

    This is intentionally conservative. It avoids mixing models that would trade under different execution assumptions.

    Workflow

    1. Generate the Plan

    Use one or more real run summary CSVs:

    • scripts/thesis/script_2_thesis/results/<RUN_TAG>/script2_experiment_ids.csv

    Recommended mental model:

    • the planner chooses the pool
    • the plan also defines which ensemble merge rules will be executed later

    2. Execute the Plan

    The runner consumes plan.json.

    That plan already contains:

    • selected source trials
    • merge rules
    • the comparable stage-3 execution range
    • ranking rationale

    3. Read or Regenerate the Report

    The runner writes all raw artifacts. The renderer can rebuild report.md later without rerunning backtests.

    Planner

    What It Selects

    The planner selects trials for:

    • one required fold
    • optional tier
    • optional model subset
    • one or more source runs

    It keeps:

    • completed trials only
    • checkpointed trials only
    • candidates that survive the chosen ranking-mode eligibility
    • only one strict compatibility bucket in the final plan

    Inside the chosen bucket it keeps the top N trials per model family.

    With --max-prob-corr and/or --max-signal-agreement, it still targets top-per-model, but only accepts a candidate when it is not too redundant with the pool selected so far.

    Planner Parameters

    generate_ensemble_backtest_plan.py

    • --summary-csv Repeatable. Path to one source script2_experiment_ids.csv.

    • --db-path Path to database/rainbow_trader.sqlite. Default: database/rainbow_trader.sqlite

    • --fold Required. Validation fold to compare.

    • --top-per-model Required. Number of selected trials to keep for each model family.

    • --tier Optional. Restrict to one tier like A or B. If omitted, the planner may mix tiers as long as strict compatibility still holds.

    • --models Optional comma-separated model filter. Example: logistic,mlp,vit,tcn

    • --rank-by Optional. Default: promise

    Supported values: - promise - friction - objective - pnl - stage2

    • --min-stage2 Optional lower bound on stage2_result.

    • --min-stage3 Optional lower bound on stage3_result.

    • --min-sharpe Optional lower bound on Sharpe.

    • --min-trades Optional lower bound on stage-3 trades.

    • --max-trades Optional upper bound on stage-3 trades.

    • --max-prob-corr Optional maximum allowed positive Pearson correlation between a candidate and each already selected member.

    The planner computes three correlations on the aligned prediction streams: - corr(prob_none_left, prob_none_right) - corr(prob_long_left, prob_long_right) - corr(prob_short_left, prob_short_right)

    Then it uses the largest of those three values as the redundancy check.

    When this is enabled: - the planner may load reusable prediction parquets during planning - if predictions are missing, it may generate them during planning - candidates are still tried in the normal ranking order inside each model family - redundant candidates are skipped and replaced by the next ranked candidate

    • --max-signal-agreement Optional maximum allowed signal agreement between a candidate and each already selected member.

    Signal agreement is the share of aligned timestamps where both models emit the same final signal: - LONG - SHORT - NONE

    This is a discrete complement to --max-prob-corr.

    When this is enabled: - the planner still keeps the same bucket/ranking/top-per-model structure - candidates are skipped if they agree too often with the already selected pool - it can be used alone or together with --max-prob-corr

    • --merge-rules Optional. Defaults to the current recommended 8-rule exploration set:
    • average
    • majority_vote
    • all_must_agree
    • majority_vote:0.75
    • majority_vote:0.85
    • weighted_average
    • weighted_majority_vote:0.75
    • weighted_majority_vote:0.85

    Use comma-separated syntax. Thresholded vote rules are written as: - majority_vote:0.75 - weighted_majority_vote:0.85

    • --range-start

    • --range-end Optional exact range override to persist in plan.json. If omitted, the planner uses the common selected stage-3 validation range.

    • --output-root Parent folder for timestamped planner outputs.

    • --output-dir Exact planner output directory.

    Ranking Modes

    promise

    Default ranking mode. This is the general ensemble-oriented composite.

    Weights:

    • objective: 1.5
    • stage2: 2.5
    • stage3: 1.0
    • sharpe: 1.25
    • win_rate: 1.25
    • trades_inverse: 0.75
    • total_pnl: 0.75

    Use this when you want a balanced shortlist of individually decent models.

    friction

    This reuses the rescue lens from:

    • scripts/thesis/script_2_thesis/extra_backtests/generate_extra_backtest_plan.py

    Additional eligibility:

    • PnL < 0
    • Total Gross PnL > 0

    Use this when you suspect the model family may contain friction victims and want to see whether an ensemble can recover latent gross edge.

    Raw Metric Modes

    • objective
    • pnl
    • stage2

    Use these when you want a more controlled experiment based on one single selection signal.

    Correlation-Aware Selection

    This is the main prediction-diversity extension in the current planner.

    Without --max-prob-corr and without --max-signal-agreement, selection is pure ranking:

    • bucket first
    • then ranked top-per-model

    With --max-prob-corr and/or --max-signal-agreement, selection becomes:

    • same bucket
    • same ranking order
    • same top-per-model
    • plus one extra rule: a candidate must not be too correlated with the already selected pool

    Important:

    • this is not a proxy metric
    • it uses real prediction streams
    • it compares the real merge-relevant probabilities:
    • prob_none
    • prob_long
    • prob_short
    • it computes Pearson correlation separately for each of those three series
    • the effective redundancy score is the largest of the three correlations
    • signal agreement can also be constrained as a separate discrete threshold
    • the threshold is applied against the whole selected pool, not only against the same model family

    Practical example:

    • top-per-model=2
    • max-prob-corr=0.85
    • max-signal-agreement=0.60

    Then the planner behaves like this for one model family:

    1. try the best ranked candidate
    2. compare it against everything already selected
    3. if it passes, keep it
    4. if it fails, skip it
    5. try the next ranked candidate from that same model family

    This preserves the existing planner shape while avoiding obviously redundant members.

    Runner

    Runner Behavior

    run_ensemble_backtests.py

    1. loads the selected source trials from the plan
    2. prepares predictions for each source trial
    3. reuses existing prediction parquets when the schema is complete
    4. regenerates predictions when the parquet is missing or legacy
    5. uses original stage-3 baselines by default
    6. runs one merged backtest per merge rule
    7. writes raw artifacts and regenerates report.md

    Batch Runner

    Batch Runner Behavior

    run_ensemble_backtest_batch.py

    This is the orchestration layer for thesis campaigns.

    Use it when:

    • you already defined a campaign block in JSON
    • you want one command to launch many planner + runner executions
    • you want one root folder that contains every execution subfolder
    • you want resumable batch behavior for long campaigns

    For each execution inside the JSON batch spec, the batch runner:

    1. creates one dedicated subfolder
    2. runs generate_ensemble_backtest_plan.py
    3. optionally runs run_ensemble_backtests.py
    4. records the execution status in batch_results.json

    Batch JSON Structure

    The batch spec has two main sections:

    1. defaults Shared values that apply to every execution unless overridden.
    2. executions The list of concrete planner/runner combinations to launch.

    Real example shape:

    {
  "batch_name": "20260403_phase1_block_a",
  "output_dir": "scripts/thesis/script_2_thesis/ensemble_backtests/results/20260403_phase1_block_a",
  "defaults": {
    "summary_csvs": [
      "scripts/thesis/script_2_thesis/results/20260325_tierB_mlp_ex/script2_experiment_ids.csv",
      "scripts/thesis/script_2_thesis/results/20260328_tierB_corrected/script2_experiment_ids.csv"
    ],
    "tier": "B",
    "merge_rules": [
      "average",
      "majority_vote",
      "all_must_agree",
      "majority_vote:0.75",
      "majority_vote:0.85",
      "weighted_average",
      "weighted_majority_vote:0.75",
      "weighted_majority_vote:0.85"
    ]
  },
  "executions": [
    {

    Batch Parameters

    run_ensemble_backtest_batch.py

    • --batch-json Required path to the JSON batch specification.

    • --output-dir Optional batch root override.

    Resolution order: 1. CLI --output-dir 2. JSON output_dir 3. fallback: scripts/thesis/script_2_thesis/ensemble_backtests/results/<batch_name>

    Important: - the batch runner requires run folders to include a full timestamp - if the resolved folder name does not already start with YYYYMMDD_HHMMSS, the batch runner automatically inserts HHMMSS into the folder name - when the requested folder already starts with YYYYMMDD_, the runner keeps that date and transforms it into: - YYYYMMDD_HHMMSS_<rest> - otherwise it creates: - YYYYMMDD_HHMMSS_<original_name>

    Example: - requested root: 20260403_phase1_block_a - effective root: 20260403_021530_phase1_block_a

    • --only-execution Optional repeatable execution-name filter.

    Use this when: - you want a smoke test before launching the full block - one execution failed and you want to rerun only that one - you want to compare off vs mild vs tight for a single family first

    • --rerun-existing Force planner and runner re-execution even when the execution folder already contains results.json.

    Without this flag: - completed executions are skipped - this makes long batches resumable

    • --plan-only Generate planner artifacts only.

    This means the batch will run: - generate_ensemble_backtest_plan.py

    and it will not run: - run_ensemble_backtests.py

    Each execution folder will contain: - ensemble_plan.log - plan.md - plan.json

    Each execution folder will not contain: - ensemble_backtests.log - results.json - results.csv - comparisons.csv - report.md

    Use this when: - you want to inspect selected source pools before paying the backtest cost - you want to audit diversity-aware selection behavior first - you want to split a large campaign into plan first, run later

    • --fail-fast Stop the whole batch on the first planner or runner failure.

    Without this flag: - the failure is recorded in batch_results.json - the batch continues with the remaining executions

    Runner Parameters

    • --plan-json Recommended mode. Load the full recipe from the planner output.

    • --source-trial-id Technical/manual mode. Repeatable. Use only when you want to bypass the planner.

    • --merge-rules Optional override. If omitted, the runner uses the merge rules already stored in plan.json.

    Important: - no hidden fallback at execution time - if neither CLI nor plan.json provides merge rules, the runner fails

    • --range-start

    • --range-end Optional execution override.

    • --initial-equity Initial backtest equity. Default: 10000

    • --include-source-baselines Also rerun the original-trial baselines on the exact execution range. If omitted, the runner compares against the original stage-3 backtests stored in the DB.

    • --output-dir Exact runner output directory. If omitted and --plan-json is provided, the runner writes into the same folder that contains plan.json. If omitted and no plan is used, the runner creates a new timestamped folder.

    Merge Rules

    This is the most important practical section.

    The merge rule is what transforms multiple model predictions into one merged signal at each timestamp.

    1. average

    What it does:

    • averages prob_none, prob_long, prob_short
    • averages strength and quality
    • derives the final signal from the averaged probabilities

    Practical example:

    • model A says LONG with high confidence
    • model B also says LONG
    • model C is less convinced and leans NONE

    The ensemble still becomes LONG, but with moderated confidence.

    Use this when:

    • you want a smooth consensus signal
    • you do not want one model to dominate

    Risk:

    • it can still trade a lot
    • weak models can pull the merged probabilities toward noisy trades

    2. weighted_average

    What it does:

    • same as average
    • but each model contributes proportionally to its plan selection score

    How weights are built:

    • the runner takes each selected trial's selection_score
    • shifts the scores to positive values if necessary
    • normalizes them to sum to 1.0

    Practical example:

    • a top-ranked TCN can influence the merged probabilities more than a weaker VIT
    • but every selected model still contributes

    Use this when:

    • you trust the planner ranking
    • you want a softer ensemble than hard voting

    Risk:

    • if the selection score is a poor proxy for ensemble usefulness, this can over-trust the wrong models

    3. all_must_agree

    What it does:

    • only emits LONG if every source model says LONG
    • only emits SHORT if every source model says SHORT
    • otherwise emits NONE

    Practical example:

    • LONG, LONG, LONG, LONG -> LONG
    • LONG, LONG, SHORT, LONG -> NONE

    Use this when:

    • you want maximum selectivity
    • you want to test whether the ensemble improves by trading much less

    Risk:

    • it can become so strict that it barely trades

    4. majority_vote

    What it does:

    • each model gives one vote: LONG, SHORT, or NONE
    • the signal with the most votes wins
    • ties are broken by summed probabilities
    • no minimum vote share is required

    Practical example:

    • 5 models
    • votes: LONG=2, SHORT=2, NONE=1
    • tie between LONG and SHORT
    • summed probabilities decide the winner

    Use this when:

    • you want a simple voting baseline
    • you want to compare against thresholded voting

    5. majority_vote:<threshold>

    What it does:

    1. chooses the same winner as majority_vote
    2. computes that winner's share of all votes
    3. keeps it only if the share is at least the threshold
    4. otherwise emits NONE

    Important:

    • majority_vote:0 is equivalent to the current unthresholded majority_vote
    • majority_vote:0.75 means the winning signal must control at least 75% of votes

    Practical example:

    • 8 models
    • vote winner: LONG
    • LONG share = 6 / 8 = 0.75

    Then:

    • majority_vote:0.50 -> LONG
    • majority_vote:0.75 -> LONG
    • majority_vote:0.85 -> NONE

    Use this when:

    • you want something between majority_vote and all_must_agree

    Recommended thresholds:

    • 0.75 for a strong but still practical consensus
    • 0.85 for a very conservative version

    6. weighted_majority_vote:<threshold>

    What it does:

    1. each model votes LONG, SHORT, or NONE
    2. votes are weighted by the normalized selection_score
    3. the winner is the signal with the highest total vote weight
    4. ties are broken by weighted summed probabilities
    5. the weighted winner is kept only if its total vote-weight share is at least the threshold
    6. otherwise emits NONE

    Practical example:

    Weights:

    • model A: 0.60
    • model B: 0.15
    • model C: 0.15
    • model D: 0.10

    Votes:

    • A -> LONG
    • B -> SHORT
    • C -> SHORT
    • D -> SHORT

    Count-based vote:

    • SHORT wins 3-to-1

    Weighted vote:

    • LONG weight = 0.60
    • SHORT weight = 0.40

    Then:

    • weighted_majority_vote:0.50 -> LONG
    • weighted_majority_vote:0.75 -> NONE

    Use this when:

    • you want vote-style behavior
    • but you also want the stronger selected models to matter more

    Practical Recommendations

    If you are exploring a new fold for the first time, a good default set is:

    • average
    • majority_vote
    • all_must_agree
    • majority_vote:0.75
    • majority_vote:0.85
    • weighted_average
    • weighted_majority_vote:0.75
    • weighted_majority_vote:0.85

    If your current ensembles are trading too much:

    • focus on majority_vote:0.75
    • majority_vote:0.85
    • weighted_majority_vote:0.75
    • weighted_majority_vote:0.85
    • all_must_agree

    If your current ensembles look too inactive:

    • compare average
    • weighted_average
    • majority_vote

    If the planner used rank_by=friction, the weighted modes become especially interesting because the score already encodes a friction-aware preference.

    Output Reading Guide

    • plan.md Human-readable description of how the shortlist was built and why each trial entered. When correlation-aware selection is enabled, it also includes pairwise diversity metrics for the final pool.

    • ensemble_plan.log Planner-time log file. This is where you inspect planning-time prediction reuse or regeneration progress, especially when --max-prob-corr forces the planner to prepare predictions during selection.

    • plan.json Machine-readable recipe used by the runner.

    • results.csv One row per executed run, including source baselines and ensemble runs.

    • comparisons.csv One row per ensemble rule × source trial comparison.

    • results.json Rich metadata including selection policy, prediction preparation state, and all result rows.

    • report.md Self-contained thesis report with selection context, prediction preparation, source tables, ensemble tables, ranking, pairwise comparisons, selection diversity, and appendix notes.
