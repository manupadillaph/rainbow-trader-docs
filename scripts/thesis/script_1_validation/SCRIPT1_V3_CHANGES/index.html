    Script 1 — V3 Changes Log

    This document tracks the evolution from V2 → V3 of Script 1 experiment configurations.

    • V1 run: 20260306_204834
    • V2 run: 20260307_232547
    • V2 findings: results/20260307_232547/script1_findings_and_recommendations.md
    • V3 changes applied: 2026-03-09
    • Scope: Only S1-E03 and S1-E04. All other experiments (E01, E02, E05, E06, E07, E08) keep their V2 results.
    • Follow-up: If untradeable force-pruning still blocks completion, use V3bis (SCRIPT1_V3BIS_CHANGES.md).
    • Follow-up 2: If E04 still prunes too early in V3bis, use E04-only V3bis2 (SCRIPT1_V3BIS2_CHANGES.md).

    1. V2 Summary — What Happened

    V2 Results (relevant experiments only)

    Exp Description completed/pruned entry_f1 median Status
    S1-E03 Random + ViT 0/20 NA ⚠️ All pruned
    S1-E04 Random + ViT + Leak 0/20 NA ⚠️ All pruned

    V2 Root Cause

    The V2 degenerate pruner fix (Change 3 in V2: warmup=2, patience=2) fires at epoch 3. On random walk data, ViT collapses to all-NONE by epoch 0 and never escapes within 4 epochs. Result: 20/20 trials pruned as degenerate for both E03 and E04. This produces 0 completed trials, making statistical tests impossible (SKIP).

    The pruner and model are both acting correctly:

    1. ViT on random data → no signal → all-NONE (correct model behavior)
    2. Pruner detects all-NONE → prunes (correct pruner behavior)
    3. 0 completed trials → tests can't run (unintended side effect)

    2. V3 Changes — What We're Fixing

    Change 1 — CRITICAL: Disable degenerate pruner for E03 and E04

    Before (V2):

    "degenerate": {
  "enabled": true,
  "warmup_epochs": 2,
  "patience_epochs": 2
}

    After (V3):

    "degenerate": {
  "enabled": false,
  "warmup_epochs": 2,
  "patience_epochs": 2
}

    Why: On random walk data, all-NONE IS the expected ViT behavior. The degenerate pruner correctly identifies this as degenerate but prevents trials from completing. We need completed trials to run statistical comparisons (E03 vs E06, E04 vs E03).

    Affects: S1-E03, S1-E04

    Change 2 — HIGH: Expand learning-rate search range (low + high)

    Before (V2):

    "learning_rate": { "type": "categorical", "choices": [3e-5, 5e-5, 1e-4, 2e-4, 3e-4, 5e-4] }

    After (V3):

    "learning_rate": { "type": "categorical", "choices": [1e-5, 3e-5, 5e-5, 1e-4, 2e-4, 3e-4, 5e-4, 1e-3, 2e-3] }

    Why: We should not bias the search to only large LRs. The goal is to let Optuna explore the full regime: conservative (1e-5, 3e-5), moderate (5e-5..5e-4), and aggressive (1e-3, 2e-3). This can reveal non-obvious combinations where low LR + longer training or specific window sizes produce useful behavior.

    Affects: S1-E03, S1-E04

    Change 3 — MEDIUM: Increase epochs from 15 to 25

    Before (V2):

    "epochs": 15

    After (V3):

    "epochs": 25

    Why: More training time gives the model more opportunity to escape the all-NONE basin, especially with the higher LRs from Change 2. This is particularly important for E04 where the leak signal on random walk data is weak — the model may need more epochs to detect it.

    Affects: S1-E03, S1-E04

    Change 4 — MEDIUM: Increase early stopping patience from 5 to 10

    Before (V2):

    "early_stopping_patience": 5

    After (V3):

    "early_stopping_patience": 10

    Why: With 25 epochs and higher LRs, the model's loss trajectory is more volatile. A patience of 5 might kill trials prematurely before the model has a chance to escape all-NONE. Patience 10 gives more breathing room while still stopping truly stalled trials.

    Affects: S1-E03, S1-E04

    Change 5 — LOW: Increase n_trials from 20 to 40

    Before (V2):

    "n_trials": 20

    After (V3):

    "n_trials": 40

    Why: With the wider LR range, 20 trials under-samples the search space. Increasing to 40 gives Optuna enough budget to test both low-LR and high-LR regions with meaningful coverage.

    Affects: S1-E03, S1-E04

    3. Change Matrix — What Changes in Each Experiment

    Change E03 E04
    1. Disable degenerate pruner ✅ ✅
    2. Expanded LR range (low + high) ✅ ✅
    3. Epochs 15→25 ✅ ✅
    4. Early stopping patience 5→10 ✅ ✅
    5. n_trials 20→40 ✅ ✅
    6. Mini backtest (100 tail rows) ✅ ✅

    NOT changed: E01, E02, E05, E06, E07, E08 — these keep their V2 results as-is.

    4. Expected V3 Outcomes

    Exp V2 Result Expected V3 Result Reasoning
    S1-E03 0/20 all pruned 40/0 completed, entry_f1 ≈ 0.0–0.25 Without degenerate pruner, trials complete. Wider LR search may produce diverse predictions.
    S1-E04 0/20 all pruned 35+/5 mostly completed, entry_f1 > 0.0 Normalized leak + more epochs + wider LR search should enable partial leak detection on random walk data.

    Expected test outcomes after V3 (combined with V2 results for other experiments)

    Test V2 Expected V3
    E01 criteria (entry_f1 > 0.15) ✅ PASS ✅ PASS (V2 result kept)
    E03 vs E06 criteria (similar) ⏭️ SKIP ⚠️ Depends on E03 entry_f1 distribution
    E04 vs E03 criteria (leak > no-leak) ⏭️ SKIP ⚠️ Depends on leak signal detectability
    E05 criteria (entry_f1 >= 0.98) ✅ PASS ✅ PASS (V2 result kept)
    E01 vs E02 Mann-Whitney ✅ PASS ✅ PASS (V2 result kept)
    E03 vs E06 Mann-Whitney ⏭️ SKIP ⚠️ Should be computable now
    E04 vs E03 Mann-Whitney ⏭️ SKIP ⚠️ Should be computable now
    E05 vs E03 Mann-Whitney ⏭️ SKIP ⚠️ Should be computable now

    Note on E03 vs E06 test

    Even with these changes, the E03 vs E06 comparison may FAIL:

    • If Optuna finds regimes with diverse predictions (low or high LR) → entry_f1 ≈ 0.25 ≈ E06 → PASS
    • If model still mostly collapses to all-NONE → entry_f1 ≈ 0.0 ≠ E06 ≈ 0.25 → FAIL

    A FAIL here would be a true result (not a false negative): ViT genuinely behaves differently from Random on random data due to the asymmetry described in V1/V2 findings. This would require revising the test criteria to account for the intelligent-model-vs-random-model asymmetry.

    6. Execution Plan

    Step 1: Create V3 experiment JSONs

    Create two new experiment JSONs in a new experiments_v3/ folder:

    scripts/thesis/script_1_validation/experiments_v3/s1-e03-random-vit.json
scripts/thesis/script_1_validation/experiments_v3/s1-e04-random-vit-leak.json

    These are copies of the V2 JSONs with the V3 changes applied.

    Step 2: Create V3 run script

    Create run_script1_v3.sh that:

    1. Creates a new RUN_TAG directory for logs
    2. Runs ONLY E03 and E04 (using experiments_v3/ JSONs)
    3. Saves script1_experiment_ids.csv with just E03 and E04 entries

    Step 3: Run V3

    source .venv/bin/activate
bash scripts/thesis/script_1_validation/run_script1_v3.sh

    Step 4: Create combined analysis CSV

    After V3 completes, create a combined script1_experiment_ids.csv that merges:

    • V2 results for E01, E02, E05, E06, E07, E08 (experiment IDs from V2 run)
    • V3 results for E03, E04 (new experiment IDs from V3 run)

    The V2 IDs are resolved from a baseline summary CSV:

    • Default: logs/20260307_232547/script1_experiment_ids.csv
    • Override supported: --v2-summary-csv /abs/path/to/script1_experiment_ids.csv

    The V3 IDs are extracted from the V3 create logs and summary CSV automatically.

    Step 5: Run combined analysis

    The run_script1_v3.sh script will automatically:

    1. Build a combined CSV with V2 IDs (E01, E02, E05, E06, E07, E08) + V3 IDs (E03, E04)
    2. Run analyze_script1_results.py with --summary-csv pointing to the combined CSV
    3. Output results to results/<V3_RUN_TAG>/

    This produces a single unified analysis report that covers all 8 experiments.

    7. Rollback Plan

    If V3 results are worse or unexpected:

    • V2 results remain intact in results/20260307_232547/
    • V2 experiment IDs remain in the DB untouched
    • Simply ignore V3 and reference V2 results
