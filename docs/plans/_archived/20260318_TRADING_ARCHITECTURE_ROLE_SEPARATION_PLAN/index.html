    Trading Architecture — Role and Responsibility Separation Plan

    Refactor trading config to separate Strategy, Risk Management, and Order Preferences into distinct BaseConfig sub-configs inside BotConfig, expand RiskManager as single shared authority for all risk decisions in both backtesting and live trading, remove BotLogic.decide(), and enforce the bot decision flow (generate_exit_orders() + generate_entry_signals() + RiskManager.evaluate_entries()).

    Status: ✅ COMPLETE — 100% Delivered Started: 2026-03-17 Completed: 2026-03-18 Priority: HIGH

    Predecessor plan: LIVE_TRADING_MULTI_INSTANCE_FOUNDATION_PLAN.md (100% complete, archived)

    📋 Description

    The current system mixes strategy decisions, risk management, and order preferences across BotConfig, LiveRuntimeConfig, and scattered runtime code. This plan separates all trading config into three clean layers:

    1. StrategyConfig — Signal generation and trade structure (what/when to trade)
    2. RiskManagerConfig — Position sizing, portfolio limits, circuit breakers (how much / is it allowed) — shared by backtesting AND live
    3. OrderPreferencesConfig — Order mechanics (how to fill) — already exists, stays as-is
    4. LiveRuntimeConfig — Infrastructure-only operational controls (live-only: kill switch, heartbeat, retry, observability, capital source)

    BotConfig remains the top-level composite containing strategy_config, risk_manager_config, and order_preferences_config as nested BaseConfig children. Single bot.yaml with nested sections. No separate YAML files. BotLogic.decide() is removed (no compatibility wrapper): execution paths call generate_exit_orders(), generate_entry_signals(), and RiskManager.evaluate_entries() directly. Risk persistence is explicit and typed: persist full risk_manager_state_json and keep scalar mirror fields (risk_manager_daily_pnl, risk_manager_date, risk_manager_halted) in both backtest_sessions and live_sessions. Resume/restore reads typed JSON state (no scalar fallback).

    Impacts: bot-logic, backtesting-engine, live-trading-runner, common configs, all tests, docstrings, CLI help, JSON experiment templates, and documentation.

    🧩 Rules

    Global rules (do not remove)

    • CLI + core.py: All business logic lives in core.py. CLIs and tools only parse arguments and call core functions.
    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch. Reality lives on disk (plans, docs, code), not in cached summaries.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md. Never run git add from the repository root unless explicitly specified in this plan.
    • Respect project docs: All changes must comply with the relevant docs (configuration, code generation, database schema, markdown style, etc.).
    • Language: User responses and AI interpretations in the conversation log may be in Spanish, but the work plan document itself and all generated code/docs MUST be written in English.

    Plan-specific rules

    • No legacy/backward compatibility: There is no past. All changes apply everywhere. No _old, no _deprecated, no compatibility wrappers.
    • Backtesting behavioral parity: Backtesting is tested and working. Changes MUST preserve identical output for the same inputs. The risk layer wraps existing logic, not replaces the math.
    • Live trading is the primary refactor target: Live trading code is new and untested in production; structural changes are cheaper there.
    • No hidden defaults for critical trading controls: All risk limits must be explicit in config, never silently defaulted.
    • Shared risk layer: RiskManagerConfig and RiskManager must be usable by BOTH backtesting and live trading with zero code duplication.
    • _OLD/ backup before major rewrites: Create backup copies of files before large refactors.
    • Predecessor plan is reference-only: The LIVE_TRADING_MULTI_INSTANCE_FOUNDATION_PLAN is 100% complete. This plan does not modify its tasks. It builds on top of its delivered code.
    • Implementation quality contract: No Any in runtime contracts, no dict-shaped public results, no suppression pragmas, full typing, rich documentation.
    • Toggleable risk controls: Numeric/threshold risk controls in RiskManagerConfig use Optional types (None = disabled) except the bare minimum to calculate a position size (position_sizing_method, risk_per_trade). Directional controls (allow_long, allow_short, force_flat_on_stop) are explicit booleans with defaults.
    • No compatibility wrappers in bot decision flow: BotLogic.decide() is removed. Backtesting/live loops must call the bot decision flow API directly (generate_exit_orders() + generate_entry_signals() + RiskManager.evaluate_entries()).
    • No payload migration/backward fallback: There are no legacy payloads to support in this refactor. Fail fast on invalid structure.

    Decision Flow

    1. BotLogic.generate_exit_orders(prediction, positions, price, time, strategy_config)
   → List[Order CLOSE]
   → exits bypass risk checks by design

2. BotLogic.generate_entry_signals(prediction, positions, price, time, strategy_config)
   → List[EntrySignalProposal]

3. Build projected post-exit portfolio state for risk evaluation:
   projected_portfolio_state = current_open_positions - positions_targeted_by_exit_orders
   (max_concurrent_positions/exposure checks use this projected state)

4. Resolve tradable equity (live):
   tradable_equity = equity * max_capital_allocation * (1 - reserve_buffer_percent)
   (backtesting uses full equity unless a runtime allocation policy is explicitly configured)

5. RiskManager.evaluate_entries(entry_signals, projected_portfolio_state, tradable_equity)
   → For each EntrySignalProposal:
       allow_long/short? → max_concurrent_positions? → max_trades_per_day? →
       daily_loss circuit breaker? → drawdown circuit breaker? → cooldown? →
       exposure? → calculate size → notional cap? → position value cap?
       → APPROVED(Order) | REJECTED(reason)

6. final_orders = exit_orders + approved_entry_orders

7. Fill (backtesting orders.py / live SimExecutionAdapter):
   → process CLOSE orders before OPEN orders in the same decision cycle
   → slippage + fees from OrderPreferencesConfig → Trade record

- RiskManager.restore_state(state: RiskManagerState) -> None
