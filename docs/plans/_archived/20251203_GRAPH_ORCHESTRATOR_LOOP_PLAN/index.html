    Graph Orchestrator Loop & Events Plan

    Status: 🟡 STAGE 3 — TASK PLANNING
    Started: 2025-12-01
    Completed: -
    Priority: HIGH

    📋 Description

    Restore and extend the graph-orchestrator so that it can run long-lived, event-driven workflows: tools that emit events and sometimes block until results arrive, an Event Bridge that resumes threads correctly, and simple but effective token/budget control per call. The goal is to match the expectations in MASTER-PLAN.md, PIPELINE-FLOWS.md, EVENTS.md, and LANGGRAPH_NOTES.md while keeping the current working implementation as the stable foundation.

    🧩 Rules

    Global rules (do not remove)

    • Follow the work-plan lifecycle and structure defined in docs/WORK-PLANS.md.
    • All code changes must comply with docs/CODE-GENERATION.md and docs/CONFIGURATION.md.

    MANU: configuration quedo deprecado, data fetcher, su codigo, commons, database, tienen codigo que muestra como se manejan configs ahora! - Codex Conclusion (2025-12-02): Tomamos docs/CONFIGURATION.md como referencia histórica y, para este plan, usaremos los patrones ya vigentes en modules/common, modules/database y modules/data-fetcher para cargar config_obj. Antes de cada iteración vamos a revisar esos módulos en disco para mantener el plan sincronizado con la implementación real y documentaremos cualquier delta para la futura reescritura de la doc.

    • Do not delete existing comments or content; changes must be additive or clearly refactoring-only.
    • Read this plan, the relevant docs, and the code on disk before each iteration; never rely on stale mental summaries.
    • Git workflow must follow docs/GIT-COMMITS.md; no destructive git commands unless explicitly approved.

    Plan-specific rules

    • Treat the current modules/graph-orchestrator implementation as the primary source of truth; legacy _OLD code is reference only.
    • Event flows must respect the canonical event types and payloads in docs/EVENTS.md and the pipeline stages in docs/PIPELINE-FLOWS.md.
    • The Event Bridge must not lose or re-order events in a way that breaks experiment/trial semantics; any batching or filtering must be explicit.
    • Token/budget enforcement can be simple at first, but it MUST be enforced centrally (one place) and use the configuration from config/graph-models.yaml.

    //MANU: config/graph-models.yaml. se carga automaticamente enel orchestador dentro de graph_orchestrator_config: GraphOrchestratorConfig que esta accesible siempre - Codex Conclusion (2025-12-02): Todos los límites de loops, budgets y selección de tiers se modelarán como nuevas secciones dentro de graph_orchestrator_config, tomando config/graph-models.yaml como fuente única. Ampliaremos ese YAML para exponer loop_limits, simulation, budget_alerts y otros grupos de constantes, dejando claro en el plan que cualquier cambio de estructuras implica actualizar valores por defecto y el archivo en disco simultáneamente.

    • HITL/interrupt logic must be designed so it can later integrate cleanly with the control-panel module and LangGraph Agent Server.

    🔍 Findings

    Finding 1: Graph invoke is single-shot and does not loop over tools

    • Problem: The current graph invocation stops after a single LLM call per agent. When the Director (or other agents) call tools, the workflow returns immediately with tool results and no follow-up LLM turn to interpret those results.
    • Location:
    • modules/graph-orchestrator/src/rainbow/graph_orchestrator/workflow.py (graph built with StateGraph, but invoke_graph() is a single call to graph.invoke(state) with no outer loop).
    • modules/graph-orchestrator/src/rainbow/graph_orchestrator/agents/director.py (tool handling returns messages and next_node=None after a single LLM call).
    • Other agents (planner.py, analyst.py, coder.py, doc.py, research.py, issues.py) share the same pattern (one LLM call, optional tool calls, immediate return).
    • Impact:
    • CLI invoke calls terminate after the first prompt; the "deep agent" cannot keep reasoning over tool outputs in a single run.
    • Director never produces a final natural-language answer after running tools; the last visible messages are often raw tool outputs.
    • The advanced flows in MASTER-PLAN.md and PIPELINE-FLOWS.md (multi-step planning, analysis, coding, etc.) cannot execute inside one graph run.
    • Recommendation:
    • Introduce a consistent pattern where agents:
      • Call tools.
      • Append ToolMessage results to state.messages.
      • Re-enter the same agent (or Director) in the same graph.invoke() run to interpret tool results and decide the next node.
