    Análisis Exhaustivo: Graph Orchestrator

    Fecha: 2025-12-02 Scope: modules/graph-orchestrator/, modules/common/src/rainbow/common/types/events.py, modules/database/src/rainbow/database/services/

    Resumen Ejecutivo

    Después de revisar TODO el código del módulo graph-orchestrator y sus componentes relacionados, encontré varios problemas que requieren atención. El código tiene una buena arquitectura base, pero hay inconsistencias, duplicaciones, y funcionalidades incompletas.

    1. PROBLEMAS CRÍTICOS en Events

    1.1 ❌ Eventos DUPLICADOS en events.py

    Archivo: modules/common/src/rainbow/common/types/events.py

    Los siguientes eventos están definidos DOS VECES en el enum EventType:

    Evento Primera definición Segunda definición
    ALL_TRIALS_DONE Línea 69 Línea 88
    LIVE_TRADING_STARTED Línea 75 Línea 123
    LIVE_EXEC_SUMMARY Línea 76 Línea 120
    LIVE_ANOMALY_DETECTED Línea 77 Línea 129
    LIVE_TRADING_STOPPED Línea 78 Línea 126

    Problema: Python permite esto pero el último valor sobrescribe. Los comentarios de payload son diferentes entre ambas definiciones.

    Solución: Eliminar las definiciones duplicadas y consolidar los comentarios de payload.

    MANU: pero unificando todos los eventos que vamos a necesitar!!! si no entendi mal, cada tarea asyncrona, REQUESTED -> PROCESING -> DONE, no?

    CASCADE: Exacto! El patrón correcto es:

    {TASK}_REQUESTED → {TASK}_PROCESSING → {TASK}_DONE | {TASK}_FAILED

    Por ejemplo para Training:

    TRAINING_REQUESTED → TRAINING_PROCESSING → TRAINING_DONE | TRAINING_FAILED

    Esto aplica a TODAS las operaciones async: - Data pipeline (fetch, labels, fat, small) - Training - Backtest - Code operations (tests, linter, deploy) - Research/Analysis tasks - HITL (request → decision)

async def _status_check_loop(self):
    """Monitorea operaciones activas y detecta timeouts."""
    while not self._stopped:
        operations = await self._get_active_operations()
        results = await self.check_active_operations(operations)

        for result in results:
            if result["status"] == "timeout":
                await self._handle_timeout(result)

        await asyncio.sleep(self.status_check_interval)  # 30s

    Thread ID Strategy - ✅ Bien diseñado

    El sistema de thread IDs jerárquico está bien pensado:

    rainbow_main → exp-{id} → exp-{id}.trial-{id} → exp-{id}.trial-{id}.{subtask}

    no es asi! es un abrol con hojas !

    es mas genericomtodabia!

    CASCADE: Tenés razón! El diagrama correcto es un árbol genérico:

                                  rainbow_main (root)
                                    │
           ┌────────────────────────┼────────────────────────┐
           │                        │                        │
       exp-1                   task-research            task-refactor
           │                        │                        │
    ┌──────┴──────┐          ┌──────┴──────┐          ┌──────┴──────┐
    │             │          │             │          │             │
 trial-1      trial-2     analyze      summarize    audit       implement
    │             │          │                        │             │
    │        ┌────┴────┐     │                   ┌────┴────┐        │
    │        │         │     │                   │         │        │
backtest  train    validate  │               module1  module2    test
                           deep-dive

    Características del árbol: - Raíz única: rainbow_main - Nodos intermedios: experiments, tasks genéricos - Hojas: operaciones finales (backtest, train, analyze, etc.) - Profundidad ilimitada: task → subtask → subsubtask → ...

    El formato de thread_id refleja el path desde la raíz:

    rainbow_main                           # raíz
rainbow_main.exp-1                     # hijo de main
rainbow_main.exp-1.trial-1             # hijo de exp-1
rainbow_main.exp-1.trial-1.backtest    # hijo de trial-1

rainbow_main.task-research             # otro hijo de main
rainbow_main.task-research.analyze     # hijo de research
rainbow_main.task-research.analyze.deep-dive  # hijo de analyze

    Pero podemos simplificar omitiendo rainbow_main. en la práctica.

    Los tests en test_thread_routing.py cubren bien los casos.
