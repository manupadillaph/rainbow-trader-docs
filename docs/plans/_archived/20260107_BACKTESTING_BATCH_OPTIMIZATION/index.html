    Backtesting Batch Optimization

    Status: 🟢 COMPLETE
    Started: 2026-01-07
    Priority: HIGH

    📋 Description

    Optimize backtesting engine by implementing batched window generation and batched model predictions. Currently, the engine generates one window per tick and runs one model forward pass per tick, which is extremely slow. This plan implements batch processing to achieve ~60x speedup.

    🧩 Rules

    Global rules (do not remove)

    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md.
    • Respect project docs: All changes must comply with the relevant docs.
    • Language: Work plan and all code/docs MUST be in English.

    Plan-specific rules

    • All batch sizes are config parameters in backtesting.yaml — no hardcoded values.
    • Trading logic remains tick-by-tick (required for realistic simulation).
    • prediction_batch_size must be <= window_batch_size and must divide evenly.
    • Invalid batch sizes must raise a ValueError before entering any backtest loops.

    🔍 Findings

    Finding 1: Slow Window Generation

    • Problem: stream_windows() calls build_window_row() for each timestamp, which does filter().tail() on the entire DataFrame every time — O(n×m) complexity.
    • Location: window-preparer/helpers/fat_windows.py:444-454 and build_window_row() lines 380-401
    • Impact: Major bottleneck — 100k timestamps × 3 features = 300k filter operations
    • Recommendation: Add stream_windows_batched() that yields batches and uses indexed lookups
    • User Decision: Implement batched version with window_batch_size config

    Finding 2: Single-Item Model Inference

    • Problem: PyTorchModelWrapper.predict() runs inference with batch_size=1, wasting GPU parallelism
    • Location: model-trainer/helpers/pytorch_wrapper.py:67-76
    • Impact: GPU is idle ~99% of time waiting for memory transfers
    • Recommendation: Add predict_batch() method that processes multiple windows at once
    • User Decision: Implement batched prediction with prediction_batch_size config

    Finding 3: Missing Batch Config Parameters

    • Problem: No batch size configuration exists in BacktestingConfig
    • Location: common/types/configs/backtesting_config.py and config/backtesting.yaml
    • Impact: Cannot configure batch sizes per environment (GPU/CPU)
    • Recommendation: Add window_batch_size and prediction_batch_size parameters
    • User Decision: Add both parameters to config

    Finding 4: Sequential Loop Architecture

    • Problem: run_backtest_loop() processes one tick at a time, preventing batch optimizations
    • Location: backtesting-engine/helpers/engine_loop.py:425-510
    • Impact: Cannot leverage batched windows and predictions
    • Recommendation: Replace run_backtest_loop() with a batched 3-loop implementation
    • User Decision: Implement batched loop and remove superseded single-tick loop once batched path is in use

    Finding 5: No Prediction Logging for Visualization

    • Problem: Predictions and confidence scores are computed but not persisted, making it impossible to visualize them in control panel
    • Location: backtesting-engine/helpers/engine_loop.py (predictions are used and discarded)
    • Impact: Cannot analyze model behavior, confidence distribution, or prediction patterns in control panel graphs
    • Recommendation: Create PredictionLogger helper that accumulates predictions and writes to parquet files per backtest session
    • User Decision: Implement prediction logging with new parquet type, integrate in batched loop, add control panel read method

    Finding 6: Equity Naming Ambiguity and Missing MTM Persistence

    • Problem: final_equity and current_equity overlap in meaning, and MTM/unrealized/open counts are not persisted in backtest_sessions
    • Location: database/models.py (BacktestSession), backtesting-engine/helpers/metrics_snapshot.py, backtesting-engine/helpers/engine_loop.py
    • Impact: Confusing equity semantics and missing MTM/unrealized data in control panel
    • Recommendation: Rename equity fields to equity and equity_mark_to_market, persist unrealized_pnl and open_positions_count
    • User Decision: Remove "final" from naming, keep logic (equity=realized, equity_mark_to_market=realized+unrealized), persist missing fields

    Architecture Decision: Two Batch Sizes

    • window_batch_size: Controls RAM usage — how many windows to pre-compute and hold in memory
    • prediction_batch_size: Controls GPU VRAM usage — how many predictions per forward pass
    • Typical setup: window_batch_size=2048, prediction_batch_size=256 (8 forward passes per window batch)
    • For CPU-only: Use prediction_batch_size=64 or lower

    Three Nested Loops Architecture

    The batched backtest loop will have three levels of nesting:

    Level 1 - Window Batch Loop (Outer):

    • Iterates over batches of windows (size=window_batch_size, e.g., 2048)
    • Calls stream_windows_batched() generator
    • Each iteration yields a batch of pre-computed windows

    Level 2 - Prediction Batch Loop (Middle):

    • Within each window batch, iterates over prediction batches (size=prediction_batch_size, e.g., 256)
    • Calls model_wrapper.predict_batch() for each sub-batch
    • Example: 2048 windows ÷ 256 predictions = 8 forward passes per window batch
    • Caches all predictions for the window batch

    Level 3 - Tick Processing Loop (Inner):

    • For each tick in the window batch, executes trading logic sequentially
    • Calls process_tick() with cached prediction
    • Updates equity, positions, risk manager state tick-by-tick

    Why this structure:

    • Level 1 optimizes window generation (vectorized operations)
    • Level 2 optimizes GPU utilization (batch inference)
    • Level 3 maintains realistic trading simulation (sequential decisions)

    Trading Logic Stays Tick-by-Tick

    The bot decision logic (process_tick()) must remain sequential because:

    1. Each trade affects equity for the next tick
    2. Risk manager state depends on previous trades
    3. Position tracking requires sequential updates

    The optimization is:

    1. Pre-compute N windows (vectorized)
    2. Batch-predict N signals (single/few forward passes)
    3. Execute N ticks sequentially (with cached predictions)

    Batch Size Validation

    • Enforce batch sizes before any loop starts.
    • Rules: window_batch_size >= 1, prediction_batch_size >= 1, prediction_batch_size <= window_batch_size.
    • If window_batch_size % prediction_batch_size != 0, raise ValueError and stop the run.

    Progress Bar Behavior (Batched)

    • Use a single tqdm with total = total ticks (timestamps count).
    • Update tqdm once per tick inside the innermost loop.
    • Track outer loop counters (window batch index and prediction batch index) and show them in postfix.
    • Do not reset progress between batches; one continuous bar from 0 → total.
    • Keep the 3-level loop structure even when batch sizes are 1.

    Prediction Logging Architecture

    Storage Location:

    data/parquets/predictions/{backtest_session_id}.parquet

    Parquet Schema:

    Column Type Description
    timestamp datetime Tick timestamp
    prediction int Predicted class (0=NONE, 1=LONG, 2=SHORT)
    confidence float Max probability (0.0-1.0)
    prob_none float Probability for NONE class
    prob_long float Probability for LONG class
    prob_short float Probability for SHORT class

    PredictionLogger Helper:

    • Accumulates predictions in memory buffer (list of dicts)
    • Flushes to parquet every window_batch_size ticks (after each window batch)
    • Uses append mode to avoid holding all predictions in memory
    • Final flush on backtest completion
