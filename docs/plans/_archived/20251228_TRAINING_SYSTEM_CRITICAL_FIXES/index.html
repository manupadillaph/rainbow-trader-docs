    Training System Critical Fixes

    Status: 🔴 IN PROGRESS
    Started: 2025-12-24
    Completed: (pending)
    Priority: HIGH

    📋 Description

    Fix critical bug in training system causing 27 trials to return identical objective values (0.524012806830309 = majority class baseline). Root cause: ModelTrainer silently falls back to majority class prediction when feature windowing fails due to missing offsets, instead of properly failing the trial.

    🧩 Rules

    Global rules (do not remove)

    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch. Reality lives on disk (plans, docs, code), not in cached summaries.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md. Never run git add from the repository root unless explicitly specified in this plan.
    • Respect project docs: All changes must comply with the relevant docs (configuration, code generation, database schema, markdown style, etc.).
    • Language: User responses and AI interpretations in the conversation log may be in Spanish, but the work plan document itself and all generated code/docs MUST be written in English.

    • Use .venv: All commands must be executed inside the virtual environment

    🔍 Findings

    Finding 1: Critical Path Resolution Bug - Small Windows Missing

    • Problem: Database stores relative paths (parquets/small_windows/BTCUSDT/5m/file.parquet) but ModelTrainer/SmallWindowPreparer look for absolute paths (/full/path/to/file.parquet). ALL 45 small window parquets recorded in database are missing from training system perspective, causing fallback to dummy data.
    • Location:
    • modules/database/src/rainbow/database/models.py - Parquet model path field
    • modules/window-preparer/src/rainbow/small_window_preparer/small_window_preparer.py - Path resolution logic
    • modules/model-trainer/src/rainbow/model_trainer/model_trainer.py - Training data loading
    • Impact:
    • Training system cannot find actual data files
    • Models train on fallback/dummy data leading to identical results
    • All optimization is meaningless as trials use the same dataset
    • Evidence:
    • Database: 45 SMALL parquets recorded
    • Disk: 46 actual parquet files exist
    • Filename matches: 100% (5/5 sampled files match exactly)
    • Path structure: DB uses parquets/... vs system expects /home/manuel.../data/parquets/...
    • Recommendation: Fix path resolution in data loading logic to handle relative paths correctly
    • Questions:
    • Should we convert DB paths to absolute or fix the loading logic to resolve relative paths?
    • Should we add path validation during parquet registration?
    • User Decision: (pending)
    • Action Required: Fix path resolution between database storage and file system access

    Finding 2: Identical Objective Values Across All Trials

    • Problem: 26 out of 44 trials return identical objective value (0.524012806830309) despite different hyperparameters, feature selections, and model configurations. This indicates training is not actually using different data or configurations.
    • Location:
    • modules/model-trainer/src/rainbow/model_trainer/model_trainer.py:1075-1221 - Training and evaluation logic
    • Database trials table - objective_value column shows identical values
    • Impact:
    • Hyperparameter optimization is completely broken
    • Optuna cannot learn which configurations work better
    • Wasted computational resources
    • Model selection is meaningless
    • Root Cause: Connected to Finding 1 - since small windows are "missing", training falls back to identical validation data
    • Evidence:
