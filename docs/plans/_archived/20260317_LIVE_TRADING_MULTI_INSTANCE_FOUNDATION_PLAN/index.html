    🛠️ Detailed execution blueprint (how we implement this)

    This section explains how each phase will be implemented without changing the approved task list.

    A. Runtime principles and invariants

    • Process only closed candles for decisions.
    • Keep strict parity with backtesting model input contracts (column names, ordering, window shape).
    • Run each live bot as one isolated instance with one persisted live_session_id.
    • Persist every critical step (feed -> features -> window -> decision -> order -> trade -> accounting -> cursor) for replayability.
    • Keep execution-plan-only runtime; no runtime master orchestration in this scope.

    B. Concrete startup flow (per selected execution plan)

    1. Resolve target execution_plan_id values from CLI input.
    2. Load plan + runtime requirements from DB/services.
    3. Resolve model checkpoint using the same selector semantics as backtesting (best, best_f1, best_precision, best_recall, latest, best_or_latest, epoch).
    4. Build per-instance runtime context:
    5. live_session_id
    6. symbol(s), timeframe(s), provider, market type
    7. feature requirements and selected model columns
    8. model artifact path and metadata
    9. Compute history requirement using build_experiment_requirements() from core/helpers/features.py:
    10. This recursively calculates exact warm-up ticks for every feature in the tree.
    11. Each ExtendedFeature node knows ticks_warm_up and range_start_with_warm_up (exact datetime).
    12. Warm-up accumulates: fat_window.window_size + pipeline get_warm_up_periods() + indicator get_warm_up_periods().
    13. No arbitrary lookback number — the experiment definition IS the bootstrap specification.
    14. Initialize historical data (deterministic, not configurable):
    15. For each leaf feature, get range_start_with_warm_up → this is the earliest data needed.
    16. Check local DB/parquet for existing data across that range.
    17. Fetch from provider (exchange API) to fill any gaps.
    18. Re-read merged range and validate continuity and timestamp alignment.
    19. Fail startup if coverage is still insufficient.
    20. Initialize per-instance in-memory buffers/states and start stream loop from last persisted cursor.

    C. Data initialization policy (deterministic from ExperimentRequirements)

    Data initialization is NOT configurable. It is deterministic from the experiment definition. There is no bootstrap_mode or bootstrap_lookback_candles — the system computes exact requirements automatically.

    For each (symbol, timeframe) instance dependency:

    1. Call build_experiment_requirements() with the experiment from ExecutionPlan.
    2. For each leaf ExtendedFeature, read range_start_with_warm_up — this is the exact earliest datetime needed.
    3. Compute the required close-time interval: from range_start_with_warm_up to current reference close time (now aligned to closed candle boundary).
    4. Read local history (DB/parquet) in that interval.
    5. If local coverage is complete, continue with local data only.
    6. If local coverage is missing or incomplete, fetch the full required interval from provider (exchange API).
    7. Persist fetched candles.
    8. Re-load full interval and assert:
    9. ordered by close time
    10. no duplicates
    11. expected timeframe step consistency
    12. Fail startup for that instance if coverage is still insufficient.

    D. Live feed loop mechanics (data-fetcher first-class)

    Per instance stream behavior:

    1. Build required OHLCV dependency set from experiment requirements:
    2. base decision stream (base_symbol, base_timeframe)
    3. all additional symbols/timeframes required by selected features
    4. Run a single control loop driven only by closed candles from base decision stream.
    5. Normalize base provider payload to canonical closed-candle structure.
    6. Ignore non-closed updates in the decision path.
    7. Build idempotency key using (live_session_id, base_symbol, base_timeframe, close_time).
    8. If key already processed, skip.
    9. Persist base feed event metadata and processing cursor candidate.
    10. For current base close time, synchronize dependency data:
    11. check local coverage for each required (symbol, timeframe) up to T_base_close
    12. if coverage is missing/outdated, fetch required closed-candle range from provider
    13. persist fetched dependency candles before continuing
    14. After dependency sync is complete, continue to feature/window path for the same T_base_close.
    15. On transient stream failure, reconnect with bounded retry/backoff, resume from persisted cursor, and fetch the full required backfill interval before normal loop resumes.

    E. Online feature and window path (data-processing + window-preparer)

    Per accepted closed candle:

    1. Append candle to timeframe buffer.
    2. Update online feature state:
    3. incremental update when available
    4. bounded tail recompute fallback when incremental is not available
    5. For base decision trigger T_base_close, resolve dependency snapshot using latest closed candle with close_time <= T_base_close per required (symbol, timeframe) after dependency sync.
    6. Check feature readiness gates (warm-up satisfied for all required columns at T_base_close).
    7. Build SMALL window using existing window-preparer contracts and exact selected-column order.
    8. Validate final model input tensor shape and dtype against training/backtesting contracts.
    9. If not ready, checkpoint cursor and continue without decision.

    F. Decision, execution, accounting, persistence

    When a ready window is available:

    1. Run model inference.
    2. Build deterministic decision payload and correlation IDs.
    3. Run bot-logic decision step.

    S. Data flow reference: offline training → live trading mapping

    This section documents the complete data flow for both offline (training/backtesting) and live trading, so the implementing agent understands exactly what exists, what to reuse, and what to build new.

    S.1 Offline training/backtesting data flow (existing system — reference for live)

    Phase 1 — Feature computation (offline, once per experiment):

    1. ExperimentConfig defines the feature tree: OHLCV → Indicators → Pipelines → Signals.
    2. build_experiment_requirements() builds the ExtendedFeature tree with warm-ups and dependencies.
    3. Source: modules/core/src/rainbow/core/helpers/features.py
    4. Each feature node carries (symbol, timeframe) — the system is multi-symbol and multi-timeframe (MTF).
    5. FeatureOHLCV has explicit symbol: str and timeframe: str fields.
    6. Source: modules/core/src/rainbow/core/types/composed/features.py
    7. OHLCV RAW leaf nodes represent the unique (symbol, timeframe) pairs that need source data.
    8. Each leaf has range_start_with_warm_up — the exact earliest datetime needed, accounting for ALL parent warm-ups propagated down the tree (fat_window.window_size + pipeline warm-ups + indicator warm-ups).
    9. Source: ExtendedFeature.range_start_with_warm_up in modules/core/src/rainbow/core/types/experiments.py
    10. fetch_experiment_requirements() fetches missing OHLCV from exchange → saves parquets to disk.
    11. Source: modules/core/src/rainbow/core/services/experiment_data_service.py
    12. process_experiment_requirements() computes features in topological order → one parquet per feature on disk.
    13. Uses topological_sort_process_requests() for correct dependency ordering.
    14. Source: modules/core/src/rainbow/core/services/experiment_data_service.py
    15. Pure compute functions are in modules/data-processing/:
      • Indicator dispatch: modules/data-processing/src/rainbow/data_processing/helpers/indicator_dispatch.py
      • Pipeline dispatch: inside data_processing.py → process_feature_request_pipeline_stage()
      • Signal dispatch: inside data_processing.py → process_feature_request_signal()
    16. Result: N parquets on disk, one per feature node (e.g., ohlcv_5m.pqt, ema_14.pqt, threshold_sig.pqt).

    Phase 2 — FAT window generation (offline, once per experiment):

    1. FatWindowPreparer.generate_fat_windows_for_experiment() creates FAT windows.
    2. Source: modules/window-preparer/src/rainbow/window_preparer/fat_window_preparer.py
    3. Uses LazyFeatureFramesReader: holds references to ALL feature parquets (zero data loaded on init).
    4. Source: modules/core/src/rainbow/core/helpers/lazy_feature_reader.py
    5. For each sampling timestamp: build_window_row() flattens ALL features × fat_window_size offsets.
    6. Example columns: feat_ema14__value__t-0, feat_ema14__value__t-1, ..., feat_ema14__value__t-29
    7. Source: modules/window-preparer/src/rainbow/window_preparer/helpers/fat_windows.py
    8. Output: fat_windows.parquet — ALL features × ALL offsets. This is a massive parquet with every possible column.

    Phase 3 — Optuna training loop (many trials per experiment):

    1. Each Optuna trial picks hyperparameters:
    2. window_size (e.g., 20 — must be ≤ fat_window_size)
    3. feature_columns_with_weights (e.g., {ema14: 0.8, rsi: 0.3, ...} — subset of all features)
    4. Model hyperparameters
    5. SmallWindowPreparer.prepare_small_window():
    6. Reads fat_windows.parquet.
    7. Filters to ONLY the selected columns × the chosen window_size offsets.
    8. Saves small_windows.parquet — a SUBSET of FAT (fewer columns, fewer offsets).
    9. Source: modules/window-preparer/src/rainbow/window_preparer/small_window_preparer.py
    10. Mini backtest (fast, for Optuna scoring):
    11. Reads tail_frame from SMALL parquet (last N rows).
    12. Iterates row by row directly: row → model.predict(features) → bot_logic → simulate.
    13. Source: run_mini_backtest_loop() in modules/backtesting-engine/src/rainbow/backtesting_engine/helpers/engine_loop.py

    Phase 4 — Validation backtest (full, for best trial):

    1. Does NOT read FAT or SMALL parquets. Uses lazy streaming instead.
    2. LazyFeatureFramesReader points to individual feature parquets on disk.
    3. build_lazy_windows_generator() → calls stream_windows_batched_lazy():
    4. Loads feature parquets in memory-efficient batches (only one batch in RAM at a time).
    5. For each batch of timestamps: loads feature data → build_window_row() with feature_columns_names_set (SMALL mode — only selected features from the trial's resolved_hyperparameters).
    6. Yields batches of (timestamp, window_row) tuples.
    7. Frees batch data before loading next batch.
    8. Source: modules/backtesting-engine/src/rainbow/backtesting_engine/helpers/engine_setup.py
    9. run_backtest_loop() consumes batches:
    10. model.predict_batch(batch_features) → signals.
    11. process_tick() per timestamp → bot_logic → simulate.
    12. Source: modules/backtesting-engine/src/rainbow/backtesting_engine/helpers/engine_loop.py

    Key insight: build_window_row() is the canonical function for building a single flattened window row. It takes feature_frames_by_feature_key (dict of DataFrames), window_size, and feature_columns_names_set. It does NOT care whether the DataFrames come from parquet or from in-memory buffers — it just slices and flattens.

    S.2 Live trading data flow (to implement)

    Phase A — Bootstrap (once per session start):

    1. Load ExperimentComposed from experiment_id (from the ExecutionPlan).
    2. Call build_experiment_requirements() — same function as offline.
    3. Returns ExperimentRequirements with the full ExtendedFeature tree.
    4. Collect unique (symbol, timeframe) OHLCV RAW leaves from the tree:
    5. Walk extended_features recursively → find nodes where feature_type == OHLCV and is_raw_level == True.
    6. Each leaf has range_start_with_warm_up (exact datetime, already accounts for ALL parent warm-ups).
    7. Deduplicate by (symbol, timeframe) — keep the earliest range_start_with_warm_up per pair.
    8. Example result set:
      • (BTCUSDT, 5m) — base, needs data from 2025-12-01
      • (BTCUSDT, 1h) — MTF dependency, needs data from 2025-11-15
      • (BTCUSDT, 1d) — MTF dependency, needs data from 2025-06-01
      • (ETHUSDT, 5m) — multi-symbol dependency, needs data from 2025-12-01
      • (ETHUSDT, 1h) — multi-symbol + MTF, needs data from 2025-11-15
    9. Load resolved_hyperparameters from the trial:
    10. window_size — the window size chosen by Optuna for this trial.
    11. feature_columns_with_weights — the feature columns the trained model expects as input.
    12. These define SMALL window mode (which features go into the model input).
    13. For each unique (symbol, timeframe) pair:
    14. Required range: range_start_with_warm_up → now (aligned to closed candle boundary).
    15. Load from existing parquets in DB covering the range.
    16. Fetch gap from exchange API to fill up to now.
    17. Store in ohlcv_buffers[(symbol, timeframe)] = pl.DataFrame.
    18. Validate: ordered by close time, no duplicates, expected timeframe step consistency.
    19. Compute ALL features in the tree in topological order, entirely in memory:
    20. Use topological_sort_process_requests() — same function as offline — for correct ordering.
    21. For each feature node in topological order:
      • Take input DataFrames from parent buffers (OHLCV buffers or previously computed feature buffers).
      • Call the same pure compute functions used offline:
      • Indicators: process_indicator_from_config() from indicator_dispatch.py.
      • Pipelines: pure compute functions extracted from process_feature_request_pipeline_stage().
      • Signals: pure compute functions extracted from process_feature_request_signal().
      • Store result in feature_buffers[(feature_key)] = pl.DataFrame.
    22. Critical: The pure compute functions are DataFrame in → DataFrame out. They do NOT care about parquets. The offline path wraps them with parquet I/O; the live path wraps them with in-memory buffers. Same core math.
    23. Build first SMALL window:
    24. Call build_window_row() — same function as backtesting.
    25. Pass feature_buffers as feature_frames_by_feature_key.
    26. Pass feature_columns_names_set from trial's feature_columns_with_weights (SMALL mode).
    27. Pass window_size from trial's resolved_hyperparameters.
    28. Result: one flattened dict, identical format to what backtesting produces.

    Phase B — Live loop (per base candle close):

    1. Trigger: closed candle arrives on (base_symbol, base_timeframe).
    2. Synchronize ALL dependency (symbol, timeframe) pairs:
    3. For each (sym, tf) in the dependency set:
      • Get latest closed candle where close_time ≤ T_base_close.
      • If a new candle is available → append to ohlcv_buffers[(sym, tf)].
      • If tf > base_tf, there may NOT be a new candle on every base tick (e.g., 1d candle doesn't close every 5m tick). This is expected — the feature buffer retains the last computed value.
      • Drop oldest rows if buffer exceeds max size (sliding window).
    4. Recompute features in topological order:
    5. For each feature that depends on an updated (symbol, timeframe):
      • Take tail of parent buffer (warm_up + margin rows) as input DataFrame.
      • Call pure dispatch function → result DataFrame.
      • Take LAST ROW of result → append to feature_buffers[(feature_key)].
      • Drop oldest if buffer exceeds max size.
    6. Same pure functions as offline and bootstrap. No new math.
    7. Build SMALL window:
    8. build_window_row() with feature_buffers, feature_columns_names_set, window_size.
    9. Result: one flattened dict, identical to backtesting format.
    10. Model inference: model.predict(window_row) → signal.
    11. Bot logic → execute (SIM in this scope, REAL in future scope).
    12. Persist decision/order/trade lineage + advance cursor.
    13. Repeat from step 1.
