    Batch Size Benchmarking Plan

    Status: 🔴 IN PROGRESS
    Started: 2026-01-15
    Completed:
    Priority: HIGH

    📋 Description

    Create a standalone benchmarking workflow that tunes performance-critical batch and loader parameters for FAT window generation, SMALL window generation, model training, and backtesting using a real experiment ID, with Optuna-driven search, staged execution, and machine-specific JSON outputs, while keeping all config and outputs inside scripts/benchmarks/batch_sizes/ and not mixing with the core project configuration.

    🧩 Rules

    Global rules (do not remove)

    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch. Reality lives on disk (plans, docs, code), not in cached summaries.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md. Never run git add from the repository root unless explicitly specified in this plan.
    • Respect project docs: All changes must comply with the relevant docs (configuration, code generation, database schema, markdown style, etc.).
    • Language: User responses and AI interpretations in the conversation log may be in Spanish, but the work plan document itself and all generated code/docs MUST be written in English.

    Plan-specific rules

    • Keep all benchmark code and config under scripts/benchmarks/batch_sizes/ (no edits to project-wide config defaults).
    • Use a real experiment ID provided at runtime; do not hardcode experiment IDs or ranges.
    • Benchmark steps must be explicitly selectable by CLI flags to allow staged execution (FAT → SMALL → training → backtesting).
    • Do not delete or overwrite parquets, trials, or database rows without explicit user approval.
    • ~~Prefer full pipeline runs to surface memory accumulation; allow a controlled micro-mode by config when requested.~~
    • Prefer full pipeline runs to surface memory accumulation; no micro-mode (use smaller experiment instead).
    • Use Optuna as the search strategy for parameter tuning, with clear logging of failures and resource usage.
    • Capture machine metadata (CPU/GPU/RAM) in benchmark outputs so results are machine-specific.
    • All outputs must be written under scripts/benchmarks/batch_sizes/ (including JSON results).
    • No benchmark config override; use the local benchmark config only.
    • Keep backtest cadence parameters fixed (no tuning in v1).

    🔍 Findings

    Finding 1: Benchmarking must be staged and process-selectable

    • Problem: FAT windows must exist before SMALL window preparation, training, and backtesting can run, so benchmarking must proceed in ordered stages.
    • Location: New benchmark script under scripts/benchmarks/batch_sizes/ that orchestrates existing pipeline steps.
    • Impact: Running stages out of order will fail due to missing FAT windows or invalid inputs; it blocks meaningful optimization.
    • Recommendation: ~~Add explicit CLI/config options to select which stage(s) to benchmark and enforce prerequisites.~~
    • Recommendation (updated): Provide stage selection flags without prerequisite enforcement or chaining.
    • Questions:
    • Should the script refuse to run a stage if prerequisites are missing, or offer a --force override?
    • Do you want a single run to chain multiple stages automatically once a previous stage completes successfully?
    • User Decision: ~~single run y no fail control, no compliquemos lel objetivo ni ls scripts! queremos optimizar parameotrs nada mas! yo me encargo de poner las cosas bien!!!~~
    • User Decision (confirmed): Single-stage runs only; no prerequisite enforcement or chain mode. Keep scripts minimal; user handles prerequisites.
    • Action Required: ~~Design CLI + config stage gates and prerequisite checks in the benchmark script.~~
