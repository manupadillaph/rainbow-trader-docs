    Work Plan: Module Standardization Upgrade

    Status: APPROVED Owner: Manuel Padilla / GitHub Copilot Date: 2025-12-01

    1. Overview

    This plan outlines the standardization and upgrade of ten (10) legacy modules in the rainbow-trader system. The goal is to bring all modules up to the "Gold Standard" established by the data-fetcher module. This includes consistent directory structure, configuration handling, logging, CLI implementation, error handling, and quality gates (mypy, ruff).

    Reference Module: modules/data-fetcher (DO NOT MODIFY)

    Objectives

    1. Standardize Structure: Ensure all modules follow the exact file and folder structure of data-fetcher.
    2. Unify Configuration: Implement Config and Logger patterns from rainbow-common.
    3. Implement CLI: Provide a consistent Typer-based CLI for all modules with a health-check command.
    4. Preserve Logic: Carefully migrate any existing logic from "skeleton" or old files (_OLD/), ensuring no functionality is lost.
    5. Quality Assurance: Ensure all modules pass mypy --strict, ruff linting, and build without errors.

    Conclusión Final: Todos los módulos deben alinearse 1:1 con el estándar de data-fetcher (estructura, CLI Typer, Config/Logger comunes, eventos y health-check), sin perder ninguna pieza de lógica previa y garantizando tipado estricto y calidad.

    2. Rules & Constraints

    1. NEVER Delete Logic: If code exists (even if it looks like a skeleton), preserve it or migrate it. Do not discard "old" code without verification.
    2. Strict Typing: All new and migrated code must be fully typed and pass mypy --strict.
    3. Consistent Naming: Use snake_case for files/functions, PascalCase for classes, UPPER_SNAKE_CASE for constants.
    4. Dependency Management: Each module must have its own valid pyproject.toml and requirements.txt.
    5. Health Checks: Every module MUST have a working health-check command in its CLI.
    6. Documentation: Update README.md and CHANGELOG.md for each module (ignore outdated content for now, focus on structure).
    7. Data Fetcher Integrity: Do NOT modify or delete code from modules/data-fetcher. It is the reference.
    8. Shared Types: Types/classes that will be reused across modules (like Order) go in rainbow.common.types/, NOT inside individual modules.

    Conclusión Final: Respetar todos los constraints anteriores; sólo se rompen si hay instrucción explícita del usuario. Cualquier lógica existente se migra intacta, se tipa estricto y se documenta, manteniendo data-fetcher como patrón inviolable.

    3. Findings & Analysis

    3.1 Reference Pattern (The "Gold Standard")

    Based on data-fetcher, database, and common: * Structure: src/rainbow/<module>/ containing __init__.py, __main__.py, cli.py, constants.py, py.typed, <main_class>.py, helpers/. * CLI: Uses Typer, rainbow.common.helpers.config.Config, rainbow.common.helpers.logger.Logger. * Config: Loads via Config.load_config with --config and --set flags. * Dependencies: pyproject.toml must list rainbow-common, rainbow-database (if used), and external libs.

    Conclusión Final: El “Gold Standard” es exactamente el layout y patrones de data-fetcher; cualquier módulo que difiera debe ser ajustado hasta coincidir en estructura, CLI, Config/Logger y deps declaradas.

    3.2 Legacy Code Locations

    We must migrate logic from these locations into the new src/ skeletons: * backtesting-engine: _OLD/_03/backtesting_engine/core.py * fat-window-builder: _OLD/_03/fat_window_builder/core.py * housekeeping: _OLD/_03/housekeeping/core.py * window-preparer: _OLD/_03/small_window_preparer/core.py * data-processing: _OLD/_03/core_processing_library/core.py + _REFERENCE/datasets_reference/src/ * bot-logic: modules/bot-logic/_OLD/bot_logic/core.py * control-panel: modules/control-panel/_OLD/control_panel/backend/core.py * labeling-service: _REFERENCE/datasets_reference/src/g_labels/detect_longshort.py * live-trading-runner: No legacy code found, skeleton only * model-trainer: No legacy code found, skeleton only

    Conclusión Final: Todo código legado listado debe migrarse a los skeletons actuales (incluyendo waits simulados y emisiones de eventos cuando existan), sin descartar nada salvo indicación explícita; los módulos sin legacy mantienen dummy completo pero alineado a estándar y con waits constantes.
