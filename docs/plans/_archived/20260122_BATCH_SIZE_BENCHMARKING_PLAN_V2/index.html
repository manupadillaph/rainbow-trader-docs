    Batch Size Benchmarking Plan V2 - Dedicated Benchmark Methods

    Status: 🔴 NOT STARTED
    Started: 2026-01-21
    Completed:
    Priority: HIGH

    📋 Description

    Create dedicated benchmark methods in each module (window-preparer, model-trainer, backtesting-engine) that allow measuring throughput (rows/second, epochs/second, ticks/second) without running full pipelines. These methods will process a configurable fraction of data and return timing/memory metrics, enabling fast Optuna-driven parameter tuning.

    Key difference from V1: Instead of modifying production code paths, we create isolated benchmark functions that reuse internal helpers but skip persistence, DB writes, and full validation.

    🧩 Rules

    Global rules (do not remove)

    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md.
    • Respect project docs: All changes must comply with the relevant docs.
    • Language: All generated code/docs MUST be written in English.

    Plan-specific rules

    • No production code modifications: Benchmark methods are NEW functions, not modifications to existing ones.
    • Reuse internal helpers: Benchmark methods call the same internal functions (e.g., run_epoch, build_fat_windows_frame) but with early exit.
    • No persistence: Benchmark methods do NOT write to DB, do NOT create parquet files, do NOT emit events.
    • Return throughput metrics: All benchmark methods return a standardized BenchmarkResult dataclass.
    • Configurable limits: Each method accepts max_iterations or similar to control how much data to process.

    3. Processes N ticks with model predictions (inference)
    4. Skips bot logic, DB writes, events, pause/cancel handling (not relevant to optimization targets)

    📝 Technical Details

    FAT Benchmark - Data Flow

    benchmark_fat_windows(experiment_id, batch_size, max_rows)
    │
    ├── ExperimentDataService.get_experiment_requirements()
    │       └── Returns: extended_features, fat_window_config
    │
    ├── Load feature frames (same as generate_fat_windows_for_experiment)
    │       └── Uses: FeatureLoader, DataProcessingService
    │
    ├── Load sampling frame
    │       └── Uses: LabelingService.get_sampling_frame()
    │
    ├── Create LazyParquetServiceWriter (writes to temp dir)
    │       └── writer = LazyParquetServiceWriter(...)
    │
    ├── try:
    │   └── build_fat_windows_frame(writer=writer, max_rows=max_rows)
    │           └── Writes batches to temp parquet (real work)
    │           └── Early exit after max_rows
    │
    └── finally:
        └── writer.cleanup()
                └── Removes temp dir, no DB registration

    Training Benchmark - Data Flow

    benchmark_training(experiment_id, trial_id, num_workers, prefetch_factor, max_epochs)
    │
    ├── Load SMALL parquet path from trial
    │       └── Uses: TrialRepository, ParquetRepository
    │
    ├── Create model (dummy hyperparams)
    │       └── Uses: create_model(model_type, model_params)
    │
    ├── Create DataLoaders
    │       └── Uses: LazyParquetDataset(num_workers, prefetch_factor)
    │
    └── for epoch in range(max_epochs):
            ├── run_epoch(model, train_loader, ...)
            └── evaluate_model(model, val_loader, ...)

    Backtesting Benchmark - Data Flow

    benchmark_backtesting(experiment_id, trial_id, window_batch_size, prediction_batch_size, max_ticks)
    │
    ├── Load model checkpoint from trial
    │       └── Uses: load_checkpoint()

    │       └── Uses: build_model_wrapper()
    │
    ├── Create windows generator
    │       └── Uses: build_windows_generator()
    │
    └── for window_batch in windows_generator:
            for tick in window_batch:
                └── model_predictor.predict_batch()
                └── (skip process_tick, just measure inference)
