    Graph Orchestrator Expansion Plan

    Status: 🟢 IN PROGRESS (Stage 4 — Execution) Owner: Copilot (pending user confirmation) Created: 2025-11-29 Scope: Restore and extend full LangGraph orchestration (all agents, tools, Event Bridge, CLI/API) so the module matches MASTER-PLAN, PIPELINE-FLOWS, and EVENTS expectations.

    Description

    The current orchestrator works but only implements the Director node plus store tools. The legacy _OLD folder contains richer agent mesh (planner/analyst/coder/doc/research/issues), event bridge loop, FastAPI/Typer runtime, and tooling to drive every compute module via events. This plan captures all gaps so we can merge the stable new foundation with the legacy feature surface, re-aligning with the reference docs and enabling the control panel + compute modules to rely on a fully autonomous brain.

    Rules

    1. Follow WORK-PLANS.md lifecycle: this draft only logs findings; tasks are frozen until user approval to enter Stage 3.
    2. Keep the current production-ready structure as source of truth; migrate legacy behavior without reintroducing deprecated patterns unless explicitly required.
    3. All new code must comply with CODE-GENERATION.md (PEP 8, Google docstrings, mypy/pytest/ruff/CHANGELOG updates) and respect the configuration hierarchy in CONFIGURATION.md.
    4. Event handling, tool wiring, and HITL checkpoints must match the canonical flows described in MASTER-PLAN, PIPELINE-FLOWS, and EVENTS.
    5. Never delete existing comments or content; adapt or extend while keeping historical context per repository rules.
    6. Code location rule: Reusable utilities for the orchestrator only go in graph_orchestrator/helpers/. Cross-module utilities go in common. New DB services go in database.
    7. No copy/paste from _OLD: All functionality must be reimplemented using the new code standards (see current orchestrator, database, common, data-fetcher as references).

    Findings

    F1 — Missing Agent Mesh and Routing

    • Problem: The new workflow only instantiates the Director and routes every decision to END, so no planner/analyst/coder/doc/research/issues loops are possible.
    • Location: modules/graph-orchestrator/src/rainbow/graph_orchestrator/workflow.py#L23-L118 vs. legacy multi-node graph in modules/graph-orchestrator/_OLD/graph/workflow.py#L23-L126.
    • Impact: The orchestrator cannot execute the flows required by MASTER PLAN (planning experiments, analyzing trials, coding changes, etc.).
    • Recommendation: Reintroduce all agents with updated BaseAgent API, restore routing logic, and ensure Director's next_node parsing can actually dispatch to those nodes and back.

    Stage 2 — Refinement Log:

    Date User AI Interpretation
    2025-11-29 "si, pero teniendo en cuenta que el codigo actual, mas simple, funciona, asi que debe seguir de alguna manera como esta hecho este. se entiende?" Entendido: el código simplificado actual (Director-only) funciona y es la base. Las extensiones deben ser aditivas, no destructivas. El nuevo routing multi-agente se construye sobre el patrón existente, sin romper lo que ya corre.

    F2 — Agent Tooling Gap

    • Problem: The only exported tools are store helpers; legacy agents relied on module triggers, system controls, code/test helpers, and issue management.
    • Location: Missing imports in modules/graph-orchestrator/src/rainbow/graph_orchestrator/tools/init.py#L3-L8 compared to the comprehensive tool suite in modules/graph-orchestrator/_OLD/tools.
    • Impact: Director cannot call compute modules, stop experiments, or manage execution plans; other agents would have no tools to do useful work.
    • Recommendation: Port and modernize agent_ops.py, module_triggers.py, system.py, and runtime helpers into the new namespace, ensuring they use the current common/database APIs and respect CONFIGURATION.

    Stage 2 — Refinement Log:

    Date User AI Interpretation
    2025-11-29 "si, dale" Aprobado: reconstruir todo el tool suite desde cero usando los estándares actuales.

    F3 — Event Bridge Not Implemented

    • Problem: GraphConfig still define event_bridge, but there is no runtime loop or CLI command to resume LangGraph from DB events.
    • Location: modules/graph-orchestrator/src/rainbow/graph_orchestrator/settings.py#L66-L121 references event_bridge, yet the new CLI lacks an event-bridge command and there is no equivalent to legacy modules/graph-orchestrator/_OLD/graph/event_bridge.py#L1-L170.
    • Impact: The orchestrator cannot react to DATA_FETCH_DONE/TRIAL_SCORED/etc., breaking the pipeline flows codified in PIPELINE-FLOWS and EVENTS.
    • Recommendation: Implement an async bridge compatible with the new GraphOrchestrator/LangGraph CLI setup, ensuring it polls events, resumes the correct thread, and emits EVENT_BRIDGE_ERROR on failure per documentation.

    Stage 2 — Refinement Log:

    Date User AI Interpretation

    F4 — Investigación: Forma Canónica de Exponer LangGraph (Agent Server)

    Según la documentación oficial de LangChain/LangSmith, la forma canónica de exponer un LangGraph application es mediante el Agent Server, que se ejecuta usando el LangGraph CLI.

    Arquitectura Oficial:

    ┌─────────────────────────────────────────────────────────────────────┐
│                        Agent Server                                  │
│  (LangGraph CLI: langgraph dev / langgraph up)                      │
├─────────────────────────────────────────────────────────────────────┤
│  Expone automáticamente:                                            │
│  • /assistants/* - CRUD de asistentes (graphs configurados)         │
│  • /threads/*    - Gestión de threads (conversaciones con estado)   │
│  • /runs/*       - Ejecutar graphs (stream, wait, background)       │
│  • /store/*      - Long-term memory store                           │
│  • /mcp          - Model Context Protocol endpoint                  │
│  • /a2a/*        - Agent-to-Agent protocol                          │
│  • /docs         - Swagger/OpenAPI automático                       │
│                                                                     │
│  Incluye:                                                           │
│  • Checkpointing automático (PostgreSQL o local)                    │
│  • Task queue para background runs                                  │
│  • Streaming (SSE) nativo                                           │
│  • HITL (Human-in-the-loop) via interrupts/breakpoints              │
└─────────────────────────────────────────────────────────────────────┘

    Comandos CLI oficiales:

    Comando Uso Descripción
    langgraph dev Desarrollo Server in-memory con hot-reload, sin Docker, puerto 2024 por defecto
    langgraph up Producción local Builds Docker image, corre con PostgreSQL + Redis, puerto 8124
    langgraph build CI/CD Solo construye la imagen Docker

    Configuración requerida — langgraph.json:

    {
  "dependencies": ["."],
  "graphs": {
    "orchestrator": "./src/rainbow/graph_orchestrator/workflow.py:compile_graph"
  },
  "env": ".env",
  "store": {
    "index": { "embed": "openai:text-embedding-3-small", "dims": 1536 },
    "ttl": { "default_ttl": 10080 }
  }
}

    SDK para consumir el Agent Server:

    from langgraph_sdk import get_client

client = get_client(url="http://localhost:2024")

# Crear thread
thread = await client.threads.create()

# Ejecutar graph con streaming
async for chunk in client.runs.stream(
    thread["thread_id"],
    "orchestrator",  # nombre del graph en langgraph.json
    input={"messages": [{"role": "human", "content": "Start experiment"}]},
    stream_mode="updates"
):
    print(chunk.data)

# O ejecutar y esperar resultado
result = await client.runs.wait(thread["thread_id"], "orchestrator", input={...})

    Conclusión para Rainbow Trader:

    1. NO necesitamos FastAPI custom — el Agent Server ya expone todas las APIs REST necesarias automáticamente.
    2. Nuestro CLI actual (langgraph dev/up) ya es correcto — solo falta:
    3. Agregar el comando event-bridge que no es parte de LangGraph sino nuestro propio polling loop.
    4. Comandos auxiliares (health, config) que son utilidades nuestras, no del SDK.
    5. El control panel debe usar langgraph-sdk para comunicarse con el orchestrator, no endpoints custom.
    6. El event bridge es nuestra extensión — LangGraph no tiene un mecanismo nativo para "despertar" el graph desde eventos en una tabla DB. Eso lo implementamos nosotros encima del Agent Server.

    • Agregar comando event-bridge que corre nuestro polling loop usando Database y langgraph-sdk.client.runs.wait/stream para resumir threads.

    • El control panel usa langgraph-sdk directo contra el Agent Server.
