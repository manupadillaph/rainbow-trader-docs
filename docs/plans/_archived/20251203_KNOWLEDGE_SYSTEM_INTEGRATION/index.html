        budget = self.max_tokens - self.reserve_for_response

        context = [SystemMessage(content=self.system_prompt)]

    class DecisionCache:
    """Cache decisions based on query similarity."""

    def __init__(self, ttl_seconds: int = 3600):
        self.cache: dict[str, CachedDecision] = {}
        self.ttl = ttl_seconds

    def get_cached_decision(self, query_embedding: list[float]) -> str | None:
        """Find similar decisions in cache."""
        for key, cached in self.cache.items():
            similarity = cosine_similarity(query_embedding, cached.embedding)
            if similarity > 0.95 and not cached.is_expired():
                return cached.decision
        return None

    2025-12-03 — Session 5: RAG & Agent Roles Clarification

    User requested clarification on RAG, agent roles, and where changes happen.

    What is RAG?

    RAG = Retrieval Augmented Generation

    It's the process of searching for relevant information BEFORE sending the prompt to the LLM, so the LLM has specific context.

    WITHOUT RAG:
  User: "How do experiments work?"
  LLM receives only the question → Generic or made-up response

WITH RAG:
  User: "How do experiments work?"
       ↓
  1. SEARCH in vector store for "experiments"
       ↓
  2. FIND relevant chunks:
     - "Experiments are defined in experiment_config.yaml..."
     - "ExperimentRunner executes trials..."
       ↓
  3. BUILD enriched prompt:
     "Context: [found chunks]
      Question: How do experiments work?"
       ↓
  4. LLM receives context + question → Precise response

    Who does what?

    Component Reads from KB Writes to KB Role
    BaseAgent ✅ (via ContextManager) ❌ All agents inherit RAG automatically
    doc-node ✅ ✅ Indexes code + docs Indexer + specialized retrieval
    research-node ✅ ✅ Indexes external research Web research + external knowledge
    Director ✅ (via tool) ❌ Routes, uses context
    Planner ✅ (via tool) ❌ Plans, uses context
    Analyst ✅ (via tool) ❌ Analyzes, uses context
    Coder ✅ (via tool) ❌ Codes, uses context
    ContextManager ✅ ❌ Prepares context before each invoke
    IncrementalIndexer ❌ ✅ Event-driven indexing

    1. BaseAgent.invoke() — All agents inherit this
    2. ContextManager — New class that prepares context

    ⚠️ IMPORTANTE: Pattern de BaseConfig para configs hijos

    Para que un módulo con BaseConfig tenga configs hijos (como memory, knowledge, graph_models) que: 1. Tengan valores dentro del config padre (archivo único YAML) 2. Puedan tener su propio archivo de config separado en el futuro (ej: config/graph-models.yaml)

    Se debe crear cada hijo como instancia de BaseConfig.

    Ejemplo para graph_orchestrator_config: ```python class MemoryConfig(BaseConfig): summarization: SummarizationConfig ...

    class KnowledgeConfig(BaseConfig): vector_store: VectorStoreConfig collections: CollectionsConfig ...

    class GraphOrchestratorConfig(BaseConfig): # Configs hijos como propiedades del tipo BaseConfig memory_config: MemoryConfig # Para memory.summarization knowledge_config: KnowledgeConfig # Para knowledge.vector_store graph_models_config: GraphModelsConfig # Para graph_models.llm_tiers ... ```

    Esto permite: - Valores inline en graph_orchestrator_config.yaml - Override con archivo separado: config/graph-models.yaml - Herencia de configuración automática
