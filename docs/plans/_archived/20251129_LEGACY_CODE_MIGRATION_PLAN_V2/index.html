    Legacy Code Migration Plan V2

    Status: 🔴 IN PROGRESS
    Started: 2025-11-29
    Completed: -
    Priority: HIGH

    📋 Description

    Migrate legacy code from _OLD/ folders into proper module structure, following the patterns established in the 4 reference modules (database, data-fetcher, graph-orchestrator, common). Each module must use the new config system with constants.py, have a clean CLI that only contains commands (no business logic), use a module class for business logic, and optionally have helpers.py for utilities.

    🧩 Rules

    Global rules (do not remove)

    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch. Reality lives on disk (plans, docs, code), not in cached summaries.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md. Never run git add from the repository root unless explicitly specified in this plan.
    • Respect project docs: All changes must comply with the relevant docs (configuration, code generation, database schema, markdown style, etc.).
    • Language: User responses and AI interpretations in the conversation log may be in Spanish, but the work plan document itself and all generated code/docs MUST be written in English.

    Project docs (must be re-opened when relevant)

    Document Description
    CODE-GENERATION.md Coding standards, quality gates
    CONFIGURATION.md Configuration system, env vars
    DATA-STORAGE.md Data directory structure, caching

    GIT-COMMITS.md Git workflow, commit conventions

    database modules/database/src/rainbow/database/ Full pattern with services/

    ⚠️ CRITICAL: When docs contradict reference module code, the CODE is the source of truth.

    Plan-specific rules

    1. Module class pattern: Each module has a main class (e.g., BacktestingEngine) that encapsulates all business logic. CLI creates instance and calls methods.

      • __main__.py — entry point (from .cli import app; app())
      • cli.py — Typer commands only, no business logic
      • constants.py — MODULE_NAME, DEFAULTS, ALLOWED_EVENT_TYPES, paths
      • {module_name}.py — main class with business logic
      • helpers.py — (optional) utility functions

    4. config/

      • defaults.yaml — YAML defaults for the module

    5. Config pattern (from reference modules):

    ```python # In cli.py callback: cfg = Config( env_prefix=MODULE_NAME, set_flags=set_vars, user_config_path=config, ) logger = Logger(cfg.config_obj) log = Logger.get(name, MODULE_NAME)

    # 3. Create database database = Database(config_obj)

    # 4. Create module instance module_instance = ModuleClass(database=database, config_obj=cfg.config_obj) ctx.obj = {"cfg": cfg, "database": database, "module": module_instance, ...} ```

    1. Constants pattern (from reference modules):

    python MODULE_NAME = "module-name" ALLOWED_EVENT_TYPES = get_module_events(MODULE_NAME) ALLOWED_EVENT_NAMES = [event.value for event in ALLOWED_EVENT_TYPES] PYDANTIC_MODEL_CONFIG: dict[str, Any] = {"arbitrary_types_allowed": True} DEFAULTS_YAML_PATH = Path(__file__).parent.parent.parent.parent / "config" / "defaults.yaml" DEFAULTS: dict[str, Any] = {"module_name": {...}}

    1. No defaults.py: The old defaults.py pattern is replaced by constants.py with DEFAULTS dict.

    2. No global variables: No CLI_CONFIG or similar globals. Use Typer context.

    3. Logger pattern: Use Logger.get(__name__, MODULE_NAME) after creating Logger instance.

    4. DRY - Use common module: NEVER duplicate code across modules. Use rainbow.common for all shared utilities (Config, Logger, Database, etc.). If something is needed by multiple modules, create it in common first.
