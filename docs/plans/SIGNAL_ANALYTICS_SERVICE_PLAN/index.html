    Signal Analytics Service Plan

    Status: 🟡 STAGE 1 — Initial Draft
    Started: 2025-01-20
    Completed: —
    Priority: HIGH

    📋 Description

    Implement a comprehensive SignalAnalyticsService for computing advanced Stage-3 signal metrics from completed backtest runs. The service will analyze prediction parquets, trades, and OHLCV data to produce detailed JSON reports stored alongside trial checkpoints. CLI commands will allow triggering analysis by backtest session ID. The implementation follows existing service patterns in rainbow.core.services.

    Finding 5: Metrics to Compute

    Based on previous discussion, the following metric groups are proposed:

    Group 1: Direction Accuracy Metrics

    • Accuracy per signal type (LONG, SHORT, NONE)
    • Precision/Recall per signal
    • Overall direction accuracy (predicted direction matches actual price movement)

    Group 2: Confidence Calibration Metrics

    • Confidence buckets analysis (0.0-0.5, 0.5-0.7, 0.7-0.9, 0.9-1.0)
    • Expected Calibration Error (ECE)
    • Calibration curve data for plotting

    Group 3: Confusion Matrix

    • 3x3 matrix for LONG/SHORT/NONE predictions vs actual outcomes
    • Per-class precision, recall, F1

    Group 4: Execution Metrics

    • Signal-to-trade conversion rate (how many signals became trades)
    • Average time from signal to entry
    • Slippage analysis (predicted confidence vs realized PnL)

    Group 5: Threshold Analysis

    • Optimal confidence threshold (maximize Sharpe or PnL)
    • Threshold sweep results (confidence threshold vs performance)

    Group 6: Temporal Metrics

    • Performance by hour of day
    • Performance by day of week
    • Rolling accuracy windows

    Group 7: Risk-Adjusted Metrics

    • Signal Sharpe Ratio (per signal type)
    • Maximum consecutive losses

    • Recovery time after losses

    • Questions:

    • Which metrics are must-have for V1 vs nice-to-have for V2?
    • Should temporal metrics use UTC or configurable timezone?
    • For threshold analysis, what's the sweep range? 0.5-1.0 in 0.05 increments?
    • User Decision: —
    • Action Required: —

    Finding 6: CLI Commands Required

    • Problem: Need CLI commands to trigger signal analytics independently.
    • Location:
    • Backtesting CLI: modules/backtesting-engine/src/rainbow/backtesting_engine/cli.py
    • Core CLI: modules/core/src/rainbow/core/cli.py
    • Impact: Users need to analyze existing backtest sessions without re-running backtest.
    • Recommendation: Add commands:
    • rainbow-backtesting-engine analyze --session-id <ID> — compute analytics for session
    • rainbow-core signal-analytics --session-id <ID> — same, available in core module
    • Questions:
    • Should both modules have the command, or just one?
    • Should we support batch analysis (--experiment-id to analyze all sessions)?
    • User Decision: —
    • Action Required: —

    Finding 7: Missing Predictions Handling

    • Problem: CLI must handle case where prediction parquet doesn't exist.
    • Location: Predictions are stored in ParquetCategory.PREDICTIONS linked by backtest_session_id
    • Impact: Some old backtests may not have predictions saved.
    • Recommendation:
    • CLI fails with clear error: "Prediction parquet not found for session {id}"
    • Include TODO note: "Re-run backtest with --save-predictions to generate"
    • Questions: None — behavior is specified.
    • User Decision: Fail with TODO note for regeneration.
    • Action Required: Implement error handling as specified.

    • Summary row in signal_analytics table with key metrics for querying
