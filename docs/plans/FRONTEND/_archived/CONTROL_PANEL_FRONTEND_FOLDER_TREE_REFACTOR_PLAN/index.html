    Control Panel Frontend Folder Tree and File Refactor Plan

    Status

    Draft for folder-tree and file-level approval only.

    This document freezes:

    • architecture laws
    • current folder inventory
    • current to target folder transformations
    • target folder tree
    • current file to target file mapping
    • final folder and file tree

    This document does not yet approve code execution.

    Locked naming decision

    The target architecture keeps:

    • training
    • trainings
    • trainingId

    There will be no study naming in the target architecture.

    Any current legacy route using [studyId] is treated as a current-state path that must become [trainingId] in the target tree.

    Scope

    • project area: modules/control-panel/frontend/src
    • focus: folders, naming, role separation, target structure, file-level planning
    • excluded for now:
      • imports rewrite
      • code cleanup inside files
      • behavior changes

    Architecture Laws

    Law 1. Same role, same pattern

    If two entities fulfill the same role, they must use the same folder pattern, naming pattern, and responsibility split.

    Law 2. Same role, same naming

    Equivalent hooks, pages, components, and lib files must use parallel names.

    Examples:

    • useBacktestCharts
    • useExperimentCharts
    • useTrainingCharts
    • useTrialCharts

    Law 3. Split by responsibility

    • app/ is routing only
    • components/ is rendering only
    • hooks/ is React state and behavior only
    • lib/ is pure logic only
    • types/ is cross-cutting contracts only
    • styles/ is global styles only

    Law 4. No page components mixed with UI primitives

    Page shells, reusable UI primitives, and Next dynamic wrappers are different roles and must be separated.

    Law 5. No pure logic inside components/

    Parsers, builders, constants, toggle state models, requirements extractors, and config definitions must live in lib/.

    Law 6. Shared means truly shared

    Only code reused by more than one entity may live in shared areas.

    Law 7. Chart infrastructure and chart domain logic are different layers

    Shared chart infrastructure goes to:

    • components/ui/charts/
    • hooks/charts/
    • lib/charts/

    Entity-specific chart logic goes to:

    • components/{entity}/detail/charts/
    • hooks/{entity}/
    • lib/{entity}/

    Law 8. Entity hooks must be explicit

    Hooks under hooks/{entity}/ should carry the entity prefix whenever the role is entity-specific.

    Examples:

    • useBacktestChartDataFetcher
    • useBacktestChartToggles
    • useExperimentChartDataFetcher

    Law 9. Remove redundant dashboard/ path level

    The whole application is already the dashboard. The extra prefix only adds noise.

    Law 10. Replace client-pages naming

    Those files are not page implementations. They are lazy client wrappers and should be treated as loaders.

    Law 11. Route trees must be complete

    If an entity supports list and detail routes with ids, the route tree must show every intermediate level and id explicitly.

    Law 12. Folder tree approval comes before file-level execution

    No file moves are approved until the folder tree itself is approved.

    Current Structural Problems

    1. Redundant components/dashboard/ prefix

    Entity UI is buried under components/dashboard/... even though the app itself is already the dashboard.

    2. Root-level component mix

    DashboardCard.tsx, Panel.tsx, IndexPage.tsx, and HitlPage.tsx live side by side even though they are different kinds of artifacts.

    3. Misleading client-pages/

    These files are lazy wrappers, not page implementations.

    4. Hooks are split inconsistently

    Ownership is fragmented between:

    • hooks/backtests/
    • hooks/experiments/
    • hooks/dashboard/
    • root hooks

    5. lib/backtests/helpers/ is overloaded

    That folder mixes backtest-only logic, chart-generic logic, and shared utility logic.

    6. Pure logic still exists inside components/

    Examples already identified:

    • components/dashboard/experiments/detail/charts/markers.ts
    • components/dashboard/experiments/detail/charts/requirements.ts
    • components/dashboard/experiments/detail/charts/types.ts
    • components/dashboard/experiments/detail/charts/plots/types.ts
    • components/dashboard/experiments/detail/config/configSections.ts

    7. Legacy route files still coexist with modern routes

    Examples:

    • app/experiments/detail/page.tsx
    • app/trainings/detail/page.tsx
    • app/trials/detail/page.tsx

    8. Legacy dynamic ids still use [studyId]

    The target naming must be [trainingId].

    Folder-Level Transformation Map

    app/

    Current

    • app/trainings/
    • app/experiments/[experimentId]/trainings/[studyId]/...
    • legacy detail/page.tsx routes
    • nested trial routes are incomplete at some levels

    Target

    • keep trainings/
    • rename dynamic ids to [trainingId]
    • delete legacy detail/page.tsx routes after migration
    • make nested trial routes explicit everywhere needed

    Folder moves

    • app/experiments/[experimentId]/trainings/[studyId] -> app/experiments/[experimentId]/trainings/[trainingId]
    • app/trainings/[studyId] -> app/trainings/[trainingId]
    • app/experiments/detail -> delete after migration
    • app/trainings/detail -> delete after migration
    • app/trials/detail -> delete after migration

    components/

    Current

    • entity components live under components/dashboard/
    • shared reusable UI also lives under components/dashboard/shared/
    • page shells and primitives are mixed at component root
    • client-pages/ hides wrapper files under a misleading name

    Target

    Split components/ into the following stable roles:

    • components/ui/
    • components/pages/
    • components/loaders/
    • components/backtests/
    • components/experiments/
    • components/trainings/
    • components/trials/

    Folder moves

    • components/dashboard/backtests -> components/backtests
    • components/dashboard/experiments -> components/experiments
    • components/dashboard/trainings -> components/trainings
    • components/dashboard/trials -> components/trials
    • components/charts -> components/ui/charts
    • components/client-pages -> components/loaders
    • root reusable primitives -> components/ui/...
    • root page shells -> components/pages/...
    • components/dashboard/shared -> split between components/ui/... and lib/shared/...

    hooks/

    Current

    • entity hooks exist for backtests and experiments
