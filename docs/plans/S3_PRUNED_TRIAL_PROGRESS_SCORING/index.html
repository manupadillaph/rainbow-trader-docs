    S3 Pruned Trial Scoring — Progress-Weighted Penalty Idea

    Idea surfaced 2026-03-24. Record for future implementation consideration.

    Problem

    Currently, in scoring.py → compute_stage3_value():

    if stage3_metrics.pruned:
    return 0.0

    Every pruned backtest returns 0.0 regardless of how far it got before being cut. A trial pruned at 30% progress and a trial pruned at 90% progress are identical to Optuna. This destroys gradient signal — TPE cannot learn that one hyperparameter set was "almost viable" vs "completely broken from the start".

    Current Pruning Mechanisms (for context)

    Defined in engine_loop_pruning.py → check_pruning_thresholds():

    1. min_trades_after_progress_pct — kills non-operating models early (no signals at all)
    2. max_loss_pct — kills after minimum progress if return is terrible
    3. max_drawdown_pct — always active, kills on catastrophic drawdown
    4. min_sharpe_ratio — kills if Sharpe is below threshold (requires min trades)
    5. min_win_rate — kills if win rate is below threshold (requires min trades)

    BacktestMetrics.pruned = True and pruning_reason are already set at pruning time. Partial metrics (return_pct, sharpe, win_rate, alpha) are also already captured.

    Idea: Incorporate Progress into Pruned Score

    Instead of flat 0.0, give pruned trials a negative score proportional to progress.

    Variant 1 — Simple (original proposal)

    score = -1.0 * (1.0 - progress)
    Progress Score
    30% -0.70
    50% -0.50
    70% -0.30
    90% -0.10

    Clean, simple, always negative (never competes with completed trials).

    Variant 2 — Progress + partial metrics (more signal)

    Use partial metrics that are already captured at pruning time:

    partial_composite = compute_composite(partial_metrics)   # same formula as completed
score = progress * partial_composite - 1.0

    Examples: - progress=0.30, partial_composite=0.20 → 0.06 - 1.0 = -0.94 (early failure) - progress=0.70, partial_composite=0.50 → 0.35 - 1.0 = -0.65 (went somewhere) - progress=0.95, partial_composite=0.70 → 0.67 - 1.0 = -0.34 (almost viable)

    Still always negative. Requires calling _compute_stage3_scoring_components() with partial metrics (already stored in BacktestMetrics).

    Key Properties of Any Valid Variant

    • Score must always be negative for pruned trials — never competes with completeds
    • Score must be differentiable across progress values — no flat zones
    • Must not require new data — BacktestMetrics already has progress and partial metrics

    Where to Implement

    Single change point:

    modules/model-trainer/src/rainbow/model_trainer/helpers/optuna/scoring.py
  → compute_stage3_value()
  → lines ~467 (the `if stage3_metrics.pruned: return 0.0` block)

    Progress is available via stage3_metrics.backtest_progress (verify field name in BacktestMetrics before implementing).

    Risks

    1. Score range change — objective values shift from [0.0, ~1.0] to [-1.0, ~1.0]. Config/thresholds that assume positive range may need review.
    2. Historical incompatibility — existing DB trials stored with 0.0 are not comparable to new trials after the change. New runs only.
    3. Stage weight interaction — if stage3_backtest weight is high, pruned trials now pull the overall objective negative, which may alter Optuna's exploration strategy in unexpected ways. Worth monitoring on first run after change.
    4. Not all progress is equal — a trial pruned by min_trades_after_progress (never operated) at 30% is arguably worse than one pruned by min_sharpe at 30% (operated but underperformed). Variant 1 ignores this distinction; Variant 2 captures it via partial metrics.
