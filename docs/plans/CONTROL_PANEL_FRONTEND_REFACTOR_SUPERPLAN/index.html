    Control Panel Frontend Refactor — Super Plan

    Decision Update — Study Naming

    On 2026-03-24, the canonical naming decision was revised. study / studies / studyId now supersede the earlier training / trainings / trainingId frontend-target wording in this plan. This supersession exists because the real project domain, database, model-trainer, and control-panel backend are already study-centric (OptunaStudy, study_id, study_name). Any instruction in this document that previously pushed the opposite rename direction must be read as obsolete and replaced by the canonical study naming decision.

    Status

    Approved for execution. All 283 source files audited. Target tree verified. U-marks defined.

    This document is the canonical architecture reference for the refactor. The only per-file execution source of truth is CONTROL_PANEL_FRONTEND_REFACTOR_SUPERPLAN_FINAL_TREE_MATRIX.md.

    Scope

    • Project area: modules/control-panel/frontend/src/
    • Total source files: 283
    • Nature of the refactor: This is NOT a cosmetic rename/move. It is a deep structural and behavioral refactor that reorganizes the entire frontend architecture while simultaneously creating new code, new patterns, and new shared infrastructure.

    What this refactor includes

    1. Folder restructure — file moves, renames, deletes per the File Action Matrix (283 rows)
    2. Splits creating NEW files — ~20+ new hook files extracted from components (useHitlPage, useExecutiveHomePage, useBacktestDashboard, useBacktestChartsPanel, useBacktestTradeBrowser, useTrainingData, useExperimentData, useExperimentChartsPanel, useChartLegendBox, useChartPanel, useChartToolbox, useExperimentFeaturesExplorer, useParquetPlotBuilder, useBacktestDetail, ExperimentsFilters, PredictionHealthPanel, etc.)
    3. New shared infrastructure — usePathContext (replaces useDashboardPathContext), useResolvedId (replaces useResolvedDashboardId), useQuery (replaces useDashboardQuery)
    4. New filter pattern — ExperimentsFilters extracted for symmetry (U4/U9). All entities get XxxFilters.tsx + useXxxFilters.ts
    5. Naming normalization — trainingId→studyId, backtestSessionId→backtestId across routes, params, hooks, API calls
    6. Data-fetch pattern unification (U1-U12) — new code: backtest query extends BaseListQuery (U1), backtest list returns PaginatedResponse (U2), new useBacktestDetail hook (U3), sort defaults as exported constants (U6), unified context resolution pathId ?? parsedFilter everywhere (U7), uniform list page hook return contract (U8), API parameter naming normalization (U10), path context returns trainingIdFromPath (U11), context flags unified in shared hook (U12)
    7. Route context vs Filter context system — new behavioral contract enforced across all list pages, filters, and launch forms
    8. Merges — hooks being absorbed (useTrainingLaunchForm→useTrainingLaunchPanel, useBacktestsLaunchPanel→useBacktestLaunchPanel, useTrialBacktestLaunchPanel absorbed)
    9. Import rewrites — done during/after moves

    What this refactor does NOT include

    • No new features or screens
    • Backend contract alignment may be required when public route or API naming still exposes the superseded training terminology
    • No changes to external API contracts

    Architecture Laws (L1–L12)

    L1. Same role, same pattern

    If two entities fulfill the same role, they must use the same folder pattern, naming pattern, and responsibility split.

    L2. Same role, same naming

    Equivalent hooks, pages, components, and lib files must use parallel names across entities.

    L3. Split by responsibility

    • app/ — routing only
    • components/ — rendering only
    • hooks/ — React state and behavior only
    • lib/ — pure logic only
    • types/ — cross-cutting contracts only
    • styles/ — global styles only

    L4. No page components mixed with UI primitives

    Page shells, reusable UI primitives, and Next dynamic wrappers are different roles and must be separated.

    L5. No pure logic inside components/

    Parsers, builders, constants, toggle state models, requirements extractors, and config definitions must live in lib/.

    L6. Shared means truly shared

    Only code reused by more than one entity may live in shared areas.

    L7. Chart infrastructure and chart domain logic are different layers

    Shared chart infrastructure → components/ui/charts/, hooks/charts/, lib/charts/. Entity-specific chart logic → components/{entity}/detail/charts/, hooks/{entity}/, lib/{entity}/.

    L8. Entity hooks must be explicit

    Hooks under hooks/{entity}/ should carry the entity prefix whenever the role is entity-specific.

    L9. Remove redundant dashboard/ path level

    The whole application is already the dashboard. The extra prefix only adds noise.

    L10. Replace client-pages naming

    Those files are lazy client wrappers, not page implementations. They become loaders/.

    L11. Route trees must be complete

    If an entity supports list and detail routes with ids, the route tree must show every intermediate level and id explicitly.

    L12. Folder tree approval comes before file-level execution

    No file moves are approved until the folder tree itself is approved. (DONE — tree is approved.)

    Naming Conventions

    Locked naming decisions

    • Target architecture uses study / studies / studyId. Prior training / trainings / trainingId wording is superseded.
    • Target architecture uses backtest / backtests / backtestId. No backtestSession naming in target.
    • Entity corridors are plural: experiments/, studies/, trials/, backtests/
    • Collection pages stay plural, detail pages stay singular
    • Launch capability: components/<entity>/launch/ with singular naming (BacktestLaunchPanel, not BacktestsLaunchPanel)
    • One component = one colocated hook (C1). No separate form hooks surviving alongside panel hooks.

    File colocation rules

    • Page owner: Page.tsx + usePage.ts + Page.module.scss — colocated under components/pages/{entity}/{list|detail}/
    • Feature owner: Panel.tsx + usePanel.ts + Panel.module.scss — colocated under components/{entity}/{capability}/
    • Filter owner: Filters.tsx + useFilters.ts + Filters.module.scss — colocated under components/{entity}/list/filters/

    Route Context vs Filter Context

    The distinction

    The frontend must distinguish between:

    • Route context — the user is inside a parent entity page hierarchy (e.g., /experiments/:experimentId/trainings)
    • Filter context — the user is on a global collection page narrowing results via query params (e.g., /trainings?experiment_id=10)

    Rules

    1. When route context exists: show contextual header, lock/hide parent filters, do not pretend context came from a free search filter
    2. Filter context decides search narrowing only, not page chrome
    3. Effective fetch precedence: pathId ?? parsedFilter (nullish coalescing). Path always wins.
    4. UI semantics must preserve the distinction even though data-fetch uses the same resolved ID

    Affected families

    All list page hooks, all filter components, and all launch forms that accept contextual parent IDs must explicitly preserve this split:

    • useBacktestsListPage.ts, useTrainingsListPage.ts, useTrialsListPage.ts
    • BacktestsFilters.tsx, TrainingsFilters.tsx, TrialsFilters.tsx, ExperimentsFilters.tsx
    • Launch forms that accept contextual parent IDs

    Unification Marks (U1–U12)

    These marks define the data-fetch pattern normalization that must be applied across all four entities during or after the structural refactor. They ensure symmetric, predictable behavior.

    U1 — Backtest query type extends BaseListQuery

    ListBacktestSessionsQuery must extend BaseListQuery with page, page_size, search like the other entities instead of using a bespoke shape.

    U2 — Backtest list API returns PaginatedResponse

    listBacktestSessions() must return PaginatedResponse<T> (with total, page, page_size) instead of a raw array, matching experiments/trainings/trials.

    U3 — Create useBacktestDetail hook

    Extract backtest session detail loading from useBacktestData into a thin useBacktestDetail.ts wrapper (wraps useDetailResource + getBacktestSession), symmetric with useExperimentDetail, useTrainingDetail, useTrialDetail. Already in target tree.

    U4 — Create ExperimentsFilters component

    Extract inline filter UI (search + status inputs) from ExperimentsListPage.tsx into ExperimentsFilters.tsx + useExperimentsFilters.ts + .module.scss for structural symmetry with backtests/trainings/trials. SYMMETRY IS MORE IMPORTANT THAN AVOIDING EXTRACTION OF SIMPLE COMPONENTS. Already in target tree.

    U5 — Rename studyId → trainingId everywhere

    All routes [studyId] → [trainingId]. All params, hooks, API calls, path context. No study naming survives in the target frontend.

    U6 — Sort defaults as exported constants per entity

    Each entity should export its default sort configuration as a named constant rather than hardcoding inline in hooks.

    U7 — Unified context resolution pattern

    All entities must use pathId ?? parsedFilter (nullish coalescing) for context resolution. Eliminate trainings' inconsistent ternary pattern that diverges from the other three entities.

    U8 — Uniform list page hook return contract

    All list page hooks must return the same shape: { page, pageSize, filters, sort, context, data, setters, goToPage }. Entity-specific fields are additive, not structural deviations.

    U9 — Every entity has XxxFilters + useXxxFilters

    Every filter component follows the XxxFilters.tsx + useXxxFilters.ts pattern. No inline filters in page components.

    U10 — API parameter naming normalization

    getTraining(studyId) → getTraining(trainingId). All API client functions use canonical entity names, not legacy Optuna names.

    U11 — Path context returns trainingIdFromPath

    useDashboardPathContext (→ usePathContext) returns trainingIdFromPath not studyIdFromPath. All consumers updated.

    U12 — Context flags unified in shared path context hook

    isXxxContextRoute flags are calculated in the shared path context hook (usePathContext), not scattered across individual list page hooks.

    Key Audit Findings (Actionable)

    These findings emerged during the systematic audit and carry user-annotated priority. All are resolved in the matrix.

    AF-1 — Experiments filters must be extracted (PRIORITY: HIGH)

    User annotation: "EXTRAER!!!! SIMETRIA Y ORDEN ES MAS IMPORTANTE!!!" The experiments list page has inline filters while backtests/trainings/trials all have dedicated filter components. This is resolved by U4 and the matrix already has the split target.

    AF-2 — Lib folder asymmetry: RESOLVED (justified, no action needed)

    lib/backtests/ has 8 files, lib/experiments/ has 7 files after splits. lib/trainings/ and lib/trials/ have only api.ts each. This asymmetry is justified and correct. Trainings and trials are genuinely simpler entities: no WebSocket, no chart type definitions, no marker transformation pipelines, no domain formatters. Their complexity lives in types/ (cross-cutting contracts) and hooks/ (behavior). Adding lib files for cosmetic symmetry would be wrong.

    AF-3 — Entity orchestrator hooks: DECIDED

    useBacktestData (684 lines) is a massive orchestrator combining session list, dashboard state, chart fetching, and control actions. The asymmetry in size vs other entity hooks is justified (backtests have real-time WebSocket, chart sync, trade navigation). The following orchestrator decisions are already final in the matrix — not pending verification:

    • Backtests: useBacktestData.ts → Split → useBacktestData.ts (stays as pure orchestrator) + hooks/backtests/useBacktestDetail.ts (thin detail wrapper, symmetric with other entities)
    • Trainings: TrainingDetailPage.tsx → Split → TrainingDetailPage.tsx + hooks/trainings/useTrainingData.ts (new orchestrator extracting inline training detail + trials sorting + experiment context logic)
    • Experiments: ExperimentChartsPage.tsx → Split → ExperimentChartsPage.tsx + hooks/experiments/useExperimentData.ts (new orchestrator extracting inline experiment detail + requirements + features + chart state logic)
    • Trials: No orchestrator extraction needed — useTrialDetailPage.ts already exists as the page hook

    AF-4 — Trainings detail: simpler but still gets orchestrator extraction

    Trainings detail page is simpler than other entities (no chart viewer, no prediction health). This is NOT an asymmetry bug — trainings are genuinely simpler entities. However, the matrix does mandate a split: TrainingDetailPage.tsx → Split+Move → components/pages/trainings/detail/TrainingDetailPage.tsx + hooks/trainings/useTrainingData.ts. The orchestrator extracts inline training detail loading, trials sorting, and experiment context resolution — consistent with React best practices (components = presentation, hooks = logic) and symmetric with the other entity orchestrator pattern (AF-3).

    AF-5 — HitlPage must be wrapped in AdminShell

    HitlPage currently does NOT use AdminShell — it renders standalone without the standard navigation bar. During refactor, HitlPage must be wrapped in AdminShell so HITL gets the same navigation chrome as all other pages. This is a behavioral change (new wrapper), not just a move.

    Execution Notes for Refactor Agent

    Preferred execution mode

    • Do not treat the current src/ tree as the place to refactor in place
    • Preserve the current tree as src_REFERENCE/
    • Assemble the final architecture in a new clean src/
    • Read legacy sources from src_REFERENCE/; create final targets under src/
    • Temporary intermediate breakage is acceptable during execution; the required guarantee is a correct final result that passes final verification

    Order of operations

    See CONTROL_PANEL_FRONTEND_REFACTOR_IMPLEMENTATION_GUIDE.md for the full phase-by-phase checklist with ⬜/✅ tracking per task.

    1. Delete dead files first — IndexPage, DashboardCard, orphaned stylesheets, legacy detail routes, barrel index
    2. Create new directories — all target folders that don't exist yet
    3. Move simple files (Keep/Move/Rename) — bulk of the 283 files, no content changes. Mark each matrix row ✅ as completed.
    4. Execute splits — files that produce multiple targets. ⚠️ CRITICAL: Splits create NEW files that do not appear as source rows in the matrix. When the Decision column says Split, read the Proposed Target column carefully — every +-separated path after the first is a NEW file you must create. Back up source files to _OLD/ first.
    5. Execute merges — hooks being absorbed. ⚠️ CRITICAL: Three merge groups exist where TWO source rows share ONE target. See the Merge Pairs Warning below — process the primary file first, then absorb the secondary into it. Never create the target file twice.
    6. Apply naming normalization — studyId→trainingId, backtestSessionId→backtestId. In Next.js, dynamic segment renaming means renaming the folder (e.g., [studyId]/ → [trainingId]/), not just the page.tsx file inside it. Use git mv for folder renames. Then search all .ts/.tsx files for the legacy identifiers — see studyId Rename Scope below.
    7. Apply U-marks — data-fetch pattern unification (U1-U12). These are behavioral code changes with concrete file-level tasks listed in the implementation guide. They are NOT simple moves — each requires modifying or creating code.
    8. Update all imports — After each phase (not only at the end), search for old import paths and update them. Strategy: for each moved file, run grep -r "<old-path>" src/ to find all consumers, then update each import to the new path. Verify with a final grep -r sweep for any remaining old paths. The matrix Proposed Target column is your reference for every new path.
    9. SCSS global style cleanup — extract shared chart-infrastructure rules from backtests.scss, extract shared panel rules from dashboard.scss (see concrete targets below)

    Agent Non-Negotiable Rules

    These rules are extracted from AGENTS.md and apply to every single action during this refactor. Violations of these rules have caused real damage in past sessions.

    R1. Read the full file before editing

    Before touching any file — for a Split, Merge, or content change — read the entire file first. Understand its structure, naming conventions, and all existing comments. Do not start editing without this step.

    R2. Comments are absolutely protected

    • NEVER delete existing comments, docstrings, or commented-out code during moves
    • NEVER rewrite, shorten, paraphrase, or relocate comments — copy them line-for-line to their equivalent location in the target
    • NEVER "clean up" or simplify code that is outside the scope of the current operation
    • Before finishing any Split or Merge, review the diff specifically for removed comment lines and restore any that were lost
    • If a refactor makes an existing comment look outdated or misplaced, STOP and ask before changing it

    R3. Scope discipline

    When moving a file: touch only that file and its direct import references. Do not refactor unrelated code "while you're at it".

    R4. No error suppressors

    Never use // @ts-ignore, // @ts-expect-error, /* eslint-disable */, or any suppression pragma. If TypeScript raises an error after a move, fix the root cause.

    R5. Create _OLD/ backup before large rewrites

    For every Split and Merge operation, keep a backup copy before making changes. Any consistent _OLD/ location is acceptable; there is no required standard path.

    R6. Explain before executing

    Before starting each phase, state what you are about to do, which files will be affected, and what the expected result is. Do not execute without this pre-announcement.

    R7. Never commit without explicit instruction

    Do not run git commit or git push at any point. Only run git for status/diff inspection.

    Merge Pairs Warning

    The matrix has three groups where TWO source rows merge into ONE target file. If you process matrix rows sequentially without knowing this, you will either create the target twice or overwrite the first merge result with the second.

    How to process a merge pair:

    1. Process the PRIMARY row first (the one whose logic is the main body of the target)
    2. Read the SECONDARY row's source file and absorb its logic into the target already created in step 1
    3. Delete the secondary source file
    4. Mark both matrix rows ✅
    Group Primary row (Move+Rename) Secondary row (Move+Merge) Target
    Backtests launch useBacktestsLaunchForm.ts → Rename to useBacktestLaunchPanel.ts useBacktestsLaunchPanel.ts → Absorbed into target components/backtests/launch/useBacktestLaunchPanel.ts
    Trainings launch useTrainingLaunchPanel.ts → Move+Merge (absorbs form) useTrainingLaunchForm.ts → Merge source components/trainings/launch/useTrainingLaunchPanel.ts
    Trials launch useBacktestLaunchForm.ts → Rename to useTrialBacktestLaunchPanel.ts useTrialBacktestLaunchPanel.ts → Absorbed into target components/trials/launch/useTrialBacktestLaunchPanel.ts

    SCSS global style split targets (concrete)

    Both backtests.scss and dashboard.scss use body.page-* class scoping (global page-theme styles, not CSS Modules). They CANNOT be fully converted to component-scoped .module.scss files because they rely on body-class matching. The concrete split strategy:

    styles/backtests.scss (1371 lines)

    • Stays as: styles/backtests.scss — all body.page-backtests-scoped page chrome, layout, panel primitives, session cards, chart controls, trade browser/details, metrics sections
    • Extract to: styles/charts-shared.scss — rules that match BOTH body.page-backtests AND body.page-experiments: legend items/markers, overview-range overlays/shades/handles/lines, overview scrollbar, charts-unified sections. These are shared chart infrastructure, not backtests-owned.
    • During execution: any rules that duplicate what already exists in component .module.scss files are removed from the global file (dedup, not copy)

    styles/dashboard.scss (660 lines)

    • Stays as: styles/dashboard.scss — all body.page-dashboard-scoped admin shell chrome, dashboard grid, stat cards, tables, forms, filters, messages
    • Extract to: styles/experiments-panels.scss — rules that match body.page-experiments: panel/panel-header/panel-body/panel-toggle/panel-action/panel-collapsed, resolution selector/buttons. These are experiments-owned panel styling, not dashboard-owned.
    • During execution: same dedup rule as backtests.scss
