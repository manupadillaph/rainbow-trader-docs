    Brain Foundation Audit — Events, CLIs, and Process Control

    Status: ✅ STAGE 1 — Audit Completed Started: 2026-03-19 Completed: 2026-03-19 Priority: HIGH

    📋 Description

    Full audit of all 13 modules to establish the real baseline before building the AI supervisor (brain). Three audit dimensions per module: (1) event emissions — are they complete, consistent, Pydantic-validated? (2) CLI commands — are implementations complete and production-ready, not just stubs? (3) process control — can the module be started, stopped, paused, resumed programmatically? The output is a verified inventory of what works, what's missing, and what's broken. No code changes in this plan — findings here feed a separate execution plan.

    Predecessor research: docs/researchs/BRAIN_ARCHITECTURE_ANALYSIS.md (Sections 3-5, 8.5)

    🧩 Rules

    Global rules (do not remove)

    • README template: All module READMEs must follow docs/README_MODULE_TEMPLATE.md.
    • Read from disk before each iteration: Before starting work, re-open this plan, the relevant docs, and the code you will touch. Reality lives on disk (plans, docs, code), not in cached summaries.
    • Git workflow: Use the MCP GitHub tools and follow docs/GIT-COMMITS.md. Never run git add from the repository root unless explicitly specified in this plan.
    • Respect project docs: All changes must comply with the relevant docs (configuration, code generation, database schema, markdown style, etc.).
    • Language: User responses and AI interpretations in the conversation log may be in Spanish, but the work plan document itself and all generated code/docs MUST be written in English.

    Plan-specific rules

    • Audit only — no code changes. This plan produces findings. Code changes go in a separate execution plan derived from these findings.
    • Per-module granularity. Each module gets its own finding with a standardized audit checklist.
    • Evidence required. Every claim must cite file path and line numbers. No "probably" or "should be."
    • Three dimensions per module:
    • Events: List every emit_event call, verify payload is Pydantic-typed (not raw dict), check for missing STARTED/DONE pairs, verify emit parameter defaults.
    • CLI: Read every Click command implementation end-to-end. Document: what it does, whether it's complete, whether it handles errors, whether it has proper exit codes.
    • Process control: Can it be started/stopped/paused/resumed programmatically (not just via CLI)? Document the Python API surface.
    • Cross-cutting audits. After per-module audits, two cross-cutting findings: (1) EventConsumer/Event Bridge status, (2) control-panel backend consumption.
    • Reference doc: docs/EVENTS.md is the canonical event reference. Flag any discrepancies between that doc and actual code.
