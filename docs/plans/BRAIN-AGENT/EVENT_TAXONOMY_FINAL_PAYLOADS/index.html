    Event Taxonomy — Final Payloads

    Status: DRAFT v3 — Process-grounded rewrite Generated: 2026-03-24 Source of truth for event names: EVENT_TAXONOMY_SIMPLE_VIEW.md Source of truth for payloads: This document (once approved) Grounded in: Actual code business logic of every module (function signatures, configs, data flows)

    Format Convention

    • Fields marked with ? are optional. All others are required.
    • IDs always go first in every payload, even when optional for that event.
    • IDs have the same optionality across the entire family — if experiment_id is optional in one event, it's optional in all.
    • Every payload inherits BaseEventPayload (provides source?, note? — not listed below).
    • Rules applied to one event in a family apply to all events in that family.

    Payload Design Rules

    1. Identifiers first — experiment_id, then study_id / trial_id / backtest_session_id / etc.
    2. Required = the emitter ALWAYS has this value at emission time. Not "should have" — HAS.
    3. Optional = the emitter might not have this value depending on the invocation path.
    4. No duplicate sources of truth — if it's in the experiment config, don't repeat it. Only the ID + what changed.
    5. REQUESTED = what the caller passes — resolved parameters known at request time.
    6. STARTED = REQUESTED + newly created IDs — session IDs, validated/aligned ranges.
    7. PROGRESS = IDs + position in the work — where am I, how far along.
    8. DONE = IDs + outcome — what happened, key results. Heavy data goes to DB.
    9. Consistent within a family — same fields, same optionality, same naming.

    A. Data Fetch

    Process: Downloads historical OHLCV candles for ONE symbol+timeframe combination. Can be invoked from experiment (orchestrator) or standalone (CLI). Provider is ALWAYS resolved before any work starts — either from ExperimentConfig.data_provider or from active DataProviderConfig.

    DATA_FETCH_REQUESTED:  experiment_id?, symbol, timeframe, provider, range_start, range_end
DATA_FETCH_STARTED:    experiment_id?, symbol, timeframe, provider, range_start, range_end
DATA_FETCH_PROGRESS:   experiment_id?, symbol, timeframe, provider, fetched_rows, total_rows, progress_percent
DATA_FETCH_DONE:       experiment_id?, symbol, timeframe, provider, rows_fetched, parquet_id?

    Why each field:

    • experiment_id? — Optional across the entire family. When fetch(experiment_id=123) is called, it's present. When fetch_symbol(symbol="BTCUSDT", ...) is called standalone or from CLI, there's no experiment.
    • symbol — Always singular. Code operates on base_symbol (one symbol per fetch call). Never plural.
    • timeframe — Always known at request time. You can't request a fetch without knowing what timeframe you need.
    • provider — Always resolved before any event fires. Either from ExperimentConfig.data_provider (with overrides applied via apply_experiment_data_provider()) or from the active DataProviderConfig. Never unknown.
    • range_start, range_end — Explicit dates. From ExperimentConfig.range.start/end or CLI --start/--end. STARTED may have adjusted/aligned values (last_tick_start logic, provider earliest clipping).
    • parquet_id? — Only present when save_parquet=True and DB registration succeeds.

    Changes from v2: symbols (plural) → symbol (singular, matching code reality). Added timeframe to REQUESTED (can't request without it). provider required everywhere (always resolved). time_range → explicit range_start/range_end. Removed stream_mode (streaming is a separate code path, not a mode flag).

    B. Data Processing

    Process: Computes feature DAG for an experiment — transforms raw OHLCV through pipelines (CLEAN_PRICE, RELATIVE_PRICE) and indicators (RSI, EMA, etc.). Each ExtendedFeature node is processed independently. The orchestrator calls this once per experiment; internally it iterates over all feature nodes in dependency order.

    DATA_PROCESSING_REQUESTED:  experiment_id
DATA_PROCESSING_STARTED:    experiment_id, features_total
DATA_PROCESSING_PROGRESS:   experiment_id, features_processed, features_total, progress_percent, current_feature?
DATA_PROCESSING_DONE:       experiment_id, features_processed, features_total, features_failed, duration_seconds

    Why each field:

    • experiment_id — Always required. Data processing always runs for an experiment (features come from ExperimentRequirements.build_experiment_requirements()).
    • features_total — Count of ExtendedFeature nodes in the dependency tree. Known at start after build_experiment_requirements().
    • features_processed — How many nodes completed successfully so far.
    • features_failed — How many nodes failed (partial processing is possible).
    • current_feature? — Human-readable name of the feature being processed now (e.g., "RSI(14) on BTCUSDT:5m"). Optional because the last PROGRESS before DONE may not have a "current" one.
    • duration_seconds — Total wall time for all features.

    Changes from v2: Removed per-feature detail from events (feature_name, symbol, timeframe, feature_hash, pipeline_hash, parquet_path, parquet_id, stage_name, status, rows). That's per-node logging, not event-level. Events track the high-level job: "how many of N features are done?" Individual feature results and validation go to structured logs and DB.

    C. Labels

    Process: Generates LONG/SHORT/NONE labels for each candle based on future price movement over a horizon window. Always runs per experiment. Uses OHLCV + ATR (if ATR-based thresholds). One labeling config per experiment, three label classes, no "label families".

    LABELS_REQUESTED:  experiment_id
LABELS_STARTED:    experiment_id, label_config_hash, threshold_type, horizon
LABELS_PROGRESS:   experiment_id, processed_rows, total_rows, progress_percent
LABELS_DONE:       experiment_id, label_config_hash, count_long, count_short, count_none, count_total

    Why each field:

    • experiment_id — Always required. generate_labels_for_experiment(experiment_id) is the only entry point.
    • label_config_hash — Computed from LabelingConfig (threshold_type + horizon + thresholds + drawdown limits). Identifies which config version produced these labels. Known after config loads (STARTED), and carried through to DONE for traceability.
    • threshold_type — "percentage" / "atr_based" / "atr_based_absolute". The single most important parameter that changes labeling behavior. Worth surfacing in STARTED so you immediately know what kind of run this is.
    • horizon — Look-ahead candles (e.g., 24). Defines how many future candles the label evaluates. Critical parameter.
    • processed_rows / total_rows — Labeling is row-by-row (vectorized, but progress is per-row). total_rows = len(ohlcv) - horizon (last horizon rows dropped since no future data).
    • count_long, count_short, count_none, count_total — The result. Code today emits these as nested dict — flatten.

    Changes from v2: Removed label_config_hash from REQUESTED (caller just passes experiment_id — hash is computed by the service). Added threshold_type and horizon to STARTED (key behavior parameters). Removed label_family? (doesn't exist — no label families). Removed range_start?/range_end? (range comes from experiment config, no need to repeat). Removed warnings?/errors? from DONE (validation details go to logs/DB). Made label_config_hash required in both STARTED and DONE (always computable).

    D. FAT Windows

    Process: Builds flattened feature windows for each sampling timestamp. One row per potential trade entry, containing window_size historical values for every feature. Input: sampling parquet (timestamps + labels) + all feature parquets. Output: one large FAT parquet.

    FAT_WINDOWS_REQUESTED:  experiment_id
FAT_WINDOWS_STARTED:    experiment_id, window_size, features_count, sampling_rows
FAT_WINDOWS_PROGRESS:   experiment_id, windows_built, total_windows, progress_percent
FAT_WINDOWS_DONE:       experiment_id, fat_hash, n_windows, n_columns, fat_parquet_id?

    Why each field:

    • experiment_id — Always required. generate_fat_windows_for_experiment(experiment_id) is the only entry.
    • window_size — From experiment config. Defines how many historical candles each window contains (e.g., 30). Determines the shape of the output.
    • features_count — Number of features being expanded into windows. Known after build_experiment_requirements().
    • sampling_rows — Total rows in the sampling parquet (= total windows to build). Known at start.
    • windows_built / total_windows — Progress through the sampling timestamps. total_windows = sampling_rows.
    • fat_hash — Deterministic hash of window_size + window_features + output_columns + feature hashes. Identifies this FAT configuration for caching/change detection.
    • n_windows — Final count of windows built (may differ from sampling_rows if some timestamps are invalid).
    • n_columns — Total columns in the FAT parquet (e.g., features_count × window_size + label columns).
    • fat_parquet_id? — DB parquet record ID. Only present when save_parquet=True.

    Changes from v2: Added window_size, features_count, sampling_rows to STARTED (the parameters that define the FAT build). Renamed feature_count → features_count (consistent naming). Added n_columns to DONE (critical info — how wide is the FAT?). Moved fat_hash to DONE (it's a result identifier). Removed range_start?/range_end? (from experiment config). Removed warnings?/errors? (logs/DB).

    E. SMALL Windows

    Process: Creates a trial-specific subset of the FAT parquet. Selects specific feature columns (from Optuna suggestion) and optionally reduces rows (via data_fraction). Can be reused across trials if the cache_key matches. Called per-trial from the training loop.

    SMALL_WINDOWS_REQUESTED:  experiment_id, trial_id
SMALL_WINDOWS_STARTED:    experiment_id, trial_id, cache_key, feature_subset_size, data_fraction, reused
SMALL_WINDOWS_PROGRESS:   experiment_id, trial_id, cache_key, columns_processed, total_columns, progress_percent
SMALL_WINDOWS_DONE:       experiment_id, trial_id, cache_key, reused, small_parquet_id?

    Why each field:

    • experiment_id, trial_id — Always required. prepare_small_window(experiment_id, trial_id, ...) takes both.
    • cache_key — Hash of fat_parquet_id + window_size + feature_columns_with_weights. If a SMALL with this key exists with enough rows → cache hit. Known after parameter resolution.
    • feature_subset_size — How many feature columns were selected for this trial (e.g., 15 out of 60). From Optuna's feature_columns_with_weights suggestion.
    • data_fraction — From PhaseConfig.data_fraction (e.g., 0.3 = use 30% of rows). Defines how much of the FAT to keep.
    • reused — True if cache hit (existing SMALL reused), False if built from scratch. Known at STARTED because the cache check happens before any work.
    • columns_processed / total_columns — If building from scratch, tracks column selection progress. If reused=True, PROGRESS may not fire at all (skip straight to DONE).
    • small_parquet_id? — DB record ID. Only if registered in DB.

    Changes from v2: Added data_fraction to STARTED (the key parameter — what fraction of data this trial sees). Made feature_subset_size required in STARTED (always known from Optuna suggestion). Made reused required in both STARTED and DONE (always known — cache check happens before STARTED fires).

    F. Training (container level)

    Process: Top-level orchestration of multi-phase progressive training. Resolves all phases from config, creates the OptunaStudy DB record, then runs phases sequentially. Knows total_phases and total_trials_planned from resolve_study_phase_totals().

    TRAINING_REQUESTED:  experiment_id
TRAINING_STARTED:    experiment_id, total_phases, total_trials_planned, resume
TRAINING_PROGRESS:   experiment_id, phases_completed, total_phases, trials_completed, total_trials_planned, progress_percent
TRAINING_DONE:       experiment_id, phases_completed, total_phases, trials_completed, total_trials_planned, status, duration_seconds, best_trial_id?, best_score?

    Why each field:

    • experiment_id — Always required. Training always runs for an experiment.
    • total_phases — From resolve_study_phase_totals().phases_total. Fixed at creation. Never changes.
    • total_trials_planned — Sum of n_trials across all phases. From resolve_study_phase_totals().trials_planned. Known at start.
    • resume — True if --resume flag was passed (continuing from checkpoint). Always known (defaults to False).
    • phases_completed — How many phases finished so far.
    • trials_completed — Total trials completed across all phases so far.
    • status — Final outcome: "completed" / "failed" / "cancelled".
    • best_trial_id?, best_score? — Optional because training may fail before any trial completes.

    Changes from v2: Renamed trials_completed_overall → trials_completed and total_trials_overall → total_trials_planned (cleaner, matched to resolve_study_phase_totals() output). Made resume required (always known, defaults False). Removed starting_phase? (if resuming, the starting phase is internal detail). Added total_trials_planned to STARTED (known from phase resolution). Added trials_completed and total_trials_planned to DONE (final tally). Replaced completed_phases with phases_completed (consistent verb form).

    G. Phase (nested inside Training)

    Process: One Optuna optimization run within a training session. Creates/resumes an Optuna study with n_trials trials. Uses data_fraction to control how much training data trials see (progressive training). Can warm-start from previous phase's best trials.

    PHASE_STARTED:   experiment_id, study_id, phase_number, total_phases, planned_trials, data_fraction
PHASE_PROGRESS:  experiment_id, study_id, phase_number, total_phases, trials_completed, planned_trials, progress_percent,
                 current_trial_id?, current_trial_number?, last_trial_outcome?, last_trial_score?
PHASE_DONE:      experiment_id, study_id, phase_number, total_phases, n_completed, n_pruned, n_failed, planned_trials,
                 best_trial_id?, best_score?, duration_seconds

    Why each field:

    • experiment_id — Always required.
    • study_id — OptunaStudy DB primary key. Created before the phase starts. Uniquely identifies this optimization run.
    • phase_number — Which phase in the sequence (1, 2, 3...). From PhaseConfig.phase_number.
    • total_phases — Total phases in this training run. Always known (from training-level resolution).
    • planned_trials — How many trials this phase will run (PhaseConfig.n_trials). Fixed at phase start.
    • data_fraction — The defining parameter of progressive training (e.g., 0.3 = 30% of data). From PhaseConfig.data_fraction. This is what makes phase 1 different from phase 2.
    • trials_completed — Trials finished in this phase so far (completed + pruned + failed).
    • n_completed, n_pruned, n_failed — Breakdown by outcome. From OptunaStudy counters.
    • current_trial_id?, current_trial_number? — Optional because between trials there's no "current" one.
    • last_trial_outcome? — "completed" / "pruned" / "failed". Optional because first PROGRESS before any trial finishes has no last outcome.
    • last_trial_score? — Score of the last finished trial. Optional (pruned/failed trials may not have scores).
    • best_trial_id?, best_score? — Optional because all trials may fail.
    • duration_seconds — Required on DONE. Phase always has a measurable duration.

    Changes from v2: Added data_fraction to STARTED (THE key progressive training parameter — you can't understand a phase without knowing its data fraction). Added planned_trials to DONE (for final tally context). Made duration_seconds required on DONE (always measurable). Removed resumed_from_checkpoint? (internal detail, warm-start info is not needed in events).

    H. Trial (nested inside Phase)

    Process: One Optuna trial. Samples hyperparameters (or inherits from parent trial if warm-starting), builds SMALL windows, trains a model for total_epochs epochs, evaluates through 3 stages (train → validation → mini-backtest), reports objective value. Each epoch produces EpochSnapshotMetrics.

    TRIAL_STARTED:   experiment_id, study_id, trial_id, trial_number, phase_number
TRIAL_PROGRESS:  experiment_id, study_id, trial_id, trial_number, phase_number,
                 epoch, total_epochs, progress_percent,
                 train_loss, val_loss, f1_macro, epoch_objective_value,
                 val_accuracy?, recall?, precision?, entry_count?, entry_f1?, best_score_so_far?
TRIAL_DONE:      experiment_id, study_id, trial_id, trial_number, phase_number,
                 outcome, duration_seconds,
                 score?, reason?, step_pruned_at?, error_message?, error_type?

    Why each field:

    • experiment_id, study_id, trial_id — The full ID chain. Always required. Trial always exists inside a study inside an experiment.
    • trial_number — Optuna's trial sequence number (0-indexed per study). If phase 1 has 10 trials and phase 2 has 7, phase 2 trials are numbered 10–16.
    • phase_number — Which phase this trial belongs to. Always known.
    • epoch / total_epochs — Current epoch (1-indexed) and total. From training loop.
    • train_loss, val_loss — Always computed every epoch. From EpochSnapshotMetrics.
    • f1_macro — Always computed. Primary classification metric.
    • epoch_objective_value — The objective function value for this epoch (what Optuna optimizes). Always computed.
    • val_accuracy? — Optional because some model configurations may not compute it.
    • recall?, precision? — Optional for same reason.
    • entry_count?, entry_f1? — Entry-specific metrics. Optional because they depend on prediction distribution (if model predicts all NONE, these are zero/undefined).
    • best_score_so_far? — Best epoch objective seen so far in this trial. Optional (may not be tracked in all configurations).
    • outcome — Literal["completed", "pruned", "failed", "cancelled"]. Always known when trial ends.
    • score? — Final objective value. Optional because pruned/failed trials may not produce one.
    • reason? — Why the trial ended (e.g., pruning reason, cancellation reason). Optional for completed trials.
    • step_pruned_at? — Epoch at which pruner triggered. Only for pruned trials.
    • error_message?, error_type? — Only for failed trials.

    Changes from v2: Renamed loss → train_loss (disambiguating — there's also val_loss). Removed entry_rate_over_total_predictions? (too granular for events — belongs in DB/logs). Removed is_recoverable? from DONE (internal error handling detail, not useful for event consumers).

    I. Backtest

    Process: Simulates trading using a trained model over a historical range. Three types: MINI (during trial, last N rows from SMALL), COMPLETE (after phase, full range), MANUAL (user CLI). Creates a BacktestSession DB record. Runs tick-by-tick: model predicts, bot decides, orders execute with slippage/commission.

    BACKTEST_REQUESTED:  experiment_id, study_id?, trial_id, backtest_type
BACKTEST_STARTED:    experiment_id, study_id?, trial_id, backtest_session_id, backtest_type,
                     range_start, range_end, initial_equity
BACKTEST_PROGRESS:   experiment_id, study_id?, trial_id, backtest_session_id, backtest_type,
                     processed_timestamps, total_timestamps, progress_percent,
                     equity?, open_positions?, closed_trades_count?
BACKTEST_DONE:       experiment_id, study_id?, trial_id, backtest_session_id, backtest_type,
                     outcome, duration_seconds,
                     total_pnl, return_pct, sharpe_ratio, trades_count, win_rate,
                     winning_trades?, losing_trades?, avg_win?, avg_loss?,
                     equity?, unrealized_pnl?, equity_mark_to_market?,
                     benchmark_return?, alpha?, prediction_health?

    Why each field:

    • experiment_id — Always required. Backtest always runs for an experiment.
    • study_id? — Optional. MINI/COMPLETE backtests inside training have it. MANUAL backtests from CLI may not.
    • trial_id — Always required. BacktestSession always has trial_id (you're backtesting a specific trial's model).
    • backtest_session_id — DB primary key. Created at start. Required from STARTED onwards (multiple sessions per trial possible).
    • backtest_type — "MINI" / "COMPLETE" / "MANUAL". Required across ALL events in the family. Always known at request time. You always know WHAT KIND of backtest you're requesting.
    • range_start, range_end — The historical range being simulated. Known at start.
    • initial_equity — Starting equity (default 10,000). From config.
    • outcome — "completed" / "failed". Required on DONE (backtest always has a final status).
    • total_pnl, return_pct, sharpe_ratio, trades_count, win_rate — The 5 key metrics. Required because even a failed backtest produces partial metrics (or zeroes).
    • Optional metrics — The full BacktestMetrics detail. Present when backtest completes normally.
    • prediction_health? — PredictionHealthMetrics (model prediction quality analysis). Optional, present when enabled.

    Changes from v2: Added backtest_type to ALL events (always known, critical for understanding what's happening). Added study_id? to ALL events (optional — manual backtests don't have it). Added outcome to DONE (replaces the weird pruned: bool — consistent with Trial pattern). Removed checkpoint_selection_mode?/checkpoint_selection_epoch? from STARTED (internal model-loading detail). Removed pruned/pruning_reason? (subsumed by outcome). Removed processed_timestamps?/total_timestamps? from DONE (already in PROGRESS, DONE tells outcome not position).

    J. Live Trading

    Process: Runs a live trading session from an ExecutionPlan. Each plan defines one symbol+timeframe+model combination. The runner spawns a LiveInstance, creates a LiveSession DB record, subscribes to a candle stream, and runs the bot loop: receive candle → model predicts → bot decides → execute orders.

    ExecutionPlan always carries: experiment_id, trial_id, mode, target_symbol, target_timeframe.

    LIVE_TRADING_REQUESTED:          execution_plan_id, execution_plan_uuid, experiment_id, trial_id,
                                 mode, target_symbol, target_timeframe
LIVE_TRADING_STARTED:            execution_plan_id, execution_plan_uuid, experiment_id, trial_id, live_session_id,
                                 mode, target_symbol, target_timeframe
LIVE_TRADING_PROGRESS:           execution_plan_id, execution_plan_uuid, experiment_id, trial_id, live_session_id,
                                 target_symbol, target_timeframe,
                                 equity, realized_pnl_total, unrealized_pnl_total,
                                 open_positions_count, closed_trades_count, executed_orders_count,
                                 decision_id?, candle_id?
LIVE_TRADING_DONE:               execution_plan_id, execution_plan_uuid, experiment_id, trial_id, live_session_id,
                                 reason, duration_seconds
LIVE_TRADING_ANOMALY_DETECTED:   execution_plan_id, execution_plan_uuid, experiment_id, trial_id, live_session_id,
                                 target_symbol, target_timeframe,
                                 anomaly_type, details, severity?, recommended_action?

    Why each field:

    • execution_plan_id — DB primary key of ExecutionPlan. Primary routing ID for the live family.
    • execution_plan_uuid — Stable UUID that survives session restarts. Required across ALL events (always available from the ExecutionPlan record).
    • experiment_id, trial_id — From ExecutionPlanConfig. Always present (live trading always runs a specific experiment's trial). Required across ALL events.
    • live_session_id — LiveSession DB primary key. Created at spawn time. Required from STARTED onwards. One ExecutionPlan can have many sessions (restarts, retries).
    • mode — "LIVE_SIM" / "LIVE_REAL". Always known. Changed from optional in some events to required in REQUESTED and STARTED (you always know the mode).
    • target_symbol, target_timeframe — The trading pair and candle timeframe. From ExecutionPlanConfig. Required where relevant (REQUESTED, STARTED, PROGRESS, ANOMALY). Consistent naming throughout (not symbol/timeframe in some and target_symbol/target_timeframe in others).
    • equity, realized_pnl_total, unrealized_pnl_total — Live P&L state. From InstanceManager tick loop.
    • open_positions_count, closed_trades_count, executed_orders_count — Trading activity counters.
    • decision_id? — Globally unique ID for one bot decision on one candle (live:{plan_id}:{session_id}:{candle_ts_ms}). Optional because PROGRESS may fire before any decision is made.
    • candle_id? — Source data identifier ({symbol}:{timeframe}:{ts_ms}). Optional for same reason.
    • reason — Why the session ended (e.g., "manual_stop", "stream_runtime_failure", "data_gap_auto_pause"). Always present on DONE.
    • duration_seconds — Required on DONE (session always has measurable duration).
    • anomaly_type — "data_gap_auto_pause" / "stream_runtime_failure". From code.
    • details — Human-readable anomaly description.
    • severity?, recommended_action? — Not yet implemented in code. Planned for future.

    Changes from v2: Added experiment_id and trial_id to ALL events (always available from ExecutionPlan — they were inexplicably missing). Made execution_plan_uuid required across ALL events (always available). Made duration_seconds required on DONE. Removed scope? from DONE (not implemented, undefined meaning). Consistent target_symbol/target_timeframe naming everywhere (was symbol/timeframe in PROGRESS and ANOMALY). Removed requested_by? from REQUESTED (not in the current ExecutionPlanConfig contract).

    K. HITL (Human-in-the-Loop)

    Process: Governance mechanism for sensitive actions. Partially implemented — payloads and DB audit table defined, but the orchestrator interruption mechanism is not wired yet. Flow: system requests approval → control panel displays → human decides → system proceeds or stops.

    HITL_REQUEST:   experiment_id?, trial_id?, action, description, context, allowed_responses
HITL_DECISION:  experiment_id?, trial_id?, action, decision, approver, reason?
HITL_APPROVED:  experiment_id?, trial_id?, action, approver, reason?
HITL_REJECTED:  experiment_id?, trial_id?, action, approver, reason?

    Why each field:

    • experiment_id?, trial_id? — Optional. HITL gates may apply to non-experiment actions (e.g., budget extensions, master plan updates). When triggered from within a trial, both are present. IDs always go first.
    • action — What's being gated (e.g., "update_master_plan", "experiment_create", "budget_extension"). From HITLRequestPayload.action. Always present — it's the primary key of the governance request.
    • description — Human-readable explanation of what the system wants to do. Always present on REQUEST.
    • context — Dict of contextual data (experiment_id, budget_amount, plan_changes, etc.). Always present on REQUEST.
    • allowed_responses — What the human can choose (default: ["approve", "reject"]). Always present on REQUEST.
    • decision — "approve" / "reject". Always present on DECISION.
    • approver — Who made the decision (user ID / name). Always present on DECISION/APPROVED/REJECTED.
    • reason? — Why the human decided this way. Optional (human may not provide a reason).

    Open question: HITL_APPROVED and HITL_REJECTED carry the same semantic as HITL_DECISION with decision="approve"/"reject". Consider consolidating to just HITL_DECISION — same pattern as TRIAL_DONE with outcome.
