    Event System — DB Schema & Traceability Design

    Status: DRAFT v3 Date: 2026-03-24 Depends on: EVENT_TAXONOMY_SIMPLE_VIEW.md (canonical event list), EVENT_TAXONOMY_FINAL_PAYLOADS.md (payload contracts) Scope: Database schema changes, naming conventions, traceability architecture, and Task lifecycle design for the event system refactor

    1. Naming Conventions

    Suffixes

    Every field name in the system follows these suffix rules:

    Suffix Meaning Example Points to
    _id Integer FK to a DB table PK experiment_id, task_id, trial_id table.id column
    _uuid UUID string, self-owned by this record task_uuid, event_uuid, execution_plan_uuid No FK — generated at creation, stable across restarts
    _ref String referencing an external system (NOT a DB table) thread_ref External routing key (e.g., LangGraph thread)
    _hash Deterministic hash of a configuration label_config_hash, fat_hash, cache_key Content-addressable — same config = same hash

    Rule

    If you see _id → there MUST be a table with that PK. If there's no table → it's NOT _id.

    Why thread_ref and not thread_id

    thread_ref references a LangGraph conversation thread. There is no threads table in our DB. LangGraph manages its own thread state via its checkpointer (SQLite/Postgres). Our system only stores the thread reference string for routing — "when this task completes, resume THIS thread."

    2. Current DB Schema (Before Changes)

    Table: tasks

    tasks
├── id                          INT PK AUTO
├── task_uuid                   VARCHAR(36) UNIQUE INDEXED — stable UUID
├── task_type                   VARCHAR(50) NOT NULL — TaskType enum
├── thread_id                   VARCHAR(255) NOT NULL INDEXED — ⚠️ TO RENAME → thread_ref
├── experiment_id               INT FK → experiments.id (nullable)
├── trial_id                    INT FK → trials.id (nullable)
├── status                      VARCHAR(20) NOT NULL DEFAULT "pending" INDEXED — TaskStatus enum
├── expected_duration_seconds   FLOAT NOT NULL DEFAULT 60.0
├── started_at                  DATETIME(tz) NOT NULL
├── completed_at                DATETIME(tz) nullable
├── created_at                  DATETIME(tz) server default
└── updated_at                  DATETIME(tz) server default

    Relationships: - events → 1:N with events.task_id - experiment → N:1 with experiments.id - trial → N:1 with trials.id - parquets → 1:N with parquets.task_id

    Table: events

    events
├── id              INT PK AUTO
├── event_uuid      VARCHAR(36) UNIQUE INDEXED — stable UUID
├── event_type      ENUM(EventType) NOT NULL
├── payload         JSON nullable — event-specific data
├── processed       BOOL NOT NULL DEFAULT false — Event Bridge consumption flag
├── task_id         INT FK → tasks.id (nullable)
├── experiment_id   INT FK → experiments.id (nullable)
├── trial_id        INT FK → trials.id (nullable)
├── created_at      DATETIME(tz) server default
└── updated_at      DATETIME(tz) server default

    Enum: TaskType

    class TaskType(StrEnum):
    DATA_FETCH = "data_fetch"
    DATA_PROCESS = "data_process"
    LABELING = "labeling"
    FAT_WINDOWS = "fat_windows"
    SMALL_WINDOWS = "small_windows"
    TRAINING = "training"
    BACKTEST = "backtest"

    Enum: TaskStatus

    class TaskStatus(StrEnum):
    PENDING = "pending"
    RUNNING = "running"
    COMPLETED = "completed"
    FAILED = "failed"
    CANCELLED = "cancelled"
    TIMEOUT = "timeout"

    What exists and what's missing

    Component Exists? Notes
    tasks table ✅ Model defined in models.py
    events table ✅ Model defined in models.py
    EventRepository ✅ Full CRUD + emit_event(), get_unprocessed_events(), mark_event_processed()
    TaskRepository ❌ Referenced in MCP memory but file does not exist
    parent_task_id in tasks ❌ No hierarchy support between orchestrated Tasks
    thread_ref naming ❌ Currently named thread_id (misleading)
    TaskType.LIVE_TRADING ❌ Missing from enum
    TaskType.DATA_PROCESSING ⚠️ Named DATA_PROCESS — inconsistent with event family name DATA_PROCESSING_*

    3. Proposed Schema Changes

    3.1 Table tasks — Modifications

      tasks
  ├── id                          INT PK AUTO
  ├── task_uuid                   VARCHAR(36) UNIQUE INDEXED
  ├── task_type                   VARCHAR(50) NOT NULL — TaskType enum
- ├── thread_id                   VARCHAR(255) NOT NULL INDEXED
+ ├── thread_ref                  VARCHAR(255) nullable INDEXED
  ├── experiment_id               INT FK → experiments.id (nullable)
  ├── trial_id                    INT FK → trials.id (nullable)
+ ├── parent_task_id              INT FK → tasks.id (nullable) INDEXED
  ├── status                      VARCHAR(20) NOT NULL DEFAULT "pending" INDEXED
  ├── expected_duration_seconds   FLOAT NOT NULL DEFAULT 60.0
  ├── started_at                  DATETIME(tz) NOT NULL
  ├── completed_at                DATETIME(tz) nullable
  ├── created_at                  DATETIME(tz) server default
  └── updated_at                  DATETIME(tz) server default

    Changes explained:

    Change Why
    thread_id → thread_ref Not a FK — it's an external reference to a LangGraph thread. Follows naming convention.
    thread_ref nullable (was NOT NULL) Tasks launched from CLI or standalone scripts have no graph thread. Only orchestrator-spawned tasks have a thread_ref.
    + parent_task_id FK → tasks.id Self-referencing FK for chaining orchestrated Tasks. E.g., LABELING task spawned as part of a pipeline that started with DATA_FETCH.

    New relationships on Task model:

    # Self-referencing hierarchy for orchestrated task chains
parent_task_id: Mapped[int | None] = mapped_column(ForeignKey("tasks.id"), nullable=True, index=True)
parent_task: Mapped[Task | None] = relationship("Task", remote_side=[id], back_populates="child_tasks")
child_tasks: Mapped[list[Task]] = relationship("Task", back_populates="parent_task")

    Important: parent_task_id links orchestrated Tasks only (the 8 TaskTypes). Phase/Trial are NOT Tasks — their hierarchy is tracked via domain IDs in payloads (§4.2).

    3.2 Table events — No Column Changes

    The events table does NOT change. Rationale:

    • thread_ref is NOT duplicated here. To find the thread for an event: event.task → task.thread_ref. One JOIN, trivial at our volume.
    • task_uuid and parent_task_uuid live in the JSON payload (via BaseEventPayload), not as DB columns. The structured column task_id (FK) already provides the DB-level link.
    • No denormalization. Schema stays normalized.

    3.3 Enum TaskType — Additions

      class TaskType(StrEnum):
      DATA_FETCH = "data_fetch"
-     DATA_PROCESS = "data_process"
+     DATA_PROCESSING = "data_processing"
      LABELING = "labeling"
      FAT_WINDOWS = "fat_windows"
      SMALL_WINDOWS = "small_windows"
      TRAINING = "training"
      BACKTEST = "backtest"
+     LIVE_TRADING = "live_trading"

    Changes:

    Change Why
    DATA_PROCESS → DATA_PROCESSING Consistent with event family name DATA_PROCESSING_*
    + LIVE_TRADING Live trading sessions are orchestrator-dispatched work units

    Why NO PHASE, TRIAL, or HOUSEKEEPING?

    A Task represents work dispatched by the orchestrator (or CLI) that the Event Bridge needs to track for routing/timeout. Phases and Trials are internal steps of the model-trainer module — nobody dispatches them, nobody awaits them via Event Bridge. Their events exist for observability, not for routing. The hierarchy Training → Phase → Trial → Epoch is expressed through domain IDs in payloads (study_id, phase_number, trial_id, trial_number, epoch), not through Task records.

    3.4 BaseEventPayload — Traceability Fields

    class BaseEventPayload(BaseModel):
    """Base class for all event payloads."""

    # Traceability — links this event to its task lifecycle
    task_uuid: str | None = Field(default=None, description="UUID of the task this event belongs to")
    parent_task_uuid: str | None = Field(default=None, description="UUID of the parent task in the pipeline chain")

    # Existing fields
    source: str | None = Field(default=None, description="Module that emitted the event")
    note: str | None = Field(default=None, description="Optional debug note")

    Why UUIDs in payload (not IDs)?

    • The payload JSON travels outside the DB context — into LangGraph agent state, into MCP messages, into logs, into webhooks.
    • Integer IDs are DB-internal. UUIDs are globally meaningful.
    • The DB-level FK (events.task_id → tasks.id) handles SQL queries. The payload UUID handles everything else.

    What task_uuid and parent_task_uuid answer:

    Field Question it answers Who uses it
    task_uuid "This event belongs to which task lifecycle?" Event Bridge (resume the right thread), Agents (group events), Dashboard (show lifecycle)
    parent_task_uuid "This task was spawned by which bigger task?" Agents (understand pipeline chain), Dashboard (drill-down), Debugging (trace root cause up)

    Note: parent_task_uuid links orchestrated Tasks (the 8 TaskTypes). Phase/Trial events do NOT use parent_task_uuid — they all share the TRAINING task's task_uuid and express hierarchy via domain IDs (§4.2).

    4. Task Lifecycle & Event Flow

    4.1 How Tasks and Events Relate

    Task (DB record)                           Events (DB records)
┌──────────────────────┐                   ┌──────────────────────┐
│ task_uuid: "abc-123" │                   │ LABELS_REQUESTED     │
│ task_type: LABELING  │◄──────────────────│   task_id: 42        │
│ thread_ref: "exp-1"  │                   │   payload.task_uuid: │
│ parent_task_id: 40   │                   │     "abc-123"        │
│ status: running      │                   │   payload.parent_    │
│ experiment_id: 5     │                   │     task_uuid: "p-1" │
│                      │                   ├──────────────────────┤
│                      │                   │ LABELS_STARTED       │
│                      │◄──────────────────│   task_id: 42        │
│                      │                   ├──────────────────────┤
│                      │                   │ LABELS_PROGRESS      │
│                      │◄──────────────────│   task_id: 42        │
│                      │                   ├──────────────────────┤
│                      │                   │ LABELS_DONE          │
│                      │◄──────────────────│   task_id: 42        │
└──────────────────────┘                   └──────────────────────┘

One Task → N Events (all sharing the same task_id FK)
All 4 events share task_uuid="abc-123" in their payload

    4.2 Training Task — All Events Under One Task

    Phase, Trial, and Epoch are internal steps of the model-trainer module, not separate Tasks. All their events point to the same TRAINING Task via task_id. The internal hierarchy is expressed through domain IDs in payloads.

    Task: TRAINING (task_uuid="t-001", task_type=TRAINING, thread_ref="exp-main")
  │
  │  All events below share: task_id=42, payload.task_uuid="t-001"
  │  Internal hierarchy via domain IDs: study_id, phase_number, trial_id, trial_number, epoch
  │
  ├── Event: TRAINING_REQUESTED   experiment_id=5
  ├── Event: TRAINING_STARTED     experiment_id=5, total_phases=3, total_trials_planned=27
  │
  ├── Event: PHASE_STARTED        experiment_id=5, study_id=10, phase_number=1, total_phases=3, planned_trials=10, data_fraction=0.3
  │   ├── Event: TRIAL_STARTED    experiment_id=5, study_id=10, trial_id=101, trial_number=0, phase_number=1
  │   ├── Event: TRIAL_PROGRESS   experiment_id=5, study_id=10, trial_id=101, epoch=1, total_epochs=50, train_loss=0.8, ...
  │   ├── Event: TRIAL_PROGRESS   experiment_id=5, study_id=10, trial_id=101, epoch=2, total_epochs=50, train_loss=0.6, ...
  │   ├── Event: TRIAL_DONE       experiment_id=5, study_id=10, trial_id=101, outcome="completed", score=0.75
  │   ├── Event: TRIAL_STARTED    experiment_id=5, study_id=10, trial_id=102, trial_number=1, phase_number=1
  │   ├── Event: TRIAL_DONE       experiment_id=5, study_id=10, trial_id=102, outcome="pruned", step_pruned_at=5
  │   └── Event: PHASE_PROGRESS   experiment_id=5, study_id=10, phase_number=1, trials_completed=2, planned_trials=10
  ├── Event: PHASE_DONE           experiment_id=5, study_id=10, phase_number=1, n_completed=8, n_pruned=2
  │
  ├── Event: TRAINING_PROGRESS    experiment_id=5, phases_completed=1, total_phases=3, trials_completed=10
  │
  ├── Event: PHASE_STARTED        experiment_id=5, study_id=11, phase_number=2, ...
  │   └── ...                     (same pattern: trials, epochs, progress)
  ├── Event: PHASE_DONE           experiment_id=5, study_id=11, phase_number=2, ...
  │
  ├── Event: TRAINING_PROGRESS    experiment_id=5, phases_completed=2, total_phases=3, trials_completed=20
  │
  └── Event: TRAINING_DONE        experiment_id=5, phases_completed=3, total_phases=3, status="completed", best_trial_id=101

    How to query the Phase/Trial hierarchy (via domain IDs in payloads, not parent_task_id):

    Question Query
    All events for this Training WHERE task_id = 42
    All events for Phase 1 WHERE task_id = 42 AND payload->>'phase_number' = '1'
    All events for Trial 101 WHERE task_id = 42 AND payload->>'trial_id' = '101' (or WHERE trial_id = 101 via DB FK)
    Phases completed so far Last TRAINING_PROGRESS event → phases_completed field
    Trials completed in Phase 1 Last PHASE_PROGRESS with phase_number=1 → trials_completed field

    4.3 Event Bridge Flow

    Orchestrator (LangGraph)              Module                    Event Bridge
│                                     │                         │
│  1. Create Task (status=pending)    │                         │
│     task_uuid="abc-123"             │                         │
│     thread_ref="exp-main"           │                         │
│                                     │                         │
│  2. Emit LABELS_REQUESTED           │                         │
│     task_id=42                      │                         │
│     payload.task_uuid="abc-123"     │                         │
│                                     │                         │
│  3. graph.interrupt()  ─────────────┼───────────────────────→ │
│     (waiting for resume)            │                         │
│                                     │                         │
│                                     │  4. Pick up task        │
│                                     │     Update status=running│
│                                     │                         │
│                                     │  5. Emit LABELS_STARTED │
│                                     │     task_id=42          │
│                                     │                         │
│                                     │  6. Do work...          │
│                                     │     Emit LABELS_PROGRESS│
│                                     │                         │
│                                     │  7. Emit LABELS_DONE    │
│                                     │     task_id=42          │
│                                     │     Update task         │
│                                     │     status=completed    │
│                                     │                         │
│                                     │                         │  8. Poll: get_unprocessed_events()
│                                     │                         │     Finds LABELS_DONE
│                                     │                         │     event.task → task.thread_ref
│                                     │                         │     → "exp-main"
│                                     │                         │
│  9. graph.invoke(                   │                         │
│       Command(resume=payload),      │                  ←──────│
│       config=thread_ref             │                         │
│     )                               │                         │
│                                     │                         │  10. mark_event_processed()
│  10. Director wakes up              │                         │
│      Reads payload                  │                         │
│      Decides next action            │                         │
│                                     │                         │

    Key point: The Event Bridge resolves thread_ref by doing event.task_id → task.thread_ref. One JOIN. No denormalization needed.

    5. Payload 4-Layer Pattern

    Every event payload follows this layered structure:

    Layer 1: Identity (ALL events in the family)

    Fields that route and identify the work unit. Present in REQUESTED, STARTED, PROGRESS, and DONE.

    task_uuid, parent_task_uuid          ← from BaseEventPayload
experiment_id, study_id?, trial_id   ← domain-specific routing IDs

    Layer 2: Config Fingerprint (STARTED + DONE)

    Deterministic hash of the configuration that defines what this task does. For families where the config matters for caching or change detection.

    label_config_hash    ← Labels family
fat_hash             ← FAT Windows family
cache_key            ← SMALL Windows family
pipeline_hash?       ← Data Processing (if implemented)

    Families that DON'T need config hashes:

    Family Why no hash
    Data Fetch Identified by symbol + timeframe + range — hash would be redundant
    Training Identified by experiment_id + training config is immutable per experiment
    Phase Identified by study_id + phase_number
    Trial Identified by trial_id (DB PK)
    Backtest Identified by backtest_session_id (DB PK)
    Live Trading Identified by execution_plan_id + live_session_id

    Layer 3: Key Parameters (STARTED)

    Human-readable parameters that define the behavior of this task. Not for machines (that's what the hash is for) — for humans reading logs, dashboards, and agent summaries.

    threshold_type, horizon              ← Labels: "what kind of labeling run is this?"
window_size, features_count          ← FAT Windows: "how big is this FAT build?"
data_fraction, feature_subset_size   ← SMALL Windows: "how much data, how many features?"
total_phases, total_trials_planned   ← Training: "how big is this training run?"

    Layer 4: Variable Data (PROGRESS + DONE)

    Data that changes during execution or represents the final outcome.

    PROGRESS: position counters + live metrics

    processed_rows, total_rows, progress_percent    ← where am I?
train_loss, val_loss, f1_macro                  ← how is it going? (Trial)

    DONE: outcome + key results

    status/outcome, duration_seconds                ← how did it end?
count_long, count_short, count_none             ← what was produced? (Labels)
total_pnl, return_pct, sharpe_ratio             ← what was the result? (Backtest)

    Layer Summary

    Layer What REQUESTED STARTED PROGRESS DONE
    1. Identity IDs, routing ✅ ✅ ✅ ✅
    2. Config hash Fingerprint ❌ ✅ ❌ ✅
    3. Key params Human-readable config ❌ ✅ ❌ ❌

    REQUESTED is special: It carries Layer 1 (identity) + the parameters the caller passes to trigger the work. It does NOT carry config hashes (not computed yet) or key params (not resolved yet). But it MUST be self-contained — enough data for the Event Bridge to trigger the module.

    7. emit_event() — Updated Signature

    The existing EventRepository.emit_event() already accepts task_id as a parameter (it's used to set the FK column). No signature change needed for the DB-level link.

    However, the caller code needs to:

    1. Create the Task FIRST (via TaskRepository.create_task())
    2. Pass task_id to emit_event() when emitting REQUESTED/STARTED/PROGRESS/DONE
    3. Include task_uuid and parent_task_uuid in the payload dict

    Example flow in module code:

    # 1. Orchestrator creates the Task (chained from a previous DATA_FETCH task)
task = task_repo.create_task(
    task_type=TaskType.LABELING,
    thread_ref="exp-main-thread",
    experiment_id=experiment_id,
    parent_task_id=previous_task.id,  # chain to parent task in the pipeline
)

# 2. Emit REQUESTED with task linkage
event_repo.emit_event(
    event_type=EventType.LABELS_REQUESTED,
    payload={
        "task_uuid": task.task_uuid,
        "parent_task_uuid": previous_task.task_uuid,
        "experiment_id": experiment_id,
    },
    experiment_id=experiment_id,
    task_id=task.id,
)

# 3. Module picks up, emits STARTED
event_repo.emit_event(
    event_type=EventType.LABELS_STARTED,
    payload={
        "task_uuid": task.task_uuid,
        "parent_task_uuid": previous_task.task_uuid,
        "experiment_id": experiment_id,
        "label_config_hash": config_hash,
        "threshold_type": "atr_based",
        "horizon": 24,
    },
    experiment_id=experiment_id,
    task_id=task.id,
)

    10. Query Examples

    "Show me all events for a specific task lifecycle"

    SELECT e.* FROM events e
WHERE e.task_id = (SELECT id FROM tasks WHERE task_uuid = 'abc-123')
ORDER BY e.created_at;

    "Find the thread to resume for this DONE event"

    SELECT t.thread_ref
FROM events e
JOIN tasks t ON e.task_id = t.id
WHERE e.id = :event_id;

    "Get all events for a Training run (including Phase/Trial/Epoch events)"

    -- All events under the TRAINING task — includes Phase, Trial, Epoch events
SELECT e.* FROM events e
WHERE e.task_id = (SELECT id FROM tasks WHERE task_uuid = 't-001')
ORDER BY e.created_at;

    "Get all Trial events for Phase 1"

    -- Filter by domain IDs in payload (Phase/Trial hierarchy is in the payload, not in Task records)
SELECT e.* FROM events e
WHERE e.task_id = 42
  AND e.event_type IN ('TRIAL_STARTED', 'TRIAL_PROGRESS', 'TRIAL_DONE')
  AND json_extract(e.payload, '$.phase_number') = 1
ORDER BY e.created_at;

    "Trace a Task up its pipeline chain"

    -- Walk parent_task_id chain: LABELING → DATA_PROCESSING → DATA_FETCH
WITH RECURSIVE lineage AS (
    SELECT * FROM tasks WHERE task_uuid = 'abc-123'
    UNION ALL
    SELECT parent.* FROM tasks parent
    JOIN lineage child ON child.parent_task_id = parent.id
)
SELECT * FROM lineage;

    "Find timed-out tasks"

    SELECT * FROM tasks
WHERE status = 'running'
AND (CURRENT_TIMESTAMP - started_at) > (expected_duration_seconds || ' seconds')::interval;

    11. Open Questions

    1. Task creation responsibility: Who creates the Task record — the orchestrator (before emitting REQUESTED) or the module (upon receiving the work)? Proposal: orchestrator creates it because only the orchestrator knows the thread_ref.

    2. CLI/standalone tasks: When a module runs from CLI (no orchestrator), should it still create a Task record? Proposal: yes, with thread_ref=None. This ensures ALL work is traceable, not just orchestrated work.

    3. HITL tasks: HITL governance gates are not really "tasks" in the compute sense. Should they create Task records? Proposal: no — HITL events use their own action field as the primary key, not task_uuid.

    4. SMALL_WINDOWS tasks: SMALL windows are called internally per-trial by model-trainer, not dispatched by the orchestrator. Should they create Task records? If so, they'd use thread_ref=None. Or should SMALL_WINDOWS events just share the TRAINING Task's task_id? This affects whether SMALL_WINDOWS stays in TaskType.

    5. Retry semantics: When a task fails and is retried, should the retry be a NEW task with parent_task_id pointing to the failed task, or a NEW task with the SAME parent_task_id? Proposal: same parent — the retry is a sibling, not a child of the failure.
