    Brain Architecture Analysis v3 — Active Supervisor System

    Author: Cascade Date: 2026-03-19 Status: DRAFT v3 — complete CLI/API/event/process-control audit Method: Read every cli.py, every emit_event call, every pause/resume path, full MASTER-PLAN, both remote repos

    1. Experiment Lifecycle — What Actually Happens (from code)

    Source: scripts/thesis/script_2_thesis/run_script2.sh

    Script2 runs a bash loop per experiment code from a CSV manifest:

    [1/7] create experiment (DB row: Experiment + OptunaStudy)
[2/7] fetch-requirements (DataFetcher: OHLCV parquets)
[3/7] process-requirements (DataProcessing: indicators → parquets)
[4/7] generate-labels (LabelingService: label parquets)
[5/7] generate-sampling (SamplingConfig from label distribution)
[6/7] generate-fat-windows (FatWindowPreparer: FAT parquet)
[7/7] run-training (ModelTrainer.run_optimization → Optuna loop)
       ├── Per trial: suggest hyperparams → build SMALL window → train epochs → stage eval → backtest
       └── Post-optuna: optional full backtest on top-N trials

    Each step calls Python CLIs. No LLM. No orchestrator. Pure deterministic code.

    2. What Gets Stored — The Real Data (from DB models + training code)

    2.1 Per Trial (DB: trials table + training_metrics JSON)

    Source: modules/database/src/rainbow/database/models.py:209-274 Source: modules/model-trainer/src/rainbow/model_trainer/types.py:418-473

    DB columns (fast query/sort): objective_value, objective_value_raw, last_epoch_objective_value, best_epoch_objective_value, stage1_result, stage2_result, stage3_result, stage3_return_pct, stage3_total_pnl, stage3_trades_count, window_size, feature_columns_count, status, error_message, checkpoint_path, model_profile_json, param_importances_json, resolved_hyperparameters (full JSON)

    training_metrics JSON blob (deep detail):

    • Final objectives: objective_value, objective_value_raw, objective_direction
    • Epoch tracking: current_epoch, epochs_ran, resume_from_epoch, best_epoch
    • Per-metric bests: best_f1_epoch/value, best_recall_epoch/value, best_precision_epoch/value
    • Stage results: stage1_result/metrics, stage2_result/metrics, stage3_result/metrics
    • Stage3 is a BacktestMetrics with PnL, sharpe, trades, win_rate, alpha, benchmark_return
    • Warm-start metadata: warm_start_applied, warm_start_trial_id, warm_start_checkpoint_path
    • Pruning/failure: pruned, pruned_reason, pruned_at_epoch, failed, failed_reason
    • Full scoring_params with formulas for reproducibility
    • last_epoch_metrics and best_epoch_metrics (EpochSnapshotMetrics)
    • last_eval_metrics and best_eval_metrics (full EvalMetrics)

    resolved_hyperparameters JSON blob: Complete hyperparameter set: window_size, feature_selection (columns + weights), training_hyperparameters (lr, batch_size, weight_decay), model_type, model_hyperparameters (architecture-specific), loss_type, loss_hyperparameters, loss_weights, combined_confidence_weights

    2.2 Per Backtest Session (DB: backtest_sessions table)

    Source: modules/database/src/rainbow/database/models.py:394-476

    backtest_type, status, range_start/end, bot_config_hash/json, backtesting_config_hash/json, backtest_metrics (JSON), checkpoint_selection_mode_requested/effective, initial_equity, equity, equity_mark_to_market, unrealized_pnl, total_pnl, sharpe_ratio, trades_count, win_rate, benchmark_return, alpha, prediction_parquet_id, full resume state (RNG, risk_manager, processed_count)

    2.3 Per Trade (DB: trades table)

    Source: modules/database/src/rainbow/database/models.py:325-391

    side, size, entry_price, exit_price, stop_loss, take_profit, pnl, entry_reason (structured JSON with signal, sizing, levels, kelly_stats, adaptations, execution), exit_reason (structured JSON with exit_code, levels, execution, pnl), entry_confidence, entry_strength, entry_quality, entry_combined_confidence, trailing_stop_distance/activation, highest_price, lowest_price

    2.4 Per Experiment (DB: experiments table)

    Source: modules/database/src/rainbow/database/models.py:133-164

    name, experiment_type, status, plan_hash, experiment_config_json (full config), trials_planned/total/completed/pruned/failed/cancelled, best_trial_id, best_score

    2.5 Disk Artifacts

    • trial.json: Full snapshot per trial (written at start, updated at completion)
    • experiment_config.json: Config snapshot per trial checkpoint folder
    • Checkpoints: best.pt, per-epoch .pt files with model weights + optimizer + scheduler state
    • Prediction parquets: Stored per backtest session when save_predictions_parquet=True
    • TensorBoard: Per-trial training reporter writes epoch metrics, hparams, histograms

    3. Event System Audit — The Real State (It's a Mess)

    Source: modules/common/src/rainbow/common/types/events.py:1-452 Source: grep of all emit_event calls across 18 files

    3.1 What's Actually Emitted (verified from code)

    Module Events Actually Emitted Source File
    model-trainer TRIAL_STARTED, TRIAL_COMPLETED, TRIAL_PRUNED, TRIAL_FAILED, ALL_TRIALS_DONE, TRAINING_PROGRESS objective.py, trials.py, training_processing.py
    backtesting-engine BACKTEST_TRIAL_DONE engine_loop.py:1293-1317
    data-fetcher DATA_FETCH_DONE data_fetcher.py
    data-processing DATA_PROCESSED_DONE data_processing.py
    labeling-service LABELS_DONE labeling_service.py
    window-preparer FAT_WINDOWS_DONE, SMALL_WINDOWS_PROGRESS fat_window_preparer.py, small_window_preparer.py
    housekeeping HOUSEKEEPING_DONE housekeeping.py
    live-trading-runner LIVE_TRADING_STARTED, LIVE_TRADING_STOPPED, LIVE_ANOMALY_DETECTED, LIVE_EXEC_SUMMARY live_trading_runner.py, instance_manager.py
    core MIGRATION_GENERATED, MIGRATION_APPLIED migration_service.py
    database DEMO_SEEDED (test only) cli.py

    3.2 What's DEFINED but NEVER Emitted (dead EventTypes)

    These exist in the EventType enum but NO code emits them:

    • Missing STARTED events: DATA_FETCH_STARTED, LABELS_STARTED, FAT_WINDOWS_STARTED, TRAINING_STARTED, BACKTEST_STARTED, SMALL_WINDOWS_STARTED
    • Missing PROGRESS events: DATA_FETCH_PROGRESS, LABELS_PROGRESS, FAT_WINDOWS_PROGRESS, BACKTEST_PROGRESS
    • Missing DONE: SMALL_WINDOWS_DONE (only PROGRESS exists!)
    • Missing lifecycle: TRIAL_CONFIG_READY, TRIAL_SCORED
    • Missing control: EXPERIMENT_STOPPING, TRIAL_STOPPING, TRIAL_CANCELLED, EXPERIMENT_STOPPED
    • Missing plan: EXPERIMENT_PLAN_CREATED, EXECUTION_PLAN_CREATED, MASTER_EXEC_PLAN_UPDATED
    • All orchestrator events: GRAPH_THREAD_*, AGENT_*, STORE_*, BUDGET_*, HITL_* (obviously — orchestrator never ran)
    • All REQUESTED events: DATA_FETCH_REQUESTED, LABELS_REQUESTED, etc. (interrupt/resume pattern never used)

    3.3 Specific Problems Found

    1. No lifecycle consistency: Some steps emit DONE but not STARTED. No step emits both STARTED and DONE consistently.
    2. SMALL_WINDOWS_DONE is never emitted — only SMALL_WINDOWS_PROGRESS (per-column) exists.
    3. Payloads are inconsistent: TRIAL_STARTED payload is {experiment_id, trial_id} while TRIAL_COMPLETED adds {objective_value, duration_seconds}. No standard payload schema enforced at emission time.
    4. The emit flag is threaded through the code but controlled by the caller. Script2 may or may not pass --emit.
    5. ~50 EventTypes defined, ~15 actually emitted. The rest are aspirational for the orchestrator that never ran.
    6. No event for experiment creation — EXPERIMENT_PLAN_CREATED is defined but never emitted.
    7. No event for data pipeline completion — DATA_PIPELINE_DONE is defined but never emitted.
    8. Payload schemas exist in Pydantic (BaseEventPayload and subclasses in events.py:436+) but are never used at emission time — payloads are ad-hoc dicts.

    3.4 Verdict

    The event system is NOT fine. It's a skeleton with good intentions but inconsistent, incomplete execution. Before any agent can consume events, this needs a systematic overhaul:

    • Every pipeline step must emit STARTED + DONE (minimum)
    • Payloads must use the existing Pydantic schemas (they're already defined!)
    • Dead EventTypes must be either implemented or removed
    • The emit flag should default to True (events are cheap, silence is expensive)

    4.2 The --emit Flag Pattern

    Most pipeline commands have --emit/--no-emit flag (default: False). This controls event emission.

    Modules with --emit: data-fetcher, labeling-service, window-preparer, model-trainer, backtesting-engine, housekeeping.

    Problem: Default is --no-emit. For the active supervisor, this MUST default to --emit (or always emit).

    5. Process Control Matrix — What Can Be Paused/Stopped Today

    5.1 Current Capabilities (verified from code)

    Module Pause Resume Cancel/Stop Status Query Mechanism
    backtesting-engine YES YES (--resume session_id) YES YES DB flag polling (pause_requested/cancel_requested in BacktestSession)
    live-trading-runner YES YES YES (stop) YES CLI commands → in-memory instance manager
    model-trainer NO YES (study-level + per-trial checkpoint) SIGINT/SIGTERM only NO (no CLI cmd) Optuna load_if_exists=True + checkpoint resume + orphan trial cleanup. Stop via OS signal only.
    data-fetcher NO NO NO NO Fire-and-forget
    labeling-service NO NO NO NO Fire-and-forget
    window-preparer NO NO NO NO Fire-and-forget
    data-processing NO NO NO NO Library only
    housekeeping NO NO NO NO One-shot operations

    5.2 model-trainer — What It Already Has (corrected)

    The model-trainer has robust resume at multiple levels:

    • Study-level resume: optuna.create_study(load_if_exists=True) + remaining_trials = n_trials - (completed + pruned) → only runs what's missing
    • Phase-level skip: If all trials in a phase are finalized → "Skipping already completed phase"
    • Per-trial checkpoint resume: --resume flag (default True) → resolve_resume_checkpoint_path() finds latest epoch .pt → load_checkpoint() restores model + optimizer + scheduler + epoch + best metrics
    • Orphan cleanup: On re-entry, cancels any trials left in running state from interrupted runs
    • Graceful SIGINT/SIGTERM: Signal handler → cancels running trials → updates all DB counters → marks study STOPPED

    5.3 model-trainer — What's Actually Missing (for active supervisor)

    The remaining gaps are narrower than originally stated:

    1. No DB-flag-based stop — Stop requires sending SIGTERM to the OS process. An agent would need to know the PID. Adding stop_requested flag polled in handle_trial_complete callback would let the agent stop via DB write.
    2. No status CLI command — Cannot query progress without reading DB directly. A model-trainer status --experiment-id <n> command would expose counters + current trial info.
    3. No pause — Only full stop exists. Pause (checkpoint current trial + halt without cancelling) would be a new feature.

    5.4 What Else Needs Process Control

    Module What's needed Priority Difficulty
    model-trainer stop_requested DB flag + callback check CRITICAL Medium
    model-trainer status CLI command HIGH Easy
    data-fetcher cancel_requested for long multi-symbol fetches Medium Easy
    window-preparer cancel_requested for long FAT window builds Medium Easy

    5.5 Live Trading Supervision — Deep Audit (CRITICAL)

    This is the most critical module for active supervision. Live trading involves real money. Every failure mode must be anticipated and monitored.

    Source: modules/live-trading-runner/src/rainbow/live_trading_runner/ (all files read in full)

    5.5.1 Control Surface — What Exists Today

    The live-trading-runner has the richest control surface of any module. All 8 CLI commands map to Python API methods on LiveTradingRunner:

    CLI Command Python Method State Transition What It Does
    start --execution-plan-id <n> start() → STARTING → RUNNING Load execution plan, create LiveSession DB row, bootstrap model/features/window, start stream
    stop [--execution-plan-id <n>] stop() → STOPPING → STOPPED Halt stream, persist final state, emit LIVE_TRADING_STOPPED, cleanup loop
    pause [--execution-plan-id <n>] pause() → PAUSING → PAUSED Halt decision processing, keep stream alive, positions stay open
    resume [--execution-plan-id <n>] resume() → RESUMING → RUNNING Resume decision processing from cursor checkpoint
    status [--execution-plan-id <n>] status() (no transition) Return InstanceSnapshot with state, mode, symbol, timeframe, timestamps
    reload [--execution-plan-id <n>] reload() (no transition) Hot-reload runtime config without restarting session
    force-close [--execution-plan-id <n>] force_close() (no transition) Queue force-close request; executed on next closed candle tick
    positions positions() (no transition) Return all open trades across active sessions

    State Machine (state_machine.py): Enforces legal transitions via next_state_for_command(). States: STARTING, RUNNING, PAUSING, PAUSED, RESUMING, STOPPING, STOPPED, FAILED. Illegal transitions raise ValueError.

    5.5.2 Events Emitted — The Supervisor's Eyes

    The live-trading-runner emits 4 event types, more than any other module:

    1. LIVE_TRADING_STARTED — Emitted per session on start(). Payload: {execution_plan_id, execution_plan_uuid, live_session_id, mode, target_symbol, target_timeframe}

    2. LIVE_TRADING_STOPPED — Emitted per session on stop(). Also emitted with {scope: "no_active_sessions"} when stop is called with nothing running. Payload: {execution_plan_id, execution_plan_uuid, live_session_id, reason}

    3. LIVE_ANOMALY_DETECTED — Emitted on two conditions:

    • Data gap auto-pause: When auto_pause_on_data_gap=True and observed candle gap exceeds data_gap_tolerance_candles × timeframe_delta. Payload includes {anomaly_type: "data_gap_auto_pause", observed_gap_seconds, tolerated_gap_seconds}.
    • Stream runtime failure: When stream processing raises an unrecoverable error. Payload includes {anomaly_type: "stream_runtime_failure", details: <error_reason>}. Instance transitions to FAILED state.

    4. LIVE_EXEC_SUMMARY — Emitted every metrics_snapshot_every_n_ticks ticks AND on every tick with execution activity (orders executed, positions opened/closed). Payload: {live_session_id, execution_plan_id, symbol, timeframe, decision_id, candle_id, equity, realized_pnl_total, unrealized_pnl_total, open_positions_count, closed_trades_count, executed_orders_count}

    This is the primary heartbeat the supervisor should monitor. If LIVE_EXEC_SUMMARY events stop arriving for a session, something is wrong.

    5.5.3 Safety Guards Already Implemented

    Kill Switch (LiveRuntimeConfig.kill_switch_enabled):

    • Checked at the TOP of process_closed_candle() in runtime_loop.py:527-534
    • When kill_switch_enabled=False → decision processing is completely blocked, candle is silently skipped
    • This is a runtime config field — can be toggled via reload command without restarting

    Auto-Pause on Data Gap (LiveRuntimeConfig.auto_pause_on_data_gap):

    • Checked in runtime_loop.py:551-581 after candle ingestion
    • Compares consecutive candle timestamps against data_gap_tolerance_candles × timeframe_delta
    • If gap exceeds tolerance → emits RuntimeControlAction(action="auto_pause_data_gap") → InstanceManager catches it and transitions session to PAUSED + emits LIVE_ANOMALY_DETECTED
    • This is automatic — no supervisor intervention needed for detection, but supervisor should be notified to investigate WHY the gap occurred

    Risk Manager Force-Flat (RiskManager.should_force_flat()):

    • Checked in runtime_loop.py:694-719 after regular order generation
    • When risk halt is active (e.g., daily loss exceeded) → ALL open positions are force-closed with exit_reason.code="RISK_FORCE_FLAT"
    • Risk manager state is persisted per-tick: risk_manager_daily_pnl, risk_manager_date, risk_manager_halted, risk_manager_state_json

    Force-Close on Specific Position (force_close --position-id <n>):

    • Queued via _pending_force_close_by_execution_plan_id map
    • Executed on the NEXT closed candle tick (not immediately — deterministic)
    • Can target a single position or all positions
    • Generates exit_reason.code="MANUAL_FORCE_CLOSE"

    Heartbeat Timeout (LiveRuntimeConfig.heartbeat_timeout_seconds):

    • Configured but enforcement depends on the stream service polling loop
    • Value validated as > 0 at config time

    5.5.4 Runtime Audit Trail — What's Persisted Per Tick

    Every tick persists structured data into live_sessions.notes.runtime_audit (JSON in DB):

    CommandHistoryEntry: {command, timestamp, from_state, to_state} — every control command ever applied. Bounded to last 500 entries.

    LiveDecisionEntry: {decision_id, correlation_id, candle_id, candle_timestamp_ms, signal, confidence, requested_orders_count, executed_orders_count, opened_positions_count, closed_positions_count, realized_pnl_delta} — every trading decision. Bounded to last 1,000 entries.

    LiveMetricsEntry: {timestamp, equity, realized_pnl_total, unrealized_pnl_total, open_positions_count, closed_trades_count} — portfolio snapshots. Bounded to last 1,000 entries.

    LiveCursorCheckpoint: {last_processed_candle_timestamp_ms, processed_decision_ids_tail, pending_orders_count, open_trade_ids} — idempotency checkpoint for resume.

    DB-level per-tick persistence: live_sessions.status, live_sessions.risk_manager_daily_pnl, live_sessions.risk_manager_date, live_sessions.risk_manager_halted, live_sessions.risk_manager_state_json.

    Mode Adapter Real Money? Risk
    LIVE_SIM SimExecutionAdapter NO — simulated fills against market data Low — good for testing
    LIVE_REAL LiveRealExecutionAdapter YES — real orders to exchange CRITICAL — requires maximum supervision

    Both modes use the same LiveInstanceLoop, RiskManager, and BotLogic. The only difference is whether orders are simulated or sent to a real exchange.

    5.5.6 What the Active Supervisor MUST Watch

    The supervisor must implement these watchdog rules for live trading:

    Watch Target Detection Method Action Priority
    Session heartbeat No LIVE_EXEC_SUMMARY event for > heartbeat_timeout_seconds Alert + investigate (stream dead? exchange down?) CRITICAL
    Equity drawdown LIVE_EXEC_SUMMARY.equity drops below threshold (% of initial) Auto-pause + alert + HITL for resume decision CRITICAL
    Daily PnL limit risk_manager_halted=True in DB Alert (RiskManager already halts entries, but supervisor should verify and log) HIGH
    Anomaly events LIVE_ANOMALY_DETECTED event consumed Log + alert + investigate root cause (data gap? stream failure?) CRITICAL
    Position count explosion LIVE_EXEC_SUMMARY.open_positions_count exceeds max Auto-pause + alert (possible bug in entry logic) HIGH
    Stale positions Open positions with no activity for > N candles Alert (position may be stuck without SL/TP) MEDIUM
    Model degradation Signal distribution shift (too many HOLD signals, or all same direction) Alert + suggest investigation MEDIUM
    Session stuck in transition Instance in STARTING, PAUSING, RESUMING, STOPPING for > N seconds Alert (possible deadlock or crash) HIGH
    Multiple sessions same symbol Two sessions trading same symbol:timeframe simultaneously Alert (position conflicts possible) HIGH
    Kill switch disabled kill_switch_enabled=False detected in config Alert (safety guard bypassed) CRITICAL

    5.5.7 Supervisor Intervention Matrix — What It Can Do Today

    Situation Supervisor Action CLI Command Requires HITL?
    Emergency stop all trading Stop all sessions live-trading-runner stop YES (destructive)
    Emergency stop one session Stop specific session live-trading-runner stop --execution-plan-id <n> YES (destructive)
    Pause for investigation Pause sessions live-trading-runner pause --execution-plan-id <n> No (non-destructive, positions preserved)
    Resume after investigation Resume paused sessions live-trading-runner resume --execution-plan-id <n> YES (resumes real trading)
    Force close positions Close all or specific position live-trading-runner force-close [--position-id <n>] YES (destructive, realizes PnL)
    Disable new entries Toggle kill switch via reload live-trading-runner reload (after config change) YES (blocks all decisions)
    Check current status Query all instances live-trading-runner status No (read-only)
    Check open positions Query positions live-trading-runner positions No (read-only)

    5.5.8 What's Missing for Full Active Supervision

    1. No equity threshold auto-stop — RiskManager handles daily PnL limits, but there's no configurable "stop session if equity drops below X%" at the runtime level. The supervisor must implement this by watching LIVE_EXEC_SUMMARY events.

    2. No session-level PnL alerting — The LIVE_EXEC_SUMMARY event has all the data, but there's no built-in threshold that triggers an alert event. The Event Bridge watchdog must implement this.

    3. No cross-session correlation — Each session is independent. No built-in check for "two sessions trading same symbol" or "total portfolio exposure across sessions". The supervisor must aggregate across sessions.

    4. No automatic recovery from FAILED state — Once a session hits FAILED, it requires manual stop + start to recover. The supervisor could automate retry with exponential backoff for transient failures.

    5. Heartbeat enforcement is passive — heartbeat_timeout_seconds is configured but the CONSUMER of heartbeats must enforce it. The Event Bridge watchdog must check last LIVE_EXEC_SUMMARY timestamp per session.

    6. No structured anomaly classification — LIVE_ANOMALY_DETECTED events are free-form. A severity classification (INFO/WARNING/CRITICAL) and standardized anomaly taxonomy would help the supervisor prioritize.

    7. No DB-flag-based kill switch — Current kill switch is config-driven (requires reload). A kill_switch_requested DB flag polled per-tick would let the supervisor kill instantly via DB write without CLI.

    Two Operating Modes

    Mode 1: Daemon (primary) — Event Bridge runs continuously, monitors events, invokes Director when intervention needed. User doesn't need to be present.

    Mode 2: Chat (interactive) — User talks to Director via chat. Director uses same tools, same agents. Can ask: "What's running?", "Stop that experiment", "Run a new experiment with VIT on BTCUSDT".

    Both modes use the SAME LangGraph, SAME agents, SAME tools. The difference is who initiates: an event (daemon) or the user (chat).

    Agents — 4 Initial

    Agent Role Tools LLM Tier
    Director Routes, decides, executes direct actions ALL pipeline CLIs, stop/pause/resume, DB queries, KB search Tier 3 (Genius)
    Analyst Interprets metrics, compares trials, detects anomalies DB queries, KB search, write insights to Store Tier 2 (Smart)
    Research Searches KB, explores patterns, proposes hypotheses KB search, web search, write findings to Store Tier 2 (Smart)
    Planner Generates experiment configs, proposes plans KB search, create_experiment_config, read analyst summaries Tier 3 (Genius)

    Coder/Doc/Issues deferred — Copilot/Windsurf + exposed MCP tools handle those needs today.

    Key Principles

    1. No manual CLI — User talks to chat, agent executes CLIs via MCP tools
    2. Events drive everything — Every module action emits events, events trigger agent evaluation
    3. Active supervision — Watchdog detects stuck/failing processes and intervenes
    4. HITL configurable — Destructive actions (stop, delete) require approval. Queries don't.
    5. Research runs in parallel — Agent can start background research, notify user when done
    6. Knowledge is institutional memory — All experiment results indexed for future queries
