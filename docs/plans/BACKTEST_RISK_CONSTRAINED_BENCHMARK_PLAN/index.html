    Backtest Risk-Constrained Benchmark Plan

    Status: STAGE 4 — Final Task List (Frozen for review)
    Started: 2026-04-08
    Completed: —
    Priority: CRITICAL

    Description

    Implement a new model-free, risk-constrained benchmark for backtests without replacing the current benchmark_return / alpha fields.

    The new metric family will compare the bot return against a passive LONG-only deployment that:

    • opens at t0
    • opens up to max_concurrent_positions
    • uses the persisted experiment/session risk policy
    • ignores model-driven gates and ignores pyramiding
    • never closes
    • is marked to market at range_end

    The rollout must include:

    • new nullable columns on backtest_sessions
    • natural persistence for all future backtests
    • JSON persistence inside backtest_metrics
    • manual SQL schema change artifacts instead of Alembic
    • one robust SQLite backfill script with backup, validation, and atomic rollback behavior
    • test coverage for math, persistence, serialization, SQL verification, and rollback safety

    This document freezes scope, decisions, touched modules, operational workflow, and failure policy so implementation can proceed without hidden assumptions or mid-flight design changes.

    Rules

    Global rules

    • Re-open this plan and the touched source files before each implementation iteration.
    • Keep all code, comments, docstrings, and documentation in English.
    • Do not replace or reinterpret the current benchmark_return / alpha semantics in V1.
    • Do not use YAML defaults as a historical source of truth.
    • Do not use prediction parquet, signal streams, or model outputs for this benchmark.
    • Do not use Alembic for this change.
    • Do not allow silent behavior drift. If implementation reveals a missing prerequisite not covered here, stop and update this plan first.

    NO LEGACY / NO BACKWARD COMPATIBILITY

    This is a clean implementation. No legacy fallbacks, no backward compatibility hacks.

    • The new fields are added to the schema and code as if they always existed.
    • Old rows simply have NULL in the new columns until backfill runs.
    • No "if old format then..." branching in runtime code.
    • No deprecated code paths, no compatibility wrappers.
    • The kelly_fallback_method default for old sessions ("risk_based") is a backfill-only convenience, not a runtime compatibility layer.

    Source-of-truth rules

    • Historical and runtime benchmark inputs must come from backtest_sessions.bot_config_json.
    • If session JSON is missing or invalid, the only allowed fallback is persisted experiment/session config already stored in the database.
    • File-based config under config/**/*.yaml is explicitly out of scope for the benchmark calculation.

    Benchmark definition rules

    • Benchmark is LONG-only.
    • Benchmark opens only at t0.
    • Benchmark opens up to max_concurrent_positions.
    • Benchmark does not use pyramid_decay_factor.
    • Benchmark does not use min_pyramid_ratio.
    • Benchmark does not use max_trades_per_day.
    • Benchmark does not use model thresholds, model quality gates, cooldown logic, or signal-change logic.
    • Benchmark reuses risk sizing math and risk caps only.

    Operational safety rules

    • Manual SQL must be written as standalone .sql files under scripts/sql/.
    • Backfill must create a physical backup before the first write.
    • Backfill is atomic per session: each session update (both backtest_sessions and linked trials rows) is one transaction. If a session fails, that session gets NULLs and the script continues to the next session.
    • Re-running the backfill processes only sessions with benchmark_return_risk_constrained IS NULL.
    • Validation queries must exist before rollout and after rollout.

    Findings

    Finding 1: Current alpha is full-equity buy-and-hold and must remain untouched in V1

    • Problem: The current alpha is based on a 100% market buy-and-hold baseline and is already wired into scoring and reporting.
    • Evidence:
    • metrics_snapshot.py computes:
      • benchmark_return = (last_price - first_price) / first_price
      • alpha = return_pct - benchmark_return
    • engine_loop.py persists those fields into backtest_sessions and backtest_metrics.
    • scoring.py consumes alpha inside Stage 3 scoring.
    • Decision: Keep existing benchmark_return and alpha unchanged. Add a second metric family instead of replacing the existing one.
    • Impact: This avoids changing Stage 3 behavior, historical comparability, and current UI/report semantics.

    Finding 2: The new benchmark must be model-free and must not depend on predictions

    • Problem: A benchmark that reads predictions or signals is no longer neutral.
    • Evidence:
    • User direction for this plan explicitly rejected prediction-driven or model-aware benchmark variants.
    • The current bot entry path depends on model thresholds and gating logic that are intentionally excluded here.
    • Decision: The new benchmark will not read prediction parquet, labels, confidence, strength, quality, or signal history.
    • Impact: The benchmark can be backfilled from persisted config + market data + DB state only.

    Finding 3: The benchmark needs a finite slot count, so max_concurrent_positions is a required benchmark input

    • Problem: Without a finite number of initial slots, a t0-only passive benchmark has no bounded definition.
    • Evidence:
    • bot_config.py defines max_concurrent_positions as optional.
    • The benchmark opens only once at t0; therefore an unbounded slot count is not computable.
    • Current workspace DB sample shows all persisted backtest sessions use risk_based and max_concurrent_positions = 3.
    • Decision: max_concurrent_positions is mandatory for this benchmark. If it is absent or invalid for a selected session, the benchmark is treated as unsupported for that session.
    • Impact:
    • New code must compute the new fields as NULL for unsupported sessions, without mutating current alpha.
    • Backfill script must fail fast in strict mode when a selected session is unsupported.

    Finding 4: Risk sizing can be reused, but model-driven size multipliers must be neutralized

    • Problem: The sizing helper currently accepts strength, quality, and quality attenuation config, which are model-dependent.
    • Evidence:
    • position_sizing.py applies:
      • strength_multiplier = 1 + strength
      • optional quality attenuation via sizing_quality_weight and sizing_quality_floor
    • risk_manager.py feeds live signal strength/quality into _calculate_order_size().
    • Decision: The benchmark uses the same sizing formulas but with neutralized model inputs:
    • strength = 0.0
    • quality = 1.0
    • sizing_quality_weight = 0.0
    • sizing_quality_floor = 1.0
    • pyramid_ratio = 1.0
    • Impact: Sizing remains risk-manager compatible while staying model-free.

    Finding 5: risk_based sizing requires baseline stop-loss distance at t0

    • Problem: risk_based sizing depends on risk_per_unit, and that value comes from stop-loss distance, not directly from ATR.
    • Evidence:
    • bot_logic.py builds baseline stop-loss from:
      • strategy_config.stop_loss_config
      • strategy_config.atr_period
      • market ATR at runtime
    • It then computes risk_per_unit = abs(current_price - stop_loss_price).
    • Decision: The benchmark must derive risk_per_unit from the baseline stop-loss formula only:
    • for atr_based, use current_price +/- atr * stop_loss_config.value
    • for fixed_percent, use current_price +/- current_price * stop_loss_config.value
    • do not apply quality-based stop-loss tightening
    • do not use TP or trailing-stop logic
    • Impact: Backfill must reload the first market row and its ATR for each selected session.
    • Definition of t0: t0 is exactly backtest_sessions.range_start. The market price and ATR at this timestamp are used for initial benchmark position sizing.
    • ATR source: ATR must be read from the experiment's base_features parquet at range_start. If ATR is unavailable, the session is marked unsupported (NULLs) and the backfill continues to the next session.

    Finding 6: Included risk controls are narrower than the full runtime risk manager

    • Problem: The runtime RiskManager contains more gates than the benchmark should inherit.
    • Evidence:
    • risk_manager.py includes:
      • directional gates
      • concurrent-position cap
      • daily entry cap
      • pyramiding ratio
      • exposure cap
      • venue minimum notional gate
      • trading halt / cooldown logic
    • Decision: The benchmark includes only these controls:
    • position_sizing_method
    • risk_per_trade
    • max_concurrent_positions
    • max_position_value_pct
    • max_notional_per_trade
    • max_exposure_percent
    • min_order_notional
    • sizing_price_buffer_pct
    • estimated slippage / estimated commission from order preferences
    • allow_long
    • Kelly statistics inputs when sizing method is kelly
    • Decision: The benchmark excludes:
    • entry_threshold
    • exit_threshold
    • signal_flip_threshold
    • min_quality_threshold
    • same_direction_cooldown_ticks
    • require_higher_confidence
    • allow_on_signal_change
    • pyramid_decay_factor
    • min_pyramid_ratio
    • max_trades_per_day
    • max_daily_loss
    • max_drawdown_percent_stop
    • cooldown_after_loss_seconds
    • force_flat_on_stop
    • take-profit / trailing-stop exits
    • Impact: The benchmark helper should not call the full runtime evaluate_entries() path directly.

    Finding 7: allow_long = false must explicitly disable the benchmark, not silently turn it into cash

    • Problem: A LONG-only benchmark is not semantically valid if the persisted risk policy forbids LONG entries.
    • Evidence:
    • bot_config.py defines allow_long.
    • Benchmark side is intentionally fixed to LONG.
    • Decision: If allow_long is false, the three new benchmark fields must be NULL.
    • Impact: No hidden “flat cash benchmark” will be introduced. Unsupported sessions remain explicit.

    Finding 8: Kelly requires a fallback method (BUG FIX INCLUDED)

    • Problem: The current Kelly implementation has a chicken-and-egg bug: Kelly sizing requires historical trades, but if there are no trades yet, the bot raises ValueError and can never open its first trade.
    • Evidence:
    • risk_manager.py line 224-225 raises ValueError("Kelly sizing requested but insufficient trade history for stats") when get_kelly_stats_from_db() returns None.
    • kelly_stats.py returns None when there are fewer than kelly_min_trades closed trades.
    • This means a bot configured with position_sizing_method="kelly" can never open its first position.
    • Current workspace DB shows kelly_sessions = 0, so this bug has not been hit in practice, but it must be fixed.
    • Decision: V1 introduces a new config field kelly_fallback_method: Literal["fixed", "risk_based"] | None.
    • Required when position_sizing_method == "kelly".
    • When Kelly stats are unavailable, the bot (and benchmark) uses kelly_fallback_method instead of raising.
    • For the benchmark at t0, there are never trades, so kelly_fallback_method is always used.
    • For backfill of old sessions without kelly_fallback_method in bot_config_json, default to "risk_based".
    • Impact:
    • New config field in RiskManagerConfig.
    • Fix in risk_manager.py to use fallback instead of raising.
    • Benchmark always uses kelly_fallback_method for Kelly sessions (since t0 has no trades).
    • Old Kelly sessions (if any existed) can still be backfilled with default risk_based fallback.

    Finding 9: Both BacktestMetrics and TrialSummaryMetrics JSON must be updated

    • Problem: Existing backtest_metrics JSON rows do not contain the new fields, and the control-panel service validates that JSON via Pydantic. Additionally, trials.training_metrics contains stage3_metrics: BacktestMetrics which also needs the new fields.
    • Evidence:
    • types.py defines BacktestMetrics and its to_dict().
    • backtest_service.py parses backtest_metrics with BacktestMetrics.model_validate(value).
    • trials.training_metrics JSON contains stage3_metrics which is serialized from BacktestMetrics.
    • Decision:
    • The three new risk-constrained fields must be added to BacktestMetrics as nullable optional fields with explicit None defaults.
    • Backfill must update BOTH backtest_sessions.backtest_metrics JSON AND trials.training_metrics.stage3_metrics JSON.
    • New summary columns added to trials table: stage3_benchmark_return_risk_constrained, stage3_alpha_risk_constrained.
    • Impact: Old rows remain parseable before backfill completes, new rows persist the new data naturally, and both tables stay consistent.

    Finding 10: Reporting and API surfaces must expose the new fields, even if UI adoption comes later

    • Problem: The benchmark is not useful if it only exists in DB internals.
    • Evidence:
    • models.py defines BacktestSession.
    • backtest_session_service.py updates persisted session metrics.
    • admin.py defines BacktestSessionSummary.
    • backtest_service.py builds summary payloads.
    • reports.py exposes backtest session values to analytics/report scripts.
    • backtest_monetary_metrics.py reads alpha / benchmark_return from backtest_sessions.
    • Decision: V1 must expose the new fields through:
    • ORM model
    • session update service
    • BacktestMetrics JSON
    • control-panel summary/detail payloads
    • report/query helpers that already surface benchmark metrics
    • Impact: The change is cross-module even without touching Stage 3 scoring.

    Finding 11: The benchmark should use risk-aware sizing, but benchmark return remains a passive market PnL on synthetic positions

    • Problem: Runtime sizing uses an effective_price that includes cost assumptions and a safety buffer. The benchmark must be realistic without becoming a full execution simulator.
    • Evidence:
    • risk_manager.py computes effective_price for sizing from:
      • current market price
      • estimated slippage
      • worst-case commission
      • sizing_price_buffer_pct
    • Decision: V1 benchmark sizing uses the same cost-aware effective price for position sizing and cap checks, but benchmark PnL is computed as passive mark-to-market on synthetic LONG holdings:
    • benchmark_pnl_risk_constrained = sum(size_i * (last_price - first_price))
    • where size_i was computed using the cost-aware sizing input
    • benchmark_return_risk_constrained = benchmark_pnl_risk_constrained / initial_equity
    • alpha_risk_constrained = return_pct - benchmark_return_risk_constrained
    • Impact: The benchmark becomes materially more realistic than full-equity buy-and-hold without introducing a separate execution simulator in V1.

    Finding 12: Backfill must be atomic per session to avoid intermediate DB state

    • Problem: The user explicitly wants rollback protection so the database does not remain half-updated if something fails mid-run.
    • Evidence:
    • Backfill affects persisted historical benchmark values across many rows.
    • Existing repository history already prefers manual SQL and conservative SQLite scripts for similar operations.
    • With 4500+ sessions, a single-transaction approach risks losing all progress on any single failure.
    • Decision: Backfill is atomic per session: each session update (backtest_sessions row + linked trials rows) is one transaction.
    • If a session fails: log error, set NULLs for that session, commit, continue to next session.
    • No all-or-nothing for the full set — partial progress is preserved.
    • Re-running processes only sessions with benchmark_return_risk_constrained IS NULL.
    • --session-id / --limit / --resume-from-id can scope smaller runs for debugging.
    • Impact: Recovery is deterministic. Failed sessions can be retried individually without losing progress on other sessions.

    Finding 13: Persistence timing — backtest_sessions during progress, trials only at end

    • Problem: Currently benchmark_return and alpha are only persisted at the end of a backtest, not during progress. User wants ALL metrics (including new ones) visible in real-time during backtest execution.
    • Evidence:
    • metrics_snapshot.py persist_metrics_snapshot() updates total_pnl, equity, sharpe_ratio, trades_count, win_rate, progress_pct during progress.
    • engine_loop.py finalize_backtest() adds benchmark_return, alpha, and full backtest_metrics JSON only at the end.
    • trials stage3 fields are only updated when the trial completes (in objective.py / training.py).
    • Decision:
    • backtest_sessions: Update ALL fields during progress (every N ticks + on trades) — including benchmark_return, alpha, and all 6 new fields.
    • trials: Keep current behavior — stage3 summary fields updated only when backtest completes.
    • Rationale: The control panel can show live progress directly from backtest_sessions. The trials table is a high-level summary that makes sense to update only at completion.
    • Impact: Users can monitor benchmark/alpha/friction metrics in real-time during backtest execution via the control panel or DB queries.

    Finding 14: Benchmark is deterministic and safe to recompute on checkpoint resume

    • Problem: The backtesting engine supports resume from checkpoint. When a backtest resumes mid-stream, the benchmark state at t0 needs to be available.
    • Evidence: The benchmark synthetic positions depend only on static inputs: bot_config_json, first_price, ATR at range_start, and initial_equity. None of these change during the backtest.
    • Decision: The benchmark positions are recomputed deterministically at resume. They do NOT need to be persisted as part of checkpoint state.
    • Impact: No checkpoint format changes required. The benchmark helper is called once at engine setup (or on resume) and produces identical results every time.

    Finding 15: Backfill parquet resolution uses existing core services

    • Problem: The backfill script needs ATR from the experiment's base_features parquet at range_start, running outside the normal engine context.
    • Evidence:
    • experiment_data_service.py provides load_feature_frames_for_experiment() to load all feature parquets.
    • parquets.py find_parquet_by_filters() performs two-stage DB query with coverage validation.
    • features.py find_base_atr_in_tree() locates the base ATR indicator parquet.
    • parquet_service.py _resolve_absolute_path() converts relative DB paths to absolute filesystem paths.
    • Decision: The backfill script reuses the existing ExperimentDataService → find_parquet_by_filters() → Parquet.path resolution chain. No custom parquet path logic needed.
    • Flow: experiment_id → ExperimentService.get_by_id() → ExperimentComposed → ExperimentRequirements (parquet coverage metadata) → find_parquet_by_filters() → read ATR at range_start.
    • If the parquet file is missing, corrupted, or has no data at range_start, the session is marked unsupported (NULLs) per the unsupported-session policy.
    • Impact: No new file-lookup logic required. Backfill benefits from the same coverage validation the engine uses.

    Finding 16: Prune handler does not persist top-level benchmark columns (BUG — must fix in Task 5)

    • Problem: The BacktestPrunedError handlers in backtesting_engine.py compute BacktestMetrics (including benchmark_return and alpha) but only persist the JSON blob via backtest_metrics=metrics.to_dict(). They do NOT pass benchmark_return=, alpha=, or any other top-level column keywords to update_backtest_session().
    • Evidence:
    • backtesting_engine.py line ~780 (validation prune handler) and line ~1254 (prediction prune handler) both call: python backtest_session_service.update_backtest_session( backtest_session_id=exc.backtest_session_id, backtest_metrics=metrics.to_dict(), # Only JSON blob )
    • No benchmark_return=, alpha=, equity=, total_pnl=, or any other top-level field is passed.
    • A one-time backfill (scripts/tmp-utils/bakfills/backfill_pruned_benchmark_alpha.py) already fixed 1015 historical pruned sessions by copying values from JSON to top-level columns. But the runtime bug remains.
    • Decision: Task 5 must patch ALL prune handlers to pass top-level fields (existing ones like benchmark_return, alpha, equity, total_pnl, etc. AND all 6 new fields) in the same update_backtest_session() call.
    • Impact: Without this fix, any future pruned backtest will still have NULLs in top-level columns, requiring repeated backfills.

    Locked Benchmark Definition

    New fields

    The V1 schema change adds:

    backtest_sessions table:

    • benchmark_pnl_risk_constrained (REAL, nullable)
    • benchmark_return_risk_constrained (REAL, nullable)
    • alpha_risk_constrained (REAL, nullable)
    • net_per_trade (REAL, nullable)
    • gross_per_trade (REAL, nullable)
    • friction_retention_ratio (REAL, nullable)

    trials table:

    • stage3_benchmark_return_risk_constrained (REAL, nullable)
    • stage3_alpha_risk_constrained (REAL, nullable)
    • stage3_net_per_trade (REAL, nullable)
    • stage3_gross_per_trade (REAL, nullable)
    • stage3_friction_retention_ratio (REAL, nullable)

    No additional version column is included in V1.

    Exact benchmark algorithm

    For one selected backtest_session:

    1. Load persisted bot config from backtest_sessions.bot_config_json.
    2. Parse the persisted bot config into typed config models.
    3. Validate prerequisite inputs:
    4. initial_equity > 0
    5. first_price and last_price available
    6. allow_long = true
    7. max_concurrent_positions present and > 0
    8. baseline stop-loss inputs available for the configured sizing method
    9. Load market data at t0 to obtain the ATR needed by baseline stop-loss logic when the stop type is atr_based.
    10. Compute baseline risk_per_unit from the strategy stop-loss config only.
    11. For slot_index from 0 to max_concurrent_positions - 1:
    12. determine effective sizing method:
      • if position_sizing_method == "kelly": use kelly_fallback_method (default "risk_based" for old configs)
      • otherwise: use position_sizing_method directly
    13. compute neutral sizing using the effective method
    14. neutralize model inputs (strength=0.0, quality=1.0, no quality attenuation)
    15. do not apply pyramiding
    16. apply max_position_value_pct
    17. apply max_notional_per_trade
    18. apply cumulative max_exposure_percent (caps are cumulative across slots)
    19. apply min_order_notional
    20. if the slot cannot be opened after those caps, stop opening further slots
    21. Hold all opened synthetic LONG positions until range_end.
    22. Mark them to market using last_price.
    23. Compute:
    24. benchmark_pnl_risk_constrained
    25. benchmark_return_risk_constrained
    26. alpha_risk_constrained = return_pct - benchmark_return_risk_constrained

    Per-trade friction metrics algorithm

    For each backtest_session, calculate from ALL closed trades in that session:

    1. Query all trades for the session: SELECT pnl, gross_pnl FROM trades WHERE backtest_session_id = ? AND status = 'CLOSED'
    2. Count closed trades: closed_trades_count
    3. Sum net PnL: sum_net_pnl = SUM(trade.pnl)
    4. Sum gross PnL: sum_gross_pnl = SUM(trade.gross_pnl)
    5. Calculate:
    6. net_per_trade = sum_net_pnl / closed_trades_count (NULL if no closed trades)
    7. gross_per_trade = sum_gross_pnl / closed_trades_count (NULL if no closed trades)
    8. friction_retention_ratio = net_per_trade / gross_per_trade (NULL if gross_per_trade == 0 or no closed trades)

    Interpretation: - net_per_trade: Average realized profit per trade after all friction (commissions, slippage) - gross_per_trade: Average profit per trade before friction - friction_retention_ratio: What fraction of gross edge survives friction (e.g., 0.50 = 50% retained)

    Unsupported-session policy

    If any required benchmark prerequisite fails, V1 does not invent a fallback behavior. Instead:

    • the three new fields remain NULL
    • the current benchmark_return / alpha remain untouched
    • new-backtest runtime logs the failure reason clearly
    • strict backfill aborts and rolls back the selected transaction

    Unsupported examples:

    • allow_long = false
    • max_concurrent_positions is NULL or <= 0
    • unsupported stop-loss type
    • missing first_price, last_price, or required ATR at t0
    • missing ATR in experiment's base_features parquet at range_start

    Note: kelly configured is NOT unsupported — the benchmark uses kelly_fallback_method (or defaults to risk_based for old sessions).

    Backfill Plan

    Scope

    The backfill script updates historical backtest_sessions AND linked trials rows. For each session:

    • Updates backtest_sessions columns + backtest_metrics JSON
    • Updates linked trials columns + training_metrics.stage3_metrics JSON

    Both DB columns and JSON blobs are kept consistent.

    Preflight

    The script must:

    1. verify the target DB file exists
    2. verify the three columns exist
    3. create a timestamped physical backup before the first write
    4. discover the selected sessions (--session-id, --limit, --resume-from-id, or full set)
    5. validate that each selected session has the minimum inputs required for computation

    Transaction policy

    • Atomic per session: each session update (backtest_sessions row + linked trials rows) is one transaction.
    • If a session fails: log error, set NULLs for that session, commit, continue to next session.
    • No all-or-nothing for the full set — partial progress is preserved.
    • Re-running processes only sessions with benchmark_return_risk_constrained IS NULL.
    • Dry-run performs the same read/compute/validate phases without writing.

    Script controls

    The script must support at least:

    • --db-path
    • --dry-run
    • --backtest-session-id
    • --limit
    • --resume-from-id
    • --verbose

    Validation before commit

    Before commit, the script must verify for every updated selected row:

    • the three DB columns are non-contradictory
    • benchmark_return_risk_constrained == benchmark_pnl_risk_constrained / initial_equity within tolerance
    • alpha_risk_constrained == return_pct - benchmark_return_risk_constrained within tolerance
    • the JSON payload, if rewritten, matches the DB fields

    Failure policy

    • Per-session failure does NOT abort the run.
    • The script logs the precise failure reason with session ID.
    • The failed session gets NULLs and the script continues to the next session.
    • At the end, the script prints a summary: X succeeded, Y failed, Z skipped.
    • Re-run the script to retry failed sessions (they still have NULLs).
    • The original backup remains the recovery source for full restore if needed.
