Absolute net PnL remains useful, but it is not enough for thesis interpretation because a losing market can make a good defensive ensemble look bad in raw dollars. The primary thesis metric for the redesigned ensemble cycle is therefore alpha_rc: ensemble return minus the risk-constrained benchmark return for the same fold period.

    rank_by=promise is not risk-constrained today. It is an ensemble-oriented weighted percentile score with these components:

    • objective: 1.5;
    • stage2: 2.5;
    • stage3: 1.0;
    • sharpe: 1.25;
    • win_rate: 1.25;
    • trades_inverse: 0.75;
    • total_pnl: 0.75.

    • objective: 1.5;
    • stage2: 2.5;
    • stage3: 1.0;
    • sharpe: 1.25;
    • win_rate: 1.25;
    • trades_inverse: 0.75;

    9. Appendix A — Inherited Workflow Contract From The Old Cycle

    The new cycle keeps the following workflow elements from the old cycle. Only rank_by and the source-run set change. Everything below remains valid because it is not part of the discarded economic lens.

    A.1 Strict compatibility bucketing

    Trial-level compatibility is enforced before any top-N or merge selection. A candidate enters a bucket only if it shares with the bucket key all of:

    • base_symbol;
    • base_timeframe;
    • effective Stage 3 range (range_start, range_end);
    • full bot_config_json;
    • full simulation_config_json;
    • combined_confidence_weights.

    Candidates from different buckets cannot be merged. Top-N selection happens within the chosen best bucket. The ensemble planner persists the compatibility bucket choice in plan.json.

    A.2 Correlation-aware selection thresholds

    The 3-level battery off / mild / tight is inherited verbatim:

    Level --max-prob-corr --max-signal-agreement
    off not passed not passed
    mild 0.85 0.60
    tight 0.60 0.45

    Pearson correlation is computed on prob_none, prob_long, prob_short over the Stage 3 evaluation range. Signal agreement is computed on the discrete signal outputs over the same range.

    A.3 Eight-rule merge battery

    All ensemble executions in Block B run the same merge-rule battery in one pass:

    average, majority_vote, all_must_agree, majority_vote:0.75, majority_vote:0.85, weighted_average, weighted_majority_vote:0.75, weighted_majority_vote:0.85.

    Audits group findings by pool size to avoid the documented small-pool collapse between several rules.

    A.4 Synthetic experiment / study / trial persistence

    Each ensemble run materializes in DB as one synthetic experiment, one synthetic study, and one synthetic trial per merge rule. The synthetic trial stores merge_rule, the canonical source trial id, and the full source_trial_ids list inside model_profile_json. The downstream backtest session, trades, and benchmark attribution attach to the synthetic trial id through the existing trial-centric DB schema. The DB schema is not refactored.

    Standard artifacts per ensemble run:

    plan.json

ensemble_backtests.log

    The planner additionally writes ensemble_plan.log in the plan output folder when prediction reuse/regeneration runs at planning time under correlation-aware selection.

    10. Appendix B — Why The Old rank_by=friction Was Replaced

    The discarded selector was conceptually wrong for Thesis 2 for the following concrete reasons:

    • ref_gross was the decision-time / reference-price gross, not real executed closed-trade gross. It approximated what a trade would have produced if friction had been zero. Selecting candidates on ref_gross therefore rewarded models for theoretical upside under no execution friction, which is not the thesis criterion.
    • PnL + Fric is not equivalent to real closed-trade gross PnL. The persisted real gross is stage3_total_gross_pnl from the engine snapshot columns; the old proxy mis-attributed friction direction in several places.
    • stage3_total_gross_pnl > 0 used as a hard rescue gate emptied valid stress pools where real models traded enough to generate measurable friction without producing positive gross at the reference price.
    • The thesis explicitly cares about executed-trade economics, including the risk-constrained benchmark introduced in the recent backtest refactor.

    The redesigned rank_by=friction keeps the same selector name but replaces the implementation entirely with a stress pool defined by real trade count, real total friction, weak real net result, and a quality floor on stage2_result and entry_f1. No legacy or backward-compatibility code path is preserved.
