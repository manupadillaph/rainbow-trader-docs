    Script2 Thesis — Table Unification Plan

    Created: 2026-04-10 Status: In Progress Scope: 24+ tables across 6 exporter scripts under scripts/thesis/script_2_thesis/ Shared helpers: modules/core/src/rainbow/core/helpers/reports.py, modules/core/src/rainbow/core/helpers/backtest_monetary_metrics.py MCP Memory hash (confirmed headers): 26844f256551819b0e23403e52f5581731211595bb1d4760a090485958e3abb2 MCP Memory hash (field inventory): d9d7a10621a2bd0b2c0d121e2da0d29ef785f2935f13f0969cd4b1a9a3728d9e

    1. Canonical Column Sequence (Master Spec — Clarity Pass 2026-04-10)

    The canonical sequence is the union of all meaningful columns already present across the thesis tables. No column is removed from the canonical family unless it is proven to be a pure alias of another column with the same semantics.

    Every horizontal table follows this macro block ordering:

    # → Code → Tier → Model → Fold
  → [Rank / Rankings In / Count / Sel Rank / Sel Score]
  → [Trials / Prun-Fail / Prog / View / Status]
  → [Rule / Kind / Label / Variant / Thrshld Orig / Thrshld New]
  → Obj → S1 → S2 → S3
  → PnL (Ret%) → Est.Fric → PnL+Fric → Gross/trade → Benchmark → α (adj)
  → Trades (Entry%) → Win% → Sharpe
  → F1 Ent/Mac → JS Div
  → [TrueLong% / PredLong% / TrueShort% / PredShort%]
  → [Max Prob Corr / Max Signal Agr]
  → WS×Feat → Params
  → [Trainable Params / Total Params / Frozen Params / Model MB / Train Param MB / Work VRAM GB / Reserve VRAM Low GB / Reserve VRAM High GB]
  → BT Prog%
  → [Run / Runs] → [ExpID / ExpIDs] → TrialID (Opt#) → Best TrialID (Opt#) → BktID → Best BktID
  → [Orig X / New X / Δ X]
  → [Vs Best ... / Vs Avg ...]
  → [Ensemble / Source / Interpretation]

    1.1 Semantic Rules

    1. Model and Fold are always separate columns. Never use Model (Fold) in the target spec.
    2. No column loss. If a column exists today and carries distinct information, it stays in the canonical family.
    3. Prog and Status are not interchangeable by default. Prog is progress/lifecycle summary. Status is a separate operational state when it adds distinct meaning.
    4. Merged identity must be pluralized. Use Runs and ExpIDs for merged rows. Never use singular Run or ExpID for merged rows.
    5. Canonical IDs stay canonical even if current payloads are incomplete. TrialID (Opt#) and BktID remain the target shape. Missing propagation is an implementation gap, not a reason to weaken the spec.
    6. Monetary block travels together. If a table shows PnL analysis, the preferred block is: PnL (Ret%) → Est.Fric → PnL+Fric → Gross/trade → Benchmark → α (adj).
    7. Delta ordering uses one system only: Orig X → New X → Δ X when a table shows paired before/after versions. Vs Best ... / Vs Avg ... stays as a comparison tail block. Pure comparison tables keep only delta columns, in canonical semantic order.

    1.2 Composite Cells

    The following are single columns with composite cell content, not split columns:

    Column Cell meaning Example
    PnL (Ret%) pnl + return_pct +$186.42 (+1.86%)
    Trades (Entry%) trades + entry_pct 48 (12.50%)
    F1 Ent/Mac entry_f1 + f1_macro 0.1234 / 0.2871
    α (adj) alpha + alpha_adjusted +1.23% (+1.87%)
    WS×Feat window_size + feature_count 120×45
    TrialID (Opt#) trial_id + optuna_trial_number 3698 (#26)

    1.3 Block Legend

    Block Columns When present
    Identity 1 #, Code, Tier, Model, Fold Trial-like tables
    Rankings Rank, Rankings In, Count, Sel Rank, Sel Score Ranking/selection tables
    Progress / View Trials, Prun/Fail, Prog, View, Status Progress and operational dashboards
    Ensemble / Variant Rule, Kind, Label, Variant, Thrshld Orig, Thrshld New Ensemble and extra-backtest tables
    Objectives Obj, S1, S2, S3 Where stage objectives exist
    Monetary PnL (Ret%), Est.Fric, PnL+Fric, Gross/trade, Benchmark, α (adj) Where backtest PnL is discussed
    Performance Trades (Entry%), Win%, Sharpe Where backtest performance exists
    Quality F1 Ent/Mac, JS Div Where prediction quality exists
    Prediction Health TrueLong%, PredLong%, TrueShort%, PredShort% Prediction health tables
    Diversity Max Prob Corr, Max Signal Agr Ensemble selected-source tables
    Size / Profile WS×Feat, Params, profile memory columns Where model size/profile exists
    Identity 2 BT Prog%, Run/Runs, ExpID/ExpIDs, TrialID (Opt#), Best TrialID (Opt#), BktID, Best BktID Trial/session identity
    Paired Change Orig X, New X, Δ X Before/after extra backtests
    Tail Comparison Vs Best ..., Vs Avg ... Ensemble execution comparisons

    3. Critical Data-Source Notes

    3.1 No Data Loss Guardrails

    The following columns are distinct and must not be silently removed:

    • Max Prob Corr
    • Max Signal Agr
    • TrueLong%
    • PredLong%
    • TrueShort%
    • PredShort%
    • Trainable Params
    • Total Params
    • Frozen Params
    • Status when it carries operational meaning distinct from Prog

    3.2 Historical PRUNED Alpha / Benchmark Persistence Gap

    There was a confirmed persistence gap in PRUNED backtest flows:

    • trials.training_metrics.stage3_metrics.alpha and benchmark_return were populated
    • backtest_sessions.backtest_metrics.alpha and benchmark_return were populated
    • but top-level backtest_sessions.alpha and backtest_sessions.benchmark_return were often left NULL

    For the purposes of this plan, that historical data-availability issue is treated as resolved via backfill. Therefore:

    • Benchmark and α (adj) are considered available inputs for unification planning
    • they are not an active blocker for this document
    • any future write-path hardening for PRUNED persistence is tracked separately from this table-unification plan

    3.3 Required Upstream Dependencies Before Table Unification

    These dependencies must be explicitly called out in the plan:

    1. Expand TrialMetricsDetailed
    2. Expand ExperimentStatsSummary
    3. Expand merged-summary structures so merged rows can emit Runs and ExpIDs
    4. Propagate trial_id, optuna_trial_number, backtest_session_id, and source identity through ensemble/extra payloads where still missing

    Note: historical alpha / benchmark_return availability is assumed solved for this plan after backfill. Any remaining PRUNED write-path persistence hardening belongs to follow-up implementation work, not to this prerequisite list.
