    Script2 Thesis — Deep Verification Plan

    Status: Active — only open plan for Script2 thesis work. Created: 2026-04-13 Rewritten: 2026-04-18 Supersedes the verification handoff from:

    • docs/plans/archived/20260418_SCRIPT2_THESIS_TABLE_UNIFICATION_PLAN.md (implementation Complete)
    • docs/plans/archived/20260418_SCRIPT2_THESIS_CODE_AUDIT_PLAN.md (implementation Complete)

    Prior verification round (cosmetic-only, V1 baseline _NEW): docs/plans/archived/20260410_SCRIPT2_CODE_QUALITY_VERIFICATION_PLAN.md

    Current execution naming (2026-04-23):

    • Original: existing canonical .md artifact.
    • NEW: prior _NEW artifact from the previous verification round.
    • NEW1: current regenerated artifact once the script output is verified.
    • _V2, _V3, and _V4 references below are historical intermediate artifact names only.

    1. Objective

    Prove that every number in every generated thesis report is mathematically correct and traceable to its DB/CSV source. These reports are part of Manu's thesis; a single numerical error is unacceptable.

    The previous verification round (2026-04-10 → _NEW files) was a cosmetic refactor check. This round must validate that the subsequent code-audit + table-unification rounds produced correct numbers, not just that scripts run without crashing.

    Scope additions vs. previous plan version:

    • Layer 5 — Code Conventions Audit (grep hard-gates: no inline SQL/CSV/formulas, shared helpers/types used, thesis script style rules enforced).
    • Layer 6 — Thesis-Quality Sanity Checks (magnitudes, signs, no NaN/Inf, NA placement).
    • Two reusable minimal verification scripts (DB spot-check + cross-table consistency).

    2. Principles

    1. Source-of-truth rule. Every rendered value traces to (a) a DB column, (b) a summary CSV column, or (c) one of the 4 centralized derived formulas in modules/core/src/rainbow/core/helpers/backtest_monetary_metrics.py. Nothing else is allowed.
    2. No overwrite rule. Current verified regenerations use explicit NEW1 artifacts. Original (.md) and prior-round _NEW files stay intact on disk.
    3. Three-way diff rule. Wherever _NEW exists, compare Original ↔ NEW (pre-unification cosmetic deltas), NEW ↔ NEW1 (this-round audit/bug-fix deltas), and Original ↔ NEW1 (total accumulated delta). This isolates expected vs. unexpected changes.
    4. No code touches for verification. This plan verifies the current code. If a numerical bug is found, stop and open a targeted fix — do not patch inside this verification pass.

    9.4 Execution Log — 2026-04-23 Positive Edge Retention Fix

    This block resolves the Layer 6.8 metric-definition blocker from §9.3. The old visible Friction Retention% criterion was invalid for thesis reporting because the persisted DB ratio is a raw net/gross value and is not meaningful when gross edge is non-positive or net edge is negative.

    Decision

    Visible report metric:

    positive_edge_retention = net_per_trade / gross_per_trade
only when gross_per_trade > 0 and net_per_trade >= 0
otherwise NA

    Implementation rule:

    • The formula lives in rainbow.core.helpers.backtest_monetary_metrics.compute_positive_edge_retention().
    • Renderers and campaign parsers must call the helper, not implement local formula branches.
    • The helper raises ValueError when the defined positive-edge domain has net_per_trade > gross_per_trade. That case indicates data inconsistency and must not be clamped or hidden by formatting.
