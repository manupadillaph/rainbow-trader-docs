    Script2 Thesis — Deep Verification Plan

    Status: Active — only open plan for Script2 thesis work. Created: 2026-04-13 Rewritten: 2026-04-18 Supersedes the verification handoff from:

    • docs/plans/archived/20260418_SCRIPT2_THESIS_TABLE_UNIFICATION_PLAN.md (implementation Complete)

    Current execution naming (2026-04-27):

    • Original: existing canonical .md artifact.
    • NEW1: current regenerated artifact once the script output is verified.
    • _NEW, _V2, _V3, and _V4 references below are historical intermediate artifact names only and are not operational comparison targets in the current restart.

    Manual restart (2026-04-26):

    • Manu requested a clean NEW1 restart because too much changed and every generated artifact/comparator result must be inspected manually.
    • Operational inventory/cleanup commands may be executed by the agent, but regeneration scripts and comparator reviews must be presented one command at a time for Manu to run and inspect.
    • The current step is Phase 1: regenerate NEW1 artifacts one script at a time, then inspect each output with a purpose-built comparator before moving to the next script.
    • A scoped cleanup command was executed under scripts/thesis/script_2_thesis; it found no existing NEW1 paths before or after cleanup. See §9.5.

    1. Objective

    Prove that every number in every generated thesis report is mathematically correct and traceable to its DB/CSV source. These reports are part of Manu's thesis; a single numerical error is unacceptable.

    The previous verification round (2026-04-10 → _NEW files) was a cosmetic refactor check. This round must validate that the subsequent code-audit + table-unification rounds produced correct numbers, not just that scripts run without crashing.

    Scope additions vs. previous plan version:

    • Layer 5 — Code Conventions Audit (grep hard-gates: no inline SQL/CSV/formulas, shared helpers/types used, thesis script style rules enforced).
    • Layer 6 — Thesis-Quality Sanity Checks (magnitudes, signs, no NaN/Inf, NA placement).
    • Two reusable minimal verification scripts (DB spot-check + cross-table consistency).

    2. Principles

    1. Source-of-truth rule. Every rendered value traces to (a) a DB column, (b) a summary CSV column, or (c) one of the 4 centralized derived formulas in modules/core/src/rainbow/core/helpers/backtest_monetary_metrics.py. Nothing else is allowed.
    2. Persisted snapshot precondition. The manual integrity gate scripts/checks/check_monetary_metrics_consistency.py must pass before generated thesis reports are treated as verifiable. That script checks trades against the redundant monetary snapshots in backtest_sessions, trials.stage3_*, and training_metrics.stage3_metrics. After it passes, renderers/helpers must consume the persisted snapshots directly and must not re-scan trades.
    3. No overwrite rule. Current verified regenerations use explicit NEW1 artifacts. Original (.md) artifacts stay intact on disk.
    4. Comparator rule. Every regenerated report/table must have a purpose-built comparator script. The comparator parses the relevant tables/sections, matches stable identities, compares all fields in its declared scope, and reports every mismatch. The comparator reports facts; the audit decides whether each mismatch is expected, accepted, or blocking.
    5. No code touches for verification. This plan verifies the current code. If a numerical bug is found, stop and open a targeted fix — do not patch inside this verification pass.
