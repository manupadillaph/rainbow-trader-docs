    Script 2 Implementation Plan — For Coder Agent

    Status: ✅ CLOSED — Archived on 2026-03-16

    Closure decision: - AI supervisor: explicitly removed from scope. - Evolution tracking: implemented via run_script2.sh append-only evolution_log.jsonl under scripts/thesis/script_2_thesis/results/<RUN_TAG>/. - Remaining historical checklist items in this document are kept as legacy notes, not active blockers.

    Overview

    This document contains actionable implementation tasks for Script 2 of the thesis validation. It covers: implementing new model types, creating experiment JSON templates, creating the bash runner script, and creating the analysis script.

    Read these documents FIRST for full context:

    • docs/plans/EXPERIMENT_PLAN_THESIS_VALIDATION.md — Master thesis plan (sections 6, 8.6)
    • docs/researchs/_ML/script2-baseline-decisions.md — Research decisions and rationale

    Reference implementations (read these to understand patterns):

    • modules/model-trainer/src/rainbow/model_trainer/models/random.py — Non-trainable baseline pattern
    • modules/model-trainer/src/rainbow/model_trainer/models/vit.py — Trainable model pattern
    • modules/model-trainer/src/rainbow/model_trainer/models/factory.py — Model factory
    • modules/model-trainer/src/rainbow/model_trainer/models/base.py — BaseModel abstract class
    • scripts/thesis/script_1_validation/ — Script 1 structure (experiments/, run script, analyze script)

    Execution Readiness Update (2026-03-09)

    Current repository state is executable for Script 2 baseline runs:

    • generate_script2_experiments.py: implemented as a thesis orchestrator using MEGA CLI integration.
    • run_script2.sh: implemented, manifest-driven, supports --resume-from, auto-runs analysis.
    • analyze_script2_results.py: implemented and integrated with runner output.
    • New model types are present in model-trainer (rule_baseline, logistic, mlp, tcn).
    • Tier E multi-symbol generation: enabled through MEGA symbol-spec requests.
    • Versioned output snapshots: enabled (experiments/<version-tag>/...) with active manifest at experiments/script2_manifest.csv.
    • Current generator output: 50 base specs × 4 folds = 200 run JSON templates, identified by simple codes (S2-TX-MYY-FZ) and full codes (s2-tx-myy-<model>-fz-exx).

    Items still pending (non-blocking for baseline Script 2 execution):

    • Evolution log utilities — implementation pending (see Task 15)
    • AI experiment supervisor process — implementation pending (see Task 14)

    CRITICAL RULES

    1. NO hardcoded values. All thresholds, hyperparameters, and numeric values must come from configuration (experiment JSON, YAML config, or Pydantic models).
    2. Follow existing code style exactly. Read existing files before writing. Match indentation, naming conventions, docstring style, comment style, type hints.
    3. All code must pass: mypy --strict, ruff check, pylint >= 9.5, Pyright.
    4. Never use pragmas (# type: ignore, # noqa, # pyright: ignore) to suppress errors.
    5. Complete docstrings with Args, Returns, Examples where applicable.
    6. Comments explaining every block of logic using # comments.
    7. UTF-8 encoding, 4-space indentation, max 120 chars per line.
    8. All commands run inside .venv — use source .venv/bin/activate or .venv/bin/python.

    Task 1: Pass feature_names to Model Factory

    Goal: Make feature names available to models that need them (rule-based baseline).

    File to modify: modules/model-trainer/src/rainbow/model_trainer/models/factory.py

    Changes:

    1. Add parameter feature_names: list[str] | None = None to create_model() function signature.
    2. Forward feature_names to TradingRuleBaseline constructor (when model_type == "rule_baseline").
    3. Other model types can ignore it (existing models don't need it).

    File to modify: modules/model-trainer/src/rainbow/model_trainer/helpers/model/training.py

    Changes:

    1. At ~line 278, feature_columns_names_ordered is already computed as sorted(feature_columns_with_weights.keys()).
    2. Pass this list to create_model() as feature_names=feature_columns_names_ordered.

    Verification: Existing tests must still pass. No behavior change for existing models.

    Task 2: Implement TradingRuleBaseline Model

    Goal: Non-trainable model that applies expression-based rules on named features.

    Expression System: Inspired by CONDITION Signal

    Rules use expression strings parsed via Python AST, following the same philosophy as the CONDITION signal (modules/data-processing/src/rainbow/data_processing/signals/condition.py).

    Key reference: _build_polars_expr_from_condition_expression() in condition.py already implements a full AST parser supporting: <, <=, >, >=, ==, !=, and, or, not, arithmetic (+, -, *, /, **), parentheses for grouping.

    The rule-based model adapts this same approach but:

    • Instead of generating Polars expressions (for DataFrames), generates tensor operations (for the [B, W, F] input tensor).
    • Instead of column names, maps feature names to tensor indices (dim 2).
    • Adds timestep notation {T} to specify which position in the window dimension (dim 1).

    Timestep Notation: {T} with Curly Braces

    The model input is [B, W, F] — a window of W timesteps. Expressions need to specify WHICH timestep to evaluate. Curly braces {T} are used to avoid collision with parentheses () used for mathematical grouping in expressions.

    Notation rules:

    • feature_name{-1} → value at the last timestep (most recent, index -1)
    • feature_name{0} → value at the first timestep (oldest in window, index 0)
    • feature_name{-5} → value at 5 ticks back from current
    • Every feature reference in the expression MUST include {T} — there is no default

    Examples:

    # Simple threshold
- expression: "rsi_14{-1} > 70"
  signal: "SHORT"

# Combined conditions with parentheses for math grouping
- expression: "(rsi_14{-1} > 70) and (volume{-1} > 1000)"
  signal: "SHORT"

# Cross-timestep comparison (momentum)
- expression: "rsi_14{-1} > rsi_14{-5}"
  signal: "LONG"

# Arithmetic in expression
- expression: "ema_9{-1} - ema_21{-1} > 0"
  signal: "LONG"

# Feature vs feature comparison
- expression: "ema_9{-1} > ema_21{-1}"
  signal: "LONG"

    Constructor Parameters (beyond BaseModel)

    • feature_names: list[str] — ordered feature names matching tensor dim 2
    • rules: list[RuleExpressionConfig] — list of (expression, signal) pairs from config
    • default_signal: str — signal when no rule matches (typically "NONE")
    • strength_value: float — fixed output for strength head (0.0-1.0)
    • quality_value: float — fixed output for quality head (0.0-1.0)
    • sampled_logit_margin: float — logit value assigned to the predicted class

    Model Output Fields Explained

    • sampled_logit_margin: The logit value assigned to the predicted class. Since this model is non-trainable and doesn't produce real logits, it creates them synthetically: logits = [0.0, 0.0, 0.0] then logits[predicted_class] = sampled_logit_margin. After softmax, higher margin = more "confident" prediction. This is the same mechanism used by all random baselines (see random.py lines 131-132).
    • strength_value: Fixed constant output for the strength head (0.0-1.0). Since this model has no learned strength estimation, it outputs this constant for every prediction.
    • quality_value: Fixed constant output for the quality head (0.0-1.0). Same as strength.

    forward(x) Logic

    For each sample in batch:
  For each rule in priority order (list order from config):
    Parse and evaluate expression against tensor values:
      - Map feature_name{T} → x[batch_idx, T, feature_index]
      - Evaluate boolean expression
    If expression is True:
      Map rule.signal to class index ("NONE"→0, "LONG"→1, "SHORT"→2)
      logits[batch_idx, :] = 0.0
      logits[batch_idx, class_index] = sampled_logit_margin
      strength[batch_idx] = strength_value
      quality[batch_idx] = quality_value
      Break (first match wins)
  If no rule matched:
    Map default_signal to class index
    logits[batch_idx, :] = 0.0
    logits[batch_idx, default_class_index] = sampled_logit_margin

    Important Implementation Details

    • Build _feature_index: dict[str, int] in __init__ mapping feature name → tensor position
    • Parse ALL expressions via AST in __init__ (once, not every forward call)
    • Validate ALL feature names referenced in expressions exist in feature_names during __init__ (raise ValueError if a referenced feature is not found)
    • Validate ALL timestep indices are valid for the window size during __init__
    • Use _anchor pattern from TradingRandomBaseline for differentiable graph
    • Signal mapping: "NONE" → class 0, "LONG" → class 1, "SHORT" → class 2
    • The AST parser can be adapted from condition.py's _build_polars_expr_from_condition_expression but generating tensor index operations instead of Polars expressions

    Architecture:

    class TradingLogistic(BaseModel):
    def __init__(self, *, input_dim: int, seq_len: int, num_classes: int) -> None:
        super().__init__(input_dim=input_dim, seq_len=seq_len, num_classes=num_classes)
        flat_dim = seq_len * input_dim
        self.classifier = nn.Linear(flat_dim, num_classes)
        self.strength_head = nn.Sequential(nn.Linear(flat_dim, 1), nn.Sigmoid())
        self.quality_head = nn.Sequential(nn.Linear(flat_dim, 1), nn.Sigmoid())

    def forward(self, x: torch.Tensor) -> dict[str, torch.Tensor]:
        # x: [B, W, F] → flatten to [B, W*F]
        flat = x.view(x.shape[0], -1)
        logits = self.classifier(flat)
        strength = self.strength_head(flat)
        quality = self.quality_head(flat)
        return {"logits": logits, "strength": strength, "quality": quality}

    Task 5: Implement TradingTCN Model

    Goal: Temporal convolutional network — 1D convolutions over time axis.

    New file: modules/model-trainer/src/rainbow/model_trainer/models/tcn.py

    Constructor parameters (beyond BaseModel):

    • channels: int — number of conv channels (default: 64)
    • kernel_size: int — convolution kernel size (default: 3)
    • num_layers: int — number of conv layers (default: 2)
    • dropout: float — dropout rate (default: 0.1)
    • activation: str — activation function name (default: "relu")

    Architecture:

    Input: [B, W, F]
  → Transpose to [B, F, W] (Conv1d expects [B, channels_in, length])
    → Conv1d(F, channels, kernel_size, padding="same") + Activation + Dropout
    → Conv1d(channels, channels, kernel_size, padding="same") + Activation + Dropout
    → ... (num_layers total)
      → Global Average Pooling over time dim: [B, channels]
        → classifier: Linear(channels, num_classes) → logits
        → strength_head: Linear(channels, 1) + Sigmoid → strength
        → quality_head: Linear(channels, 1) + Sigmoid → quality

    Important: Use padding="same" (or calculate padding manually for PyTorch < 1.9) to keep temporal dimension constant through conv layers. Only reduce via pooling at the end.
