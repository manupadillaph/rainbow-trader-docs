    Script 2 Final Reporting Consistency Audit

    Created: 2026-05-23
    Scope: final reporting audit plus the staged implementation / regeneration / comparison guide for the current Script 2 Markdown report families in scope for this pass.
    Status: Closed — audit, code patch pass, staged regeneration, comparison, diff review, and canonical overwrite all completed (2026-05-24). B3 campaign verdict deferred to SCRIPT2_CLOSURE_AND_NEW_ENSEMBLE_PLAN.md.
    Decision boundary: this guide does not authorize canonical overwrite without Manu's explicit confirmation, does not regenerate extra-backtests in this pass, and does not reopen or close the separate redesigned-ensemble plan.

    1. Purpose

    This audit answers Manu's final reporting-review request for Script 2:

    • inventory the relevant generated report tables and their entities;
    • compare table schemas by entity family, not by raw text alone;
    • verify percentage-scale and monetary rendering against the ratio-scale contract;
    • identify field-format inconsistencies that need either normalization or explicit documentation;
    • check whether archived plans are completed, superseded, or still active through the current closure plan;
    • record the current extra-backtest state after the 2026-05-23 friction-stress run completed.

    The governing percentage contract remains:

    • internal percentage-like values are ratio-scale by default (0.20 = 20%), independent of _pct naming;
    • renderers multiply ratio-scale fields by 100 for display;
    • already_pct=True is reserved for true 0..100 progress/diagnostic fields such as BT Prog%;
    • cache_coverage_pct remains a documented 0..100 diagnostic exception.

    4. Schema Consistency Findings

    4.1 Modern monetary block is consistent in thesis-facing tables

    The current progress, merged progress, ensemble, and extra-backtest rendered reports all include the modern monetary block:

    PnL (Ret%) | Total Fric | Total Gross PnL | Net/trade | Gross/trade | Positive Edge Retention% | Benchmark RC (Ret%) | α (RC) | Benchmark (Ret%) | α (adj)

    Differences are entity-specific and acceptable:

    • source-trial summary tables place identity and model-quality columns before the monetary block;
    • ensemble execution tables start with Rule because the execution entity is a merge rule, not a source trial;
    • extra-backtest comparison tables prefix columns with Orig, New, or Δ because the entity is a baseline-vs-rerun pair;
    • statistical-test tables do not use the monetary block because their entity is a statistical group/test, not an executed backtest row.

    4.2 Run identity columns are coherent by entity

    • Single-run progress uses Run and ExpID.
    • Merged progress uses Runs and ExpIDs.
    • Ensemble and extra-backtest reports use TrialID (Opt#) and BktID tails for executed rows.

    This is acceptable and should not be normalized mechanically unless a future report-unification pass explicitly decides to make identity tails identical across all entity families.

    4.3 Rank and selection columns differ intentionally

    • Progress dashboards summarize the best current trial per experiment and therefore do not carry Rank.
    • Ensemble source tables include Sel Rank and Sel Score because they explain planner selection.
    • Extra-backtest plan/report tables include Rank because candidates were ranked per fold by the friction-stress selector.

    This is acceptable by entity.

    4.4 Label-format decisions

    These are not analytical errors. They are presentation decisions for thesis-facing Markdown reports. CSV/JSON data contracts should not be renamed just to make Markdown prettier.

    Field Decision Reason
    Model Use canonical thesis aliases in Markdown: LOG, MLP, MEX, TCN, VIT, AXV, RLA, RBO, RMK. Keeps cross-report comparison readable without changing raw data contracts.
    Fold Use F1, F2, F3, F4 in Markdown. Avoids mixing numeric fold IDs with thesis-facing fold labels.
    JS Div / JS Use the existing JS formatter consistently, including the js suffix. The formatter intentionally adds js; this is the canonical display style.
    Params Compact form is accepted and preferred in dense tables, for example 677.9k or 1.70M; full integers remain acceptable in profile/detail contexts. Dense thesis tables need compact values; profile tables can keep exact technical values.
    threshold columns Use short Thr labels in dense Markdown tables: Thr Orig, Thr Mild, Thr Rec, Thr Strict, Thr New. Thr is shorter and fits the existing compact table style better than full Threshold.
    generated legends Every current in-scope Markdown table header must have an explicit literal entry in _COLUMN_LEGEND_DESCRIPTIONS. Avoids fallback legends such as Report field named ... after label normalization.

    Recommended follow-up: a presentation-only report-label normalization pass for Markdown outputs. Do not change database fields or CSV/JSON schema names for this cosmetic pass.
