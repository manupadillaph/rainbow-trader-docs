    2. CRITICAL: Hardcoded Constants

    The Problem

    backtest_monetary_metrics.py has:

    SLIPPAGE_RATE = 0.0008
COMMISSION_RATE = 0.0004
INITIAL_EQUITY = 10_000.0

    What the DB Actually Has

    trades table columns (per trade):

    • entry_commission — real commission paid at entry
    • exit_commission — real commission paid at exit
    • gross_pnl — PnL before commissions
    • pnl — net PnL (after commissions)
    • ref_gross — reference gross value

    Data coverage: 727,284 of 731,048 backtest trades have entry_commission populated.

    backtest_sessions table columns:

    • initial_equity — real starting equity per session
    • total_pnl, trades_count, alpha, benchmark_return

    What This Means

    1. SLIPPAGE_RATE and COMMISSION_RATE are LIES. The DB already has real per-trade commission data (entry_commission, exit_commission). The current code ignores these and re-estimates friction from a hardcoded rate × notional volume.

    2. INITIAL_EQUITY is a DEFAULT that should ALWAYS come from the DB. backtest_sessions.initial_equity has the real value per session. The code already reads it from DB in fetch_monetary_metrics_for_trial but falls back to this constant.

    3. Friction estimation is fundamentally wrong: _friction_from_notionals computes (entry_notional + exit_notional) × (COMMISSION_RATE + SLIPPAGE_RATE) instead of using SUM(entry_commission + exit_commission) which is the REAL data.

    Proposed Fix

    Option A (minimal): Replace _friction_from_notionals with SUM(entry_commission + exit_commission) from trades. Keep SLIPPAGE_RATE only as a documentation reference, not used in actual computation.

    Option B (thorough): Also compute slippage from actual price impact (entry_price vs some reference price) using ref_gross and gross_pnl data. This is more accurate but requires understanding the ref_gross semantics.

    OPTION B! no quiero ada de fallbak ni legacy ni backward!

    RESPUESTA: Perfecto. Option B confirmada. Se eliminan las 3 constantes hardcodeadas (SLIPPAGE_RATE, COMMISSION_RATE, INITIAL_EQUITY). Se reemplaza con datos reales de la DB: entry_commission/exit_commission para comisiones, ref_gross/gross_pnl para slippage, initial_equity de backtest_sessions. Sin fallback, sin legacy, sin backward compatibility.

    3. CRITICAL: Regex-Based Experiment Name Inference

    The Problem

    Multiple files infer model, fold, tier, and code by parsing the experiment NAME string with regex:

    detect_model(name)   # "thesis-s2-tier-a-mlp-fold1" → "MLP"
detect_fold(name)    # "thesis-s2-tier-a-mlp-fold1" → "F1"
detect_tier(name)    # "thesis-s2-tier-a-mlp-fold1" → "A"
detect_code(name)    # → "S2-TA-MLP-F1"
parse_experiment_name(name)  # → ("A", "mlp", 1)

    The experiment name is a free-form string, not a standard. It's fragile, and all these functions are in reports.py (regex-based).

    Where It's Used

    Python regex (reports.py helpers):

    File Functions Used How
    audit_correlations.py detect_model, detect_fold L308-309: Inside fetch_trials()
    audit_gen_tables.py detect_code, detect_fold, detect_model, detect_tier L382-399: Inside fetch_trials()
    generate_extra_backtest_plan.py detect_model, detect_fold L365-366: Inside load_completed_trials()
    progress_time_metrics.py parse_experiment_name L257: Inside _load_historical_trials()

    SQL LIKE en WHERE (mismo problema, distinto sabor):

    File SQL Línea
    audit_correlations.py e.name LIKE '%thesis-s2-tier-b%' OR e.name LIKE '%thesis-s2-tier-a%' Dentro de fetch_trials()
    audit_gen_tables.py Idéntico LIKE Dentro de fetch_trials()

    Estos LIKE filtran por tier usando el nombre del experimento en vez de recibir los experiment_ids ya filtrados del CSV. Es el mismo anti-pattern que el regex Python, pero en SQL.
