    Script 2 Tier A Rescue Plan

    Scope

    • Rescue only Tier A.
    • Rescue only trainable models.
    • Include a fold when completed_trials < 10.
    • Keep thesis scoring, labeling, simulation, and pruning logic unchanged.
    • Do not add new model families in this pass.

    Core Changes

    1. Generator must support a separate rescue config

    Files: - scripts/thesis/script_2_thesis/generate_experiments/generate_script2_experiments.py

    Changes: - Add --config-path <path> CLI argument. - Keep current default behavior when --config-path is not provided. - Change _load_script2_configuration() to receive config_path: Path. - Stop hardcoding script2_tiers_config.json inside the generator.

    Reason: - Rescue must live in its own JSON file. - Rescue must not require temporary overwrites of the main Tier A-E config.

    2. Generator must support partial tier sets

    Files: - scripts/thesis/script_2_thesis/generate_experiments/generate_script2_experiments.py

    Changes: - Do not force A..E to exist in every config. - Validate only tiers present in the input file. - Preserve stable tier ordering using the known order A,B,C,D,E. - Make preflight operate only on configured tiers.

    Current blockers in code: - TIER_NAMES is used as a hard loop over all tiers. - _load_script2_configuration() requires every tier object. - _build_preflight_specs() assumes A..E x {rule_based, one trainable}. - _run_preflight() requires rule-based and default counts for every tier.

    Reason: - script2_tiers_config_v2_tierA_rescue.json should contain Tier A only.

    3. Generator must allow trainable-only tier configs

    Files: - scripts/thesis/script_2_thesis/generate_experiments/generate_script2_experiments.py

    Changes: - Remove the hard requirement that every tier must include rule_based. - Keep the requirement that every configured tier must include at least one trainable model. - Preflight must skip rule-based profile when the tier does not define rule_based.

    Reason: - Rescue config should not regenerate rule baselines or non-trainable controls.

    4. Add float_ranges and int_ranges

    Files: - modules/common/src/rainbow/common/types/configs/experiments/optuna_config.py - modules/model-trainer/src/rainbow/model_trainer/helpers/optuna/helpers.py

    New schema: - float_ranges - int_ranges

    Proposed JSON shape:

    {
  "type": "float_ranges",
  "ranges": [
    { "min": 0.00003, "max": 0.0008, "log_scale": true },
    { "min": 0.007, "max": 0.025, "log_scale": true }
  ]
}

    {
  "type": "int_ranges",
  "ranges": [
    { "min": 34, "max": 40 },
    { "min": 27, "max": 27 }
  ]
}

    Implementation details: - Add range-segment models for float and int ranges. - Extend OptunaParam discriminated union. - In suggest_param(), sample the segment first using an internal name like "<param_name>__range_index", then sample the final value under the original parameter name. - In extract_values_from_optuna_param(), return numeric boundaries from all segments.

    Reason: - Rescue spaces are not always continuous. - We want one rescue experiment per model, not multiple experiments only because the useful area is split.

    5. Add enqueue_trials

    Files: - modules/common/src/rainbow/common/types/configs/experiments/optuna_config.py - modules/model-trainer/src/rainbow/model_trainer/model_trainer.py

    New field: - optuna_config.enqueue_trials: list[dict[str, bool | int | float | str]] | None

    Example:

    {
  "enqueue_trials": [
    {
      "hp.window_size": 34,
      "hp.training.learning_rate": 0.00012979568644058307,
      "hp.training.batch_size": 256,
      "hp.training.weight_decay": 0.0000383705573847325,
      "hp.loss_type": "cross_entropy",
      "hp.loss.cross_entropy.label_smoothing": 0.1,
      "hp.model.mlp.hidden_dim": 256,
      "hp.model.mlp.num_hidden_layers": 2,
      "hp.model.mlp.dropout": 0.0,
      "hp.model.mlp.activation": "silu"
    }
  ]
}

    Implementation details: - Enqueue before the first study.optimize(...) call. - Use study.enqueue_trial(params=..., skip_if_exists=True). - Keep queued trials as normal Optuna trials. - Do not create synthetic DB rows. - Do not bypass pruning, metrics, or persistence.

    Reason: - Exact survivor replay should be explicit and auditable. - Neighbor replay should be explicit and auditable.

    6. Do not implement min_completed_trials in V1

    Reason: - It requires deeper changes in the optimization loop and worker-restart batching. - Rescue can run with a fixed n_trials. - Manual stop remains acceptable for this pass.
