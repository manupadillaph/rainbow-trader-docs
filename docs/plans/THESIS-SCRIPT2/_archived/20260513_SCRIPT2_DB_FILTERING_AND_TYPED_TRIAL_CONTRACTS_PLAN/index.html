    Script 2 DB Filtering And Typed Trial Contracts Plan

    Created: 2026-05-12
    Status: Completed — archived on 2026-05-13.
    Scope: Script 2 DB-backed report builders, backtest planners, and shared report helpers.

    1. Objective

    Fix the Script 2 planning/reporting contract at the source:

    1. Move heavy eligibility filters into the SQL/query layer before large trial lists, candidate records, JSON plans, or serialized payloads are built.
    2. Replace over-wide trial contracts with typed contracts that match each script's real data requirements.
    3. Resolve Ensemble Phase 4 fixed-trial planning without treating zero-trade completed backtests as friction-eligible candidates.
    4. Keep docs/plans/SCRIPT2_REPORT_CLEANING_AND_REGENERATION_PLAN.md as the downstream final verification and regeneration plan.

    This plan is an implementation/refactor plan. It does not authorize report deletion, report renaming, canonical report regeneration, DB mutation, or changes to generated artifacts.

    2. Non-Negotiable Guardrails

    • Filter in DB/query helpers whenever the filter reduces a heavy analytical or planning dataset.
    • Do not build a broad Python list and then filter it when the DB can express the eligibility condition.
    • Do not keep fallback post-query filters for conditions intentionally moved to DB.
    • Do not hide contract problems by making required analytical fields optional in the final typed object.
    • Do not use CompletedTrialResult when a script does not require prediction parquet metadata.
    • Do not modify generated results, caches, DB backups, or report artifacts unless a later approved plan explicitly lists the exact paths.
    • Keep every validation command inside the project virtual environment.
    • Put generic DB-backed trial types and converters in modules/core/src/rainbow/core/helpers/reports.py even when the first implementation has only one current caller.
    • Put shared prediction parquet preparation/loading in modules/core/src/rainbow/core/helpers/reports.py. Script folders such as ensemble_backtests and extra_backtests are task-specific consumers, not helper ownership boundaries.
    • Keep script-local types only for highly script-specific calculated rows, temporary report rows, CSV rows, or presentation-only structures.

    3. Current Blockers

    3.1 Original Ensemble Phase 4 parquet contract blocker

    The first blocker came from using CompletedTrialResult in the ensemble planner path. That type requires prediction_parquet_id and prediction_parquet_path, but the planner can select and score source trials without persisted prediction parquet metadata because the runner can reuse or regenerate predictions later.

    The existing local EnsemblePlanningTrial direction must be narrowed: generic DB-backed trial fields and validation belong in shared contracts, while a script-local planning type may add only ensemble-specific compatibility or ranking context that is not a reusable DB trial concept.

    3.2 Zero-trade friction eligibility blocker

    The newer blocker is:

    trial 2685: backtest_gross_per_trade is None on completed trial

    Read-only DB inspection identified the row as a completed zero-trade/pruned backtest. A zero-trade completed backtest can be valid as historical data, but it is not valid for friction-aware candidate ranking, rescue selection, or any metric that requires per-trade economics.

    The fix is not to allow backtest_gross_per_trade=None in a friction candidate. The fix is to exclude non-eligible rows in the DB query that creates the friction/backtest candidate universe.

    4. Shared Helper Design

    4.1 fetch_trials() DB filter API

    Extend fetch_trials() in modules/core/src/rainbow/core/helpers/reports.py with explicit SQL-backed eligibility parameters. Candidate names are intentionally descriptive so call sites document why the dataset is constrained.

    Proposed filters:

    Parameter SQL intent Main use
    min_stage3_trades_count stage3_trades_count >= :min_stage3_trades_count Friction/ranking/report datasets.
    max_stage3_trades_count stage3_trades_count <= :max_stage3_trades_count Bounded candidate universes.
    require_stage3_trades stage3_trades_count IS NOT NULL AND stage3_trades_count > 0 Per-trade metrics.
    require_backtest_gross_per_trade backtest_gross_per_trade IS NOT NULL Friction-aware scoring.
    require_backtest_net_per_trade backtest_net_per_trade IS NOT NULL Net per-trade scoring.
    require_backtest_total_gross_pnl total_gross_pnl IS NOT NULL Rescue/positive gross filters.
    min_backtest_total_gross_pnl total_gross_pnl >= :min_backtest_total_gross_pnl Rescue candidates.
    max_stage3_total_pnl stage3_total_pnl <= :max_stage3_total_pnl Rescue candidates.
    min_stage2_score Stage 2 score lower bound Ensemble candidate shortlist.
    min_stage3_score Stage 3 score lower bound Ensemble candidate shortlist.
    min_backtest_sharpe Backtest Sharpe lower bound Ensemble candidate shortlist.

    Implementation notes:

    • Validate incompatible filter combinations before building SQL.
    • Add SQL predicates to where_parts; do not post-filter returned rows for these conditions.
    • Use named parameters in the existing parameter dictionary.
    • Preserve existing default behavior when every new filter argument is omitted.
    • Keep status/as-of/training/checkpoint/objective filters in the same helper instead of duplicating query logic.

    4.2 Typed trial contracts

    Create generic DB-backed contracts in the shared helper layer when they describe reusable trial facts from the database, even if the first caller is a single script. Create script-local types only when they represent a specific script's calculated row, temporary report row, CSV row, or presentation-only shape.

    • TrialResult
    • Required fields: raw nullable DB projection.
    • Purpose: general fetch result and all-trial reporting.
    • CompletedBacktestTrialResult
    • Required fields: completed trial fields plus required backtest summary fields.
    • Purpose: backtest/report logic that needs completed backtest metrics without prediction parquet metadata.
    • FrictionEligibleBacktestTrialResult
    • Required fields: completed backtest fields, positive trade count, and required per-trade economics.
    • Purpose: friction-aware ranking, rescue, and ensemble candidate scoring.
    • CompletedPredictionTrialResult or existing CompletedTrialResult
    • Required fields: prediction parquet fields.
    • Purpose: only scripts that truly need persisted prediction parquet metadata.

    Converter functions:

    • to_completed_backtest_trial_result(...)
    • Input: TrialResult.
    • Output: CompletedBacktestTrialResult.
    • Notes: no prediction parquet requirement.
    • to_friction_eligible_backtest_trial_result(...)
    • Input: TrialResult.
    • Output: FrictionEligibleBacktestTrialResult.
    • Notes: assumes DB filters ran, then validates hard.
    • Existing to_completed_trial_result(...)
    • Input: TrialResult.
    • Output: CompletedTrialResult.
    • Notes: keep only for persisted prediction-backed use cases.

    The converters are defensive validation, not the primary filtering mechanism. If a converter fails because an eligible query returned an invalid row, that indicates a query/schema contract bug that must be fixed at the DB predicate layer.

    4.3 Shared prediction parquet helpers

    Prediction parquet preparation/loading belongs in reports.py when it is shared by Script 2 tasks. Task scripts may decide when predictions are needed, but they must not own reusable parquet resolution, regeneration, or loading logic.

    • prepare_prediction_parquet_for_trial(...) exposes reusable-or-regenerated prediction parquet preparation.
    • load_prediction_frame(...) exposes full prepared prediction-frame loading for tasks that need all prediction fields.
    • load_prediction_parquet_columns(...) loads selected prepared prediction columns through the shared parquet reader.
    • load_prediction_signal_quality_frame(...) loads the signal/quality subset used by threshold-band analysis.
