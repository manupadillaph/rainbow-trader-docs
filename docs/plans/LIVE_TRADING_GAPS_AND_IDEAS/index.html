    Live Trading — Gaps, Ideas & Recommendations

    What's missing, what could be better, and what production live trading systems typically need.

    Gaps (verified against real code)

    G1 — Provider submit_order() not implemented

    data_fetcher.submit_order() raises NotImplementedError for all providers. LIVE_REAL generates orders correctly but they always end as REJECTED. Need at least one provider (Binance) implemented.

    G2 — No exchange equity sync

    Equity starts from runtime_config.initial_equity and only tracks internal operations. No fetch of real balance from exchange. Diverges if there are external trades, manual interventions, or partial fills different from requested size.

    G3 — No position reconciliation on startup

    When the runner restarts, it loads open positions from local DB only. No check against exchange to detect orphaned positions (opened outside runner) or stale positions (closed on exchange but still open in DB).

    G4 — No fill confirmation / async fill handling

    LIVE_REAL assumes provider accepted = filled at requested price. Real exchanges return fills asynchronously, potentially at different prices and partial sizes. No websocket listener for fill events.

    G5 — Force-close not implemented for LIVE_REAL

    Plan T3 covers LIVE_SIM force-close. LIVE_REAL needs either a proper implementation or an explicit fail-fast.

    G6 — No circuit breaker / kill switch

    If the model starts generating bad signals or the market goes haywire, there's no automated emergency stop beyond risk_manager.should_force_flat(). No external kill switch endpoint that immediately halts all order submission.

    Ideas & Recommendations

    R1 — Exchange balance sync on startup + periodic

    On startup and every N ticks, query exchange balance and compare to internal equity. Log warning if drift exceeds threshold. Optional: auto-correct internal equity to match exchange.

    R2 — Position reconciliation service

    On startup: fetch open positions from exchange, compare to DB, log discrepancies. Options: auto-close orphans, auto-record unknowns, or halt and notify.

    R3 — Async fill listener (websocket)

    For LIVE_REAL: subscribe to exchange fill/order websocket. Update internal state with actual fill price, actual fill size, and partial fill handling. This is the hardest piece but critical for production.

    R4 — Order status tracking

    Track each submitted order through lifecycle: SUBMITTED → ACKNOWLEDGED → PARTIALLY_FILLED → FILLED / CANCELLED / REJECTED. Persist in DB for audit trail.

    R5 — Circuit breaker / emergency stop

    • Max consecutive losses → auto-halt
    • Max drawdown in N minutes → auto-halt
    • External API endpoint /emergency-stop → immediate halt, cancel all pending, optionally close all positions
    • Cooldown period before auto-resume

    R6 — Heartbeat & health monitoring

    • Runner emits periodic heartbeat (to DB or external service)
    • Alert if heartbeat stops (runner crashed without clean shutdown)
    • Alert if no candle received in 2x expected timeframe (stream stalled)
    • Dashboard integration via control-panel

    R7 — Order rate limiting

    Prevent accidental order storms. Max N orders per minute/hour per instance. Configurable per mode.

    R8 — Slippage monitoring (LIVE_REAL)

    Compare requested price vs actual fill price. Track slippage statistics over time. Alert if slippage exceeds threshold consistently.

    R9 — Paper trading mode

    A middle ground: submit to exchange paper/testnet API instead of real. Same code path as LIVE_REAL but no real money. Useful for validation before going live.

    R10 — Notification system

    Alerts on: trade opened/closed, force-close triggered, risk halt, error, daily P&L summary. Channels: Telegram, email, webhook.

    R11 — Graceful shutdown with position protection

    On SIGTERM/SIGINT: stop accepting new signals, wait for pending orders to resolve, persist final state, log clean exit. Never leave the system in an ambiguous state.

    R12 — Multi-exchange support

    Abstract provider interface to support Binance, Bybit, etc. Provider-specific implementations behind a common protocol.
