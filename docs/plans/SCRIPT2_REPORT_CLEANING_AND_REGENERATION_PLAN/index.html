    Script 2 Report Cleaning And Regeneration Plan

    Created: 2026-05-06
    Status: Closed on 2026-05-16. Cleanup scope is complete; any future manual rewrites or new ensemble work require a separate follow-up plan.
    Scope: scripts/thesis/script_2_thesis/**, excluding _OLD, OLD, _BK, BK, _archived, and archived unless an exact path is listed here.

    1. Objective

    Clean the active Script 2 report tree and leave one canonical output per report.

    Target state:

    1. Every non-ensemble result root being republished is preserved as <name>_LEGACY/.
    2. Inside those preserved legacy roots, the preserved published report files are renamed with *_LEGACY.
    3. Every temporary regenerated variant (NEW, NEW1, NEW2, V2, V3, V3b, V3c, V3d, V4) is removed.
    4. Every non-ensemble script-generated result root is regenerated clean at the original canonical path; legacy and regenerated outputs never coexist inside the same root.
    5. The two historical standalone timing reports are preserved only as legacy files; the fresh canonical run roots keep only the timing sidecar emitted by script2_progress_tracker.py.
    6. Every manual Script 2 document in scope is renamed to *_LEGACY.md, receives an opening legacy/scope note, and is tracked explicitly in §8.1. Manual rewrites are deferred; Manu will decide later which fresh documents to create.
    7. No destructive step uses wildcards for report files or report folders.

    Clarification: in this cleanup plan, _LEGACY is the archival-preservation label applied to preserved result roots and preserved report files before rewrite/regeneration. It does not automatically mean that every statement inside that artifact is conceptually invalid; some roots/files are renamed because they are superseded originals, while others are legacy in the stronger historical/business-logic sense.

    2. Guardrails

    • Exact paths only for rm, mv, git mv, and canonical regeneration outputs.
    • For non-ensemble results and extra-backtest publish roots, preserve the old root by renaming it to <name>_LEGACY; regenerate a fresh clean canonical root at the original path instead of mixing legacy and regenerated outputs in one folder.
    • Protected inputs are never deleted. For non-ensemble results and extra backtests they can move only because the whole legacy root is renamed; individual file renames inside those roots are limited to the preserved published report artifacts listed in §7 and §8. Manifests, configs, summary CSVs, event logs, persisted results.csv / results.json, and raw experiment trees are otherwise not renamed individually. Current ensemble batch specs are the explicit exception in §9 because that cycle is being frozen as _LEGACY.
    • Every baseline report stays read-only during Phase 3. Probes write only under refactor/tmp_regen/.
    • All non-ensemble reports in scope must be regenerated, and all regenerated reports must be compared against their baseline.
    • Existing compare scripts are mandatory first-pass tools. If a compare script does not cover all visible sections, that is a tool limitation, not a scope reduction; the remaining visible data still must be checked before approval.
    • Manual documents are not regenerated automatically. They remain legacy until Manu chooses which fresh manual replacements to write.
    • Final clean run roots rely on the single-run progress exporter timing sidecar. Historical standalone timing reports are not compared or regenerated in the final publish.
