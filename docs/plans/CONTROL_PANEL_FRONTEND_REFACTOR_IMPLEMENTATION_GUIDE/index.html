    Control Panel Frontend Refactor — Implementation Guide

    Decision Update — Study Naming

    On 2026-03-24, the canonical naming decision was revised. study / studies / studyId now supersede the earlier training / trainings / trainingId target wording in this guide. The change is intentional and reflects the real project domain (OptunaStudy, study_id, study_name). Any older execution note that still mentions the opposite rename direction must be interpreted as obsolete.

    Purpose

    This is the operational playbook for executing the refactor. It is a companion to the architecture reference (CONTROL_PANEL_FRONTEND_REFACTOR_SUPERPLAN.md) and the per-file decision log (CONTROL_PANEL_FRONTEND_REFACTOR_SUPERPLAN_FINAL_TREE_MATRIX.md).

    • Architecture laws, naming conventions, U-marks: see the Superplan
    • Per-file decisions (283 rows): see the Matrix
    • How to execute, phase by phase, with concrete tasks: this document

    Execution Mode And Working Directory

    • All shell commands in this guide assume the working directory is modules/control-panel/frontend/
    • The only per-file source of truth is CONTROL_PANEL_FRONTEND_REFACTOR_SUPERPLAN_FINAL_TREE_MATRIX.md
    • CONTROL_PANEL_FRONTEND_REFACTOR_SUPERPLAN_FINAL_FILE_TREE.txt is the final target tree companion artifact and must remain synchronized with the matrix
    • Preferred execution mode is clean-build mode: preserve the current tree as src_REFERENCE/ and assemble the final architecture in a new clean src/
    • Temporary intermediate breakage during execution is acceptable; the success criterion is the final verified result, not preserving buildability at every intermediate step

    Non-Negotiable Agent Rules

    Read these before writing a single line. These rules have caused real damage in past sessions when violated.

    R1 — Read the full file before editing

    For every Split, Merge, or content-change operation: read the entire source file first. Understand its structure, naming, and all existing comments. Do not start editing without this step.

    R2 — Comments are absolutely protected

    • NEVER delete, rewrite, shorten, paraphrase, or relocate existing comments
    • NEVER clean up or simplify code that is outside the scope of the current operation
    • When moving code: copy comments line-for-line to their equivalent position in the target
    • Before finishing any Split or Merge: review the diff specifically for removed comment lines and restore any that were lost
    • If a refactor makes an existing comment look outdated or misplaced: STOP and ask

    R3 — Scope discipline

    When moving a file: touch only that file and its direct import references. Do not refactor unrelated code at the same time.

    R4 — No error suppressors

    Never use // @ts-ignore, // @ts-expect-error, /* eslint-disable */, or any suppression pragma. If TypeScript raises an error after a move, fix the root cause.

    R5 — Create _OLD/ backup before large rewrites

    For every Split and Merge:

    • In clean-build mode, src_REFERENCE/ is the primary full-tree backup
    • If you need an additional local safety copy while extracting a large file into the new src/, create an _OLD/ copy in any consistent path before continuing

    R6 — Explain before executing

    Before starting each phase: state what you are about to do, which files will be affected, and what the expected result is. Do not execute without this pre-announcement.

    R7 — Never commit without explicit instruction

    Do not run git commit or git push at any point. Use git only for status/diff inspection.

    R8 — No Any, no untyped dicts, no untyped returns

    • Never use Any as a type annotation
    • Never use raw dict without explicit TypedDict or generic parameters (dict[str, T])
    • Never leave function returns untyped — always specify the exact return type
    • Prefer typed classes: Pydantic models, dataclasses, or TypedDict over raw dictionaries
    • If you must use Any, document why with a comment explaining the external constraint

    Example of what to avoid:

    // WRONG
function parseData(response: any): any { ... }
const config: dict = loadConfig();

// CORRECT
interface Config {
  apiUrl: string;
  timeout: number;
}
function parseData<T>(response: ApiResponse<T>): T { ... }
const config: Config = loadConfig();

    Merge Pairs — Critical Warning

    Three merge groups exist where TWO matrix rows share ONE target file. Processing rows sequentially without this knowledge will either create the target twice or overwrite the first result with the second.

    How to process a merge pair:

    1. Process the PRIMARY row first (the row whose logic is the main body of the target)
    2. Read the SECONDARY row's source file and absorb its logic into the target from step 1
    3. Delete the secondary source file
    4. Mark BOTH matrix rows ✅

    Phase 2 — Pure Moves (Keep / Move / Rename, no content changes) ⬜

    Process all matrix rows with Decision = Keep, Move, Rename, or Move+Rename (no Split, no Merge). File content is not modified — only path changes. After moving each file, mark the matrix row ✅.

    In clean-build mode, these are copy/create operations from src_REFERENCE/ into the new src/ target path, not in-place filesystem moves.

    Route folder renames ([studyId] → [trainingId]) are handled in Phase 5, not here. Do not process those rows in this phase.

    After moving each batch of files, run the import update strategy below for that batch. Do not defer all import fixes to the end.

    Import update strategy (apply after each file move)

    # For each file moved from OLD_PATH to NEW_PATH:
grep -r "OLD_PATH" src/ --include="*.ts" --include="*.tsx" -l
# Then update each found file to import from NEW_PATH instead

    After completing all moves in this phase:

    # Sweep for any remaining stale old paths
grep -r "client-pages/" src/ --include="*.ts" --include="*.tsx"
grep -r "hooks/dashboard/" src/ --include="*.ts" --include="*.tsx"
grep -r "components/dashboard/shared/" src/ --include="*.ts" --include="*.tsx"

    Verification: No broken import paths remain for moved files. grep sweeps return empty.

    Phase 3 — Execute Splits ⬜

    For each Split row:

    1. Create a backup copy of the source file in an _OLD/ location if you want an additional local safety checkpoint
    2. Read the entire source file
    3. Create the target files (both the renamed primary and all new secondary files from the New-File Manifest above)
    4. Move logic to each new owner, copying all comments line-for-line
    5. Delete or hollow out the source (if it is not itself a target)
    6. Mark matrix row ✅
    7. Run import update strategy for the new paths

    In clean-build mode, read source files from src_REFERENCE/ and create all targets under the new src/.
