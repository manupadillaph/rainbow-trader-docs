    Quick Start

    # 1. Copy environment template
cp .env.example .env

# 2. Edit your values
nano .env

# 3. Run any module (config loads automatically)
python -m database migrate

    Priority Order

    Configuration values are merged from multiple sources. Higher priority wins.

    Priority Source Example
    1 kwargs to Config class LogConfig.from_config(config_obj, level="DEBUG")
    2 CLI --set flags --set log.level=DEBUG
    3 Config file (-c/--config) --config custom.yaml
    4 Environment variables LOG__LEVEL=DEBUG
    5 YAML file per config class config/log.yaml
    6 DEFAULTS class variable DEFAULTS = {"level": "INFO"}

    Configuration Sources

    1. Environment Variables (.env)

    Located at project root. Use __ (double underscore) for nesting.

    # .env
LOG__LEVEL=INFO
LOG__FORMAT=TXT
LOG__TARGET=TERMINAL
DATABASE__SQLITE_PATH=rainbow_trader.sqlite
FOLDERS__CONFIG=./config
FOLDERS__DATABASE=./database
FOLDERS__DATA=./data

# API Keys
KEYS__OPENAI_API_KEY=sk-...

    Mapping:

    Env Variable Becomes In YAML
    LOG__LEVEL {"log": {"level": ...}} log.level
    LOG__FORMAT {"log": {"format": ...}} log.format
    LOG__TARGET {"log": {"target": ...}} log.target
    DATABASE__SQLITE_PATH {"database": {"sqlite_path": ...}} database.sqlite_path
    FOLDERS__DATA {"folders": {"data": ...}} folders.data
    KEYS__OPENAI_API_KEY {"keys": {"openai_api_key": ...}} keys.openai_api_key

    2. YAML Config Files

    Each BaseConfig subclass automatically loads its YAML file from config/.

    Config Class YAML File Section Key
    LogConfig config/log.yaml log
    DatabaseConfig config/database.yaml database
    DataProviderConfig config/data-provider.yaml data_provider

    Example config/log.yaml:

    log:
  level: INFO
  format: TXT

    3. CLI Flags

    Override any value at runtime:

    # Single override
python -m database --set log.level=DEBUG migrate

# Multiple overrides
python -m database --set log.level=DEBUG --set log.format=json migrate

# Use custom config file
python -m database --config custom.yaml migrate

# Inline JSON
python -m database --config '{"log": {"level": "DEBUG"}}' migrate

    4. DEFAULTS Class Variable

    Lowest priority. Defined in each config class:

    class LogConfig(BaseConfig):
    DEFAULTS: ClassVar[dict[str, Any]] = {
        "level": "INFO",
        "format": "txt",
    }

    level: str
    format: str

    BaseConfig Classes

    All configuration classes inherit from BaseConfig (Pydantic BaseModel).

    Creating a Config Class

    from typing import Any, ClassVar
from rainbow.common.types.configs.base_config import BaseConfig

class MyConfig(BaseConfig):
    """My module configuration."""

    # Default values (lowest priority)
    # Fields listed here will NOT fail validation
    DEFAULTS: ClassVar[dict[str, Any]] = {
        "timeout": 30,   # ← Has default, won't fail
        "retries": 3,    # ← Has default, won't fail
    }

    # Fields (type hints required)
    timeout: int         # ← Covered by DEFAULTS
    retries: int         # ← Covered by DEFAULTS
    api_key: str         # ← NOT in DEFAULTS = ValidationError if missing!

    What happens:

    • timeout → Uses 30 from DEFAULTS if not provided elsewhere
    • retries → Uses 3 from DEFAULTS if not provided elsewhere
    • api_key → Raises ValidationError if not in env, YAML, CLI, or config_obj

    Using a Config Class

    from rainbow.common.helpers import Config

# Load merged config from all sources
config_obj = ctx.obj = {
        ()

# Create typed config instance
my_config = MyConfig.from_config(config_obj)

# Access values with type safety
print(my_config.timeout)  # int
print(my_config.api_key)  # str

    Automatic Naming

    Class names are automatically converted:

    Class Name Section Key YAML File
    LogConfig log log.yaml
    DatabaseConfig database database.yaml
    DataProviderConfig data_provider data-provider.yaml
    MyCustomConfig my_custom my-custom.yaml

    Nested Configs

    Parent configs can contain child configs:

    class ChildConfig(BaseConfig):
    DEFAULTS: ClassVar[dict[str, Any]] = {"value": 100}
    value: int

class ParentConfig(BaseConfig):
    DEFAULTS: ClassVar[dict[str, Any]] = {"name": "parent"}
    name: str
    child: ChildConfig  # Automatically initialized

    Dict of Configs with Inheritance

    For dict[str, BaseConfig] fields, use InheritFrom to share defaults and KeyToType for specific types:

    from typing import Annotated
from rainbow.common.types.configs.base_config import BaseConfig, InheritFrom, KeyToType

# Base with shared fields
class BaseProviderConfig(BaseConfig):
    retry_max_attempts: int
    retry_backoff_secs_base: int

# Specific providers can add their own fields
class BinanceProviderConfig(BaseProviderConfig):
    api_key: str | None = None

class BybitProviderConfig(BaseProviderConfig):
    testnet: bool = False

# Map YAML keys to specific types
PROVIDER_TYPE_MAP = {
    "binance": BinanceProviderConfig,
    "bybit": BybitProviderConfig,
}

class DataProviderConfig(BaseConfig):
    base_provider: BaseProviderConfig  # Shared defaults

    # InheritFrom: each provider inherits from "base_provider"
    # KeyToType: "binance" → BinanceProviderConfig, "bybit" → BybitProviderConfig
    providers: Annotated[
        dict[str, BaseProviderConfig],
        InheritFrom("base_provider"),
        KeyToType(PROVIDER_TYPE_MAP)
    ]

    YAML example:

    data_provider:
  base_provider:
    retry_max_attempts: 5
    retry_backoff_secs_base: 2

  providers:
    binance:
      retry_max_attempts: 3  # Override base
      api_key: "xxx"         # BinanceProviderConfig-specific
    bybit:
      testnet: true          # BybitProviderConfig-specific
      # retry_* inherited from base_provider

    Result:

    • binance: BinanceProviderConfig(retry_max_attempts=3, retry_backoff_secs_base=2, api_key="xxx")
    • bybit: BybitProviderConfig(retry_max_attempts=5, retry_backoff_secs_base=2, testnet=True)

    Naming Conventions

    Environment Variables

    # Pattern: SECTION__KEY=value
LOG__LEVEL=DEBUG
DATABASE__SQLITE_PATH=rainbow_trader.sqlite
FOLDERS__DATA=./data
KEYS__OPENAI_API_KEY=sk-...

    CLI Flags

    # Pattern: --set section.key=value
--set log.level=DEBUG
--set database.sqlite_path=rainbow_trader.sqlite
--set keys.openai_api_key=sk-...

    Missing Fields = Error

    CRITICAL: If a required field has no value from ANY source, Pydantic raises ValidationError.

    class MyConfig(BaseConfig):
    # NO DEFAULTS defined

    required_field: str      # MUST come from somewhere
    optional_field: str | None = None  # Has Python default, OK

    # This FAILS with ValidationError
MyConfig.from_config()
# Error: "required_field" is missing

# This WORKS
config_obj = {"my": {"required_field": "value"}}
MyConfig.from_config(config_obj)

    Why this matters:

    • No silent failures with undefined values
    • Users know exactly which fields need configuration
    • Configuration is explicit, never implicit

    Sources that satisfy a required field:

    1. kwargs to constructor
    2. CLI --set flag
    3. Config file (YAML/JSON)
    4. Environment variable
    5. YAML file for the config class
    6. DEFAULTS class variable
    7. Python default value on the field itself

    deep_merge Behavior

    When merging configs, deep_merge has specific rules:

    Scenario Result
    Empty dict {} Does NOT clear existing values
    Explicit None DOES overwrite (when intentional)
    Missing key Preserves existing value

    Why this matters for Optional fields:

    class SimulationConfig(BaseConfig):
    random_seed: int | None  # None = use random, int = reproducible

    # DEFAULTS
{"simulation": {"random_seed": 123}}

# User config with explicit None
{"simulation": {"random_seed": None}}

# Result: random_seed = None (user's choice wins)

    Examples

    Example 1: Simple Module Config

    # my_module/config.py
from typing import Any, ClassVar
from rainbow.common.types.configs.base_config import BaseConfig

class MyModuleConfig(BaseConfig):
    DEFAULTS: ClassVar[dict[str, Any]] = {
        "batch_size": 32,
        "learning_rate": 0.001,
    }

    batch_size: int
    learning_rate: float
    model_path: str  # Required - no default

    # config/my-module.yaml
my_module:
  model_path: ./models/default.pt

    # Override at runtime
python -m my_module --set my_module.batch_size=64 train

    Example 2: Enum Conversion

    from enum import Enum

class LogLevel(str, Enum):
    DEBUG = "DEBUG"
    INFO = "INFO"
    WARNING = "WARNING"
    ERROR = "ERROR"

class LogConfig(BaseConfig):
    DEFAULTS: ClassVar[dict[str, Any]] = {"level": "INFO"}

    level: LogLevel  # Automatic case-insensitive conversion

    # All of these work:
LOG__LEVEL=debug
LOG__LEVEL=DEBUG
LOG__LEVEL=Debug

    Example 3: Full Priority Chain

    # 1. DEFAULTS (lowest)
DEFAULTS = {"level": "INFO"}

# 2. YAML file
# config/log.yaml
log:
  level: WARNING

# 3. Environment variable
LOG__LEVEL=ERROR

# 4. CLI flag (highest)
--set log.level=DEBUG

# Result: level = "DEBUG"
