    1. System Identity

    Rainbow Trader is an autonomous research–training–execution system for quantitative trading.

    Core Philosophy

    1. Brains vs. Brawn: We separate Cognitive Logic (LangGraph Agents) from Heavy Compute (Optimized Python Binaries).
    2. Stateful Orchestration: The system is a persistent LangGraph, not a fragile script loop. It survives crashes and resumes exactly where it left off via Checkpointers.
    3. Event-Driven Bridge: Compute modules signal completion via DB events, which wake up the suspended Graph.
    4. Reproducibility: Everything is reproducible via commit hashes, plan hashes, and immutable datasets.

    Core Ideas

    • Experiments are defined in structured configs (ExperimentPlan)
    • Data pipelines build reusable FAT and SMALL windows
    • Optuna-based training explores features and hyperparameters
    • Models are evaluated via backtesting and live execution
    • Multi-agent LLMs (director, planner, analyst, etc.) decide what to try next
    • Everything is reproducible via commit hashes, plan hashes, and metadata

    Experiment Types

    • experiment_type = "trading" → model outputs trading signals
    • experiment_type = "regime" → model outputs market regime labels

    2. Architecture Overview

    The system is divided into three layers:

    2.1 Cognitive Layer (The Brain)

    3.3 Graph State (Short-term Memory)

    The shared state across all nodes in a single thread (conversation/workflow).

    from typing import TypedDict, Annotated, List, Optional, Dict, Any
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

class TodoItem(TypedDict):
    task: str           # e.g., "Analyze experiment results"
    status: str         # "pending", "in_progress", "completed"

class ActiveOperation(TypedDict):
    task_type: str       # "training", "data_fetch", "labeling", etc.
    experiment_id: str
    trial_id: Optional[str]
    started_at: str     # ISO 8601

class AgentState(TypedDict):
    # Core conversation history (auto-appended by LangGraph)
    messages: Annotated[List[BaseMessage], add_messages]

    # Current context
    current_experiment_id: Optional[str]
    current_trial_id: Optional[str]
    current_plan_id: Optional[str]

    # Active async operations (modules running in background)
    active_operations: List[ActiveOperation]

    # Agent's task list (within a single invocation)
    todos: List[TodoItem]

    # Budget tracking snapshot
    budget_stats: Dict[str, float]  # {"daily_spend": 1.23, "limit": 5.0}

    # Explicit routing override (if set, next node is forced)
    next_node: Optional[str]

    # Error tracking for retry logic
    error_count: int

    Thread Policy:

    Rainbow Trader uses a single global thread for the entire system:

    RAINBOW_THREAD_ID = "rainbow_main"  # Global thread for all operations

    Why one thread:

    • The system is ONE autonomous agent, not multiple users/sessions
    • Graph State tracks active_experiments[] to manage multiple experiments
    • Event Bridge always resumes the same thread
    • Simpler architecture: all state in one place

    Parallelism Architecture:

    Component Instance Count Notes
    Orchestrator 1 (singleton) Single LangGraph instance
    Agent Nodes 1 each (singleton) director, planner, analyst, coder, etc.
    Compute Modules Multiple (parallel) Each can spawn multiple processes

    Compute Module Parallelism:

    • data-fetcher: Multiple symbols/timeframes in parallel (with file locking)
    • labeling-service: Multiple experiments in parallel
    • window-preparer: Multiple experiments and trials in parallel
    • model-trainer: Multiple trials in parallel (via max_concurrent_trials)
    • backtesting-engine: Multiple trials in parallel
    • live-trading-runner: Multiple bots in parallel (one per ExecutionPlan)

    Example: The director-node (singleton) calls trigger_training(exp_123), which spawns 5 parallel training processes in model-trainer.

    Hash algorithms for cache lookup:

    All hashes are deterministic and do NOT include experiment_id to enable reuse across experiments.

    def compute_label_cache_key(labeling_config: dict, base_symbol: str, base_timeframe: str, time_range: dict) -> str:
    """
    Hash for labels cache. Same labeling config + symbol + timeframe + range = reusable labels.
    """
    data = {
        "labeling_config": labeling_config,  # includes horizon, thresholds (sampling_config is separate)
        "base_symbol": base_symbol,
        "base_timeframe": base_timeframe,
        "time_range": time_range  # {"start": ..., "end": ...}
    }
    json_str = json.dumps(data, sort_keys=True)
    return hashlib.sha256(json_str.encode()).hexdigest()[:16]


def compute_fat_cache_key(features_config: List[dict], window_size: int, label_cache_key: str) -> str:
    """
    Hash for FAT window cache.
    FAT windows depend on: all features, max window_size, and which labels (timestamps).
    """
    data = {
        "features": sorted([f["name"] for f in features_config]),
        "window_size": window_size,
        "label_cache_key": label_cache_key  # links to specific labels/timestamps
    }
    json_str = json.dumps(data, sort_keys=True)
    return hashlib.sha256(json_str.encode()).hexdigest()[:16]


def compute_small_cache_key(feature_subset: List[str], window_size: int, batch_size: int) -> str:
    """
    Hash for SMALL window cache.
    SMALL windows depend on: feature subset, window size, AND batch_size (affects file organization).
    """
    data = {
        "features": sorted(feature_subset),
        "window_size": window_size,
        "batch_size": batch_size
    }
    json_str = json.dumps(data, sort_keys=True)
    return hashlib.sha256(json_str.encode()).hexdigest()[:16]

    5.1 director-node (Router/Supervisor)

    The Boss. Coordinates the high-level workflow. Decides "What next?".

    Model Tier: Tier 3 (Genius) — requires maximum intelligence.

    Triggered when:

    • User sends a message to the Graph
    • Event Bridge resumes the Graph after a module completes
    • Another node routes back to Director

    Responsibilities:

    • Reads incoming messages/events
    • Decides next action:
    • New ExperimentPlans → delegates to planner-node
    • Analyze results → delegates to analyst-node
    • Fix code issue → delegates to coder-node
    • Stop or modify running work → uses tools directly
    • Improve system → delegates to research-node then coder-node
    • Uses summaries from The Store for context
    • Reads high-level state from long-term memory

    Self-Improving Code Flow: See PIPELINE-FLOWS.md Section 8 for the complete flow diagram.

    Tools available:

    • stop_experiment(experiment_id, reason) → sets experiments.status = "stopping" + emits event
    • stop_trial(experiment_id, trial_id, reason) → sets trials.status = "stopping" + emits event
    • prioritize_experiment(experiment_id) → adjusts experiment priority
    • update_master_execution_plan(changes) → modifies MasterExecutionPlan (HITL required)
    • read_from_store(namespace, query) → semantic search in long-term memory
    • write_to_store(namespace, key, value) → save insight to long-term memory

    Routing (Conditional Edges):

    11. Memory, Persistence & LLM Integration

    11.1 Graph State (Short-term Memory)

    • Mechanism: LangGraph StateGraph with messages list.
    • Content: Current conversation history, active plan, recent tool outputs.
    • Persistence: Checkpointed to DB after every step via Checkpointer.
    • Benefit: Safe to restart orchestrator at any time. Resumes exactly where it left off.

    11.2 The Store (Long-term Memory)

    • Mechanism: LangGraph BaseStore (KV Store with optional semantic search).
    • Namespaces:
    • ("insights", "experiments"): Lessons learned from experiments.
    • ("insights", "features"): Which features work best.
    • ("user", "preferences"): User-specified preferences.
    • ("system", "errors"): Common failure patterns.
    • Usage: Agents query the Store to retrieve context before acting.
    # Writing to Store
store.put(("insights", "experiments"), experiment_id, {
    "summary": "XGBoost outperformed LSTM on BTC/5m",
    "best_features": ["RSI_14", "MACD_signal"],
    "regime": "high_volatility"
})

# Reading from Store (semantic search)
results = store.search(("insights", "experiments"), query="What works in high volatility?")

    11.3 Checkpointer (Persistence)

    • Tool: SqliteSaver (dev) / PostgresSaver (prod).
    • Function: Saves the entire Graph state after every node execution.
    • Benefit: "Time Travel" debugging, resumable workflows, crash recovery.

    11.4 Human-in-the-Loop (HITL)

    • Mechanism: interrupt_before on sensitive tools.
    • Workflow:
    • Agent proposes action.
    • Graph hits interrupt, state saved.
    • Control Panel shows approval request.
    • User approves/rejects.
    • Control Panel sends Command(resume=decision).
    • Graph executes or skips tool.

    11.5 Context Window Management

    Before invoking any agent node:

    1. Calculate available tokens: max_context - reserve_for_response
    2. Priority-based loading: Current task > Recent messages > Summaries > Historical
    3. If exceeds budget: Summarize older items using Tier 1 model.
    4. Large tool results: Write to file, return reference.

    11.6 Error Recovery

    Orchestrator crash recovery:

    1. Checkpointer has saved state after every step.
    2. On restart, read last checkpoint from DB.
    3. Resume from exact point of failure.
    4. Event Bridge resumes processing unprocessed events.

    Agent error handling:

    • Retry up to max_consecutive_agent_errors.
    • If still failing: Create alert for USER, pause workflow.

    14. Final Notes

    • bot-logic is explicitly responsible for transforming model outputs + bot_config into concrete orders through the bot decision flow (generate_exit_orders -> generate_entry_signals -> RiskManager.evaluate_entries).

    • ExperimentPlan and ExecutionPlan both contain bot_config (for backtesting and live/Sim).

    • OHLCV and indicators are stored in the structured layout under data/OHLCV and data/indicators.

    • FAT and SMALL windows are cached to enable re-use across trials and experiments:

    • FAT windows: per experiment, full feature set

    • SMALL windows: per trial config (feature_subset, window_size, batch_size)

    • All major steps emit explicit events so the Event Bridge can resume the Graph.

    • batch_size is a tunable parameter in dataset_config, and Optuna can optimize it via batch_size_range.

    • Feature hash applies to both FAT and SMALL windows for cache lookup.

    Key Architectural Points:

    1. The Graph is the Brain: LangGraph manages all agent workflows, persistence, and HITL.
    2. Modules are the Muscle: Heavy compute runs as separate processes (subprocesses), communicates via events.
    3. Events are the Bridge: events table + Event Bridge connects async modules to the stateful Graph.
    4. The Store is the Memory: Long-term knowledge lives in LangGraph Store, agents query it before acting.
    5. Checkpointer is the Safety Net: Crash recovery is native, no custom checkpointing needed.
    6. Live-trading-runner is Orchestrated: The Graph starts/stops/reloads it, not manual.

    Director-node is the decision maker:

    • Triggered by Event Bridge or user input.
    • Reads messages/events, analyzes, decides.
    • Uses tools to execute decisions.
    • Delegates to specialist nodes via routing edges.
    • Can trigger self-improving code flow (research → coder → deploy).

    Self-Improving System: See PIPELINE-FLOWS.md Section 8 for the complete flow.

    Trainer-engine polls DB for status:

    • Checks trials.status periodically during training.
    • If status = "stopping" → graceful stop → status = "cancelled".
    • No need to listen to events for stop signals.

    The system is ready for the orchestrator and agents to start generating code and documentation per repo.
