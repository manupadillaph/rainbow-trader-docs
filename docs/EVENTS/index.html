    Sections

    1. Overview

    All events are stored in events table with JSON payload. Event types MUST use the names below (prefer *_DONE for completion signals). Payloads should include the minimal keys listed so downstream consumers and the control panel stay stable. Dummy runs are acceptable as long as the payload shape is correct.

    2. Canonical Event Types

    Use exactly these names (case-sensitive):

    Data Preparation:

    • DATA_FETCH_DONE, LABELS_DONE, FAT_WINDOWS_DONE, SMALL_WINDOWS_DONE

    Per-Trial:

    • TRIAL_CONFIG_READY, SMALL_WINDOWS_PROGRESS, SMALL_WINDOWS_DONE, TRAINING_PROGRESS, TRIAL_TRAINING_DONE, TRIAL_PRUNED, TRIAL_SCORED, ANALYST_TRIAL_SUMMARY_READY

    Backtest:

    • BACKTEST_TRIAL_DONE

    Experiment Closure:

    • ALL_TRIALS_DONE, ANALYST_EXPERIMENT_SUMMARY_READY

    Live Execution:

    • EXECUTION_PLAN_CREATED, MASTER_EXEC_PLAN_UPDATED, LIVE_EXEC_SUMMARY, LIVE_TRADING_STARTED, LIVE_TRADING_STOPPED, LIVE_ANOMALY_DETECTED

    Stop/Cancel:

    • EXPERIMENT_STOPPING, TRIAL_STOPPING, TRIAL_CANCELLED, EXPERIMENT_STOPPED

    System:

    • MODULE_RESTARTED, MIGRATION_DONE, CODE_DEPLOYED

    Housekeeping:

    • HOUSEKEEPING_DONE, EVENT_BRIDGE_ERROR

    Do NOT use deprecated/alias names:

    Deprecated Use Instead
    DATA_FETCH_COMPLETE DATA_FETCH_DONE
    LABELING_COMPLETE LABELS_DONE
    BACKTEST_COMPLETE BACKTEST_TRIAL_DONE
    TRAINING_COMPLETE TRIAL_TRAINING_DONE (or TRIAL_SCORED as the final signal)

    3. Event Structure

    class Event:
    id: int                 # PK (autoincrement)
    event_uuid: str | None  # Optional UUID for dedupe
    event_type: str         # e.g., "TRIAL_SCORED"
    experiment_id: int | None  # For routing
    trial_id: int | None       # For trial events
    thread_id: str | None      # LangGraph thread to resume
    created_at: str            # ISO 8601
    payload: dict              # event-specific data
    processed: bool            # whether Event Bridge has processed it
    processed_at: str | None   # when bridge handled it

    4. Payload Expectations

    Minimal payload expectations (applies to dummy and real):

    • Include identifiers required by the event type (experiment_id, trial_id, etc.).
    • Include the numeric metrics listed per type (e.g., n_windows, sharpe_ratio, final_score).
    • For timing fields, use ISO 8601 strings (see DATABASE-SCHEMA.md for timestamp conventions).
    • Add note/source in payload for debugging when emitting from stubs.

    5. Event Types (Detailed)

    Use these exact event_type strings (case-sensitive). Prefer *_DONE for completion markers.

    5.1 Data Preparation (Sequential)

    Event Payload Description
    EXPERIMENT_PLAN_CREATED {experiment_id, experiment_type, n_trials_planned} New experiment plan created
    DATA_FETCH_DONE {experiment_id, symbols_fetched, time_range} OHLCV data fetched
    LABELS_DONE {experiment_id, count_long, count_short, count_none, count_total} Labels generated
    FAT_WINDOWS_DONE {experiment_id, n_windows, feature_count, fat_hash} FAT windows built

    5.2 Per-Trial (Parallel Across Trials)

    Event Payload Description
    TRIAL_CONFIG_READY {experiment_id, trial_id, model_type, feature_count, window_size, batch_size} Trial config generated
    SMALL_WINDOWS_PROGRESS {experiment_id, trial_id, progress_percent, columns_processed, total_columns, cache_key} SMALL windows progress
    SMALL_WINDOWS_DONE {experiment_id, trial_id, cache_key, reused: bool} SMALL windows ready
    TRAINING_PROGRESS {experiment_id, trial_id, epoch, total_epochs, loss, val_loss, val_accuracy, progress_percent} Training progress
    TRIAL_TRAINING_DONE {experiment_id, trial_id, objective_value, duration_seconds} Training complete
    TRIAL_PRUNED {experiment_id, trial_id, reason, step_pruned_at} Trial pruned early (NOT scored)
    BACKTEST_TRIAL_DONE {experiment_id, trial_id, sharpe_ratio, total_trades, profit_factor} Backtest complete
    TRIAL_SCORED {experiment_id, trial_id, final_score, rank} Final score computed
    ANALYST_TRIAL_SUMMARY_READY {experiment_id, trial_id, summary_path, highlights: list} Analyst summary for individual trial

    5.3 Experiment Closure

    Event Payload Description
    ALL_TRIALS_DONE {experiment_id, n_scored, n_pruned, n_cancelled, best_trial_id, best_score} All trials finished
    ANALYST_EXPERIMENT_SUMMARY_READY {experiment_id, summary_path, recommended_trials: list, regime_tags: list} Analyst summary for complete experiment

    Note on pruned trials:

    • n_trials_completed counts only status == "completed" trials
    • Pruned trials are NOT counted in n_trials_completed and do NOT receive a final_score
    • Use derived output-only totals when needed: finished = completed + pruned + failed
    • Pruned trials are excluded from ranking and best trial selection

    5.4 Stop/Cancel (Triggered by Director Tools)

    Event Payload Description
    EXPERIMENT_STOPPING {experiment_id, reason, requested_by} Stop requested
    TRIAL_STOPPING {experiment_id, trial_id, reason, requested_by} Trial stop requested
    TRIAL_CANCELLED {experiment_id, trial_id, final_status: "cancelled"} Trial cancelled
    EXPERIMENT_STOPPED {experiment_id, n_completed, n_cancelled, reason} Experiment stopped

    5.5 Live Execution

    Event Payload Description
    EXECUTION_PLAN_CREATED {execution_plan_id, experiment_id, trial_id, mode, target_symbol} New execution plan
    MASTER_EXEC_PLAN_UPDATED {version_id, state, changed_slots: list} MasterExecutionPlan updated
    LIVE_EXEC_SUMMARY {execution_plan_id, period_id, period_start, period_end, n_trades, pnl, sharpe} Periodic live summary
    LIVE_TRADING_STARTED {execution_plan_id, mode} Live trading started
    LIVE_TRADING_STOPPED {execution_plan_id, reason} Live trading stopped
    LIVE_ANOMALY_DETECTED {execution_plan_id, anomaly_type, details} Anomaly detected in live trading

    5.6 System Events

    Event Payload Description
    MODULE_RESTARTED {module_name, reason} Module service restarted
    MIGRATION_DONE {note} All DB migrations completed (emitted by database module)
    CODE_DEPLOYED {module_name, commit_hash} Code changes deployed

    6. The Event Bridge

    6.1 How It Works

    A background process in the orchestrator that:

    1. Polls events table for unprocessed events.
    2. Finds the corresponding Graph thread.
    3. Resumes the Graph: graph.invoke(Command(resume=event_data), config={"thread_id": ...})
    4. Marks the event as processed.

    6.2 Implementation

    async def event_bridge_loop(graph, poll_interval=1.0):
    while True:
        with get_db_session() as db_session:
            events = db_session.query(Event).filter_by(processed=False).order_by(Event.created_at).all()
            for event in events:
                try:
                    await graph.ainvoke(
                        Command(resume={"event_type": event.event_type, "payload": event.payload}),
                        config={"configurable": {"thread_id": event.thread_id}}
                    )
                    event.processed = True
                    db_session.commit()
                except Exception as e:
                    log.error(f"Failed to process event {event.id}: {e}")
                    # Mark as processed to avoid infinite retry, create error event
                    event.processed = True
                    create_error_event(event, e)
                    db_session.commit()
        await asyncio.sleep(poll_interval)

    6.3 Implementation Notes

    Polling:

    • Query: SELECT * FROM events WHERE processed = false ORDER BY created_at ASC
    • Optional: filter by whitelist of event types

    Thread Routing:

    • Use event.thread_id if present, otherwise default to "rainbow_main"
    • Multi-threading support planned for future

    After Processing:

    • On success: set processed = true and processed_at = now()
    • On error: set processed = true (to avoid infinite retry) and emit EVENT_BRIDGE_ERROR event

    Testing Mode:

    • Modules can insert DONE events directly (with correct payload shape) to unlock graph flow without doing real work

    Database Schema:

    • See DATABASE-SCHEMA.md Section 2.3 for the complete events table definition
