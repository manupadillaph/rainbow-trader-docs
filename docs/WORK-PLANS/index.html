    1. Overview

    1.1 What is a Work Plan?

    A work plan is a structured document that:

    • Defines the objective of a change
    • Defines the rules that control how work is executed
    • Lists findings from analysis
    • Contains questions for the user
    • Provides a task list to execute
    • Tracks progress through completion

    1.2 Why Use Work Plans?

    Benefit Description
    Clarity Both human and AI know exactly what to do
    Traceability Every change is documented and trackable
    Coordination Complex changes across files stay organized
    Quality Prevents forgotten steps or incomplete work
    Recovery If interrupted, work can resume from last checkpoint

    2. When to Create a Plan

    2.1 Required Triggers

    Create a plan when the change involves:

    Trigger Threshold
    Multiple files >5 files affected
    Multiple modules Cross-module changes
    Architecture changes New patterns, major refactoring
    Breaking changes API changes, migrations
    Coordinated updates Docs + code + tests + configs together
    Audit/Review tasks Systematic review of codebase

    2.2 Examples

    Requires a plan:

    • Variable name refactoring across all modules
    • Directory restructuring
    • New configuration system
    • Database schema migration
    • CLI pattern standardization
    • Adding a new module to the system

    Does NOT require a plan:

    • Bug fix in a single file
    • Adding a new function to existing module
    • Documentation typo fix
    • Single test addition

    2.3 When NOT to Create a Plan

    • Simple, isolated changes
    • Changes affecting ≤5 files in a single module
    • Routine maintenance (unless systematic)
    • Quick fixes with clear scope

    3. Plan Lifecycle: Five Stages

    Work plans always go through five clearly separated stages:

    1. Stage 1 — Initial Draft: Create the document with Header, Description, Rules, and the first version of the Findings (no tasks yet).
    2. Stage 2 — Findings Refinement: Iterate on the Findings and Rules together with the user, logging the full conversation until everything is aligned (still no tasks).
    3. Stage 3 — Task Planning and Refinement: Derive and refine the task list per finding, keeping previous tasks intact and appending changes (historical planning view).
    4. Stage 4 — Final Task List (Clean): Build the final, clean checklist for execution, derived from the planning history.
    5. Stage 5 — Execution (LOCKED): Execute the plan exactly as written, with tasks and progress frozen.

    3.1 Stage 1: Initial Draft (First Findings)

    Goal: Create the initial plan document with the first snapshot of the problem, without tasks.

    Steps:

    1. Create plan file in docs/plans/PLAN_NAME.md
    2. Fill Description — one clear paragraph about the goal
    3. Define Rules — keep global rules from the template and add plan-specific rules
    4. Analyze the problem — gather and document the first version of the Findings

    During this stage:

    • Plan is a living document
    • Only Header, Description, Rules, and initial Findings are filled
    • No tasks are created yet
    • Previous findings must not be deleted or rewritten; if something changes, add new findings or updates

    3.2 Stage 2: Findings Refinement (with User)

    Goal: Refine Findings and Rules with the user until they are stable and ready to drive tasks (but without creating tasks yet).

    Steps:

    1. Iterate with user — refine Description, Rules, and Findings based on multiple rounds of questions and answers
    2. Log the conversation — after each user response, write down what the AI understood and which findings or rules are impacted, creating new findings or updates instead of editing away previous ones
    3. Check consistency — ensure that Rules and Findings do not contradict each other
    4. Get user confirmation — "Findings and Rules look good, derive tasks"

    During this stage:

    • Plan is still a living document, but changes are made by adding new findings/updates and conversation log entries
    • Multiple questions per finding are allowed and expected
    • A single user response can affect multiple findings and rules, and this must be recorded under each affected finding
    • Description, Rules, and Findings are finalized before creating tasks

    3.3 Stage 3: Task Planning and Refinement

    Goal: Derive and refine the task list per finding, using the same append-only methodology as for Findings.

    Steps:

    1. Create initial tasks per finding — for each finding, create a detailed, actionable task list based on the approved Rules and Findings, plus an optional extra-tasks section if needed.
    2. Review tasks with user — present the tasks, grouped by finding, and ask for adjustments or additions.
    3. Refine tasks append-only — when something needs to change:
    4. Do not delete or rewrite existing tasks.
    5. Add new tasks that supersede or complement previous ones.
    6. Optionally add a short note like "Superseded by Task X" next to older tasks instead of removing them.
    7. Log the conversation — for each user response that affects tasks, append:
    8. A User line with the relevant part of your answer.
    9. An AI Interpretation line explaining what changed, and which findings/tasks are impacted. These entries go under the "Task planning history" subsection for each affected finding (and can also be summarized in Notes).
    10. Get user confirmation — "Task list is final, lock the plan and execute it".

    During this stage:

    • Plan is still flexible for tasks, but history is preserved (no destructive edits).
    • Tasks remain grouped by finding, and extra tasks are clearly marked.
    • Task list can be reorganized or extended until the plan is locked for execution.

    3.4 Stage 4: Final Task List (Clean)

    Goal: Produce a clean, final task list for execution, based on the planning/refinement history.

    Steps:

    1. Synthesize the planning history — read the task planning/refinement log from Stage 3.
    2. Build a clean checklist — create a concise, non-duplicated task list (still grouped by finding, plus extra tasks) that represents what will actually be executed.
    3. Keep history separate — do not delete or rewrite the planning history; the clean checklist is a new, separate view.
    4. Review with user — present the clean checklist and ask for confirmation that it matches the agreed plan.
    5. Apply changes via Stage 3 — if the user wants changes, add or adjust tasks in the Stage 3 planning log first, then regenerate the clean checklist from that updated history.
    6. Get user confirmation — "Final task list ready, lock and execute it".

    During this stage:

    • The planning history remains append-only.
    • The clean checklist is allowed to be edited/reordered as needed, but always as a projection of the planning history.

    3.5 Stage 5: Plan Execution (LOCKED)

    Goal: Execute the plan exactly as defined in the final clean task list.

    Steps:

    1. Lock the plan — task list is FROZEN
    2. Execute tasks — work through the clean checklist one by one
    3. Mark completed — add [x] to completed tasks
    4. Update progress — increment percentage

    During this stage:

    • ✅ CAN: Mark tasks done, update progress %, add notes
    • ❌ CANNOT: Modify task descriptions, add/remove tasks, change order

    4. Plan Template

    4.1 Required Structure

    Every plan MUST have these sections:

    Section Purpose
    Header Status, dates, priority
    Description What the plan achieves (1 paragraph max)
    Rules Global + plan-specific rules for execution
    Findings Analysis results, each stated ONCE
    Tasks Task planning history + final clean checklist
    Progress Completion tracking
    Notes Log of decisions, weirdness, and conversation during execution

    4.2 Template File

    # Plan Title

**Status:** 🔴 IN PROGRESS / 🟢 COMPLETED  
**Started:** YYYY-MM-DD  
**Completed:** YYYY-MM-DD (when done)  
**Priority:** HIGH / MEDIUM / LOW

## 🔍 Findings

Each finding stated ONCE, clearly:

### Finding 1: [Title]

- **Problem:** What's wrong
- **Location:** Where it is (files, modules)
- **Impact:** Why it matters
- **Recommendation:** Suggested solution
- **Questions:** One or more questions for the user
- **User Decision:** [User fills this in during review]
- **Action Required:** Short summary of what must be done based on the decisions (tasks will be listed in the Tasks section)

If understanding changes after further discussion, do **not** overwrite this finding. Instead, either:

- Add a new finding that references this one and explains what changed, or
- Add a dated "Update" bullet under this finding that explains the change in understanding.

For every user response that affects this finding, append to its conversation log, for example:

- **2025-11-28 — User:** [short quote or link to the user message]
- **2025-11-28 — AI Interpretation:** [what was understood and what changed, including which findings or rules are impacted]

### Finding 2: [Title]

- **Problem:** ...
- **Location:** ...
- **Impact:** ...
- **Recommendation:** ...

## ✅ Tasks

Two separate subsections are used for tasks:

1. **Task planning history (Stage 3)** — append-only log of how tasks evolved.
2. **Final clean task list (Stage 4)** — concise checklist used for execution.

### 3.A — Task planning history (Stage 3)

#### Task planning for Finding 1: [Title]

- [ ] Task 1: Do X in file Y
- [ ] Task 2: Change Z from A to B
- [ ] Task 3: Update documentation
- [ ] Task 4 (supersedes Task 2): Change Z from A to C  <!-- keep Task 2 for history -->

**Task planning conversation for Finding 1:**

    5. Findings Section

    5.1 Structure

    Each finding should be:

    • Unique — stated only once
    • Specific — exact files, functions, lines
    • Actionable — leads to a task
    • Questioned — can contain multiple questions for the user over time
    • Stable — previous findings are not deleted or rewritten; changes in understanding are recorded as updates or new findings that reference the original
    • Conversation-complete — by the end of the plan, the updates and conversation log under each finding should capture the full back-and-forth relevant to that finding

    5.2 Example Finding

    ### Finding 3: Inconsistent Config Loading

- **Problem:** Some modules use `load_config()`, others use `get_config()`
- **Location:** 
  - `data-fetcher/cli.py` (line 45)
  - `model-trainer/cli.py` (line 32)
  - `backtesting-engine/cli.py` (line 28)
- **Impact:** Maintenance burden, potential bugs
- **Recommendation:** Standardize on `load_cli_config()` from common
- **Questions:**
  - Should we also update the function signature to match new pattern?
  - Do we need a deprecation path for `get_config()`?
- **User Decision:** Yes, update signature. Use `load_cli_config(module_name, defaults, config_arg, cli_overrides)`
 - **Action Required:** Standardize all CLIs on `load_cli_config()` and update signatures accordingly. Derive one task per file in the Tasks section.

    5.3 User Decision Process

    1. AI presents finding with question
    2. User reviews and provides decision
    3. AI updates "User Decision" field without rewriting the original problem/impact description
    4. Task is created based on decision
    5. For each user response, the AI appends an interpretation line under every affected finding, and may create new findings that reference the response
    6. The same user response can affect multiple findings and rules; all of them must be updated or cross-referenced explicitly
    7. Plan moves to execution phase

    6. Task List Rules

    6.1 Task Format

    Good task format:

    - [ ] Update `data-fetcher/cli.py`: Change `load_config()` to `load_cli_config()`

    Bad task format:

    - [ ] Fix config loading  # Too vague
- [ ] Update all the files  # Not specific

    Tasks MUST be derived from the approved Rules and Findings. Do not create tasks that are not anchored in them.

    Each finding MUST have its own task subsection in the Tasks section. Only tasks that truly do not belong to any single finding should go into the "Extra tasks" subsection.

    The Tasks section has two views:

    • Task planning history (Stage 3): append-only list showing how tasks evolved (including superseded tasks).
    • Final clean task list (Stage 4): the checklist that will actually be executed in Stage 5.

    For each finding, the Task planning history MUST include a small conversation log:

    • One or more User (tasks) lines quoting your answers about those tasks.
    • Matching AI Interpretation (tasks) lines stating what was understood and which findings/tasks are affected.

    6.2 Task Granularity

    Scope Granularity
    Single file change One or more tasks under the corresponding finding
    Multi-file pattern One task per file, grouped under the relevant finding(s)
    Complex change Break into sub-tasks under the affected findings

    Example — Good granularity:

    ### Tasks for Finding 3: Standardize --emit flag

- [ ] Update `labeling-service/cli.py`: Add `--emit` flag

    It is OK for different findings to have tasks that touch the same file, as long as each task clearly states which finding it belongs to and what specific change it performs.

    6.3 Task Dependencies

    • Tasks within a phase can be done in any order
    • Phases should be done in order
    • If task B depends on task A, put A first

    If you think a task is wrong:

    4. DO NOT modify the task yourself
    5. DO NOT "interpret" what it should be

    • Do not modify the existing locked plan

    12. Human-AI Collaboration Patterns

    12.1 Communication During Planning

    AI should:

    • Present findings clearly
    • Ask specific questions
    • Propose solutions with rationale
    • Wait for user decisions
    • Record important decisions and clarifications as new findings or updates, instead of overwriting previous findings
    • After each user response, append a short "AI Interpretation" note under every affected finding, stating what changed and which findings or rules are impacted

    User should:

    • Answer questions directly
    • Provide decisions in "User Decision" field
    • Confirm when plan is ready for execution
    • Flag any concerns before locking

    12.2 Communication During Execution

    AI should:

    • Report progress after each task
    • Ask immediately if something is unclear
    • Never assume or interpret
    • Keep Notes updated with relevant observations and decisions

    User should:

    • Monitor progress
    • Answer questions promptly
    • Approve any deviations
    • Confirm completion

    12.3 Handling Blockers

    Blocker Type Action
    Unclear task Ask user for clarification
    Missing dependency Record in Progress or comments, ask user
    Task impossible Stop, explain why, ask user
    Found better approach Record it, but execute as written unless user approves change
    External blocker Document, move to next task if independent

    13. Common Mistakes and How to Avoid Them

    Mistake How to Avoid
    Modifying locked plan Remember: Stage 5 (Execution) = FROZEN
    Skipping tasks All tasks must be done or explicitly approved to skip
    Not reading plan before work Always read_file() first

    Duplicate findings State each finding ONCE only
    Overwriting previous findings or decisions Add updates or new findings/notes that reference the original instead of editing away history

    Not updating progress Update after EVERY task
