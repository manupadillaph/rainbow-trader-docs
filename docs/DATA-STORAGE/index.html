    Overview

    The data/ directory holds all heavy artifacts and runtime data. It is NOT versioned in Git.

    • Location: ./data/ by default (configurable via FOLDERS__DATA env var)
    • Purpose: Store OHLCV, indicators, labels, windows, checkpoints, logs
    • Managed by: housekeeping module for cleanup and archival

    Directory Structure

    data/                                 # Default: ./data/ (configurable via FOLDERS__DATA)
├── OHLCV/                            # GLOBAL cache — raw market data
│   └── {symbol}/{timeframe}/         # e.g., BTCUSDT/5m/raw.parquet
├── indicators/                       # GLOBAL cache — computed indicators
│   └── {symbol}/{timeframe}/         # e.g., BTCUSDT/5m/RSI_14_raw.parquet
├── labels/                           # Labels per labeling config
│   └── {label_cache_key}/            # hash(labeling_config, symbol, timeframe, range)
├── fat_windows/                      # FAT windows per experiment
│   └── {fat_cache_key}/              # hash(features, window_size, labeling_config_hash)
├── small_windows/                    # SMALL windows per trial config
│   └── {small_cache_key}/            # hash(feature_subset, window_size, batch_size)
├── checkpoints/                      # Model checkpoints + Optuna storage
│   └── {timestamp}-experiment-{id}/  # One folder per experiment
│       ├── optuna_study.sqlite       # One Optuna DB per experiment
│       └── {suffix}/phase-{N}/       # Suffix + phase namespaces
│           └── trial_{id}/            # Checkpoint files per trial
│               ├── trial.json         # Trial metadata snapshot (ordered fields)
│               └── epoch_*.pt, best.pt
├── plans/                            # Plan JSON files (SOURCE OF TRUTH)
│   ├── experiments/{experiment_id}.json
│   ├── execution/{execution_plan_id}.json
│   └── master_execution_plan.json
├── live/                             # Live trading logs and data
│   └── {execution_plan_id}/{live_session_id}/
├── archived/                         # Old experiments (managed by housekeeping)
└── logs/                             # System logs

database/                             # Folder configurable via FOLDERS__DATABASE
└── rainbow_trader.sqlite             # The actual database file
                                      # Filename configurable via DATABASE__SQLITE_PATH

    Artifact Caching Strategy

    1. Base Data (OHLCV)

    Unique, update-only. One file per symbol/timeframe.

    • Path: OHLCV/raw/{symbol}/{timeframe}/data.parquet
    • Behavior: When new data arrives: append or merge. Never duplicate.

    2. Derived Features (Indicators/Norm)

    Versioned by config hash.

    • Path: indicators/{symbol}/{timeframe}/{indicator_name}_{hash}.parquet
    • Reusability: Reusable across experiments if config matches.

    3. Labels

    Versioned by labeling config hash.

    • Path: labels/{label_cache_key}/
    • Cache key: label_cache_key = hash(labeling_config, base_symbol, base_timeframe, time_range)
    • Reusability: Reusable across experiments with same labeling config.

    4. Windows (FAT/SMALL)

    LRU Cache, purgeable by housekeeping.

    • FAT: fat_windows/{fat_cache_key}/
    • Cache key: fat_cache_key = hash(features, window_size, label_cache_key)
    • SMALL: small_windows/{small_cache_key}/
    • Cache key: small_cache_key = hash(feature_subset, window_size, batch_size)
    • Both are reusable across experiments with matching config.
    • Cleanup: Housekeeping task can delete old windows not accessed in X days.

    Global Parquet Cache

    OHLCV and indicators are GLOBAL (not per-experiment):

    • Can be reused, extended, completed by ANY process
    • Each feature has its own parquet file(s)
    • Same indicator with different params = different parquet files

    Parquet Naming Convention

    data/OHLCV/{symbol}/{timeframe}/{pipeline}.parquet
  e.g., OHLCV/BTCUSDT/5m/raw.parquet
  e.g., OHLCV/BTCUSDT/5m/clean.parquet
  e.g., OHLCV/BTCUSDT/5m/relative.parquet
  e.g., OHLCV/BTCUSDT/5m/normalized.parquet

data/indicators/{symbol}/{timeframe}/{indicator}_{params}_{pipeline}.parquet
  e.g., indicators/BTCUSDT/5m/RSI_14_raw.parquet
  e.g., indicators/BTCUSDT/5m/RSI_14_normalized.parquet
  e.g., indicators/BTCUSDT/1h/MACD_12_26_9_raw.parquet

    Feature to Parquet Lookup

    Each feature in features_config contains enough info to find its parquet:

    • symbol, timeframe, type, source, params, pipeline

    Process:

    1. Check if parquet exists and covers required time range
    2. If missing or incomplete: fetch data, compute, save to global cache

    Database Registration

    All parquets are tracked in the parquet_files table with time ranges covered.

    This allows efficient lookup: "do I have BTCUSDT/5m/RSI_14 for 2024-01-01 to 2024-06-01?"

    See DATABASE-SCHEMA.md for table schema.
