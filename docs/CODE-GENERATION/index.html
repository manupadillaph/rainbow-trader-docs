    1. Language & Style

    1.1 General Rules

    Aspect Requirement
    Language All code, comments, docstrings, and documentation in English
    Style Guide PEP 8
    Line Length Maximum 120 characters
    Indentation 4 spaces (no tabs)
    Encoding UTF-8

    1.2 Python Style Guide

    # Good: Clear, descriptive names
def calculate_moving_average(prices: list[float], window_size: int) -> float:
    """Calculate the simple moving average for a price series."""
    ...

# Bad: Abbreviated, unclear names
def calc_ma(p, w):
    ...

    Naming Conventions:

    Type Convention Example
    Functions snake_case calculate_rsi()
    Variables snake_case window_size
    Classes PascalCase ExperimentPlan
    Constants UPPER_SNAKE_CASE MAX_RETRIES
    Private _leading_underscore _internal_helper()
    Modules snake_case data_fetcher.py

    1.3 Docstrings (Google Style)

    All public functions, classes, and modules MUST have docstrings:

    def compute_rsi(series: pl.Series, period: int = 14) -> pl.Series:
    """Compute the Relative Strength Index (RSI) for a price series.

    Args:
        series: Price series (typically close prices).
        period: Lookback period for RSI calculation. Defaults to 14.

    Returns:
        Series containing RSI values (0-100 scale).

    Raises:
        ValueError: If period is less than 2.

    Example:
        >>> prices = pl.Series([44, 44.5, 45, 44.8, 45.2])
        >>> rsi = compute_rsi(prices, period=3)
    """
    ...

    1.4 Type Hints

    Required for ALL function signatures. Enforced by mypy --strict.

    from typing import Optional, List, Dict, Any
import polars as pl

def process_data(
    df: pl.DataFrame,
    columns: List[str],
    config: Optional[Dict[str, Any]] = None,
) -> pl.DataFrame:
    ...

    2. Quality Gates

    2.0 CRITICAL: Always Use .venv

    ⚠️ ALL commands MUST be run inside the .venv virtual environment!

    # Activate .venv FIRST (from project root)
source .venv/bin/activate

# Or prefix commands with .venv/bin/
.venv/bin/python -m pytest
.venv/bin/pip install -e .
.venv/bin/mypy src --strict

    NEVER use system Python, conda, or any other environment!

    2.3 Logging Standards

    Use structlog for structured logging:

    from common import get_logger

logger = get_logger(__name__)

# Log levels
logger.debug("Processing batch", batch_size=100, duration_ms=45)
logger.info("Experiment started", experiment_id="exp_123")
logger.warning("Approaching budget limit", current=4.5, limit=5.0)
logger.error("Failed to fetch data", symbol="BTCUSDT", error=str(e))
logger.critical("Database connection lost")

    Rules:

    • All module entry/exit points: INFO level
    • All exceptions: ERROR level with stack trace
    • Performance-critical operations: DEBUG level with duration
    • CLI entrypoints MUST route logging config through the shared --config/--set pattern and honor the LOG__LEVEL env var as default.
    • Before configuring logging, always call common.ensure_env_loaded() once per process. This guarantees .env.local / .env are loaded hierarchically (submodule first, then repo root).
    • Wrap error-prone sections in try/except and log with log.error(..., exc_info=True) before re-raising or translating the exception. Add log.debug breadcrumbs so failures can be reconstructed from logs.
    • Do not swallow exceptions silently—every catch block must emit a log entry.
    • Block Comments: Write code in logical task blocks, each starting with a comment explaining what that block does. This makes code readable from start to finish. Don't be afraid to add many comments—the more, the better.
    • Extensive try/except: Wrap risky operations in try/except blocks with log.debug breadcrumbs throughout all code paths.

    Note for coder: dont put results of commands in Verification section, just the commands to run

    Note for coder: we are using triple backslashes \`` above to escape the backticks in this document. In actualCHANGELOG.md`, use standard triple backticks.

    5.2 Rules

    1. One entry per change session — Group related changes in a single entry
    2. Timestamp required — Use ISO format: YYYY-MM-DD HH:MM
    3. Include verification commands — Tests or CLI commands to validate the change
    4. Be specific — List exact files, functions, and classes modified
    5. Link to context — Reference experiment IDs, issue numbers, or research findings if applicable
    6. First line must be a heading — Start file with # Changelog (MD041 compliance)
    7. Newest entries go on top — Append every new ## CHANGE [...] section immediately after the file header so the changelog is reverse chronological.
    8. Bulk changes from git — If many commits were made during a session, use git to reconstruct the changelog:

    ```bash # Get commits from last session (adjust date/count as needed) git log --oneline -20

    # Get detailed changes for a commit range git log --pretty=format:"- %s" HEAD~10..HEAD

    # See files changed git diff --stat HEAD~10..HEAD

    # See full diff for specific files git diff HEAD~10..HEAD -- src/module/file.py ```

    Then consolidate all commits into a single CHANGELOG entry with the most important changes summarized.

    12. Critical Rule: Read Before Editing

    NEVER edit a file without reading it completely first.

    This applies to: - Documentation files (README, CHANGELOG, etc.) - Configuration files (YAML, .env, etc.) - Code files (Python, etc.)

    Why: - Prevents duplication - Maintains coherence - Preserves context - Avoids conflicts - Ensures logical flow

    Process: 1. Read entire file using read_file tool 2. Understand structure and existing content 3. Identify where new content fits logically 4. Make coherent edit that integrates smoothly

    Bad example:

    # Just append to end without reading
grep -n "def main" file.py  # Find line number
sed -i '50a new_function()' file.py  # Insert blindly

    Good example:

    # Read entire file first
read_file("file.py")
# Understand: main() is at line 50, imports at top, helpers at bottom
# Edit: Add new import at top, new helper before main()
edit(file_path, old_string=..., new_string=...)

    13. Critical Rule: No Legacy/Deprecated Code

    NEVER leave old, deprecated, or legacy code in the codebase unless explicitly requested by the user.

    Rules:

    • The codebase must be as clean as possible at all times
    • Every single function must have a use (no dead code)
    • Do not keep code "for compatibility" or "just in case" unless user explicitly asks
    • When refactoring, remove the old implementation completely
    • If replacing a function, delete the old one (don't rename to _old or _deprecated)

    Exceptions (ONLY if user explicitly requests):

    • User says "keep the old version for reference"
    • User says "leave it for backward compatibility"
    • User explicitly asks to preserve something

    When in doubt: Ask the user if old code should be kept or removed.

    14. Opportunistic Improvements

    When viewing code for a task, you may notice improvements unrelated to the current task.

    DO make the improvement if:

    • It's a small, obvious fix (typo, formatting, naming consistency)
    • It aligns with patterns established elsewhere in the codebase
    • It doesn't contradict any documented rules
    • You're confident it's correct

    ASK the user first if:

    • The change is significant or architectural
    • You're unsure if it contradicts something
    • It might have side effects
    • It's a recommendation but you have doubts

    DON'T:

    • Make large changes without asking
    • Change things without understanding why
    • Refactor entire files when the task was small
    • Make changes you can't explain

    Balance: Finding improvements while viewing code is good, but don't go overboard changing things the user didn't ask for.
