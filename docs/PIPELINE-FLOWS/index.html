    Sections

    1. Experiment Flow (Stage by Stage, Parallel-Ready)

    START
│
│ (User message or automatic trigger)
▼
director-node
│ decides that a new experiment is needed
│ routes to planner-node
▼
planner-node
│ creates ExperimentPlan (file + database record)
│ includes:
│   - time ranges (download, train, backtest)
│   - base symbol/timeframe (for labeling)
│   - features_config (see Section 4.4 for schema)
│   - labeling_config (trading / regime)
│   - repo commits snapshot
│   - bot_config
│   - dataset_config with batch_size
│ Returns to director-node
▼
director-node
│ uses Tool: trigger_data_fetch(experiment_id)
│ Graph SUSPENDS (waiting for DATA_FETCH_DONE)
▼
data-fetcher (subprocess)
│ for (experiment_id, symbol, timeframe):
│   - downloads OHLCV → data/OHLCV/raw
│   - can also precompute clean/relative/normalized if desired
│   - registers files in database
│ when all required data for this experiment is ready:
│ EMIT EVENT: DATA_FETCH_DONE(experiment_id)
▼
Event Bridge → Graph RESUMES
▼
director-node
│ uses Tool: trigger_labeling(experiment_id)
│ Graph SUSPENDS (waiting for LABELS_DONE)
▼
labeling-service (subprocess)
│ reads:
│   - labeling_config from ExperimentPlan (thresholds, horizon)
│   - sampling_config from ExperimentPlan (training_none_ratio, validation_none_ratio)
│   - OHLCV data for base_symbol/base_timeframe from data
│
│ For trading tasks:
│   1. Compute directional labels:
│      - identifies all long/short timestamps
│      - writes: labels_directional.parquet
│   2. Compute none candidates:
│      - all timestamps not in long/short
│   3. Apply sampling_config:
│      - selects NONE per period (training_none_ratio, validation_none_ratio)
│      - keeps tail timestamps without sampling when configured
│   4. Generate final labels:
│      - combines: all long + all short + sampled none
│      - writes: labels_selected.parquet ← THIS IS USED FOR FAT WINDOWS
│
│ For regime tasks:
│   - computes regime label for all timestamps
│   - writes: labels_selected.parquet (all timestamps)
│
│ stores in database:
│   - label scheme hash
│   - timestamps T (list of selected timestamps from labels_selected)
│   - count of long/short/none
│ EMIT EVENT: LABELS_DONE(experiment_id)
▼
Event Bridge → Graph RESUMES
▼
director-node
│ uses Tool: trigger_data_fetch(experiment_id)
│ Graph SUSPENDS (waiting for DATA_FETCH_DONE)
▼
data-fetcher (subprocess)
│ downloads OHLCV/indicators per plan
│ registers files + metadata in DB
│ EMIT EVENT: DATA_FETCH_DONE(experiment_id, symbols_fetched, time_range)
▼
Event Bridge → Graph RESUMES to director-node
▼
director-node
│ uses Tool: trigger_labeling(experiment_id)
│ Graph SUSPENDS (waiting for LABELS_DONE)
▼
labeling-service (subprocess)
│ generates labels_selected.parquet + stats
│ registers label metadata in DB
│ EMIT EVENT: LABELS_DONE(experiment_id, counts, label_cache_key)
▼
Event Bridge → Graph RESUMES to director-node
▼
director-node
│ uses Tool: trigger_fat_build(experiment_id)
│ Graph SUSPENDS (waiting for FAT_WINDOWS_DONE)
▼
window-preparer (subprocess)
│ reads:
│   - labels_selected.parquet from data/labels/{label_cache_key}
│   - timestamps T from labels_selected (one FAT window per timestamp)
│   - OHLCV/indicator parquets from data (GLOBAL cache)
│   - cleaning_config, relative_config, normalization_config, features_config
│ for each timestamp T in labels_selected:
│   - builds one FAT window (N rows x M features)
│   - each feature looks up its parquet in global cache
│   - fetches N candles looking backward from T
│ uses data-processing for all transformations
│ generates:
│   - FAT windows → data/fat_windows/{fat_cache_key}
│   - one FAT window per label in labels_selected
│   - metadata in database (files, hashes, ranges, count)
│ EMIT EVENT: FAT_WINDOWS_DONE(experiment_id)
▼
Event Bridge → Graph RESUMES to director-node
▼
director-node
│ uses Tool: trigger_training(experiment_id)
│ Graph SUSPENDS (waiting for ALL_TRIALS_DONE) — but remains interruptible by per-trial events
▼
model-trainer (subprocess)
│ reads ExperimentPlan and FAT+labels metadata
│ creates an Optuna study for experiment_id
│ for each trial (parallel within limits):
│   - Optuna selects parameters
│   - generates trial_config (model_type, hyperparameters, window_size, feature_subset, batch_size)
│   - stores trial_config in database
│   - EMIT EVENT: TRIAL_CONFIG_READY(experiment_id, trial_id, model_type, feature_count, window_size, batch_size)
│   - triggers window-preparer (internal SMALL flow, `prepare_small_window`):
│       * checks cache: if SMALL windows exist for this selection (source FAT parquet + window_size + selected features) → reuse
│       * otherwise: slices from FAT, organizes into batches
│       * EMIT EVENT: SMALL_WINDOWS_DONE(experiment_id, trial_id, cache_key, reused)
│   - trains model with SMALL windows + labels
│   - polls trials.status periodically (for graceful stop)
│   - saves checkpoint, metrics
│   - if backtest needed: triggers backtesting-engine
│       * EMIT EVENT: BACKTEST_TRIAL_DONE(experiment_id, trial_id, sharpe_ratio, total_trades, profit_factor)
│   - computes final_score (deterministic)
│   - EMIT EVENT: TRIAL_SCORED(experiment_id, trial_id)
│   - analyst-node generates summary → EMIT EVENT: ANALYST_TRIAL_SUMMARY_READY(experiment_id, trial_id, summary_path, highlights)
│
│ when all trials done (scored + pruned + cancelled):
│ EMIT EVENT: ALL_TRIALS_DONE(experiment_id)
▼
Event Bridge → Graph RESUMES with ALL_TRIALS_DONE
▼
director-node
│ routes to analyst-node
▼
analyst-node
│ reads:
│   - all trials for experiment_id (with final_score, rank)
│   - all associated backtests
│   - labeling and regime-related info
│ analyzes (LLM-based insights):
│   - patterns across trials (which features/models work best)
│   - regime tags for the experiment period
│   - comparison with past experiments (from The Store)
│   - recommendations
│ stores:
│   - structured summary in analyst_summaries
│   - regime tags in regimes table
│   - insights in The Store
│ EMIT EVENT: ANALYST_EXPERIMENT_SUMMARY_READY(experiment_id, summary_path, recommended_trials, regime_tags)
│ returns to director-node with summary
▼
director-node
│ with summaries:
│   - may create new ExperimentPlans
│   - may create ExecutionPlans for best trials
│   - may update MasterExecutionPlan

    2. Per-Trial Flow (SMALL windows + Training + Backtest)

    model-trainer (for each trial)
│ emits events per stage; Event Bridge can route TRIAL_SCORED/TRIAL_SUMMARY_READY to analyst-node while director waits for ALL_TRIALS_DONE
│
▼
window-preparer (OFF / batch mode)
│ INPUT: FAT windows (pre-computed matrices from data)
│ reads:
│   - FAT windows + labels for experiment_id
│   - trial_config (feature subset, window_size, batch_size, etc.)
│ checks in database:
│   - if SMALL windows for this small_cache_key (feature_subset, window_size, batch_size)
│     already exist and are valid:
│       → can reuse them for this trial
│   - otherwise:
│       → SLICE columns (features) and rows (timestamps) from FAT
│       → organize into batches (batch_0.parquet, batch_1.parquet, etc.)
│       → build SMALL windows now
│ SMALL windows may be:
│   - in memory (for immediate training), and/or
│   - parquet files in data/small_windows/{small_cache_key}/
│ registers new SMALL window entries in database
▼
model-trainer (training phase)
│ reads:
│   - SMALL windows (in memory or from data)
│   - trial_config
│   - labels from data/labels/{label_cache_key}
│ trains model:
│   - iterates through batches per epoch
│   - writes checkpoint → data/checkpoints/{experiment_id}/{trial_id}.*
│   - writes training metrics → database (trials table)
│   - polls trials.status for graceful stop
▼
model-trainer (mini-backtest phase)
│ checks scoring_config from ExperimentPlan:
│   - if stage3_backtest > 0 and tail timestamps exist:
│       → calls backtesting-engine.run_mini_backtest()
│   - if no tail timestamps:
│       → stage3 metrics default to 0
▼
model-trainer (scoring phase - DETERMINISTIC)
│ reads:
│   - training metrics from trials table
│   - scoring_config from ExperimentPlan
│ computes final_score (deterministic formula):
│   - weighted sum of training metrics and stage metrics
│   - NO LLM involved - pure math
│ stores:
│   - final_score in trials table
│   - updates rank within experiment
│ EMIT EVENT: TRIAL_SCORED(experiment_id, trial_id)
▼
analyst-node (trial-level)
│ triggered by TRIAL_SCORED
│ reads:
│   - trial metrics (training + backtest)
│   - config and labels metadata
│ generates human-readable trial summary
│ stores summary in The Store
│ EMIT EVENT: ANALYST_TRIAL_SUMMARY_READY(experiment_id, trial_id, summary_path, highlights)

    3. From Experiment to Execution (ExecutionPlan + MasterExecutionPlan)

    director-node
│ after analyzing experiment summary:
│ decides to create ExecutionPlan for best trial(s)
│ routes to planner-node
▼
planner-node
│ generates ExecutionPlan(s) (file + DB row), each with:
│   - experiment_id
│   - trial_id
│   - mode: LIVE_REAL or LIVE_SIM
│   - features_config for SMALL windows
│   - bot_config
│   - bot_config (strategy_config + risk_manager_config + order_preferences_config)
│   - exchange settings
│ returns to director-node
│ EMIT EVENT: EXECUTION_PLAN_CREATED(execution_plan_id)
▼
director-node
│ decides for each slot (symbol_TF):
│   - which ExecutionPlan becomes LIVE (0 or 1 per slot)
│   - which ExecutionPlans run in SIM (0 to N per slot)
│ uses tool: update_master_execution_plan(changes)
│
│ ** HITL INTERRUPT ** (if configured)
│ Graph SUSPENDS, waits for user approval
▼
Control Panel
│ shows approval request
│ User approves/rejects
▼
Graph RESUMES (if approved)
│
│ tool updates MasterExecutionPlan (file + DB):
│   - sets state = UPDATING
│   - increments version_id
│   - updates slots[symbol_TF].live_plan_id
│   - updates slots[symbol_TF].sim_plan_ids
│   - sets state = ACTIVE
│ EMIT EVENT: MASTER_EXEC_PLAN_UPDATED
▼
director-node
│ uses tool: start_live_trading()
│ live-trading-runner subprocess starts

    4. Live Trading with MasterExecutionPlan

    Update (2026-03-17): current delivered runtime foundation is ExecutionPlan-first. MasterExecutionPlan orchestration remains a future scope and this section is kept for historical context.

    4.A Current Live Flow (ExecutionPlan-first, implemented)

    director-node / CLI
│ starts live runner with explicit execution_plan_id(s) or query mode
▼
live-trading-runner
│ loads ExecutionPlan rows
│ resolves checkpoint selector
│ creates one LiveSession per execution_plan_id (STARTING -> RUNNING)
│ emits LIVE_TRADING_STARTED
▼
per execution plan instance
│ runtime_stream consumes DataFetcher.stream(closed candles + dedup)
│ runtime_loop tick:
│   1) OnlineFeatureEngine updates online feature state
│   2) OnlineWindowEngine.build_row creates ordered model input row
│   3) model.predict returns typed prediction payload
│   4) BotLogic.generate_exit_orders(...)
│   5) optional force-close close orders
│   6) optional risk force-flat close orders
│   7) projected post-exit portfolio state
│   8) BotLogic.generate_entry_signals(...)
│   9) RiskManager.evaluate_entries(...)
│  10) execution branch:
│      - LIVE_SIM  -> SimExecutionAdapter.process_decision(...)
│      - LIVE_REAL -> LiveRealExecutionAdapter.process_decision(...)
│  11) persist runtime_audit + risk state fields
│  12) emit LIVE_EXEC_SUMMARY periodically
▼
runtime controls
│ pause/resume/reload/force-close operate per execution_plan_id or live_session_id
│ data-gap auto-pause emits LIVE_ANOMALY_DETECTED and requires manual resume
▼
stop command
│ transition STOPPING -> STOPPED
│ emit LIVE_TRADING_STOPPED

    Execution notes:

    • force-close is consumed on the next processed closed candle.
    • On LIVE_REAL, provider order submission is routed through data-fetcher.
    • If provider submission returns rejected, the decision is recorded as rejected and the loop continues.

    5. Stop Experiment Flow (via Director Tools)

    analyst-node (triggered by Event Bridge after TRIAL_SCORED)
│ detects problem:
│   - multiple trials failing
│   - all trials showing very bad metrics
│   - experiment seems hopeless
│ returns recommendation to director-node
▼
director-node
│ analyzes context:
│   - how many trials completed vs remaining
│   - resources being consumed
│   - other experiments waiting
│ DECIDES: approve stop
│ CALLS TOOL: stop_experiment(experiment_id=X, reason="...")
▼
stop_experiment tool (executed by director-node)
│ 1. UPDATE experiments SET status = "stopping" WHERE experiment_id = X
│ 2. UPDATE trials SET status = "stopping"
│      WHERE experiment_id = X AND status IN ("pending", "training", "backtesting")
│ 3. EMIT EVENT: EXPERIMENT_STOPPING(experiment_id) ← for audit/logs
│ returns: "Experiment X marked as stopping, N trials affected"
▼
model-trainer (in each active trial)
│ in training loop, polls DB: SELECT status FROM trials WHERE trial_id = Y
│ sees status = "stopping"
│ graceful stop:
│   - finishes current epoch/iteration
│   - cleans up resources
│   - does NOT save checkpoint (incomplete)
│   - UPDATE trials SET status = "cancelled" WHERE trial_id = Y
│ EMIT EVENT: TRIAL_CANCELLED(experiment_id, trial_id)
▼
Event Bridge (monitoring)
│ tracks TRIAL_CANCELLED events
│ when all trials for experiment X are cancelled or completed:
│   - UPDATE experiments SET status = "stopped"
│ EMIT EVENT: EXPERIMENT_STOPPED(experiment_id)
▼
Event Bridge → Graph RESUMES
▼
director-node
│ routes to analyst-node for partial summary
▼
analyst-node
│ generates summary:
│   - reason for stop
│   - trials completed before stop
│   - any useful insights from partial results
│ returns to director-node
▼
director-node
│ uses information to:
│   - learn from failed experiment
│   - adjust future ExperimentPlans
│   - possibly retry with different config

    6. Stop Individual Trial Flow

    director-node (or based on analyst recommendation)
│ DECIDES: stop specific trial Y
│ CALLS TOOL: stop_trial(experiment_id=X, trial_id=Y, reason="poor intermediate results")
▼
stop_trial tool
│ 1. UPDATE trials SET status = "stopping" WHERE trial_id = Y
│ 2. EMIT EVENT: TRIAL_STOPPING(experiment_id, trial_id) ← for audit/logs
│ returns: "Trial Y marked as stopping"
▼
model-trainer (trial Y)
│ polls DB, sees status = "stopping"
│ graceful stop → status = "cancelled"
│ EMIT EVENT: TRIAL_CANCELLED(experiment_id, trial_id)

    7. USER Manual Intervention

    USER (human operator)
│ via Control Panel or CLI:
│   rt message --to director --content "Please stop experiment X"
│
│ This sends a message to the Graph
▼
Graph receives user input
▼
director-node
│ receives message as input
│ analyzes and decides
│ takes action using tools (stop_experiment, prioritize_experiment, etc.)

    8. Self-Improving Code Flow

    research-node
│ identifies improvement opportunity:
│   - new model architecture from paper
│   - new feature engineering technique
│   - better normalization method
│ returns findings to director-node
▼
director-node
│ validates improvement is worth implementing
│ routes to coder-node with implementation request
▼
coder-node
│ reads existing code and documentation
│ implements changes to appropriate module
│ runs tests:
│   - unit tests
│   - property tests
│   - integration tests
│   - e2e tests (if touching critical paths)
│
│ if all tests pass:
│   - git_commit(message)
│   - run_migrations() if DB schema changed
│   - restart_module(module_name)
│   - check_module_health(module_name)
│
│ returns result to director-node
▼
director-node
│ verifies system is healthy after deployment
│ may trigger new experiment to validate improvement

    Steps summary:

    1. research-node suggests new model architecture, feature, or processing improvement
    2. Director validates the suggestion is worth implementing
    3. Director delegates to coder-node to implement the change
    4. coder-node:
    5. Writes code changes to the appropriate module
    6. Runs unit tests, property tests, integration tests
    7. If all pass → commits changes
    8. Director triggers deployment tools to apply changes:
    9. run_migrations() → apply any DB schema changes
    10. restart_module(module_name) → restart affected service
    11. reload_live_config() → hot-reload live-trading-runner if needed
    12. Director verifies system is healthy after deployment
