    ⚠️ CRITICAL: Commands to NEVER Use

    ❌ NEVER use git reset --hard

    # ❌ WRONG - Destroys untracked files permanently
git reset --hard HEAD~1

# ✅ CORRECT - Use soft reset instead
git reset --soft HEAD~1

    Why: git reset --hard DESTROYS:

    • All untracked files
    • All uncommitted changes
    • Hours of work with NO way to recover

    When to use: ONLY if user explicitly requests it with full understanding of data loss.

    🚫 Problem: MCP Git Tool Commits Submodule Internal Files

    What Happened

    When using MCP's mcp2_git_add and mcp2_git_commit tools, they can accidentally commit: - .git files inside submodules (should be directories, not files) - .gitignore, .mypy_cache, .pytest_cache from submodules - Other internal submodule files

    This breaks the submodule structure - git sees them as regular files instead of submodule references.

    Symptoms

    git add rainbow-*
# Error: You have both modules/common and modules/common/.git
# Error: You have both modules/common and modules/common/.gitignore

    ✅ Correct Process: Committing Submodules

    MCP Tools: Which to Use

    ✅ SAFE MCP Tools (Use These)

    # These work correctly with submodules
mcp2_git_status(repo_path="/path/to/submodule")
mcp2_git_add(repo_path="/path/to/submodule", files=["."])
mcp2_git_commit(repo_path="/path/to/submodule", message="...")
mcp2_git_log(repo_path="/path/to/submodule")

    ❌ AVOID for Main Repo with Submodules

    # These can commit internal submodule files by mistake
mcp2_git_add(repo_path="/main/repo", files=["rainbow-*"])  # ❌ Risky!

# Instead, use shell commands for main repo:
run_command("git add modules/common modules/database")      # ✅ Safe

    Rule: Use MCP git tools for individual submodules, use shell commands for main repo.

    Step 1: Analyze Changes First

    # See what changed
git status

# Review diffs to understand changes
git diff path/to/file

# Group related changes mentally
# Example groups:
# - Configuration files (.env.example, CONFIGURATION.md)
# - Documentation (README.md, MASTER-PLAN.md)
# - Core code (config_loader.py, runtime.py)
# - Tests (test_*.py)

    Step 2: Commit Submodules in Logical Groups

    Group by functionality, not "all at once"!

    # Group 1: Core config system (modules/common)
cd modules/common
git add src/common/config_loader.py src/common/log_format.py
git commit -m "feat(config): add automatic env var mapping"

git add src/common/runtime.py src/common/__init__.py
git commit -m "feat(config): update runtime with LogFormat enum"

git add tests/test_config.py tests/test_paths.py
git commit -m "test: update tests for new variable names"
cd ..

# Group 2: Database module
cd modules/database
git add src/database/session.py
git commit -m "docs: update comments to use DATABASE__SQLITE_PATH"

git add src/database/cli.py src/database/defaults.py
git commit -m "feat(config): add --set flag and empty defaults"
cd ..

# Group 3: Similar modules (batch if changes are identical)
for module in backtesting-engine bot-logic window-preparer; do
  cd rainbow-$module
  git add src/*/cli.py src/*/defaults.py .env.example
  git commit -m "feat(config): standardize configuration files"
  cd ..
done

    Step 3: Commit Main Repo in Logical Groups

    # Group 1: New documentation
git add CONFIGURATION.md CONFIG_CLEANUP_PLAN_V3.md
git commit -m "docs: add comprehensive configuration guide"

# Group 2: Update existing docs
git add README.md README_MODULE_TEMPLATE.md MASTER-PLAN.md CODE-GENERATION.md
git commit -m "docs: update documentation with configuration links"

# Group 3: Root config files
git add .env.example .gitignore
git commit -m "chore: update root configuration files"

# Group 4: Submodule references (LAST!)
git add modules/common modules/database modules/backtesting-engine
git add modules/bot-logic modules/window-preparer
git commit -m "chore: update submodule references to config changes"

# Group 5: More submodule references
git add modules/labeling-service modules/live-trading-runner
git add modules/model-trainer modules/data-fetcher
git commit -m "chore: update remaining submodule references"

    Key: When you git add modules/common, git should only add the commit hash reference, NOT the internal files.

    🔧 How to Fix When Things Go Wrong

    Scenario: Accidentally Committed Submodule Internal Files

    Step 1: Soft Reset (NEVER --hard!)

    # Undo the bad commits but keep all changes
git reset --soft HEAD~2  # Go back 2 commits (adjust number as needed)

    Step 2: Clear the Staging Area

    # Remove ALL staged files
git reset HEAD .

    Step 3: Remove Problematic Files from Git Cache

    # Remove internal submodule files from git's index
git rm --cached -r rainbow-*/.git
git rm --cached -r rainbow-*/.gitignore
git rm --cached -r rainbow-*/.mypy_cache
git rm --cached -r rainbow-*/.pytest_cache
git rm --cached -r rainbow-*/.coverage
git rm --cached -r rainbow-*/.env
git rm --cached -r rainbow-*/src/*.egg-info

    Step 4: Re-add Files Correctly

    # Add root files
git add .env.example CONFIGURATION.md README.md MASTER-PLAN.md CODE-GENERATION.md

# Add submodule references (NOT their internal files)
git add modules/backtesting-engine modules/bot-logic modules/common
git add modules/control-panel modules/data-processing modules/database
git add modules/window-preparer modules/graph-orchestrator modules/housekeeping
git add modules/labeling-service modules/live-trading-runner modules/model-trainer
git add modules/data-fetcher

    Step 5: Verify Before Committing

    # Check what's staged - should NOT see internal submodule files
git status --short

# Should see:
# M  modules/common          ✅ (just the reference)
# NOT:
# A  modules/common/.git     ❌ (internal file - BAD!)

    Step 6: Commit Correctly

    git commit -m "feat: your commit message"

    📋 Checklist: Before Every Commit

    • [ ] All submodules committed first
    • [ ] git status shows only submodule references (M rainbow-*)
    • [ ] NO internal files staged (no .git, .gitignore, .mypy_cache, etc.)
    • [ ] Verified with git status --short
    • [ ] Using git reset --soft if need to undo (NEVER --hard)

    🎯 Quick Reference

    Safe Git Commands

    git reset --soft HEAD~1   # Undo commit, keep changes
git reset HEAD .          # Unstage everything
git rm --cached file      # Remove from staging, keep file
git status --short        # Quick status check

    Dangerous Git Commands (Ask First!)

    git reset --hard          # ⚠️ DESTROYS untracked files
git clean -fd             # ⚠️ DELETES untracked files
git checkout -- .         # ⚠️ DISCARDS uncommitted changes

    🔍 Verification Commands

    # Check submodule status
git submodule status

# Check what's staged
git status --short

# Check submodule commits
git log --oneline -5

# Check if submodules are clean
for module in rainbow-*; do 
  echo "=== $module ===" 
  cd $module && git status --short && cd ..
done

    💡 Best Practices

    Commit Philosophy

    1. Small, logical commits - Group related changes, not everything at once
    2. Understand before committing - Use git diff and git status to analyze
    3. Tell a story - Each commit should represent one logical change
    4. Commit messages matter - Clear, descriptive messages help future you

    Commit Grouping Examples

    ❌ BAD: One giant commit

    git add .
git commit -m "fix: update everything"  # What changed? No idea!

    ✅ GOOD: Logical groups

    # Commit 1: Core functionality
git add src/common/config_loader.py
git commit -m "feat(config): add automatic env var mapping with __"

# Commit 2: Supporting changes
git add src/common/runtime.py src/common/__init__.py
git commit -m "feat(config): integrate LogFormat enum in runtime"

# Commit 3: Tests
git add tests/test_config.py tests/test_paths.py
git commit -m "test: update for new variable naming convention"

# Commit 4: Documentation
git add CONFIGURATION.md README.md
git commit -m "docs: add configuration guide and update README"

    General Best Practices

    1. Always commit submodules first, then main repo
    2. Never use --hard reset without explicit permission
    3. Verify staging area before committing (git status --short)
    4. Use git rm --cached to remove files from staging without deleting them
    5. Keep untracked files (.mcp/, CONFIG_CLEANUP_PLAN.md, etc.) untracked
    6. Test in a branch if unsure about git operations
    7. Use MCP git tools for submodules, shell commands for main repo
    8. Review diffs before committing (git diff --staged)

    📝 Commit Message Format

    Follow Conventional Commits:

    <type>(<scope>): <description>

[optional body]

[optional footer]

    Commit Types

    Type Description Example
    feat New feature feat(config): add automatic env var mapping
    fix Bug fix fix(database): handle empty query results
    docs Documentation only docs(readme): update installation guide
    test Adding or updating tests test(config): add precedence tests
    refactor Code restructuring refactor(core): simplify indicator pipeline
    style Formatting, no logic change style: fix indentation in cli.py
    chore Maintenance tasks chore: update submodule references

    Examples

    # Good commit messages
feat(config): add --set flag support with dot notation
fix(model-trainer): handle empty feature subset gracefully
docs(git): add comprehensive commit guide
test(backtesting): add property tests for trade simulation
chore: update all submodule references to latest

# Bad commit messages
fix: stuff
update files
changes
WIP

    🌿 Branch Strategy

    Branch Types

    Branch Purpose Protection
    main Production-ready code Protected, requires PR
    develop Integration branch Semi-protected
    feature/<name> New features Delete after merge
    fix/<name> Bug fixes Delete after merge
    docs/<name> Documentation Delete after merge

    When to Branch

    Change Size Strategy
    Small fixes (< 50 lines) Commit to current branch locally
    Medium features (50-200 lines) Create feature/<name> branch, commit locally
    Large features (> 200 lines) Create feature/<name> branch + PR required
    Architecture changes Create feature/<name> branch + PR + review required

    PR Requirements

    • Major changes (new features, architecture changes) require PRs
    • All commits should pass quality gates before push
    • User reviews and pushes manually (never auto-push)

    ✅ Quality Gates Before Commit

    Every code change MUST pass these checks without errors or warnings:

    # 1. Tests with 100% coverage
pytest --cov=src --cov-fail-under=100

# 2. Type checking (strict mode)
mypy src --strict

# 3. Linting and formatting
ruff check src

    All three must pass before committing.

    Remember: When in doubt, ask before running destructive git commands!
