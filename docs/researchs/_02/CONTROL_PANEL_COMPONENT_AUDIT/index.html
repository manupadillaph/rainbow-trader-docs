    Control Panel Component Unification Audit

    Status: In Progress
    Started: 2026-02-18
    Priority: High

    Objective

    Systematically audit and unify all list and detail components across the Control Panel to eliminate duplication and create a consistent, maintainable component architecture.

    Current Problems

    1. Multiple trial list implementations - Same data shown differently across pages
    2. Multiple training list implementations - Duplicated table logic
    3. Multiple context/header components - Similar experiment/training summaries repeated
    4. Inconsistent table structures - Different columns, different actions for same entity
    5. No shared component library - Each page reimplements similar functionality

    Findings

    1. TRIALS LISTS - 3 Different Implementations

    A. TrialsTable.tsx (Full-featured list page)

    Location: @/components/dashboard/trials/list/table/TrialsTable.tsx
    Used in: /experiments/{id}/trials/, /experiments/{id}/trainings/{id}/trials/, /trials/
    Features:

    • Full pagination
    • Sortable columns (ID, Objective)
    • 9 columns: ID, Optuna #, Status, Objective, Stages (1/2/3), Eq/PnL/Trades, Phase/Study, Best Flags, Actions
    • Detailed stage metrics (s1, s2, s3)
    • Equity/PnL/Trades calculations
    • Phase/Study hierarchy info
    • Best flags (experiment + study)
    • Actions: Detail, Backtests, "Use in launch" button
    • Table density toggle (compact/comfortable)
    • Context-aware routing (experiment/study context)

    B. ExperimentTrialsTable.tsx (Simplified for experiment detail)

    Location: @/components/dashboard/experiments/detail/shared/ExperimentTrialsTable.tsx
    Used in: /experiments/{id}/ detail page
    Features:

    • NO pagination (shows limited set)
    • NO sorting
    • 7 columns: ID, Optuna #, Status, Objective, Study, Best Flags, Actions
    • Simplified view - no stage details, no metrics
    • Best flags (experiment + study)
    • Actions: Detail, Backtests
    • Fixed to experiment context

    C. Experiments List Page (inline table)

    Location: @/components/dashboard/experiments/list/ExperimentsListPage.tsx
    Features:

    • Inline table definition (NOT a separate component)
    • Shows experiment-level trial counts and best trial info
    • NOT a trials list, but shows trial summary data

    DUPLICATION ISSUE: TrialsTable and ExperimentTrialsTable show the same entity (trials) with different columns and features. Should be ONE unified component with configuration options.

    2. TRAININGS LISTS - 3 Different Implementations

    A. TrainingsTable.tsx (Full-featured list page)

    Location: @/components/dashboard/trainings/list/table/TrainingsTable.tsx
    Used in: /experiments/{id}/trainings/, /trainings/
    Features:

    • Full pagination
    • Sortable columns (ID)
    • 10 columns: ID, Study, Status, Backend, Progress, Trials, Phase, Best Flag, Experiment, Actions
    • Progress percentage
    • Trial completion counts
    • Phase tracking
    • Best flag (only experiment-level)
    • Actions: Detail, Trials, Backtests

    B. ExperimentTrainingsTable.tsx (Simplified for experiment detail)

    Location: @/components/dashboard/experiments/detail/shared/ExperimentTrainingsTable.tsx
    Used in: /experiments/{id}/ detail page
    Features:

    • NO pagination (shows limited set)
    • NO sorting
    • 7 columns: ID, Study, Status, Progress, Trials, Best Flag, Actions
    • Simplified view - no backend, no phase, no experiment name
    • Best flag (only experiment-level)
    • Actions: Detail, Trials
    • Fixed to experiment context

    DUPLICATION ISSUE: TrainingsTable and ExperimentTrainingsTable show the same entity with different columns. Should be ONE unified component with configuration.

    3. EXPERIMENTS LIST - 1 Implementation (inline)

    ExperimentsListPage.tsx

    Location: @/components/dashboard/experiments/list/ExperimentsListPage.tsx
    Features:

    • Inline table definition (NOT extracted to separate component)
    • Full pagination
    • Sortable columns (ID, Name)
    • 8 columns: ID, Name, Status, Market, Range, Trainings, Trials, Actions
    • Complex cell rendering (name + type, counts + best info)
    • Actions: Detail, Trainings, Trials, Backtests

    ISSUE: Should be extracted to ExperimentsTable.tsx component for consistency.

    4. CONTEXT/HEADER COMPONENTS - Multiple Similar Implementations

    A. Experiment Context Components (3 copies)

    1. TrialsExperimentContext.tsx
    2. Location: @/components/dashboard/trials/list/context/
    3. Shows experiment summary in trials list page
    4. Uses ContextSummaryCard with experiment metrics

    5. Links: Experiment, Trainings, Trials, Backtests

    6. TrainingExperimentContext.tsx

    7. Location: @/components/dashboard/trainings/list/context/
    8. Shows experiment summary in trainings list page
    9. Uses ContextSummaryCard with experiment metrics

    10. Links: Experiment, Trainings, Trials, Backtests

    11. ExperimentDetailHeader.tsx

    12. Location: @/components/dashboard/experiments/detail/shared/
    13. Shows experiment summary in experiment detail page
    14. Uses ContextSummaryCard with experiment metrics
    15. Links: Experiment, Trainings, Trials, Backtests, Requirements

    DUPLICATION ISSUE: All three components show the SAME experiment data with nearly identical structure. Should be ONE unified ExperimentContextCard component.

    B. Training Context Components (2 copies)

    1. TrialsTrainingContext.tsx
    2. Location: @/components/dashboard/trials/list/context/
    3. Shows training summary in trials list page
    4. Uses ContextSummaryCard with training metrics

    5. Links: Training, Backtests

    6. Training detail pages likely have similar headers (need to verify)

    DUPLICATION ISSUE: Training context shown in multiple places with similar structure.

    C. Backtests Context Components

    1. BacktestsExperimentContext.tsx
    2. Location: @/components/dashboard/backtests/list/context/
    3. Shows experiment summary in backtests list page
    4. Likely similar to other experiment contexts

    5. SHARED COMPONENTS (Good - Already Unified)

    These components are CORRECTLY unified and reused:

    • ✅ ContextSummaryCard - Generic card for showing context summaries
    • ✅ StatusBadge - Status display
    • ✅ BestFlags - Best trial/study flags
    • ✅ PaginationControls - Pagination UI
    • ✅ SortableTableHeader - Sortable column headers
    • ✅ TableDensityToggle - Density switcher
    • ✅ AdminShell - Page layout wrapper

    Proposed Unified Architecture

    1. Unified Table Components

    Create configurable table components that can be used in multiple contexts:

    UnifiedTrialsTable

    Props:

    interface UnifiedTrialsTableProps {
  trials: TrialSummary[];
  variant: "full" | "compact" | "minimal";
  pagination?: PaginationProps;
  sorting?: SortingProps;
  columns?: TrialColumn[];  // Allow column customization
  actions?: TrialAction[];  // Allow action customization
  context?: {
    experimentId?: number;
    studyId?: number;
    rangeStart?: string;
    rangeEnd?: string;
  };
  density?: "compact" | "comfortable";
  launchForm?: UseBacktestLaunchFormResult;
}

    Variants:

    • full - All columns, all features (for list pages)
    • compact - Reduced columns, essential info (for detail pages)
    • minimal - Minimal columns, basic info (for summaries)

    UnifiedTrainingsTable

    Props:

    interface UnifiedTrainingsTableProps {
  trainings: TrainingSummary[];
  variant: "full" | "compact" | "minimal";
  pagination?: PaginationProps;
  sorting?: SortingProps;
  columns?: TrainingColumn[];
  actions?: TrainingAction[];
  context?: {
    experimentId?: number;
  };
}

    UnifiedExperimentsTable

    Extract from inline definition to reusable component.

    2. Unified Context Components

    Create ONE context component per entity type:

    ExperimentContextCard

    Props:

    interface ExperimentContextCardProps {
  experiment: ExperimentDetail | ExperimentSummary;
  variant: "full" | "compact";
  actions?: "all" | "subset" | CustomAction[];
  showRequirements?: boolean;
}

    Replaces:

    • TrialsExperimentContext
    • TrainingExperimentContext
    • ExperimentDetailHeader
    • BacktestsExperimentContext

    TrainingContextCard

    Props:

    interface TrainingContextCardProps {
  training: TrainingDetail | TrainingSummary;
  variant: "full" | "compact";
  experimentId?: number;
  actions?: "all" | "subset" | CustomAction[];
}

    Replaces:

    • TrialsTrainingContext
    • Training detail headers

    3. Column Configuration System

    Create a column configuration system to allow flexible table customization:

    Questions for User

    1. How to control which columns show in each table?

    Option A - Simple Variants (RECOMMENDED):

    <UnifiedTrialsTable variant="full" />      // Shows ALL 9 columns (for /trials/ page)
<UnifiedTrialsTable variant="compact" />   // Shows 7 columns (for /experiments/{id}/ page)
<UnifiedTrialsTable variant="minimal" />   // Shows 4 columns (for summaries)
    • ✅ Simple to use
    • ✅ Consistent across pages
    • ❌ Less flexible

    Option B - Granular Column Selection:

    <UnifiedTrialsTable 
  columns={["id", "status", "objective", "bestFlags", "actions"]} 
/>
    • ✅ Maximum flexibility
    • ❌ Each page can show different columns (inconsistent)
    • ❌ More complex to maintain

    Which do you prefer?
