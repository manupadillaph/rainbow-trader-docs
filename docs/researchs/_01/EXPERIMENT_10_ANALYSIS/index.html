    Experiment 10 — Analysis & Decisions for Next Run

    Date: 2026-02-10 Experiment: 10 — "LA_QUINTA_ES_LA_VENCIDA" Study: 81 trials total, 49 COMPLETE, 32 pruned/failed Model: ViT, 3 classes (NONE/LONG/SHORT), BNB 5m candles

    1. Initial Observation

    All trials peaked between epoch 1-5 (91.7% of trials had best epoch ≤ 5, median = 2). This seemed suspicious for 15-epoch training runs.

    2. Root Cause: LR Schedule Too Aggressive

    The cosine annealing LR scheduler with warmup_steps_ratio=0.05 finished warmup at ~epoch 2.5. After warmup, LR was at maximum and started decaying immediately. Combined with warmup_start_lr=1e-5 (too high), the model learned fast, overfit fast, and peaked early.

    Evidence: LR simulation showed peak LR reached at epoch ~2.5, which matches the best epoch distribution.

    3. Bug Found: best.pt Not Reloaded Before Stage Evaluation

    After training loop breaks (early stopping), compute_stage_results(model=model) evaluates the last epoch model (overfit), NOT the best checkpoint.

    • save_checkpoint() correctly saves best.pt when objective improves
    • But after the loop breaks, there's NO load_checkpoint(best.pt) before stage evaluation
    • All 3 stages (s1, s2, s3 backtest) evaluated with the wrong model

    Impact measured:

    Metric Best Epoch Stage2 (last epoch) Degradation
    F1 0.188 avg 0.161 avg -16%
    val_loss 0.49 avg 1.46 avg 3× worse

    Fix: Manu implementing — reload best.pt between training loop break and compute_stage_results().

    4. Sampler & Pruner Never Activated

    • sampler_config.n_startup_trials = 50 → TPE needs 50 random trials before intelligent sampling starts
    • pruner_config.n_startup_trials = 50 → Pruner needs 50 completed trials before pruning
    • Only 49 trials completed → TPE never started, pruner never acted
    • All 49 trials were pure random exploration

    5. Statistical Analysis: TOP 15 vs BOTTOM 15 Trials

    Decoded all 49 completed trials from Optuna SQLite DB with full hyperparameters.

    Key Findings

    Parameter TOP 15 BOTTOM 15 Signal
    focal_gamma=3.0 73% 7% STRONGEST — most important param
    num_heads=8 100% 47% MANDATORY — 8 always wins
    dropout=0.2 87% 40% Sweet spot
    hidden_dim=192 60% 13% Clear winner
    batch_size=256 7% 47% 256 = bad
    weight_decay avg 0.028 0.016 Higher is better
    feature_columns_count avg 185 247 Fewer features = better
    window_size avg 72 83 Shorter windows = better

    6. Loss Function: Focal vs Cross Entropy

    Problem

    If mixing both in Optuna search space, val_loss values are NOT comparable:

    • Focal loss (γ=3): val_loss ~0.15 (down-weighted by $(1-p_t)^\gamma$)
    • Cross entropy: val_loss ~0.80

    This creates a ~0.10 bias in the objective favoring focal (via -0.15 × val_loss term).

    Solution

    Set val_loss.weight = 0.0 in training_weights. The other 4 metrics (accuracy, F1, precision, recall) are percentages [0,1] and independent of loss function — perfectly comparable.

    val_loss only flows through compute_training_objective(), which feeds pruning, early stopping, best checkpoint, and stages 1-2. Setting weight=0 cleanly removes it from all decisions without breaking anything. The model still trains with the loss function normally.

    Conditional Hyperparameters

    Verified in loss_selection.py: when Optuna chooses focal, it never suggests label_smoothing, and vice versa. TPE treats missing params as conditional — no confusion.

    7. Config Changes Applied

    LR Schedule

    Param Before After Reason
    warmup_steps_ratio 0.05 0.10 Double warmup to delay peak LR
    warmup_start_lr 1e-5 1e-7 Start lower to avoid early overshoot
    min_lr 1e-6 1e-5 Higher floor to prevent dead learning late

    Hyperparameter Search Space

    Param Before After Reason
    gamma [1.0, 3.0] continuous [2.5, 3.0, 3.5, 4.0] categorical 3.0 dominated top; narrow around it
    num_heads [4, 8, 11] [8, 10, 12] 8 was 100% of top; drop 4
    dropout [0.1, 0.5] continuous [0.1, 0.15, 0.2, 0.25, 0.3] categorical 0.2 sweet spot; cap at 0.3
    hidden_dim [120, 384] [168, 192, 216, 240] categorical 192 was 60% of top
    batch_size [256, 512, 1024] [512, 1024] 256 = 47% of bottom
    window_size {type: int, min: 60, max: 100} {type: int, min: 50, max: 75} Shorter windows correlated with top
    epochs 15 30 More room to train with gentler LR
    early_stopping_patience 7 12 Match longer epochs

    Sampler & Pruner

    Param Before After Reason
    sampler n_startup_trials 50 40 TPE starts by trial 41 (was never starting)
    pruner n_startup_trials 20 20 Pruner starts after 20 completed trials
    pruner n_warmup_steps 10 6 Start pruning after epoch 6 (not 10)
    n_trials 81 150 More budget for TPE to converge

    Loss Function

    Param Before After Reason
    loss_space ["focal"] ["focal", "cross_entropy"] Explore CE as alternative
    loss_hyperparameters gamma only gamma + label_smoothing [0-0.15] CE-specific regularization
    val_loss weight 0.15 0.00 Make focal/CE comparable in objective

    8. Expected Impact

    1. Bug fix (best.pt reload) → stages evaluate the actual best model, not overfit last epoch
    2. LR schedule → model trains longer before peak, should push best epoch to ~5-10
    3. Narrowed search space → Optuna explores a pre-validated region, faster convergence
    4. TPE actually works → intelligent sampling from trial 41 instead of never
    5. Pruner works → kills bad trials after epoch 6, saves compute
    6. Loss diversity → discover if CE + label_smoothing regularizes better than focal
