    Feature Weight Strategy for Experiments

    Overview

    This document describes the strategy for assigning weight_range and mandatory values to features in experiment configurations.

    These values control how Optuna selects and weights features during model training.

    Key Concepts

    What is weight_range?

    weight_range: [min, max] defines the bounds for the weight Optuna can assign to a feature column during feature selection.

    • min > 0: Feature will always have some weight (cannot be excluded)
    • min = 0: Feature can be excluded (weight = 0)
    • Higher max: Feature can have more influence relative to others

    What is mandatory?

    mandatory: true means the feature MUST be selected by Optuna.

    It cannot have weight = 0.

    Important: mandatory has NOTHING to do with indicator calculation.

    All indicators are calculated regardless of mandatory.

    This only affects feature selection for model training.

    How Multiple Timeframes Work

    Understanding this is CRITICAL for weight assignment.

    The Fat Window Mechanism

    For each sampling timestamp (base timeframe, e.g., 5m), the system:

    1. Takes EACH feature (regardless of its timeframe)
    2. Filters all data up to that timestamp
    3. Takes the last window_size records of THAT feature

    Example with window_size=100

    Feature Timeframe History covered
    bnb_5m_close 5m 100x5m ~ 8.3h
    bnb_15m_close 15m 100x15m = 25h
    bnb_1h_close 1h 100x1h ~ 4d
    bnb_4h_close 4h 100x4h ~ 16d
    bnb_1d_close 1d 100x1d ~ 3m

    Why This Matters

    The model sees SIMULTANEOUSLY:

    • Micro-movements (5m) - immediate price action
    • Short trends (15m, 1h) - intraday patterns
    • Medium trends (4h) - multi-day patterns
    • Long trends (1d) - weekly/monthly context

    Higher timeframes are NOT less important than the base timeframe.

    They provide essential context that the base timeframe cannot see.

    Weight Strategy by Category

    1. Core Base Timeframe (5m) - Price

    Feature weight_range mandatory Reasoning
    bnb_5m_high [1.0, 1.5] true Core price data, always needed
    bnb_5m_low [1.0, 1.5] true Core price data, always needed
    bnb_5m_close [1.0, 1.5] true Core price data, always needed

    Reasoning: These are the fundamental price points.

    The model needs to see the actual price action.

    2. Core Base Timeframe (5m) - Volume

    Feature weight_range mandatory Reasoning
    bnb_5m_volume [0.5, 1.5] true Volume confirms price movements

    Reasoning: In crypto, volume is crucial. A price move without volume is suspect.

    3. Core Base Timeframe (5m) - Return

    • Feature: bnb_5m_return
    • weight_range: [0.5, 1.5]
    • mandatory: true
    • reasoning: Universal cross-asset metric

    Reasoning: Percentage change is the most universal metric.

    It allows comparing movements across different price levels and assets.

    4. Multi-Timeframe OHLCV (15m, 1h, 4h, 1d)

    Feature Type weight_range mandatory Reasoning
    MTF close [0.5, 1.5] false Trend context - very important
    MTF volume [0.3, 1.2] false Volume in higher TF
    MTF return [0.5, 1.5] false Momentum in higher TF
    MTF hl_pct [0.3, 1.2] false Volatility in higher TF

    Reasoning: Higher timeframes provide trend context.

    The 15m is almost as important as 5m because it shows ~25 hours of trend vs ~8 hours.

    5. Correlation Symbols (BTCUSDT, ETHUSDT)

    Feature Type weight_range mandatory Reasoning
    BTC/ETH close [0.3, 1.2] false Market context
    BTC/ETH volume [0.3, 1.2] false Market-wide volume

    Reasoning: BNB is highly correlated with BTC and ETH.

    When BTC/ETH move, BNB often follows. This provides market-wide context.

    5b. Correlation symbols multi-timeframe coverage

    We treat BTC/ETH as market context, not as secondary data.

    In the super experiment we include:

    • OHLCV for BTC and ETH in multiple timeframes
    • 5m, 15m, 1h, 4h, 1d
    • with key columns like close, volume_base, return_close, and hl_pct
    • Indicators for BTC and ETH per timeframe
    • EMA 21
    • RSI 14
    • MACD (line, signal, histogram)
    • Signals for BTC and ETH per timeframe
    • RSI overbought/oversold thresholds
    • MACD histogram bull/bear thresholds

    This gives the model explicit multi-horizon information about the broader crypto market.

    6. Trend Indicators (EMA, ADX)

    Feature weight_range mandatory Reasoning
    EMA (all periods) [0.5, 1.5] false Proven trend indicators
    ADX [0.5, 1.5] false Trend strength
    plus_di, minus_di [0.5, 1.5] false Directional movement

    Reasoning: EMAs and ADX are battle-tested trend indicators used by traders worldwide.

    7. Momentum Indicators (RSI, MACD, Stochastic)

    Feature weight_range mandatory Reasoning
    RSI (all periods) [0.3, 1.2] false Momentum oscillator
    MACD line/signal [0.3, 1.2] false Trend momentum
    MACD histogram [0.3, 1.2] false Momentum acceleration
    Stochastic K/D [0.3, 1.2] false Overbought/oversold

    Reasoning: Momentum indicators help identify when trends are losing steam or gaining strength.

    8. Volatility Indicators (ATR, Bollinger Bands, Volatility)

    Feature weight_range mandatory Reasoning
    ATR [0.3, 1.5] false Used by bot for stop-loss
    Bollinger Bands [0.3, 1.5] false Volatility bands
    Volatility [0.3, 1.5] false Price volatility

    Reasoning: Volatility is crucial in crypto.

    ATR is especially important because the bot uses it for position sizing and stop-loss.

    9. Volume Indicators (OBV)

    Feature weight_range mandatory Reasoning
    OBV [0.5, 1.5] false Volume-price relationship

    Reasoning: OBV shows if volume is flowing into or out of the asset.

    10. Relationships (Correlation, Ratio)

    Feature weight_range mandatory Reasoning
    Correlation BNB/BTC [0.0, 1.0] false Experimental
    Correlation BNB/ETH [0.0, 1.0] false Experimental
    Ratio BNB/BTC [0.0, 1.0] false Experimental
    Ratio BNB/ETH [0.0, 1.0] false Experimental

    Reasoning: These are more experimental.

    They might be useful or might be noise. Let Optuna decide.

    11. Signals

    Feature weight_range mandatory Reasoning
    All signals [0.3, 1.2] false Pre-processed expert knowledge

    Reasoning: Signals are expert knowledge codified (RSI > 70 = overbought, EMA crossover, etc.).

    They:

    • Simplify the model's task by providing pre-processed information
    • Represent decades of trading wisdom
    • Can be combined for powerful patterns

    Why not lower? Signals are NOT just "derived" features.

    They encode specific trading rules that might be exactly what the model needs to learn.

    Why not higher? They lose information.

    Example: RSI=71 and RSI=99 both give signal=1.

    The model should also see the raw values.

    Summary Table

    Category weight_range mandatory Count
    Core price (H/L/C) [1.0, 1.5] true 3
    Core volume [0.5, 1.5] true 1
    Core return [0.5, 1.5] true 1
    MTF close/return [0.5, 1.5] false ~10
    MTF volume/volatility [0.3, 1.2] false ~8
    Correlation symbols [0.3, 1.2] false ~10
    Trend indicators [0.5, 1.5] false ~15
    Momentum indicators [0.3, 1.2] false ~15
    Volatility indicators [0.3, 1.5] false ~6
    Volume indicators [0.5, 1.5] false ~2
    Relationships [0.0, 1.0] false ~6
    Signals [0.3, 1.2] false ~12

    Design Principles

    1. Mandatory = truly essential
    2. Only core price, volume, and return are mandatory
    3. Higher TF = equally important
    4. Don't penalize 15m/1h/4h/1d features
    5. Let Optuna explore
    6. Use [0.0, X] for experimental features
    7. Signals = medium importance
    8. They encode expert knowledge but lose information
    9. Consistent ranges
    10. Similar features get similar ranges

    Progressive Training Phases Strategy

    This section documents how to configure phases_config for the master super experiment.

    The goal is to make the experiment robust and hard to "fail" due to insufficient trials, even if it takes a long time.

    Phases are for training, not dataset warm-up

    Phases are a progressive training mechanism.

    They are about:

    1. Controlling training curriculum and evaluation stability
    2. Enabling warm-start between phases (evolutionary refinement)
    3. Structuring the search from exploration to exploitation

    They are NOT related to indicator warm-up or feature preparation.

    Reality check in this project: data_fraction is a contiguous early slice

    In Rainbow Trader, data_fraction is applied by taking the earliest timestamps.

    This means that small data_fraction values do not represent a random sample of the full market history.

    They represent a specific historical regime.

    Implication:

    • If Phase 1 uses data_fraction=0.11, it can end up training mostly on the early part of the dataset.
    • In markets, regimes shift.
    • A search that overfits early regimes can select brittle feature weights and training hyperparameters.

    Because this master experiment must be reliable, we avoid using aggressive data_fraction reduction as a final selection criterion.

    However, we still use aggressive fractions intentionally in an early elimination block to discard configurations that clearly do not learn or are unstable.

    Market regime motivation

    Crypto markets across 2021-2025 are not stationary.

    Across this range, you typically see:

    • Strong bull trends and high leverage cascades
    • Long bear markets with low liquidity
    • Sideways, mean-reverting chop
    • Volatility clustering (bursts) followed by calm regimes

    This matters because:

    • Feature selection can change per regime (trend vs mean reversion)
    • Optimal training settings can change (learning rate, regularization)
    • The Stage 3 tail backtest can represent a very different environment than the early training slice

    For the master experiment, the safest stance is to always expose the model to the full regime mixture.

    Recommended master schedule

    We use phases in two blocks:

    1. A progressive data block to quickly discard configurations that are clearly unstable or nonsensical.
    2. A full-data block to validate robustness and remove regime bias.

    This keeps the "power" of phases while still producing a final set of trials that is evaluated on the full market history.

    The phases differ in:

    • How many trials we run
    • How many top trials we carry forward (top_n_from_previous)
    • How many children each parent produces (children_per_parent)

    Notes:

    • Each parent produces children_per_parent trials.
    • Child index 0 is an exact warm-start copy.
    • Child indices 1+ are training variations.
    • Each phase also includes fresh trials (not warm-started) to preserve diversity and reduce premature convergence.

    Block A - Progressive data fractions (fast elimination)

    Because data_fraction is a contiguous early slice, this block is not a representative sample of all market regimes.

    We accept that bias intentionally to:

    • Reject configurations that do not learn at all
    • Remove unstable training settings (divergence, bad regularization)
    • Reduce obviously noisy feature-weight assignments
    Phase data_fraction n_trials top_n_from_previous children_per_parent
    1 0.05 450 0 0
    2 0.10 400 40 3
    3 0.20 350 35 3
    4 0.35 300 30 4
    5 0.55 260 25 4
    6 0.80 220 20 5

    Block B - Full data (robustness and regime generalization)

    This block eliminates the regime bias introduced by the early-slice behavior of data_fraction.

    The goal here is not speed.

    The goal is to find configurations that survive the full historical mixture of regimes.

    Phase data_fraction n_trials top_n_from_previous children_per_parent
    7 1.00 300 30 4
    8 1.00 250 25 4
    9 1.00 200 20 5
    10 1.00 150 15 6
    11 1.00 120 12 6
    12 1.00 100 10 7

    This schedule intentionally gives the earliest phases a very large budget.

    With the size of the search space (feature weights + window size + training hyperparameters), a small early budget fails to explore enough combinations to produce high-quality parents.

    CLI - Generate the Super Experiment

    The super experiment configuration file is generated programmatically to avoid manual JSON editing.

    Run

    From the repository root:

    .venv/bin/python scripts/experiments/generate/generate_super_experiment.py --mode full

    Output

    The script writes:

    • data/plans/experiments/templates/super/super-experiment-bnbusdt-full.json

    Before overwriting, it creates a backup copy under:

    • data/plans/experiments/templates/super/_OLD/

    Validation

    The script validates the generated JSON using the ExperimentConfig Pydantic model before writing the file.
