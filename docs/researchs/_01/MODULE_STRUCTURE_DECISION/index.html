    Module Structure Decision

    Date: 2025-12-03
    Status: ✅ APPROVED
    Related: KNOWLEDGE_SYSTEM_INTEGRATION_V2.md, INVOKE_FLOW_ARCHITECTURE.md

    Summary

    Decisión sobre dónde ubicar cada funcionalidad del Knowledge System y utilidades de LLM.

    Resultado: Crear nuevo módulo graph-utils para servicios standalone que graph-orchestrator usa.

    Categorías

    Categoría Descripción Módulo
    Genérico Logging, config loading, paths common
    Standalone Servicios que se pueden usar a mano (CLI) graph-utils
    Motor Agentes, grafo, ejecución graph-orchestrator

    Lógica de separación

    ¿Va en common?

    • ✅ Si es usado por MUCHOS módulos (no solo graph-*)
    • ✅ Si es muy genérico (logging, config, paths)
    • ❌ Si es específico de LLMs/agents

    ¿Va en graph-utils?

    • ✅ Si es un servicio standalone (puede correr sin el graph)
    • ✅ Si tiene CLI propio
    • ✅ Si graph-orchestrator lo importa como servicio
    • ❌ Si es parte del flow de ejecución de un agent

    ¿Va en graph-orchestrator?

    • ✅ Si es parte del invoke flow (budget, context, tier0)
    • ✅ Si es un agent
    • ✅ Si es un tool de agent
    • ✅ Si es el estado del graph
    • ❌ Si puede usarse standalone

    Notas importantes

    ~~rainbow-knowledge-mcp/~~ → graph-utils/mcp/

    El plan original (Finding 18) mencionaba crear un módulo separado rainbow-knowledge-mcp/.

    Decisión actualizada: El MCP va DENTRO de graph-utils/mcp/ porque: - Es un servicio standalone (puede correr sin el graph) ✅ - Tiene CLI propio ✅ - Reutiliza knowledge/ del mismo módulo ✅ - Mantiene todo organizado en un solo lugar ✅

    CLI paths actualizados

    ~~python -m rainbow.knowledge index --all~~ → python -m rainbow.graph_utils.knowledge index --all

    Archivos de config generados

    Archivo Generado por Descripción
    config/graph-llm-benchmark.yaml graph-utils/benchmark Benchmarks de velocidad de modelos

    Nota: Se removió "results" del nombre porque es un archivo de configuración (aunque se genere automáticamente). - La clase BaseConfig donde se va a leer va a ser GraphLlmBenchMarkConfig

    Pattern de BaseConfig para configs hijos

    Para que graph_orchestrator_config tenga configs hijos (memory, knowledge, graph_models) que: 1. Tengan valores dentro del config padre (archivo único YAML) 2. Puedan tener su propio archivo separado en el futuro (ej: config/graph-models.yaml)

    Los hijos deben ser instancias de BaseConfig:

    class GraphOrchestratorConfig(BaseConfig):
    memory_config: MemoryConfig          # Para memory.summarization
    knowledge_config: KnowledgeConfig    # Para knowledge.vector_store
    graph_models_config: GraphModelsConfig  # Para graph_models.llm_tiers
