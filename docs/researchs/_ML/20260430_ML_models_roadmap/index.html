    2. Notación y restricciones del problema

    2.1 Forma del input

    Input tensor: [B, W, F]
  B = batch size
  W = window length (lags), 30 ≤ W ≤ 100  (típico W=64-80 a 5m)
  F = feature count, 30 ≤ F ≤ 300        (multi-TF + multi-symbol context)

    2.2 Contrato de output (3 cabezas)

    Todo modelo entrenable debe producir:

    logits   ∈ R^3        → softmax → P(NONE), P(LONG), P(SHORT)
strength ∈ [0, 1]     → magnitud esperada del movimiento
quality  ∈ [0, 1]     → confianza/calidad de la predicción

    2.3 Fórmula de gating (propietaria, fija para todos)

    c        = probability ^ w_p
sq       = strength ^ w_s · quality ^ w_q
combined = c + α · (1 − c) · sq

Decisión de entrada: combined ≥ entry_threshold AND quality ≥ min_quality_threshold

    Pesos actuales en Script 2: α=0.0, w_p=w_s=w_q=1.0 (fijos).

    2.4 Régimen del problema

    Variable Valor Implicación arquitectónica
    W (ventana) corta (30–100) Atención completa es viable; bias local fuerte ayuda
    F (features) grande (hasta 300) Modelar correlaciones feature-feature es importante
    Multi-TF sí (5m, 15m, 1h, 4h, 1d) Irregularidad temporal efectiva entre features
    Multi-symbol sí (BTC + ETH + BNB) Correlaciones cross-asset relevantes
    Labels sparse sí (5–8% LONG, 5–8% SHORT, 85% NONE) Class imbalance; cuidado con baselines triviales
    Stationarity NO (regímenes cambian) Robustez fold-to-fold > performance pico

    3.1 TradingRuleBased (rule-based determinista)

    Aplica reglas TA clásicas via expresiones AST sobre nombres de features.

    flowchart TD
    A["Input [B, W, F]"] --> B["Last timestep slice<br/>[B, F]"]
    B --> C["AST rule evaluator<br/>(rsi_14 > 70 → SHORT, etc.)"]
    C --> D["Synthetic logits<br/>(margin-based)"]
    C --> E["strength = fixed_value"]
    C --> F["quality = fixed_value"]
    D --> G[3-head output]
    E --> G
    F --> G
    • Pros: interpretable; cero training; baseline TA "honesto".
    • Cons: no aprende; rendimiento techo bajo.
    • Resultado observado: generalmente entre Random-Markov y Logistic.

    3.2 TradingLogistic (lineal multi-head)

    flowchart TD
    A["Input [B, W, F]"] --> B["Flatten → [B, W·F]"]
    B --> C["Linear → logits [B, 3]"]
    B --> D["Linear + sigmoid → strength"]
    B --> E["Linear + sigmoid → quality"]
    C --> F[3-head output]
    D --> F
    E --> F
    • Pros: baseline lineal claro; si gana a Random es prueba casi irrefutable de patrones explotables.
    • Cons: ignora interacciones; capacidad limitada.

    3.3 TradingMLP / MLP_EX

    flowchart TD
    A["Input [B, W, F]"] --> B["Flatten → [B, W·F]"]
    B --> C["Linear + ReLU + Dropout"]
    C --> D["Linear + ReLU + Dropout<br/>(num_hidden_layers ∈ 1-2)"]
    D --> E["3 heads (logits/strength/quality)"]
    • Pros: captura no-linealidad; barato; sorprende en W chico.
    • Cons: no usa orden temporal; overfitting si crece mucho.
    • MLP_EX: variante con más capas / hidden_dim para Tier B.

    3.4 TradingTCN (Temporal Convolutional Network)

    flowchart TD
    A["Input [B, W, F]"] --> B["Transpose → [B, F, W]"]
    B --> C["Conv1D dilated<br/>kernel=3-7, dilation=1,2,4,8..."]
    C --> D["Residual blocks<br/>num_layers=2-4"]
    D --> E["Global avg pool over W"]
    E --> F["3 heads"]
    • Pros: bias temporal fuerte; menos params que ViT; estable; muy bueno para W chico.
    • Cons: receptive field limitado por dilations; menos flexibilidad que atención.
    • Resultado empírico: competitivo con ViT en muchos folds; mejor selectividad.

    3.5 TradingViT (Transformer encoder temporal — modelo actual "estrella")

    flowchart TD
    A["Input [B, W, F]"] --> B["Linear F → d_model<br/>(per timestep)"]
    B --> C["Add learned position<br/>embedding (over W)"]
    C --> D["TransformerEncoder<br/>(N layers, H heads)"]
    D --> E["Pool (mean over W)"]
    E --> F["3 heads"]
    • Pros: atención global; flexibilidad; alta capacidad.
    • Cons: sobreparametrizado para W chico; overfitting; sensible al pos encoding.
    • Resultado empírico: mejor effect size promedio (~0.58), pero más friction → friction paradox.

    flowchart TD

    4.1 Gradient Boosting (LightGBM / XGBoost / CatBoost) ★ NUEVO

    No estaba en docs internos previos. Es la familia que domina problemas tabulares de finanzas (incluyendo el meta-model de Numerai). Al ser tabulares, requieren flatten del tensor [B, W, F] a [B, W·F] (o feature engineering manual).

    flowchart TD
    A["Input [B, W, F]"] --> B["Flatten → [B, W·F]"]
    B --> C["Optional: feature engineering<br/>(rolling stats, lag features)"]
    C --> D{"Boosting variant"}
    D -->|LightGBM| E1["Histogram-based GBDT<br/>leaf-wise growth"]
    D -->|XGBoost| E2["GBDT con regularización L1/L2<br/>level-wise"]
    D -->|CatBoost| E3["Ordered boosting<br/>+ symmetric trees"]
    E1 --> F["Probabilidades por clase"]
    E2 --> F
    E3 --> F
    F --> G["3 heads:<br/>multi-class + 2 regresores"]

    Notas de implementación:

    • Multi-class directo (NONE/LONG/SHORT). Strength/quality requieren modelos separados o multi-output.
    • Práctica común: 1 clasificador entry (binario) + 1 regresor strength + 1 regresor quality.
    • Permite early stopping sobre validation, gestión nativa de missing values, y feature importance interpretable.

    Pros/Cons:

    • 🟢 Domina competencias tabulares de finanzas (Numerai, Kaggle financial).
    • 🟢 Entrenamiento extremadamente rápido (CPU, sin GPU); minutos vs horas de ViT.
    • 🟢 No requiere normalización; robusto a outliers.
    • 🟢 Feature importance nativa → debug y selección.
    • 🔵 Probable ganador en Tier A/B donde features son pocas y tabulares.
    • 🔵 Excelente para ensemble con NN (diversidad de errores).
    • 🔴 Pierde estructura temporal (flatten); requiere feature engineering manual para capturarla.
    • 🔴 No aprende representaciones; depende 100% de calidad de features.
    • 🟠 Puede sufrir con F=300 si muchas features son ruido (NN regulariza mejor).

    Resultado esperado vs ViT:

    • Tier A (F=2): LightGBM probable igual o mejor que ViT.
    • Tier B (F=18): LightGBM probable competitivo con ViT.
    • Tier C–E (F=64–189): incierto — depende de calidad de FE temporal.
    • Highest ROI propuesto por costo de implementación bajo y precedente en finanzas.

    4.2 PatchTST (Patch-based Transformer for Time Series)

    Segmenta la serie temporal en parches (subseries) que se usan como tokens del Transformer.

    flowchart TD
    A["Input [B, W, F]"] --> B["Channel-independence<br/>split: [B·F, W]"]
    B --> C["Patch split<br/>patch_len=8, stride=4"]
    C --> D["Patches: [B·F, N_patches, patch_len]"]
    D --> E["Linear embed → [B·F, N_patches, d_model]"]
    E --> F["Add positional embedding"]
    F --> G["Transformer encoder<br/>(self-attention over patches)"]
    G --> H["Flatten patches"]
    H --> I["Reshape back → [B, F, ...]"]
    I --> J["Aggregate features → [B, d]"]
    J --> K["3 heads"]

    Pros/Cons:

    • 🟢 Patches reducen ruido local antes de attention.
    • 🟢 Channel-independence reduce overfitting cuando features son heterogéneas.
    • 🟢 Performance fuerte en benchmarks long-horizon forecasting.
    • 🔵 Para W chico, patches forzan representación local útil (anti-noise).
    • 🔵 Reduce número de tokens (W=80, patch=8 → 10 tokens) → atención más barata.
    • 🔴 Diseñado para forecasting, no clasificación; transferencia requiere head custom.
    • 🔴 Channel-independence puede perder correlaciones entre features.
    • 🟠 Tuning de patch_len y stride agrega trial budget.

    Resultado esperado: mejora marginal sobre ViT en folds ruidosos; comparable o ligeramente peor en folds limpios.

    4.3 iTransformer (Variable-as-Tokens)

    Invierte la tokenización: cada feature es un token; cada token contiene el vector lag de longitud W.

    flowchart TD
    A["Input [B, W, F]"] --> B["Transpose → [B, F, W]"]
    B --> C["Linear W → d_model<br/>(per feature)"]
    C --> D["Tokens: [B, F, d_model]"]
    D --> E["Transformer encoder<br/>(attention over F variables)"]
    E --> F["FFN per variable"]
    F --> G["Pool over F"]
    G --> H["3 heads"]

    Pros/Cons:

    • 🟢 Modela explícitamente correlaciones feature-feature (ideal para multi-symbol).
    • 🟢 Atención sobre F (no W) → maneja bien features heterogéneas.
    • 🟢 Buenos resultados en forecasting multivariante.
    • 🔵 Para tu caso F=189 (Tier E) con BTC/ETH/BNB, calza perfecto.
    • 🔵 Complementario a PatchTST: uno modela tiempo, otro features.
    • 🔴 O(F²) en atención → con F=300 puede ser caro.
    • 🔴 Diseñado para forecasting; head transfer requerido.
    • 🟠 Pierde detalle temporal fino (todo el W comprimido en un embedding).

    Resultado esperado: mejora especialmente en Tier E (multi-symbol). Para Tier A/B con pocos features, similar a ViT.

    4.4 ConvTran (Position-Aware Transformer for Multivariate TS Classification)

    Combina conv embedding + position encoding híbrido (tAPE absoluto + eRPE relativo).

    flowchart TD
    A["Input [B, W, F]"] --> B["Conv1D embedding<br/>(over time axis)"]
    B --> C["Tokens: [B, W', d_model]"]
    C --> D["Add tAPE<br/>(time-Aware absolute PE)"]
    D --> E["Transformer encoder<br/>with eRPE<br/>(efficient relative PE)"]
    E --> F["Pool"]
    F --> G["3 heads"]

    Pros/Cons:

    • 🟢 Diseñado específicamente para clasificación (no forecasting), tu caso exacto.
    • 🟢 Position encoding híbrido reduce sensibilidad a longitud de ventana.
    • 🟢 Conv embedding aporta bias local antes de attention.
    • 🔵 Cambio "quirúrgico" mínimo sobre tu ViT actual: solo reemplazar pos embedding y entrada.
    • 🔴 Menos popular que PatchTST/iTransformer; menos código de referencia.
    • 🟠 Combinación tAPE + eRPE puede requerir tuning fino.

    Resultado esperado: mejora en estabilidad fold-to-fold (tu problema F1/F3 vs F2/F4). Improvement marginal en peak performance.

    4.5 CNN + Transformer (Híbrido)

    Pipeline secuencial: CNN reduce ruido local → Transformer modela dependencias largas.

    flowchart TD
    A["Input [B, W, F]"] --> B["Conv1D blocks<br/>(local feature extraction)"]
    B --> C["Reduced sequence: [B, W', d]"]
    C --> D["Add positional embedding"]
    D --> E["Transformer encoder<br/>(global mixing)"]
    E --> F["Pool"]
    F --> G["3 heads"]

    Pros/Cons:

    • 🟢 Citado como "muy buena combinación para trading" en docs internos.
    • 🟢 Combina sesgos: local (CNN) + global (Transformer).
    • 🟢 Reduce parámetros vs ViT puro (CNN comprime sequencia).
    • 🔵 Probable mejora vs ViT en folds ruidosos.
    • 🔴 Más bloques → más hyperparams → trial budget mayor.
    • 🟠 Tuning del balance CNN/Transformer es delicado.

    Resultado esperado: mejora consistente sobre ViT en métricas de selectividad; PnL similar.

    4.6 Crossformer (Cross-Dimension Dependency)

    Embeb input como matriz 2D y modela explícitamente dependencias cross-dimension (time × feature).

    flowchart TD
    A["Input [B, W, F]"] --> B["Dimension-Segment-Wise<br/>embedding (DSW)"]
    B --> C["2D token grid: [B, W', F', d]"]
    C --> D["Two-Stage Attention:<br/>1) attn over time<br/>2) attn over features"]
    D --> E["Hierarchical encoder<br/>(multi-scale)"]
    E --> F["Pool 2D"]
    F --> G["3 heads"]

    Pros/Cons:

    • 🟢 Modela explícitamente cross-dimension (no como side effect).
    • 🟢 Multi-scale (jerárquico) → captura patrones a varias escalas.
    • 🔵 Encaja con tu intuición de "ventana como imagen" sin pasar por GAF/MTF.
    • 🔴 Complejo de implementar (Two-Stage Attention + jerarquía).
    • 🔴 Diseñado para forecasting, transferencia no trivial.
    • 🟠 Cost de atención puede explotar con W=80 y F=200.

    Resultado esperado: mejora si correlaciones feature-feature dominan; coste de implementación alto.

    4.7 Autoformer / FEDformer / TimesNet (Decomposition / Frequency)

    Familia con priors estructurales: decomposición trend/seasonal o frecuencia.

    flowchart TD
    A["Input [B, W, F]"] --> B["Series decomposition<br/>(trend + seasonal)"]
    B --> C1["Trend branch"]
    B --> C2["Seasonal branch"]
    C1 --> D1["Encoder con<br/>autocorrelation (Autoformer)<br/>o FFT (FEDformer)"]
    C2 --> D2["Encoder con<br/>periodicity (TimesNet)"]
    D1 --> E[Fusion]
    D2 --> E
    E --> F["3 heads"]

    Pros/Cons:

    • 🟢 Priors estructurales estabilizan training con labels sparse.
    • 🟢 TimesNet es task-general (clasificación + forecasting).
    • 🔵 Decomposition encaja con multi-TF (cada TF aporta una "escala").
    • 🔴 Complejidad alta; muchos hyperparams.
    • 🔴 Diseñados para forecasting long-horizon, no clasificación de ventanas.
    • 🟠 Beneficio dudoso para W=30–100 (decomposition brilla en horizons largos).

    Resultado esperado: baja prioridad. Pasar solo si ViT/PatchTST/iTransformer fallan.

    4.8 S4 / Mamba (State Space Models)

    Backbone secuencial alternativo a atención, lineal en W, con bias temporal continuo-time.

    flowchart TD
    A["Input [B, W, F]"] --> B["Linear F → d_model"]
    B --> C{"SSM variant"}
    C -->|S4| D1["Structured State Space<br/>continuous-time params"]
    C -->|Mamba| D2["Selective SSM<br/>data-dependent dynamics"]
    D1 --> E["Sequence: [B, W, d_model]"]
    D2 --> E
    E --> F["Pool"]
    F --> G["3 heads"]

    Pros/Cons:

    • 🟢 Linear-time en W (no O(W²)); escalable.
    • 🟢 Bias temporal distinto a atención; otra dinámica de optimización.
    • 🟢 Muy estables en secuencias.
    • 🔵 Útil si ViT muestra colapso o inestabilidad (no es el caso hoy).
    • 🔴 Beneficio computacional no crítico para W=30–100.
    • 🔴 Implementación más compleja (custom CUDA kernels para Mamba).
    • 🟠 Tuning poco documentado para clasificación.

    Resultado esperado: baja prioridad. Solo si ViT colapsa.

    4.9 TabNet (Tabular NN con sequential attention)

    Arquitectura tabular con attention secuencial sobre features (Numerai-style).

    flowchart TD
    A["Input [B, W, F]"] --> B["Flatten → [B, W·F]"]
    B --> C["Feature Transformer<br/>(initial)"]
    C --> D["Step 1: Attentive<br/>feature selection (mask)"]
    D --> E["Decision step 1 output"]
    D --> F["Step 2: Attentive selection<br/>(complementary features)"]

    I --> J["3 heads"]

    Pros/Cons:

    • 🟢 Combina poder de NN + interpretabilidad de árboles.
    • 🟢 Sequential attention selecciona features dinámicamente (sparse).
    • 🟢 Probado en Numerai con resultados decentes.
    • 🔵 Bueno para F=300 con muchas features irrelevantes (atención filtra).
    • 🔴 Ignora estructura temporal (tabular).
    • 🔴 Tuning sensible (n_steps, gamma, sparsity).
    • 🟠 No supera consistentemente a LightGBM en tabulares.

    Resultado esperado: competitivo con MLP_EX; menos costo-efectivo que LightGBM.

    4.15 Time2Vec (Encoding temporal explícito)

    No es un modelo standalone — es una capa de encoding que se inserta en cualquier arquitectura.

    flowchart TD
    A["Timestamp / time delta τ"] --> B["Linear: ω·τ + φ"]

    Pros/Cons:

    • 🟢 Barato (k pequeño, e.g. 8).
    • 🔵 Probable mejora en folds donde multi-TF causa confusión.
    • 🔴 No garantiza mejora sin tuning.
    • 🟠 Requiere exponer time deltas/timeframe IDs en el dataset.

    Resultado esperado: test rápido sobre ViT actual. Si ayuda, mantener.

    Pros/Cons:

    • 🟢 Modela explícitamente irregularidad temporal.
    • 🟢 Útil si multi-TF no se puede alinear.
    • 🔵 Puede ayudar en folds con regímenes de baja densidad.
    • 🔴 Requiere rediseñar el input (exponer time deltas).
    • 🔴 Implementación compleja, optimización delicada.
    • 🟠 Beneficio dudoso si Time2Vec resuelve el problema más barato.

    Resultado esperado: baja prioridad. Time2Vec primero.

    5. Técnicas para mejorar inferencias

    Estas NO son modelos, son técnicas que se aplican sobre cualquier modelo existente. Inspiradas en Numerai (hedge fund colaborativo) y prácticas de ML financiero.

    5.1 Feature Neutralization

    Residualizar las predicciones del modelo contra features de riesgo (volatilidad, hora, momentum del símbolo, etc.) para reducir sensibilidad a régimen.

    flowchart TD
    A["Modelo entrenado:<br/>y_pred"] --> B["Risk features X_risk:<br/>volatilidad, hora, regime"]
    B --> C["Linear regression:<br/>y_pred ~ X_risk"]
    C --> D["Residuos:<br/>y_neutral = y_pred − β·X_risk"]
    D --> E["Predicciones neutralizadas"]
    • 🟢 Reduce overfitting a regímenes específicos (BTC bull/bear).
    • 🟢 Ataca directamente tu problema F1/F3 vs F2/F4.
    • 🔵 Drop-in post-training; no requiere reentrenar.
    • 🔴 Reduce edge en magnitud (trade-off pico vs robustez).

    5.2 Era-Boosting

    Boost weighting por "era" temporal (semana, régimen) que el modelo predice peor.

    flowchart TD
    A[Train ronda 1] --> B[Eval per era]
    B --> C{Performance per era}
    C --> D["Sample weight w_era ∝ 1/perf_era"]
    D --> E[Train ronda 2 con w_era]
    E --> F[Loop hasta converger]
    • 🟢 Ataca la heterogeneidad temporal directamente.
    • 🟢 Usado en top stakers Numerai.
    • 🔵 Posible fix para folds débiles (F2/F4).
    • 🔴 Aumenta overfitting si las "eras malas" son outliers genuinos.

    5.3 Target Ensembles (Multi-target Training)

    Entrenar el mismo modelo contra múltiples targets correlacionados (e.g. horizons distintos) y promediar.

    flowchart TD
    A[Input window] --> B[Modelo único]
    B --> C1[Head: target H=12]
    B --> C2[Head: target H=24]
    B --> C3[Head: target H=48]
    C1 --> D[Avg / Weighted ensemble]
    C2 --> D
    C3 --> D
    D --> E[Final prediction]
    • 🟢 Multi-task regulariza; targets correlacionados son señal compartida.
    • 🟢 Sin coste extra de inference.
    • 🔵 Mejora estabilidad cuando un target específico es ruidoso.

    5.4 Bayesian Meta-Model Ensemble

    Combinar predicciones de múltiples modelos con pesos bayesianos (Numerai meta-model).

    flowchart TD
    A["Modelo 1: y_1, σ_1"] --> E[Bayesian update]
    B["Modelo 2: y_2, σ_2"] --> E
    C["Modelo 3: y_3, σ_3"] --> E
    D["Prior: y_0, σ_0"] --> E
    E["Posterior: y_meta = Σ w_i·y_i<br/>w_i ∝ 1/σ_i²"] --> F[Final signal]
    • 🟢 Más principled que avg simple.
    • 🟢 Cada modelo aporta según su confianza.
    • 🔵 Reemplazo natural de las 8 merge rules actuales.
    • 🔴 Requiere estimar σ confiable por modelo (bootstrap o Bayesian NN).

    5.5 Stacking Jerárquico

    Ensemble de 2 niveles: modelos base → meta-learner.

    flowchart TD
    A[Train fold] --> B1[Modelo base 1: TCN]
    A --> B2[Modelo base 2: ViT]
    A --> B3[Modelo base 3: LightGBM]
    B1 --> C["Out-of-fold predictions"]
    B2 --> C
    B3 --> C
    C --> D[Meta-learner: Logistic / GBDT]
    D --> E[Final prediction]
    • 🟢 Combina diversidad (NN + GBDT errores ortogonales).
    • 🟢 Meta-learner aprende a ponderar contextualmente.
    • 🔵 Encaja con tu Phase 1 ensemble campaign.
    • 🔴 Requiere out-of-fold predictions (más compute).

    5.6 Probability Calibration (Platt / Isotonic)

    Calibra las probabilidades del modelo para que reflejen frecuencias reales.

    flowchart TD
    A["Logits / scores raw"] --> B["Sigmoid → P_raw"]
    B --> C{"Calibrator entrenado<br/>en validation"}
    C -->|Platt| D1["Sigmoid(a·P + b)"]
    C -->|Isotonic| D2["Stepwise monotonic"]
    D1 --> E["P_calibrated"]
    D2 --> E
    E --> F["Combined confidence (más fiel)"]
    • 🟢 Hace el entry_threshold más interpretable.
    • 🟢 Mejora gating sin tocar el modelo.
    • 🔵 Probable improvement directo en selectividad.
    • 🔴 Requiere validation set separado.

    9. Pros y Cons globales (síntesis)

    9.1 ¿Por qué Transformer puede no ser el camino óptimo?

    • W=30–100 es corto: la ventaja de atención (long-range) no aplica.
    • F=300 hace que pos embedding aprenda muchas relaciones no necesariamente útiles.
    • Labels sparse (5–8% LONG/SHORT) → ViT sobreparametrizado overfittea fácil.
    • Friction paradox: ViT trade más → más fees → menos PnL pese a effect size mayor.

    9.2 ¿Por qué Gradient Boosting domina finanzas?

    • Datos tabulares heterogéneos (mezcla de scales, distribuciones).
    • Robustez a outliers nativa.
    • Feature importance interpretable (importante para el modelo de negocio).
    • Coste computacional bajo → más experimentos posibles.
    • Track record probado en Numerai, Kaggle financial, hedge funds.

    9.3 ¿Por qué los foundation models no son tu solución (todavía)?

    • Entrenados para forecasting (predecir valores), no clasificación de eventos.
    • Tu target es discreto (LONG/SHORT/NONE) + 2 regresiones acotadas; transferencia requiere head custom.
    • Coste de inferencia por trial alto.
    • Datasets de pretraining no incluyen crypto 5m con tu nivel de granularidad.

    9.4 ¿Cuándo considerar S4/Mamba?

    Solo si:

    1. ViT muestra colapso de atención (todas las cabezas atienden lo mismo).
    2. Training inestable que no se resuelve con LR schedule.
    3. Sensibilidad extrema al position encoding.

    Ninguna de estas señales aparece hoy en Script 2. No priorizar.

    10. Riesgos generales y trampas

    10.1 Overfitting con W chico + F grande

    • Ratio params/datos puede explotar; ViT con d_model=128, 6 layers ya tiene >500K params.
    • Mitigación: dropout agresivo, weight decay, label smoothing, regularización per-tier.

    10.2 Multiple testing (campaña con muchos modelos)

    • Probar 10 modelos × 4 folds → 40 comparaciones; algunas serán "buenas" por azar.
    • Mitigación: Holm-Sidak correction (ya implementado), Deflated Sharpe Ratio (DSR), Probability of Backtest Overfitting (PBO).
    • Ver 20260301_thesis-literature-review.md para detalles.

    10.3 Data leakage temporal

    • Multi-TF con timestamps mal alineados puede traer info futura.
    • Normalización fit en train→val: aplicada al pipeline ✅.
    • Mitigación: purging + embargo (López de Prado); ya parcialmente aplicado.

    10.4 Cost-sensitive evaluation

    • Métricas offline (F1, accuracy) ≠ profitable trading.
    • Friction (slippage + fees) puede dominar el edge.
    • Mitigación: Stage 3 metrics como objetivo final (ya implementado).

    10.5 Survivorship bias en literatura

    • Papers reportan papers que funcionaron; los que no, no se publican.
    • Mitigación: prefiere modelos con replicación independiente y código abierto verificable.

    10.6 Régimen drift

    • BTC bull 2024 ≠ BTC bear 2025 ≠ BTC sideways 2026.
    • Modelo entrenado en un régimen puede fallar en otro.
    • Mitigación: walk-forward (4 folds Script 2), feature neutralization, era-boosting.
