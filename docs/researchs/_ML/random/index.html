    Random Baseline Guide for Thesis Validation

    1. Title and Goal

    The goal of this document is to define a rigorous, decision-complete method to validate that model performance is statistically better than calibrated random baselines.

    In this project, "random" does not mean naive uniform guessing. It means controlled stochastic policies calibrated from allowed data (training, or locked dev if explicitly defined before final testing).

    2. Why Random Baselines Are Mandatory

    A single random run is weak evidence because random outcomes have variance. One seed can look very bad or unexpectedly good by chance.

    Multiple calibrated random baselines are required because they let us estimate a distribution of outcomes under a null hypothesis:

    • "No real predictive edge."

    Fairness principle for all comparisons:

    • Same temporal split.
    • Same execution costs and slippage settings.
    • Same gate behavior (entry_threshold, min_quality_threshold).
    • Same backtest engine and rules.

    If any of these differs, the model vs random comparison is not methodologically valid.

    Implementation rule for this project:

    • Random baseline probabilities are calibrated from observed training labels.
    • They are not Optuna-suggested hyperparameters.

    3. Freeze Temporal Splits and Reporting Protocol

    What freeze temporal splits means

    Define and lock train/validation/test periods once, before running the campaign. Do not adjust boundaries after inspecting results.

    Example:

    • Train: 2021-01-01 to 2024-12-31
    • Validation: 2025-01-01 to 2025-12-31
    • Test: 2026-01-01 to 2026-02-25

    The exact dates can differ, but they must be fixed before analysis.

    What freeze report protocol means

    Define the full evaluation procedure before running experiments, then do not change it after observing outcomes:

    • Metrics to report.
    • Number of random seeds.
    • Statistical tests and p-value formula.
    • Acceptance thresholds for thesis support.
    • Ranking and tie-break rules.

    This prevents:

    • Cherry-picking windows or metrics.
    • Moving-goalpost bias ("change criteria until model wins").

    Freeze checklist

    • Split boundaries fixed.
    • Metrics fixed.
    • Seed count fixed.
    • Statistical tests fixed.
    • Pass/fail thesis criteria fixed.

    4. Always-Flat Baseline

    Always-flat is the minimum baseline and always outputs NONE.

    Definition

    At each tick:

    • signal = NONE.
    • prob_none = 1.0, prob_long = 0.0, prob_short = 0.0.

    Why keep it

    • It verifies that your strategy beats "never trade."
    • It gives a strict floor for activity-based baselines.
    • It is useful for sanity checks and pipeline regression tests.

    Limitation

    • It does not test directional skill under active trading.
    • It is usually too weak to be your only random baseline.

    5. Random-L and Random-B Full Detail

    Random-L Full Detail (Label-Prior Matched)

    Random-L is a label-prior matched random baseline. It samples directly from empirical class shares and does not explicitly model entry policy mechanics.

    Definition

    At each tick, sample one class from calibrated class priors:

    • P(NONE) = p_none
    • P(LONG) = p_long
    • P(SHORT) = p_short

    with:

    • p_none + p_long + p_short = 1.

    Calibration source rules

    • Recommended: estimate priors from training data only.
    • Optional: estimate from training + validation only if the protocol is frozen before touching final test.
    • Never estimate priors from final test.

    Random-L algorithm pseudocode

    def sample_random_l(rng, p_none: float, p_long: float, p_short: float) -> str:
    u = rng.random()
    if u < p_none:
        return "NONE"
    if u < p_none + p_long:
        return "LONG"
    return "SHORT"

    Mapping to pipeline prediction fields

    Use calibrated priors as emitted probabilities:

    • prob_none = p_none
    • prob_long = p_long
    • prob_short = p_short
    • trade_probabilities = p_long + p_short
    • signal: sampled class
    • confidence: probability of sampled class
    • strength, quality, combined_confidence: same policy family used for Random-B to keep fair comparability.

    Strengths

    • Very simple and transparent.
    • Good first stochastic baseline above always-flat.
    • Captures class imbalance in NONE/LONG/SHORT.

    Weaknesses

    • Ignores explicit entry-policy structure.
    • Memoryless and non-temporal.
    • Usually weaker null than Random-B for policy-level comparison.

    Random-B Full Detail

    Random-B is a frequency-matched random baseline. It matches entry activity and directional mix, but remains stochastic and memoryless.

    Definition

    At each tick:

    • Sample entry/no-entry with p_trade.
    • If entry occurs, sample LONG vs SHORT with p_long_cond.
    • If no entry, output NONE.

    Parameters

    • p_trade: target entry frequency.
    • p_long_cond: probability of LONG conditional on entry.
    • p_short_cond = 1 - p_long_cond.

    Calibration source rules

    • Recommended: estimate parameters from training data only.
    • Optional: use training + validation only if this choice is frozen before touching final test.
    • Never estimate from final test.

    Random-B algorithm pseudocode

    def sample_random_b(rng, p_trade: float, p_long_cond: float) -> str:
    enter = rng.random() < p_trade
    if not enter:
        return "NONE"
    is_long = rng.random() < p_long_cond
    return "LONG" if is_long else "SHORT"

    Mapping to pipeline prediction fields

    To integrate fairly with your current pipeline, the random baseline should emit the same output schema as model predictions:

    • signal: sampled label (NONE, LONG, SHORT).
    • prob_none, prob_long, prob_short: calibrated class probabilities used by the baseline.
    • trade_probabilities = prob_long + prob_short.
    • confidence: probability of the sampled class.
    • strength, quality: defined by baseline policy (see recommended options below).
    • combined_confidence: computed with the same formula used in model pipeline.

    Recommended policy for strength and quality:

    • Option A (direction-only null, simpler): fixed constants (for example 0.5, 0.5).
    • Option B (closer behavioral null, preferred): sample from empirical training distributions (optionally conditioned by sampled signal).

    Important:

    • Keep this choice fixed before test evaluation.
    • Apply the same gate and downstream logic regardless of baseline type.

    Gate interaction

    Random-B predictions must pass through:

    • entry_threshold on combined_confidence.
    • min_quality_threshold on quality.

    This is mandatory for fair comparison versus model results that also pass the same gate.

    Strengths

    • Stronger than always-flat.
    • Matches realistic activity level of the target policy.
    • Easy to implement and explain.

    Weaknesses

    • Memoryless (IID over time).
    • Does not model persistence/regime behavior by itself.

    6. Random-Markov Full Detail (Optional Advanced)

    Random-Markov extends random baselines with temporal persistence.

    Definition

    Instead of sampling each tick independently, the next signal depends on previous signal:

    • P(s_t | s_{t-1})

    with states in {NONE, LONG, SHORT}.

    Why this matters

    Financial predictions often exhibit short-run persistence in action sequences. IID random may be too weak as null. Markov random creates a harder and more realistic null baseline.

    Transition matrix

    Use a 3x3 transition matrix:

              next NONE   next LONG   next SHORT
prev NONE     a           b           c
prev LONG     d           e           f
prev SHORT    g           h           i

    Each row sums to 1.

    Calibration

    Estimate transitions from training sequence counts:

    • Count transitions count(prev -> next).
    • Apply smoothing (for example Laplace) to avoid zero-probability transitions.
    • Normalize each row.

    Sampling algorithm pseudocode

    def sample_random_markov(rng, prev_state: str, transition_probs: dict[str, dict[str, float]]) -> str:
    probs = transition_probs[prev_state]
    u = rng.random()
    if u < probs["NONE"]:
        return "NONE"
    if u < probs["NONE"] + probs["LONG"]:
        return "LONG"
    return "SHORT"

    Initialization:

    • Sample first state from empirical class prior (training), or set fixed start state and keep it fixed across seeds.

    Optional extension

    Two-level Markov:

    • Level 1: Markov over entry/no-entry.
    • Level 2: Conditional direction LONG/SHORT when entry occurs.

    This can improve interpretability and control.

    Trade-offs

    • Better realism than IID random.
    • More parameters and calibration choices.
    • More risk of methodological mistakes if protocol is not frozen.

    Positioning

    Use Random-Markov as optional advanced baseline after Random-B.

    7. Statistics to Prove Better Than Random

    Evaluate at least:

    • Stage 3 return.
    • Stage 3 Sharpe.
    • Stage 3 win rate.
    • Final objective score.

    Multi-seed requirement

    Run each random baseline with many seeds (recommended 30 to 50) to estimate its outcome distribution.

    Compute random distribution summary

    For each metric and baseline:

    • Mean.
    • Standard deviation.
    • Confidence interval (for example 95% CI for the mean).

    Model percentile vs random distribution

    Compute where the model sits relative to random outcomes:

    • Higher percentile is better for maximize metrics.
    • This gives intuitive effect interpretation.

    Empirical p-value

    For maximize metrics:

    p = (1 + #random >= model) / (N + 1)

    For minimize metrics:

    p = (1 + #random <= model) / (N + 1)

    Where:

    • N is number of random seeds.
    • #random >= model counts random runs that are at least as good as model.

    Lower p means stronger evidence against the random-null behavior.

    Effect size requirement

    Do not rely on p-value alone. Require practical effect size:

    • Absolute delta vs random mean/median.
    • Optional standardized effect (for example Cohen-style ratio):
    effect_z = (model - mean_random) / std_random

    Consistency requirement

    A thesis claim should hold across defined temporal periods/splits, not only one lucky window.

    8. Clarification on Train vs Validation vs Test Usage

    Question clarified:

    • "Does this mean not use validation data, only training?"

    Answer:

    • Calibration must never use final test data.
    • Preferred default is training-only calibration.
    • Validation can be used for development calibration only if protocol is frozen before final test is touched.

    Reason:

    • Final test must remain untouched evidence of generalization.

    9. Leakage Explained (Test-Set Calibration)

    What leakage means here

    Leakage happens when test information influences baseline calibration.

    Example of leakage:

    • Estimate p_trade or transition matrix using final test distribution.

    Why harmful

    • Baseline becomes tuned to evaluation target.
    • Random baseline may look artificially stronger or weaker.
    • Statistical significance becomes biased and unreliable.

    Wrong vs correct

    Wrong:

    • "We computed Random-B probabilities from final test labels/predictions."

    Correct:

    • "We computed Random-B probabilities from training data (or locked dev set), then evaluated once on test."

    10. Fair Comparison: Model With Gate vs Random With Same Gate

    Pitfall

    Comparing model outputs after gate to random outputs before gate is invalid.

    Correct comparison

    Both model and random baseline must run through identical decision path:

    • Same entry_threshold.
    • Same min_quality_threshold.
    • Same execution costs/slippage settings.
    • Same backtest horizon and tick set.

    Fairness checklist

    • Gate parameters identical.
    • Backtest config identical.
    • Data split identical.
    • Metrics and objective formula identical.

    11. Thesis Acceptance Criteria

    Claim support should require all of the following:

    • Positive effect size versus random distribution.
    • Statistically significant empirical p-value at the chosen threshold.
    • Consistency across defined periods/splits.

    Fail or reject support when:

    • Improvement appears in only one split.
    • P-values are not significant.
    • Gains vanish under nearby period changes or stress splits.

    12. Recommended Reporting Template

    Use one compact table per baseline family:

    Metric Random mean Random std Random 95% CI Model value Model percentile Empirical p-value Pass
    Stage3 return
    Sharpe
    Win rate
    Objective

    Conclusion template:

    Thesis verdict: supported / partially supported / not supported.
Reason summary:
1) Effect size vs random: ...
2) Empirical p-values: ...
3) Consistency across periods: ...

    13. Common Mistakes and Anti-Patterns

    • Calibrating probabilities from final test data.
    • Running only one random seed.
    • Changing metrics/rules after seeing outcomes.
    • Comparing gated model against ungated random baseline.
    • Treating distribution divergence alone as proof of predictive edge.

    Practical Defaults for This Project

    • File scope: documentation only.
    • Statistical direction default: maximize metrics unless explicitly marked minimize.
    • Seed recommendation: 30 to 50 random seeds per random baseline variant.
    • Baseline order recommendation:
    • Always-flat
    • Random-L
    • Random-B
    • Random-Markov (optional advanced)
