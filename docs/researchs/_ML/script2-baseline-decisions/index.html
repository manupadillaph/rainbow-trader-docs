    Script 2 Baseline Decisions — Progressive Model Complexity

    Date: 2026-03-08

    Context

    This document summarizes the research and decisions made for Script 2 (Thesis Proof) regarding the progressive model complexity approach for baselines. The original plan only used ViT models vs Random baselines. This revision adds intermediate model tiers to strengthen the thesis and provide more granular evidence about the nature of exploitable patterns.

    Related Documents

    • docs/plans/EXPERIMENT_PLAN_THESIS_VALIDATION.md — Master thesis plan
    • docs/researchs/_ML/baselines.md — Baseline model catalog
    • docs/researchs/_ML/random.md — Random baseline specification
    • docs/researchs/_ML/ML-roadmap-checkpoint.md — ML development roadmap
    • docs/researchs/_ML/sugerencias.md — Prioritized suggestions for ML pipeline

    1. Problem Statement

    The original Script 2 design compares ViT directly against Random baselines. This creates a complexity gap: if ViT wins, we don't know if a simple linear model would have also won. If ViT loses, we don't know if the model is too complex (overfitting) or if there truly are no patterns.

    The progressive approach fills this gap by testing models of increasing complexity:

    Random → Rule-Based → Logistic → MLP → TCN → ViT → Axial ViT

    Each step answers a specific question about the nature of exploitable patterns.

    2. The Model Progression

    Level 0: Random Baselines (Already Implemented ✅)

    • Models: random_always_flat, random_l, random_b, random_markov
    • Trainable: No
    • What they test: Null hypothesis — "No exploitable patterns exist"
    • Code: modules/model-trainer/src/rainbow/model_trainer/models/random.py
    • Status: Fully implemented and tested in Script 1

    Level 1: Rule-Based Baseline (NEW — To Implement)

    • Model type: rule_baseline
    • Trainable: No (non-trainable, like random baselines)
    • What it tests: "Do classical TA rules (e.g., RSI > 70 → SHORT) have predictive power?"
    • Input: Same [B, W, F] tensor as all models
    • Key requirement: Model must know feature names to map rules to tensor positions

    Configuration system: uses the SAME flow as all other models:

    The rule-based model uses the exact same configuration pipeline as every other model:

    1. YAML defaults: config/experiments/models/rule-baseline-model-hyperparameters.yaml
    2. Pydantic config: RuleBaselineModelHyperparametersConfig in optuna_config.py
    3. Resolution: Same resolution flow as all models. Since it's non-trainable, values are fixed (not OptunaParam). Registered in MODEL_HYPERPARAMETERS_TYPE_MAP["rule_baseline"].

    No new configuration system is needed — this is the standard model config flow.

    Expression-based rules (inspired by CONDITION signal):

    Rules use expression strings following the same philosophy as the CONDITION signal (modules/data-processing/src/rainbow/data_processing/signals/condition.py), which parses boolean expressions via Python AST:

    • CONDITION supports: <, <=, >, >=, ==, !=, and, or, not, arithmetic, parentheses for grouping, and dot notation for column references.
    • The rule-based model reuses the same AST parsing approach, but instead of generating Polars expressions (for DataFrames), it generates tensor operations (for [B, W, F] input).

    Timestep notation: {-T} with curly braces

    Since the model input is [B, W, F] (a window of W timesteps), expressions need to specify WHICH timestep to evaluate. The notation uses curly braces {T} to avoid collision with parentheses used for mathematical grouping:

    • rsi_14{-1} → RSI value at the last timestep (most recent)
    • rsi_14{0} → RSI value at the first timestep (oldest in window)
    • rsi_14{-1} > rsi_14{-5} → Compare RSI at last tick vs 5 ticks back

    Curly braces are chosen specifically because parentheses () may be needed for expression grouping (e.g., (rsi_14{-1} > 70) and (volume{-1} > 1000)).

    How it works:

    1. Constructor receives feature_names: list[str] (sorted alphabetically, matching tensor dim 2)
    2. Builds name → index mapping: self._feature_index = {name: i for i, name in enumerate(feature_names)}
    3. Rules are a list of (expression: str, signal: str) pairs from config
    4. Expressions are parsed via AST in __init__ (once, not every forward call)
    5. Feature names in expressions are mapped to tensor indices; timestep {T} is mapped to the window dimension index
    6. forward(x): for each sample in the batch, rules are evaluated in priority order. First matching rule wins. If no rule matches → default_signal (typically "NONE").

    Critical design decisions:

    • Feature names must be passed to the model. Today create_model() receives input_dim (count) and feature_weights (list of floats) but NOT the names. The names ARE available in training.py line 278: feature_columns_names_ordered = sorted(feature_columns_with_weights.keys()). This list must be forwarded to create_model().

    • Features arrive as-is from the experiment config. If the experiment includes raw RSI (0-100) without normalization, then raw RSI values arrive at the model. The user controls this via experiment configuration — whether to apply normalization pipelines or not.

    • Non-trainable pattern: Same as TradingRandomBaseline — uses a _anchor parameter to maintain a differentiable graph for the training loop. Weights don't update.

    Model output fields explained:

    • sampled_logit_margin: The logit value assigned to the predicted class. Since this model is non-trainable and doesn't produce real logits, it creates them synthetically: logits = [0.0, 0.0, 0.0] then logits[predicted_class] = sampled_logit_margin. After softmax, higher margin = more "confident" prediction. This is the same mechanism used by all random baselines (see random.py lines 131-132).
    • strength_value: Fixed output for the strength head (0.0-1.0). Since this model has no learned strength estimation, it outputs this constant for every prediction.
    • quality_value: Fixed output for the quality head (0.0-1.0). Same rationale as strength.

    These three fields are float | OptunaParam | None to follow the standard config pattern, but in practice for rule-based they should be fixed values — there is no reason for Optuna to tune them in a non-trainable baseline.

    Example YAML default config:

    # config/experiments/models/rule-baseline-model-hyperparameters.yaml
rule_baseline_model_hyperparameters:
  rules:
    - expression: "rsi_14{-1} > 70"
      signal: "SHORT"
    - expression: "rsi_14{-1} < 30"
      signal: "LONG"
    - expression: "ema_9{-1} > ema_21{-1}"
      signal: "LONG"
    - expression: "ema_9{-1} < ema_21{-1}"
      signal: "SHORT"
  default_signal: "NONE"
  strength_value: 0.5
  quality_value: 0.5
  sampled_logit_margin: 5.0

    More complex expression examples (leveraging AST flexibility):

    # Combined conditions with parentheses for math grouping
- expression: "(rsi_14{-1} > 70) and (volume{-1} > 1000)"
  signal: "SHORT"

# Cross-timestep comparison (momentum)
- expression: "rsi_14{-1} > rsi_14{-5}"
  signal: "LONG"

# Arithmetic in expression
- expression: "ema_9{-1} - ema_21{-1} > 0"
  signal: "LONG"

    Level 2: Logistic Regression (NEW — To Implement)

    • Model type: logistic
    • Trainable: Yes
    • What it tests: "Are there exploitable LINEAR patterns in the data?"
    • Architecture: Flatten [B, W, F] → [B, W*F], then nn.Linear(W*F, num_classes)
    • Heads: Same 3-head output as all models (logits + strength + quality)
    • Hyperparameters: Almost none — only window_size, learning_rate, batch_size
    • Why it matters: If logistic beats random, there IS linear signal. A reviewer cannot object "it's just overfitting" because logistic regression has minimal capacity.

    Architecture detail:

    Input: [B, W, F]
  → Flatten: [B, W*F]
    → Linear(W*F, num_classes) → logits: [B, 3]
    → Linear(W*F, 1) + Sigmoid → strength: [B, 1]
    → Linear(W*F, 1) + Sigmoid → quality: [B, 1]

    Level 3: Small MLP (NEW — To Implement)

    • Model type: mlp
    • Trainable: Yes
    • What it tests: "Are there exploitable NON-LINEAR patterns?"
    • Architecture: Flatten [B, W, F] → hidden layers → heads
    • Hyperparameters: hidden_dim (64-256), num_hidden_layers (1-2), dropout (0.0-0.3)
    • Why it matters: Captures feature interactions that logistic regression misses

    Architecture detail:

    Input: [B, W, F]
  → Flatten: [B, W*F]
    → Linear(W*F, hidden_dim) + ReLU + Dropout
    → [Optional: Linear(hidden_dim, hidden_dim) + ReLU + Dropout] × (num_layers-1)
      → Linear(hidden_dim, num_classes) → logits: [B, 3]
      → Linear(hidden_dim, 1) + Sigmoid → strength: [B, 1]
      → Linear(hidden_dim, 1) + Sigmoid → quality: [B, 1]

    Level 4: TCN / 1D-CNN (NEW — To Implement)

    • Model type: tcn
    • Trainable: Yes
    • What it tests: "Does temporal structure matter? Are there sequential patterns?"
    • Architecture: 1D convolutions over time axis, then global pooling → heads
    • Hyperparameters: channels (32-128), kernel_size (3-7), num_layers (2-4), dropout
    • Why it matters: Respects temporal ordering (unlike flatten-based models) without the complexity of attention mechanisms. If TCN beats MLP, temporal patterns exist.

    Architecture detail:

    Input: [B, W, F]
  → Transpose to [B, F, W] (Conv1D expects channels-first)
    → Conv1D(F, channels, kernel_size) + ReLU + Dropout
    → Conv1D(channels, channels, kernel_size) + ReLU + Dropout
    → ... (num_layers)
      → Global Average Pooling over time: [B, channels]
        → Linear(channels, num_classes) → logits: [B, 3]
        → Linear(channels, 1) + Sigmoid → strength: [B, 1]
        → Linear(channels, 1) + Sigmoid → quality: [B, 1]

    Level 5: ViT (Already Implemented ✅)

    • Model type: vit, vit_compact
    • Trainable: Yes
    • What it tests: "Do attention-based temporal patterns provide edge?"
    • Code: modules/model-trainer/src/rainbow/model_trainer/models/vit.py

    Level 6: Axial ViT (Already Implemented ✅)

    • Model type: axial_vit, axial_vit_compact
    • Trainable: Yes
    • What it tests: "Does 2D attention (time × features) help?"
    • Code: modules/model-trainer/src/rainbow/model_trainer/models/axial_vit.py

    3. What Each Comparison Result Means

    Comparison If Winner Beats Random (p < 0.05) Interpretation
    Rule-Based > Random Classical TA rules have some predictive power Validates that raw TA features carry information
    Logistic > Random Linear exploitable patterns exist Strongest possible evidence — minimal model, minimal overfitting risk
    Logistic > Rule-Based Learned linear combinations beat hand-crafted rules ML adds value over manual rules
    MLP > Logistic Non-linear patterns exist that linear models miss Feature interactions matter
    TCN > MLP Temporal structure contains exploitable information Sequential patterns beyond flat feature vectors
    ViT > TCN Attention mechanisms capture patterns convolutions miss Long-range temporal dependencies exist
    ViT > Random but Logistic ≈ Random Patterns exist but require complex models to detect Higher risk of overfitting, weaker evidence
    None > Random No exploitable patterns detected Honest negative result

    Key insight for the thesis: The EARLIER in the progression a model beats random, the STRONGER the evidence. If logistic regression beats random, it's nearly irrefutable proof of exploitable patterns (minimal model, minimal overfitting capacity).

    4. Progressive Phases as Discard Mechanism

    The existing progressive training phases functionality can serve as an efficient discard mechanism for all trainable model tiers, not just ViT:

    Phase 1: data_fraction=0.50 → Explore broadly on half the data
Phase 2: data_fraction=0.75 → Refine top N performers
Phase 3: data_fraction=1.00 → Final validation on full data

    Application to new model tiers:

    • Logistic/MLP (Tier A/B): Phases DISABLED — these are fast models with few hyperparameters. Single-phase optimization is sufficient.
    • TCN (Tier B/C): Phases can be ENABLED if trial count is high enough (≥60 trials). TCN has moderate hyperparameter space.
    • ViT/Axial ViT (Tier C/D): Phases ENABLED as in original plan. Complex models benefit most from progressive refinement.

    6. Feature Name Availability — Code Evidence

    The feature names are already computed in the training flow. Here is the exact code path:

    In training.py (line 225-228):

    feature_columns_with_weights = {
    selected_feature.column_name: float(selected_feature.weight)
    for selected_feature in resolved_hyperparameters.feature_selection.feature_columns_with_weights
}

    In training.py (line 278):

    feature_columns_names_ordered = sorted(feature_columns_with_weights.keys())

    This feature_columns_names_ordered is a list[str] with the feature names in the exact same order as the tensor's dim 2 (input_dim). It is computed BEFORE create_model() is called. Forwarding it to the factory requires adding one parameter.

    7. Walk-Forward Validation — ENABLED

    Walk-forward validation is ENABLED in Script 2. After Script 1 experience (4 days of iterations, fragile single-split results), walk-forward was deemed essential for robust evidence.

    Design: 4 sequential folds within a 13-month range (2025-01 to 2026-02). Each fold is a standalone 4-month experiment with training_split=0.80 (~3.2mo train, ~0.8mo test).

    Fold 1: range 2025-01-01 → 2025-05-01, training_split=0.80
Fold 2: range 2025-04-01 → 2025-08-01, training_split=0.80
Fold 3: range 2025-07-01 → 2025-11-01, training_split=0.80
Fold 4: range 2025-10-01 → 2026-02-01, training_split=0.80

    Why it's valuable: Tests if patterns persist across different market regimes. A model that only works in one time period may be overfitting to market conditions.

    Compute impact: Does NOT multiply compute by 4x — each fold trains on less data so converges faster. Net overhead is ~1.2-1.5x.

    Total: 40 base experiments × 4 folds = 160 experiment runs.

    Consequence for phases: Progressive training phases are DISABLED for all tiers. With ~3.2 months of training data per fold, Phase 1 at 50% data fraction would train on only ~1.6 months — too little data for meaningful progressive refinement.

    8. Multiple Testing Correction

    With the expanded model progression, the number of comparison pairs increases. The Holm-Sidak correction adjusts p-values to account for this.

    How Holm-Sidak works:

    1. Collect all p-values from all comparison pairs
    2. Sort them from smallest to largest
    3. For the smallest p-value, required threshold = 1 - (1 - α)^(1/N)
    4. For the next, threshold = 1 - (1 - α)^(1/(N-1))
    5. Continue until a p-value fails its threshold; all remaining also fail

    With 96 pairs (6 real × 4 random × 4 tiers):

    • Strictest threshold: p < 0.000527 (for the most significant pair)
    • Least strict: p < 0.05 (for the least significant pair)
    • This is acceptable — if patterns truly exist, the best models should have very small p-values that survive correction

    Mitigation: The progressive approach actually HELPS with multiple testing: if Logistic already beats Random with p < 0.001, adding more complex models that also beat Random reinforces the finding rather than diluting it.

    9. Decisions Summary

    Decision Rationale
    Add Rule-Based baseline Represents "human expert TA knowledge"; bridges random to trainable
    Add Logistic baseline Minimal trainable model; if it wins, strongest possible thesis evidence
    Add MLP baseline Tests non-linear patterns with minimal architecture
    Add TCN baseline Tests temporal patterns without attention complexity
    Keep ViT + Axial ViT Already implemented; test full attention-based models
    Feature names passed to factory Required for rule-based model; useful for interpretability
    Rules configured via experiment JSON No hardcoded values; user controls rules per experiment
    Raw (non-normalized) features for rule-based User decides normalization per experiment; thesis allows any price-derived features
    Progressive phases DISABLED for all tiers Incompatible with 4-month fold windows (~1.6mo at 50% data fraction)
    Walk-forward ENABLED (4 folds) Tests robustness across temporal regimes; 40 base × 4 folds = 160 runs
    Holm-Sidak for multiple testing correction Handles expanded comparison pairs correctly
