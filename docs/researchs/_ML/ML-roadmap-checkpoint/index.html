    Checkpoint de Conocimiento ML

    Roadmap de Baselines y Modelos para Rainbow Trader

    1. Objetivo de este documento

    Este documento deja un checkpoint operativo para:

    • No perder contexto de los 3 documentos de research ya generados.
    • Traducir ese research a un plan ejecutable en tu hardware real.
    • Empezar por baselines sólidos antes de escalar a modelos complejos.
    • Tener un roadmap por fases con tiempos estimados, decisiones y criterios de salida.

    Este checkpoint consolida:

    • docs/researchs/_ML/ML-deep-research-report.md
    • docs/researchs/_ML/ML-deep-research-report-2.md
    • docs/researchs/_ML/sugerencias.md
    • Hallazgos verificados en código del repo (model-trainer, backtesting-engine, window-preparer).

    2. Estado actual del pipeline (verificado en código)

    2.1 Contrato de entrada y salida

    • Input del modelo: tensor [B, W, F].
    • W: ventana temporal (lags).
    • F: número de features seleccionadas para el trial.
    • Salidas del modelo:
    • logits para clase NONE/LONG/SHORT.
    • strength en [0, 1].
    • quality en [0, 1].

    2.2 Modelo actual

    • Modelo actual en producción de entrenamiento: TradingViT (Transformer temporal).
    • No es ViT 2D de imagen con patching espacial clásico.
    • Es encoder temporal sobre tokens por timestep.

    2.3 Punto crítico de negocio

    combined_confidence es fórmula propietaria del pipeline, no estándar de ViT:

    • c = probability^w_p
    • sq = strength^w_s * quality^w_q
    • combined = c + alpha * (1 - c) * sq

    La decisión de entrada usa gate con:

    • entry_threshold
    • min_quality_threshold

    3. Dudas clave resueltas

    3.1 "Random con misma frecuencia de trades" significa

    Frecuencia = tasa de entradas efectivas sobre total de ticks evaluados.

    Se obtiene de métricas existentes del trial:

    • training_metrics.stage2_metrics.prediction_health.rates.entry_rate_over_total_predictions

    También se puede usar:

    • entry_count / total_predictions

    3.2 De dónde sacar esa frecuencia

    Fuentes válidas, en orden:

    1. Frecuencia de labels de entrenamiento (solo train).
    2. Frecuencia de un modelo referencia en train/val.
    3. Nunca calibrar con test final para evitar leakage de decisión.

    3.3 Baselines van primero

    Sí. Primero baselines. Luego arquitectura.

    Si no superamos baselines de forma robusta, cambiar modelo no agrega claridad.

    4. Protocolo experimental fijo (obligatorio)

    Antes de correr barridos grandes, congelar:

    • Split temporal fijo.
    • Costos de ejecución fijos (slippage, fees, reglas).
    • Métrica objetivo principal fija.
    • Métricas secundarias obligatorias fijas.
    • Mismo proceso de reporte para todos los modelos.

    4.1 Ventanas temporales recomendadas para empezar

    Propuesta inicial (iteración rápida y robusta):

    • Train dev: 2021-01-01 a 2024-12-31
    • Validation: 2025-01-01 a 2025-12-31
    • Test final: 2026-01-01 a 2026-02-25
    • Stress holdout opcional: 2018-01-01 a 2020-12-31

    No usar 2018-2026 full para TODO desde día 1, porque es muy costoso y mezclará preguntas.

    4.2 Métricas mínimas para decisión

    • Objetivo principal: score final de Stage 1/2/3 (o variante aprobada).
    • Secundarias obligatorias:
    • stage3_return_pct
    • stage3_sharpe_ratio
    • stage3_win_rate
    • stage3_trades_count
    • max drawdown (si está disponible en el flujo actual o agregado en baseline report).

    5. Baselines obligatorios (fase inicial)

    Baseline Qué hace Costo Para qué sirve
    Always-flat Siempre NONE Muy bajo Piso absoluto del sistema
    Random-L Random calibrado por frecuencia de labels Muy bajo Control estocástico sin depender de modelo
    Random-B Random calibrado por entry rate de modelo referencia Muy bajo Control comparativo de política de entradas
    Regla simple EMA cross / RSI threshold Bajo Baseline técnico clásico
    Lineal Logistic/linear head Bajo Saber si lo no lineal realmente aporta
    MLP chico Flatten + MLP pequeño Bajo-medio Baseline fuerte en ventanas chicas

    5.1 Especificación de Random-L y Random-B

    Para cada tick:

    • entry ~ Bernoulli(p_trade)
    • Si entry = 1:
    • LONG ~ Bernoulli(p_long_cond)
    • Si no LONG, entonces SHORT
    • Si entry = 0: NONE

    Parámetros:

    • p_trade: frecuencia objetivo.
    • p_long_cond: proporción LONG dentro de trades.

    Correr 30 a 50 seeds por baseline random para estimar media y dispersión.

    6. Restricciones reales de cómputo (tu hardware)

    Hardware reportado:

    • NVIDIA GeForce RTX 4070 Laptop GPU 8GB CUDA.

    Costo actual observado:

    • Modelo actual con W~80, F~200, 35/50 epochs: ~4 a 5 horas por trial.

    Implicación:

    • No es viable hacer "full optuna de todo" en todas las familias desde el inicio.
    • Necesitamos búsqueda jerárquica por etapas de costo.

    7. Estrategia de trials por costo (S/M/L)

    7.1 Perfiles de trial

    • Perfil S (screening):
    • 8-10 epochs
    • data_fraction ~ 0.2-0.25
    • mini-backtest
    • Perfil M (refine):
    • 18-22 epochs
    • data_fraction ~ 0.5
    • mini-backtest
    • Perfil L (confirmación):
    • 35-50 epochs
    • data_fraction = 1.0
    • full backtest

    7.2 Presupuesto ejemplo para 500 trials

    Propuesta realista:

    • 300 trials S
    • 150 trials M
    • 50 trials L

    Tiempo total estimado aproximado:

    • S: 35-45 min/trial
    • M: 90-120 min/trial
    • L: 4-5 h/trial

    Total estimado: ~626 a ~775 horas GPU.

    Interpretación:

    • 24/7: ~26 a ~32 días.
    • 12h/día: ~52 a ~64 días.

    8. Espacio de búsqueda recomendado para W y F

    No abrir extremos desde inicio.

    Fase inicial:

    • W: {32, 48, 64, 80}
    • F: {64, 128, 192}

    Regla:

    • Solo expandir a W > 80/100 o F > 192/224 si hay señal clara de mejora cerca del borde.

    9. Roadmap por fases (de simple a complejo)

    Fase Objetivo Modelos/Acciones Salida esperada
    0 Congelar protocolo Split, métricas, reporting, seeds Base comparable estable
    1 Baselines Flat, Random-L, Random-B, reglas, lineal, MLP chico Piso y referencias sólidas
    2 Mapa W/F barato Barrido rápido con MLP/TCN chico Rango útil de W y F
    3 Refinar ViT actual Capacidad, regularización, pos encoding, threshold tuning Mejor versión de referencia
    4 Primer nuevo backbone fuerte TCN/1D-CNN Comparación justa vs ViT
    5 Transformers alternativos Patch temporal, iTransformer-style Medir tokenización alternativa
    6 Híbridos y cross-dim CNN+Transformer, Crossformer-like Ver interacción local+global
    7 Secuenciales alternativos S4/Mamba (piloto) Validar sesgo temporal alternativo
    8 Pretraining/transfer TS2Vec y foundation-model transfer Potencial extra en data limitada

    10. Catálogo completo de ideas (pros, cons, implementación)

    Familia Pros Contras Implementación Prioridad
    ViT temporal refinado Cambios mínimos sobre lo actual Puede sobreparametrizar en W chico Ajustes en vit.py Alta
    TCN / 1D-CNN Muy fuerte en ventanas pequeñas, estable Menos flexible global Nuevo tcn.py + factory + Optuna Alta
    MLP flatten Barato y rápido, baseline útil Overfit si crece mucho Nuevo mlp.py simple Alta (baseline)
    Patch Transformer Patching temporal reduce ruido local Más diseño de tokenización Nuevo vit_patch.py Media
    iTransformer-style Potencia correlaciones feature-feature Costo puede subir con F alto Nuevo itransformer.py Media
    Crossformer-like Cross-dim explícito Complejo de implementar Nuevo backbone dedicado Media-baja
    CNN + Transformer Buen balance local/global Más complejidad y tuning Nuevo cnn_transformer.py Media
    ROCKET/MiniROCKET Baseline de clasificación muy fuerte Integración distinta a pipeline NN Baseline externo/adaptador Media
    S4/Mamba Sesgo temporal distinto, potencial estabilidad Beneficio incierto para W corto Nuevo backbone complejo Baja
    Time2Vec Inyección temporal barata Mejora no garantizada Feature/embedding adicional Media
    GRU-D / Neural CDE / Latent ODE Bueno para irregularidad explícita Requiere rediseño de input Nueva línea de modelado Baja
    TS2Vec Buen pretraining SSL Flujo adicional de entrenamiento Etapa de pretraining + fine-tune Baja-media
    TimesFM/Chronos/TimeGPT/Timer (transfer) Potencial transferencia Integración costosa y no directa al target Adaptación avanzada Baja

    11. Integración de un modelo nuevo en este repo

    Checklist mínimo para cualquier nuevo backbone entrenable:

    1. Crear archivo en modules/model-trainer/src/rainbow/model_trainer/models/.
    2. Heredar contrato de BaseModel y devolver siempre:
    3. logits
    4. strength
    5. quality
    6. Registrar en models/factory.py.
    7. Incluir hiperparámetros en helpers/hyperparams_training.py.
    8. Agregar entrada en config/experiments/optuna.yaml:
    9. model_space.choices
    10. model_hyperparameters.<nuevo_modelo>
    11. Ajustar planes JSON de experimento para habilitar el modelo.
    12. Agregar tests de shape/forward y smoke de entrenamiento.

    12. Política de Optuna recomendada

    No usar full-search global desde el inicio.

    Usar búsqueda por etapas:

    • Etapa A: seleccionar familia con hyper reducido.
    • Etapa B: expandir hyper dentro de top 2-3 familias.
    • Etapa C: confirmar en perfil L con full backtest.

    Congelar hiperparámetros cuando corresponda:

    • Al explorar modelo, congelar parte de loss y thresholds.
    • Al explorar loss/thresholds, congelar arquitectura top.

    13. Qué esperamos validar en cada etapa

    Etapa de baselines

    • Que cualquier modelo serio supere a random de forma robusta.
    • Que no estemos ganando por artifacts de evaluación.

    Etapa de familia de modelos

    • Si el gap viene del backbone o del gating/política.
    • Qué sesgo inductivo funciona mejor para W chico y F alto.

    Etapa final

    • Robustez temporal (validation + test + stress).
    • Estabilidad de trades y no solo mejora en un único score.

    14. Primer sprint recomendado (acción inmediata)

    Orden de ejecución para iniciar ya:

    1. Implementar baselines always-flat, random-L, random-B.
    2. Implementar baseline de regla simple (EMA/RSI).
    3. Correr torneo de baselines con seeds múltiples.
    4. Definir umbrales de "modelo aceptable" sobre esos resultados.
    5. Recién ahí iniciar fase 3 de refinamiento ViT.

    15. Decisiones abiertas (para no bloquear)

    • Definir objetivo primario exacto de ranking para esta campaña.
    • Confirmar ventana temporal oficial de train/val/test.
    • Confirmar presupuesto total de tiempo (modo 24/7 o diario).
    • Confirmar si 500 trials se mantiene como presupuesto total de campaña.
